# VastPlan

基于 LLM 的通用 Agent 系统，面向企业级客户：在线 Agent 开发 + 远程客户端本地执行脚本/工作流 + **全栈插件市场**（类似 VSCode 的在线插件机制）。

## 文档

所有项目文档从 **[docs/dev/00-index.md](docs/dev/00-index.md)** 进入——它是文档的唯一索引。开发前先查阅相关设计篇。

## 文档维护纪律（务必遵守）

1. **单一真相源**：同一件事只有一篇权威文档，发现重复立即合并。
2. **代码与文档同一次改动**：改模块就在同一次提交里改它的设计文档。
3. **决策进 ADR**：任何架构取舍写成 `docs/dev/decisions/` 下的 ADR，只追加不覆盖。
4. **过期即删**：宁可删掉过期文档，也不留误导内容。

## 插件开发语言选型

- 规划任何新插件或插件重大改造时，必须先进行开发语言选型，不默认沿用现有语言。**Go 原生、Python、Node.js** 是重点比较基线，但不是限制名单；只要插件 Runtime 和平台契约能够支持，也可以选择 Rust、Java、C# 等更适合该场景的语言。
- 选型首先看运行效率、资源占用、生态与驱动成熟度以及业务场景适配度，再综合比较并发模型、热更新需求、进程隔离与安全边界、跨平台部署、可维护性和团队工程成本。
- 规划方案必须明确给出候选语言的优劣、推荐语言及理由；不得为了语言统一牺牲性能、生态或场景适配度。必要时允许通过稳定 RPC 契约连接多语言 Provider。
- 运行方式与开发语言分别决策：同时判断插件应内嵌、进入该语言的共享 Runtime 进程，还是使用独立隔离进程。
