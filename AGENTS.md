# VastPlan

基于 LLM 的通用 Agent 系统，面向企业级客户：在线 Agent 开发 + 远程客户端本地执行脚本/工作流 + **全栈插件市场**（类似 VSCode 的在线插件机制）。

## 文档

所有项目文档从 **[docs/dev/00-index.md](docs/dev/00-index.md)** 进入——它是文档的唯一索引。开发前先查阅相关设计篇。

## 文档维护纪律（务必遵守）

1. **单一真相源**：同一件事只有一篇权威文档，发现重复立即合并。
2. **代码与文档同一次改动**：改模块就在同一次提交里改它的设计文档。
3. **决策进 ADR**：任何架构取舍写成 `docs/dev/decisions/` 下的 ADR，只追加不覆盖。
4. **过期即删**：宁可删掉过期文档，也不留误导内容。

## 已确定的方向

- 插件运行模型：**全栈插件**（前端 UI + 后端逻辑 + 客户端脚本），见 ADR-0001。
- 技术栈：**全新选型**，不沿用 testa，见 ADR-0002。
- 参考系统：`../testa`（方向一致，可借鉴设计经验，不直接复用代码）。

## Graphify 图谱维护

- 采用按需增量更新：实质性代码、文档或 ADR 改动完成并通过验证后，在提交前运行 Graphify `--update`。
- 零散小改动可合并后统一更新，避免无意义的频繁重建。
- 架构、调用链和项目内容问题优先查询现有 `graphify-out/graph.json`。
- 仅在大规模目录迁移、全局重构或图谱诊断异常时执行全量重建。
- `graphify-out/` 是本地派生数据，必须保持 Git 忽略，不纳入提交。
