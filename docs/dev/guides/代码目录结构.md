# 代码目录结构

> 状态：v0.2（随代码演进，本文是**活的目录参考**）｜最后更新：2026-07-18
> 决策与原则见《[ADR-0016 单仓与代码目录布局](../decisions/ADR-0016-单仓与代码目录布局.md)》与《[ADR-0060 五大区域仓库布局](../decisions/ADR-0060-五大区域仓库布局与根目录收敛.md)》；架构见《[系统架构](../architecture/系统架构.md)》；契约见《[插件契约与协议](../architecture/插件契约与协议.md)》。

## 顶层树

```
VastPlan/
├── core/                          # 可运行的产品核心
│   ├── kernels/                   #   backend / frontend / runner / mobile 四内核
│   └── shared/                    #   内核间共享的 Go/前端运行时能力
├── extensions/                    # 可独立演进的扩展面
│   ├── plugins/                   #   第一方全栈插件，一插件一目录
│   └── sdk/                       #   Go / Python / TypeScript 插件 SDK
├── contracts/                     # 机器可执行的契约真源
│   ├── proto/                     #   跨语言通信契约（Protobuf）
│   └── schemas/                   #   清单、组合、部署与 UI 契约（JSON Schema + Go）
├── engineering/                   # 只服务于开发、测试和交付
│   ├── arch/                      #   架构适应度测试与根目录白名单
│   ├── e2e/                       #   跨进程/跨节点验收与故障夹具
│   ├── deploy/                    #   部署、Platform Profile 与组合样例
│   ├── tools/                     #   codegen、构建、测试和发布命令
│   └── internal/                  #   仅供仓内工程入口复用的 Go 包
└── docs/dev/                      # 架构、ADR、指南与插件文档
```

> 单元测试**与源码同目录**（不建统一 tests/ 树）；跨组件 E2E 才进 `engineering/e2e/`。详见《[测试规范](测试规范.md)》。

仓库根还保留 `go.mod`、`package.json`、`pnpm-workspace.yaml`、CI、许可证等单仓级文件。`node_modules/`、`.pnpm-store/`、`bin/` 和 `graphify-out/` 是本地可重建派生目录，全部 Git 忽略，不属于产品源码布局。

## 各目录职责

| 目录 | 职责 | 不该放什么 |
|---|---|---|
| `contracts/proto/` | 契约与协议的**唯一**定义源 | 手写的重复契约结构（一律 codegen） |
| `contracts/schemas/*/vN/` | 公共标识、组合输入、App Profile、插件与部署 JSON Schema 及 Go 校验入口 | 手写的第二套契约规则 |
| `core/kernels/*` | 各内核的**最小骨架**：宿主、注册表、协议总线、横切、生命周期 | **业务功能**（一律下沉为插件） |
| `core/shared/go/*` | 跨内核复用的通用能力 | 只被单一内核用的代码（放该内核下） |
| `core/shared/go/clientcore` | Runner 与 Mobile **共用**的客户端核心 | 后端专用逻辑 |
| `extensions/plugins/<id>/*` | 一个第一方插件的全部面 | 内核能力、跨插件公共代码（提到 shared） |
| `extensions/sdk/*` | 插件开发者用的 SDK 与脚手架 | 内核内部实现细节 |
| `engineering/deploy/` | 期望态样例、环境 profile | 代码 |
| `engineering/tools/` | codegen / 构建 / 发布 | 运行时代码 |

## 关键约定

1. **服务组合是配置，不是代码**（ADR-0016 §3）：backend / workspace / rs 是**同一个 `core/kernels/backend` 二进制**部署多次 + 期望态 `service_role` 不同。Controller 仍由 `controlplane` 子命令启动；`artifact-server` 是 ADR-0049 迁移期兼容入口，仓库服务最终由预置基础插件承载。
2. **内核不含业务**：凡"换一种实现是合理需求"的能力（审计、可观测、权限策略、用户校验）都放 `extensions/plugins/`，内核只留扩展点与契约。
3. **内嵌边界已收窄**：内核只保留制品信任、种子源、可信缓存和安装强制点；仓库 HTTP/存储/索引/市场能力进入预置基础插件（ADR-0049）。进程内能力一律由插件自身的 dynamic-go 模块从签名制品加载，`core/kernels/` 不得 import 具体插件（ADR-0051）。
4. **proto 改动流程**：改 `contracts/proto/` → 跑 `engineering/tools/gen-proto.sh` → 同一次提交更新 `core/shared/go/*`、`extensions/sdk/python/{contract,pluginhost}` 与受影响 SDK；不得手写重复消息类型。
5. **插件面目录名 = 规范 ID**（ADR-0015）：`backend/` `frontend/` `runner/` `mobile/`，与清单 `contributes.<id>` 一致。
6. **clientcore 边界**：只放 Runner 与 Mobile 都要的东西（契约、鉴权、清单模型、配置）；任一方专用的不进去。
7. **Backend 横向依赖**：普通子包通过 `contracts/schemas/` DTO 或窄接口协作，不引用同级实现包；只有 `core/kernels/backend` 主包和 `commands/` 组合根可以装配多个同级组件。
8. **版本 DTO 单一真源**：跨 schema 版本完全同语义的 Go DTO 放 `contracts/schemas/common/v1`，版本包只留别名；外部 JSON 语义不同的类型不得强行合并，必须显式转换。
9. **前端 UI 边界**：Portal 内核只放可信加载、路由、Slot、恢复页和稳定 UI 契约；具体 UI 框架放 `extensions/plugins/com.vastplan.foundation.frontend.design-system.*`。普通前端插件只能使用 `extensions/sdk/ts` 的框架无关 UI 契约，不能直接绑定 Arco/MUI。
10. **服务配置分级**：Platform Profile 由平台管理员管理 foundation/platform 基线，Application Composition 只选择应用插件；`deployment/v2` 是 Resolver 输出，不是普通服务配置入口（ADR-0057）。

## "我要加东西放哪"决策树

- 加一个**业务功能** → `extensions/plugins/<新插件>/`，按面建子目录。
- 为首方 Go 插件提供**进程内能力** → 在插件自身 `dynamic/` 目录实现 ABI 入口，使用共同构建生成签名 `.so`，并由部署方显式配置 `require-dynamic-go`；不得写进 `core/kernels/`。
- 加一个**扩展点/内核能力** → 对应 `core/kernels/<内核>/`，并在《插件契约与协议》第四章登记契约。
- 加一个**跨内核复用的通用能力** → `core/shared/go/<模块>`。
- 加一个**契约字段/协议消息** → `contracts/proto/`，然后 codegen（勿手写）。
- 加一个**插件 JSON 清单/descriptor/制品字段** → `contracts/schemas/plugin/v1/`；改四内核共同组合语义 → `contracts/schemas/composition/common/v1/`；改某内核平台/应用组合载荷 → `contracts/schemas/composition/<kernel>/vN/`；改 Runner App Profile → `contracts/schemas/app/v1/`；改节点执行快照 → `contracts/schemas/deployment/v1/`；改 Backend Resolver 锁定输出或全局副本/放置 → `contracts/schemas/deployment/v2/`；先改 JSON Schema，再让各语言消费它。
- 加一个**应用后端服务** → 在 Application Composition 中新增应用 unit；增加本地基础附件或共享平台服务 → 修改 Platform Profile。两者都不新建内核分叉代码。
