# 多语言插件开发

Backend 插件协议以 Protobuf/gRPC 为语言边界。Go、Python 与 Node Runtime 从同一份 `contracts/proto/pluginhost/v1/pluginhost.proto` 生成消息，行为差异由跨语言 E2E 阻断。

## 选择运行驱动

清单用 `execution.backend` 冻结执行契约：

```json
{
  "execution": {
    "backend": {
      "driver": "python",
      "requirements": { "python": ">=3.8", "grpcio": "1.70.0" },
      "minimumIsolation": "trusted-process",
      "features": ["channel.cancel.v1", "event.publish.v1"]
    }
  }
}
```

- `driver` 支持 `native`、`python`、`node-worker`、`python-subinterpreter`；`wasm-component`、`container`、`process-sandbox` 通过同一 SPI 接入。
- `args` 不经过 shell；不得依赖 shell 展开或命令拼接。
- `features` 是启动必需协议能力，宿主无法协商时拒绝启动。
- `minimumIsolation` 只是插件提出的下限。节点发布者策略可以提高、不能降低它。

首方 Go 插件还可在保留进程入口的同时声明可选 dynamic-go 内容：

```json
{
  "execution": {
    "backend": {
      "driver": "native",
      "minimumIsolation": "trusted-process",
      "dynamicGo": {
        "entry": "backend/plugin.so",
        "abi": "vastplan.dynamic-go.v1",
        "fingerprint": "<由共同打包流程写入的 64 位 SHA-256>",
        "required": true
      }
    }
  }
}
```

源码 Manifest 可省略 `fingerprint`，但发布包必须由 `engineering/tools/build-dynamic-go.sh` 或
`pluginpackage -dynamic-go-fingerprint` 写入后再签名。Node Agent 会先验证制品并把签名
指纹交给 Go Runtime Host；Host 在 `plugin.Open` 前再次校验，避免不兼容模块先执行
`init()`。`required: true` 禁止进程回退，运维必须显式选择
`require-dynamic-go`；它不授予 Backend 进程内权限。Node Agent 仅在部署方放置
策略允许、`publisher=vastplan + cn.vastplan.*`、ABI/共同构建指纹/工具链/依赖完全匹配
时加载。第三方插件即使声明相同字段也只能按进程与隔离驱动处理。

## Go 插件

使用 `extensions/sdk/go/plugin` 创建 `Plugin`、调用 `Contribute` 登记处理器，最后 `Serve`。SDK 自动完成握手、Channel、生命周期、心跳、deadline、取消、HostCall 和资源边界。

处理器仍接收唯一的 Protobuf `CallContext`，但它已经由宿主按受众投影。业务代码优先调用 `plugin.ContextViews(callContext)`，读取 `Scope/Caller/Subject/Authorization/Trace/Request/Grant/Baggage` 防御性只读视图。不要缓存或修改原始指针来影响后续调用；dynamic-go 同样只收到独立快照。

dynamic-go 入口导出 `func VastPlanDynamicGo() protocolbus.DynamicGoModule`，返回
`EmbeddedPlugin` 定义。不要手写构建指纹；使用 `engineering/tools/build-dynamic-go.sh` 共同构建
Go Runtime Host 与 `.so`。Go 标准库 plugin 不能卸载，因此升级创建新 Host generation，
候选完全接入后切换路由并排空旧 Host；Backend 不加载 `.so`，也不需要随插件升级重启。

## Python 插件

开发环境安装：

```bash
python3 -m pip install -e extensions/sdk/python
```

使用 `vastplan_plugin.Plugin` 与 `Contribution`。处理器收到可协作取消的 `InvocationContext`，长任务应周期调用 `raise_if_cancelled()`。完整入口见 `extensions/plugins/cn.vastplan.python-hello/backend/main.py`。

Python 使用 `ContextViews.from_wire(call_context)` 获得与 Go 同构的只读语义视图。HostCall 委托引用由 SDK 绑定到当前处理线程并写入 `InvokeRequest.delegation_token`，不会进入 `CallContext.metadata`；当前 SDK 不允许把该委托扩散到自建线程，异步子任务应发布新任务或事件，由宿主重新签发上下文。

Python 清单的 `requirements.python` 声明解释器范围，其他条目只声明精确直接依赖；完整传递闭包必须使用 PEP 751 `supply-chain/pylock.toml` 1.0，并把全部 wheel 放入 `supply-chain/python-wheels/`。打包器会把锁摘要写入最终 Manifest，仓库与 Node Agent 复验每个 wheel 后才离线物化插件专属依赖 overlay。运行节点不访问 PyPI、不解析新版本、不构建 sdist。

构建环境可以用 pip、uv 等工具生成标准锁。当前安全子集只接受单一默认安装集合和本地 wheel；一个规范包名只能出现一次，允许同版本携带多个平台 wheel。`pluginpackage -source` 会拒绝 URL、VCS、editable、sdist、缺少或多余 wheel，以及摘要漂移。Python SDK、gRPC 和 Protobuf 由受信 Runtime Host 基座提供，不写入业务插件私有 requirements。

需要子解释器时使用：

```json
{
  "driver": "python-subinterpreter",
  "python": { "subinterpreterSafe": true },
  "requirements": { "python": ">=3.14" },
  "minimumIsolation": "trusted-runtime"
}
```

`subinterpreterSafe` 必须覆盖锁中的完整 Python 与原生扩展依赖；不确定时使用 `driver=python`。驱动不会把不兼容插件静默降级到另一执行形态。

子解释器 Runtime Host 位于 `core/runtimehosts/python-subinterpreter`。主解释器持有 gRPC/Protobuf 和可信协议会话，插件业务入口通过 `vastplan_subinterpreter` 的纯 Python 队列桥接运行在独立解释器中，避免要求 `grpcio` 本身支持多解释器。桥接开放静态贡献、Invoke 和处理器内同步 HostCall；HostCall 在原始主解释器调用线程执行，委托令牌不会进入子解释器。调用受父 deadline、30 秒上限、4 MiB payload 与取消约束。事件、动态贡献或状态迁移尚未开放，需要这些能力的插件继续明确使用 `driver=python`。Python 3.14 使用有 3.14 wheel 的 `grpcio>=1.82` 与 `protobuf>=6.33`；旧 Python 驱动继续使用原锁定版本。

开发环境会设置 `VASTPLAN_PYTHON_SUBINTERPRETER_HOST` 指向可信 Host；生产包必须随 Backend 一并安装该 Host。`host.py --probe` 返回机器可读能力，CPython 版本低于 3.14 或缺少 `concurrent.interpreters` 时拒绝启动。

部署需要指定非默认解释器时统一设置可信宿主环境 `VASTPLAN_PYTHON_INTERPRETER`；离线依赖物化、独立 Python 驱动和以脚本启动的子解释器 Host 使用同一解释器，避免安装与运行 ABI 漂移。未设置时使用 `python3`。

## Node 插件

Node Worker 插件入口使用 ESM，并声明：

```json
{
  "driver": "node-worker",
  "node": { "workerSafe": true, "moduleFormat": "esm" },
  "requirements": { "node": ">=20" },
  "minimumIsolation": "trusted-runtime"
}
```

插件代码运行在 Worker，不得依赖 `isMainThread=true`、修改宿主进程全局状态或调用 `process.exit()`。升级由 Runtime Host 创建新 Worker、等待协议激活、切换路由后终止旧 Worker；不能用 URL 查询参数反复 `import()` 冒充卸载。

首方参考实现位于 `extensions/sdk/node/backend-plugin`，Runtime Host 位于 `core/runtimehosts/node-worker`，示例插件为 `cn.vastplan.foundation.backend.runtime.node-worker-hello`。本地工作区先执行 `pnpm install`，`platform-dev.sh` 会把工作区生成的 `vastplan-node-worker-host` 命令仅注入开发编排器的子进程环境。生产发布物必须随 Backend 内核一并安装同名 Runtime Host，不能从插件制品临时下载宿主代码。

Node 插件制品不能依赖安装服务器上的 workspace `node_modules`。发布流水线必须把入口、本地模块和已声明 npm 依赖构建成自包含 ESM bundle，只允许保留 Node 内置模块；VastPlan 的开发 Seed 仓库由 `build-node-backend-plugins.mjs` 自动发现全部 `driver=node-worker` 清单，并通过 `pluginpackage -backend-module` 把 bundle 注入签名 `entry.backend`。CommonJS 依赖由构建器使用 `createRequire(import.meta.url)` 兼容桥，制品安装后仍不需要复制整个 `node_modules`。

## 多语言 SBOM

第一方插件使用 `pluginpackage -source` 打包时默认自动生成统一 CycloneDX 文档，取证边界与执行语言一致：

- Go 必须传入所有最终 `-backend-bin` / `-dynamic-go-bin`，依赖来自每个实际产物的 build info；
- Node Worker 必须传入构建后的 `-backend-module`，其同目录必须存在 `vastplan.node-metafile.json`；
- Frontend 必须传入构建后的 `-frontend-graph` 与 `-frontend-graph-root`，构建目录必须包含 `vastplan.browser-metafile.json`，含服务端模块时还包含 `vastplan.server-metafile.json`；
- Python 从 PEP 751 `pylock.toml` 读取完整传递依赖和 wheel 证据；`requirements` 只校验解释器范围与精确直接依赖。

自动生成器只读取最终构建证据，不通过源码 import 猜测依赖。缺少证据、Python 锁不闭合或尚未支持的运行驱动都会阻断打包。第三方流水线、工作区外源码以及 Rust/Java/C# 等语言应使用各自成熟工具生成 CycloneDX 1.5/1.6，再通过 `pluginpackage -sbom` 进入同一可信摄取路径；生成器无权绕过内核对插件 ID、版本、摘要、文档大小和组件上限的复验。

Service Hot 配置使用 `extensions/sdk/node/configuration-controller`。它与 Go SDK 共用 `configuration.v1` JSON wire、摘要 golden 和由插件 ID 派生的 `configuration.*` capability；插件只实现持久 `prepare/commit/abort/status` 状态机，不自行注册公开工具别名。动态 Profile 使用 `extensions/sdk/node/configuration-resource-controller` 与 `configuration.resource.v1`；秘密字段必须来自签名集合固定的 `managedCredentials[]`，浏览器不能直接提交 `CredentialRef`。两类 Node SDK 都不增加独立进程，控制器与业务贡献运行在同一个插件 Worker。

## Tenant/User Scoped Hot 配置

清单声明 `scope=tenant|user + applyMode=hot` 时，必须同时声明对 `configuration.scoped` 的 `remote + strong + readiness + fail` 运行时依赖。业务调用统一使用语言中立 `configuration.scoped.v1`：

- `resolve {}` 不接受配置 ID、插件 ID、tenant 或 subject；可信 resolver 从认证 caller 和 `CallContext` 精确推导目标；
- `watchRevision` 只返回 revision/digest 变化，不推送 values；观察到变化后重新 `resolve`；
- 当前只开放非敏感值。插件不得用普通字段承载密钥，也不得自行拼装或缓存 `CredentialRef`；
- 一个 caller 在当前活动 Catalog 中必须只有一个匹配的 Scoped 定义，否则 fail-closed。需要多个独立配置集合时，优先定义独立资源 Profile，而不是扩大 resolve 请求的目标选择面。

Go consumer 使用 `extensions/sdk/go/configurationscoped`，Node consumer 使用 `extensions/sdk/node/configuration-scoped`，Python consumer 使用 `vastplan_plugin.ScopedConfigurationClient`；三者都会复核闭合响应、Seed/Active 语义和值摘要。共享向量 `contracts/testdata/sdk-interop-v1.json` 同时验证规范 JSON 与 `ManagedCredentialRef`，Node 的通用引用校验位于 `extensions/sdk/node/credential-reference`，Python 使用不可变 `ManagedCredentialRef`。这些 SDK 不复制身份选择或授权逻辑，也不读取 credential material。首个参考实现位于 `extensions/plugins/cn.vastplan.hello-world/backend/configuration.go`。

## Shared State

跨实例状态使用 `state.shared.v1`，Go、Node、Python 客户端分别位于 `extensions/sdk/go/sharedstate`、`@vastplan/shared-state` 和 `vastplan_plugin.SharedStateClient`。请求不得携带 tenant、插件 ID、RuntimeScope、Leader epoch/token 或 NATS key；可信宿主从当前调用和验签启动身份派生。Create/Update/Delete 使用单 key CAS，跨 key 工作流继续使用插件自己的 Saga/outbox。需要 Leader 所有权的插件可使用 Go `NewFenced` 或 Node/Python `fenced=true`；只有当前 Unit Host 能执行 fenced mutation，Get/List 仍为普通读取。File Provider 只用于单进程开发，生产不可在 NATS/SQL Provider 失败时自动回退。

## Runtime Host 共享与专用模式

`node-worker`、`python-subinterpreter` 与 `dynamic-go` 默认按内核服务、Service Generation、Provider、隔离等级、发布者信任域和执行要求兼容摘要共享物理 Runtime Host。同一 Generation 的兼容插件可以共享 PID；活动代与候选代必须使用不同物理 Host，避免候选崩溃破坏上一可用代。插件仍拥有独立 Worker/子解释器/模块、协议 session、一次性票据和生命周期；共享 PID 不代表共享插件身份。实际态中的 PID 已去重。

开发时不要直接把工作区源码路径注入生产 Runtime。Backend 快速循环使用开发控制器生成的内容寻址临时候选，提交前使用唯一 `dev.*` 版本、签名测试仓库和 Test Release 复验；两者最终都经过相同的 Node Agent 候选状态机。附加 Delve、Node Inspector 或 debugpy 时，调试候选必须使用临时 dedicated Host，避免断点暂停同池插件。完整边界见 [ADR-0144](../decisions/ADR-0144-组合服务代际热升级与插件动态调试.md)。

只有部署方策略可以改变 Host 模式，Manifest 不能自行要求进入共享或专用故障域：

```bash
./bin/backend-kernel reconcile \
  -runtime-hosting-default shared \
  -publisher-runtime-hosting 'partner-a=dedicated' \
  -plugin-runtime-hosting 'cn.vastplan.platform.heavy-worker=dedicated' \
  -desired desired.json
```

精确插件规则优先于发布者规则，发布者规则优先于全局默认。`shared` 仍不能跨越发布者、隔离等级或 ABI/运行要求硬边界；`dedicated` 只影响托管语言驱动，不会把 native/python 隔离进程重新合池。当前每个兼容 Pool 固定最多一个 Host 且不会自动溢出扩容，真实负载形成后再通过独立 ADR 决定扩缩容阈值。

## 动态贡献、事件与取消

- 动态贡献只允许启停签名清单中已授权的条目；不能运行时创造新权限。
- 插件事件的 `source` 由宿主按会话身份覆盖，`principal_ref` 不接受插件自报。
- Go/Python/Node SDK 都会传播调用取消；处理器仍需协作退出，deadline 是最终兜底。
- 一个会话只有一条 Channel，SDK 不应自行创建第二条连接。
- `contextAccess.required/optional/baggage` 必须与实际读取字段一致；新增读取面要和清单、设计文档在同一次改动中更新。

## 第三方插件边界

Node Agent 的安全默认值仍要求未知发布者至少使用 `process-sandbox`；`native/python` 只提供 `trusted-process`，`node-worker/python-subinterpreter` 只提供 `trusted-runtime`，因此默认都会拒绝第三方制品。第三方只能使用 `process-sandbox`、`container` 或 `wasm-component` 等满足隔离下限的驱动。内核使用者可按部署信任边界配置：

```bash
./bin/backend-kernel reconcile \
  -third-party-plugin-policy require-isolation \
  -publisher-plugin-policies 'partner-a=allow-trusted,blocked-vendor=deny' \
  -default-plugin-context-access 'scope.tenant,caller,scene,subject.id,trace,request.deadline,propagation.callPath' \
  -publisher-plugin-context-access 'partner-a=scope.tenant,caller,trace;vastplan=*' \
  -desired desired.json
```

- 全局策略可取 `require-isolation`、`allow-trusted`、`deny`；
- `publisher=policy` 规则优先于全局策略和旧第一方兼容名单；
- `allow-trusted` 不能降低签名清单的 `minimumIsolation`；
- CallContext 上限独立于进程隔离策略；发布者精确规则优先于全局字段上限，`*` 仍不能绕过清单申请和扩展点上限；
- 旧 `-require-third-party-isolation=false` 等价于全局 `allow-trusted`，不得与新全局参数同时使用。

选择 `allow-trusted` 意味着部署方允许插件在宿主权限下执行，不能把它视为恶意代码沙箱。面向不可信第三方开放前仍至少需要真正的隔离驱动、文件/网络/进程/凭证策略和独立安全评估。

无论插件清单如何声明，第三方都不能自行进入 dynamic-go。Node Worker 和 Python 子解释器也不构成恶意代码沙箱；只有部署方显式信任的发布者才可使用，默认未知发布者仍被隔离下限拒绝。

### 隔离 Runtime Host 注册

Backend 支持三个由部署方显式注册的稳定隔离入口：

```bash
export VASTPLAN_PROCESS_SANDBOX_HOST=/opt/vastplan/runtimehosts/process-sandbox
export VASTPLAN_CONTAINER_HOST=/opt/vastplan/runtimehosts/container
export VASTPLAN_WASM_COMPONENT_HOST=/opt/vastplan/runtimehosts/wasm-component
```

变量为空时对应驱动不会注册，第三方插件继续 fail-closed。路径由 Backend 读取，不进入插件清单；相对命令必须能解析为绝对可执行文件。内核调用 Runtime Host 的 argv 契约为：

```text
<host> --plugin-id <signed-id> --plugin-root <installed-root> --entry <verified-entry> -- <manifest-args...>
```

参数不经过 shell，入口还会再次校验位于安装根内。Runtime Host 必须负责并接受单独安全审计：

- `process-sandbox` 至少限制文件系统、进程、系统调用、资源和网络，只放行回连协议宿主所需通道；
- `container` 必须使用固定镜像/运行参数策略、只读根、最小挂载、非特权用户、能力裁剪和网络策略；
- `wasm-component` 只授予 WIT 明示导入，不继承宿主文件、环境、网络或凭证；
- Runtime Host 自身属于信任计算基，不能从插件制品下载或由插件参数替换。

当前仓库交付的是内核侧强制策略、稳定适配器和 fail-closed 注册机制，不把未审计的通用 `bwrap/docker/wasmtime` 命令包装冒充生产隔离。每种生产 Host 应随具体部署基线独立实现和验收。
