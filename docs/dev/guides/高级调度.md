# 高级调度运行指南

## 1. 节点上报容量

生产 Node Agent 应按可分配量而不是物理总量启动，给系统和守护进程预留余量：

```bash
go run ./kernels/backend reconcile \
  -nats-url tls://nats.internal:4222 -deployment prod -tenant acme -node-id node-a \
  -labels region=cn,disk=ssd,maintenance=false \
  -capacity-cpu-millis 7600 -capacity-memory-bytes 30064771072 -capacity-gpu 1 \
  -nats-ca /etc/vastplan/pki/ca.pem -nats-cert /etc/vastplan/pki/node-a.crt \
  -nats-key /etc/vastplan/pki/node-a.key -nats-seed /secure/node-a.seed
```

容量字段都是非负整数。旧节点不带容量时视为 0，只能运行未声明资源请求的兼容规格。

## 2. Deployment v2 高级字段

```json
{
  "replicas": 2,
  "autoscaling": {
    "min_replicas": 2,
    "max_replicas": 6,
    "metric": "queue.depth",
    "target_value_per_replica": 20
  },
  "resources": {
    "requests": { "cpu_millis": 1000, "memory_bytes": 2147483648, "gpu": 0 }
  },
  "placement": {
    "nodeSelector": { "region": "cn" },
    "affinity": {
      "preferred": [{ "match_labels": { "disk": "ssd" }, "weight": 80 }]
    },
    "antiAffinity": {
      "required": [{ "match_labels": { "maintenance": "true" } }]
    }
  }
}
```

所有 required 条件是硬约束；preferred 只改变候选节点顺序，不会绕过资源检查。一个 unit 的同一副本仍不会在同一节点重复放置。

## 3. 发布伸缩指标

监控适配器使用 runtime 角色 NKey 定期写入指标；以下命令也可用于联调：

```bash
go run ./tools/autoscalingmetric \
  -nats-url tls://nats.internal:4222 -tenant acme -deployment prod \
  -unit api -metric queue.depth -value 53 \
  -nats-ca /etc/vastplan/pki/ca.pem -nats-cert /etc/vastplan/pki/metrics.crt \
  -nats-key /etc/vastplan/pki/metrics.key -nats-seed /secure/metrics-runtime.seed
```

上例 `ceil(53/20)=3`，因此目标是 3 个副本。指标必须在 2 分钟 TTL 内续报；缺失或过期时 Controller 记录未收敛原因并保留最近健康 assignment。

## 4. 运行边界

- 资源请求是调度预留，不是操作系统 cgroup 强限制；宿主级隔离需要运行时另行执行。
- 已提交的其他 Deployment 会计入节点占用，但不同 Deployment leader 同时首次放置仍是乐观并发窗口；强配额场景需要后续 CAS reservation/admission。
- unit 按 Deployment 中顺序规划；容量竞争时把业务优先级高的 unit 放在前面。
