# 本地平台管理中心

> 状态：可试用｜最后更新：2026-07-22
>
> 本文是开发者在单机启动完整平台管理中心的操作真源。生产节点接入仍以《[Linux 节点 SSH 首次引导](Linux节点SSH引导.md)》为准。

## 1. 一键启动

环境需具备 Go、C 编译器、Node.js、pnpm 和 curl。无需单独安装 NATS、Vault 或数据库。首次运行或环境变化后先执行：

```bash
./engineering/tools/platform-dev.sh doctor
```

检查通过后在后台启动：

```bash
./engineering/tools/platform-dev.sh up
```

首次启动会构建 Backend、Portal 和第一方插件，通常需要 1–3 分钟。后续启动会根据源码、插件清单和工具链版本的内容摘要复用未变化的构建结果，不再无条件重复构建。就绪后打开：

<http://127.0.0.1:18080/operations>

常用命令：

```bash
./engineering/tools/platform-dev.sh status
./engineering/tools/platform-dev.sh logs
./engineering/tools/platform-dev.sh logs --follow
./engineering/tools/platform-dev.sh restart
./engineering/tools/platform-dev.sh down
```

`status` 同时返回 `productionEquivalent: false`，避免把开发夹具误认为生产部署证明。

前端插件事务式热替换默认开启。编排器从每个插件 `vastplan.plugin.json#entry.frontend` 自动发现 UI 插件；新插件只需将源码置于 `frontend/src/index.(tsx|ts|jsx|js)`，并将产物入口声明为 `frontend/dist/` 下的 JavaScript 文件，无须修改开发工具中的插件名称列表。修改 `extensions/plugins/*/frontend/src/` 下的 TypeScript、TSX、CSS 或 JSON 后，编排器会构建一套候选模块；浏览器完成摘要校验、完整 Portal 装配和可选状态恢复后原子切换，不刷新文档。构建或装配失败时当前页面继续运行，并在页面右下角显示失败原因。`status` 的 `hotGeneration` 与 `hotError` 可用于诊断最近一次候选。

本地 Platform Profile 同时设置 `updates.mode=automatic`。经托管测试仓库和 Frontend Test Release 产生的真实新 Activation，会由 Node Portal Kernel 在完整预取后通过认证 SSE 通知已打开页面。兼容变更无刷新提交 Portal Generation；Renderer 或共享 UI 契约边界先预验证、再刷新 Host Epoch，失败时启动上一安全 Activation 而不循环刷新。生产 Profile 未显式选择时仍默认 `refresh`，浏览器不连接更新事件，也不轮询。

启动器还会校验并物化 `engineering/deploy/portal-access-profile-catalog.json`，核对其 Frontend Platform Profile 精确锁后交给 Node Portal Kernel。可用下面的只读端点检查会话前目录是否生效；响应只包含公开品牌、语言、方法策略与 Generation ID，不包含 tenant 或内部插件引用：

```bash
curl 'http://127.0.0.1:18080/auth/v1/bootstrap?returnTo=/operations'
```

开发网关在热加载开启时也会把 `/auth/*` 转发给 Node。`/auth/access` 已运行统一语义 AuthenticationFlow，会按 Access Profile 显示企业选择的认证方式。正常登录通过 Broker Assertion 建立会话；Provider 管理页的测试操作会进入隔离测试事务，完成真实认证但不签发或替换平台会话。

开发编排器同时启动两个只监听本机、职责不同的制品仓库：

- `https://127.0.0.1:18442` 是随本次运行创建的 Seed 仓库。`.vastplan/dev-platform/runs/<run-id>/repository/`、Profile、TLS 和 `local-development` 签名身份均为临时数据，只用于自举平台基础单元；
- `https://127.0.0.1:18443` 是由 `cn.vastplan.platform.artifacts.repository` 提供的托管测试仓库。File Volume 与 `local-testing` 签名身份保存在 `.vastplan/dev-platform/repositories/testing/`，普通重启后继续存在。

托管仓库的 File A/B 在线迁移状态保存在 `repositories/testing/control/repository-migration.json`，物理卷位于 `repositories/testing/volumes/`。管理 BFF 以 `prepare/sync/cutover/rollback/finalize/release` 显式阶段开放，迁移状态只以脱敏摘要进入 Workbench。`release` 会在配置已切到目标 volume 并重启后才生效，且只把旧卷移入 Provider 私有 quarantine，不立即删除。制品级 GC 另使用仓库 volume 内的 `catalog/garbage-collection.json` 恢复隔离/清扫进度；只接受先显式 yank/revoke、再 plan、再隔离至少 24 小时、最后复核 sweep 的路径。

开发 Profile 默认给托管仓库设置 10 GiB/10000 个活动制品的全仓上限，并给 `testing` channel 设置 5 GiB/5000 个的累积上限。目录、容量/配额、引用快照和 GC 已统一为 Workbench Collection 页面，通过制品仓库三级导航访问。具有 `platform.artifacts.gc` 权限的调用方才可执行隔离和到期清扫；开发身份中的 `platform.admin` 仅是在线角色迁移完成前的兼容映射，不应写入新角色设计。配额拒绝不会自动删除任何数据。

每次启动都会分别生成 Seed-only、testing-only 和只供 Node Agent 消费的组合信任快照。Seed 仓库不接受测试发布身份，托管仓库也不接受临时 Seed 身份。两个 Node Agent 先按精确 ref 查询本地签名 Seed，未命中才访问托管测试仓库，并对两类来源做相同的内核复验。消费 Inventory 不自动取得发布权限：只有平台 Node Agent 显式承担 `publish-bootstrap-references` 职责，其传输身份也只允许精确的 `bootstrap-inventory/local-seed` SYSTEM 子身份；managed-services Node Agent 只消费同一清单。测试签名私钥不会注入仓库或 Node Agent；后续发布工具只应在受控的本机测试发布路径中使用它。可用 `VASTPLAN_DEV_SEED_ARTIFACT_PORT` 和 `VASTPLAN_DEV_ARTIFACT_PORT` 覆盖两个端口，它们必须彼此不同，也不能与网关、Node Portal Kernel 和 Vault 端口冲突。

平台 Node Agent 默认启用 `-bootstrap-upgrade`；普通消费节点不启用。当仓库关键栈的目标版本不在 Seed、但已存在于托管仓库时，可信宿主会先把已验签候选与证明复制进 Seed，再启动候选；只有真实运行健康且 Assignment 引用发布成功后才推进 `seed-inventory.json` 的 LKG。Seed/LKG 引用使用当前 Deployment 的可信租户，不使用可绕过传输身份租户边界的伪全局租户。失败候选只会扩展 Seed，不会覆盖旧 LKG。该开发默认值不代替生产目录权限和 systemd 内核升级规划。

临时排查热替换自身问题时可以关闭它；该参数同时让 Portal Shell 不连接开发事件源：

```bash
./engineering/tools/platform-dev.sh restart --no-hot
```

需要在终端直接观察完整构建与运行日志时使用调试模式；`Ctrl+C` 会由编排器按依赖倒序关闭受管进程：

```bash
./engineering/tools/platform-dev.sh up --debug
```

### 1.1 发布测试制品

待测试插件必须先由对应语言构建器产出入口，并将清单、Backend 握手身份和前端 RuntimeSpec 统一为唯一 `dev.*` SemVer 预发布版本，例如 `0.4.0-dev.20260720.3.a81c33f`。随后用 `pluginpackage` 生成包，再通过受控入口上传：

```bash
go run ./engineering/tools/pluginpackage \
  -source extensions/plugins/<plugin-id> \
  -backend-bin <已构建入口> \
  -out /tmp/<plugin-id>.tar.gz

./engineering/tools/platform-dev.sh publish-test /tmp/<plugin-id>.tar.gz
```

含 Frontend 的插件先运行 `pnpm build:frontend:plugins`，再使用构建器生成的可信图打包；`-frontend-bundle` 仅保留给旧单文件迁移输入：

```bash
plugin=extensions/plugins/<plugin-id>
go run ./engineering/tools/pluginpackage \
  -source "$plugin" \
  -frontend-graph "$plugin/frontend/dist/vastplan.browser-graph.json" \
  -frontend-graph-root "$plugin" \
  -out /tmp/<plugin-id>.tar.gz
```

打包器会逐节点复算大小和 SHA-256 后把 Graph 注入最终签名 Manifest；仓库接收和每次读取制品时由 `artifacttrust` 再次扫描 tar 实际字节。缺失节点、替换内容、路径逃逸、未知外部依赖或 graph digest 漂移均 fail-closed。

`publish-test` 不接受任意仓库 URL、channel、key ID 或命令行 token。它只读取当前回环开发平台声明的持久测试仓库，使用 `local-testing` 身份签名，强制 `testing` channel，并在返回前确认 Catalog 中的精确 ref、SHA-256 和 repository revision。

只上传、不改变运行服务时使用上面的命令。要把同一份精确回执继续提交给 Backend Test Release Controller，可选择以下一种方式：

```bash
# 首次或日常开发：自动创建/校准仅限本地开发的目标绑定
./engineering/tools/platform-dev.sh publish-test /tmp/<plugin-id>.tar.gz \
  --backend-target managed-services/hello-service

# 复用平台管理员已经创建的绑定
./engineering/tools/platform-dev.sh publish-test /tmp/<plugin-id>.tar.gz \
  --backend-binding dev-or-admin-binding-id
```

`--backend-target` 的格式固定为 `deployment/unit`，插件 ID 来自已验签制品，不能由命令行替换。CLI 用本地管理员身份保存绑定，再用独立发布者身份提交 Test Release；候选失败会返回非零，并报告已经安全回滚还是仍需人工恢复。该便利入口只在状态端点证明目标为回环、`local-development` 且非生产等价模式时启用。生产环境必须由管理中心显式预授权 binding，不能复用这些固定开发会话。

Frontend 功能插件已在 Application 中时，使用：

```bash
./engineering/tools/platform-dev.sh publish-test /tmp/<plugin-id>.tar.gz \
  --frontend-target operations
```

Renderer、Shell、Workbench 或当前属于 Platform Profile 的平台管理 UI，必须显式选择专用测试 Profile：

```bash
./engineering/tools/platform-dev.sh publish-test /tmp/<plugin-id>.tar.gz \
  --frontend-target operations \
  --frontend-scope platform-profile-plugin
```

后一种方式会生成 tenant 专用的新 Profile 和 Binding revisions，不会改写共享平台基线。成功后新 Activation 已由 Backend `portaltrust` 物化，并可由 Node Portal Kernel 冷填充；当前生产语义是用户刷新时加载，测试开发态可使用自动 Generation/Host Epoch 切换。

需要丢弃旧的运行目录、持久测试仓库和构建缓存并完整重建时显式使用：

```bash
./engineering/tools/platform-dev.sh restart --fresh
```

开发阶段如果修改了某个已进入 Seed 的稳定版本源码，却没有同步修改其 SemVer，新包会因“同 ref 不同摘要”被不可变仓库正确拒绝。这种情况应使用 `--fresh` 重建开发环境，或者为可反复发布的候选包使用唯一 `dev.*` 预发布版本；不能放宽仓库不可变约束。

`--timeout <秒>` 可覆盖默认 300 秒启动等待。脚本只检查并管理 `.vastplan/dev-platform/` 对应的进程，不会像通用端口清理脚本一样终止其他项目的端口占用者；端口冲突会在耗时构建前直接失败并显示占用进程。

### 1.2 开发构建缓存

本地启动器把构建结果按内容摘要保存在 `.vastplan/dev-platform/build-cache/`，并保留共享的 Go 编译缓存 `.vastplan/dev-platform/go-cache/`。每次运行仍创建独立运行目录，但优先用硬链接装配已验证的不可变产物；文件系统不支持硬链接时自动退回复制。

缓存分为 Backend、Portal/前端插件、dynamic-go 和本地插件制品仓库四个阶段：

- 源码、插件清单、构建脚本或对应工具链版本变化时，只使受影响阶段及其下游制品失效；
- `dist/`、`node_modules/` 等派生目录不参与源码摘要，避免构建输出反过来触发重建；
- 候选产物验证通过后才原子写入完成标记；残缺或被破坏的缓存会自动重建；
- `restart --fresh` 和 `clean` 会清除缓存，适合验证真正的冷启动。

这套缓存只优化本机开发和测试启动，不定义生产制品仓库、生产数据持久化或跨节点缓存策略。

## 2. 启动内容

启动器会为每次运行装配独立、不可变的 Seed 仓库，并挂载跨运行保留的托管测试仓库；Seed 构建内容未变化时从开发构建缓存复用。平台包含：

- 嵌入式 NATS JetStream、双目标 Controller fleet、平台 Node Agent 和独立 managed-services Node Agent；
- 七个 Backend 平台服务单元：全局设置、API Exposure、凭证、数据库连接管理、单节点开发投影的 active-active Database Runtime、制品仓库和节点部署；
- 每个服务单元本地附加的平台管理权限策略；
- 以 `dynamic-go` 内嵌运行的 bootstrap-policy；
- Node Portal Kernel、Arco 设计系统、标准/顶部布局制品，以及经 Published 输入与 CAS Activation 上线的 `/operations` Portal；
- 仅绑定回环地址的前端源码监听、候选模块端点和 SSE 热替换事件；
- 全局设置、API 暴露、凭证引用、数据库连接、制品仓库和“服务与节点部署”页面；
- 平台预授权的 `local/managed-services` 目标，以及自动走完异人审批后运行的 `hello-service` 示例。

Backend 服务页面只允许配置 Application Composition。Platform Profile、平台插件、控制面 key 和真实路由目标不会出现在该表单中；最终 Deployment v2 预览会显示内核规范化和注入后的结果。Portal 管理中心则单独提供 Platform Profiles 与 Portals 工作区：Profile、Application、Binding 发布后只是可选输入，只有创建 Activation 才改变线上 Portal；切换布局和回滚都产生新的不可变 Activation。

本地默认浏览器会话是 `local-admin`。服务提交人与审批人必须不同；需要手工演练完整流程时，可在三个独立浏览器窗口的控制台分别设置开发 cookie 后刷新：

```javascript
document.cookie = "vastplan_session=vastplan-local-portal-author; path=/"
document.cookie = "vastplan_session=vastplan-local-portal-approver; path=/"
document.cookie = "vastplan_session=vastplan-local-portal-publisher; path=/"
```

这些固定 token 仅存在于绑定 `127.0.0.1` 的开发夹具，生产环境禁止使用。

## 3. 本地安全边界

这个入口只用于开发试用：

- 所有监听地址固定为 `127.0.0.1`；
- 开发网关注入本地管理员 session，并终止运行期生成的自签名 TLS；
- NATS 使用随机本机端口和显式 insecure 开发模式；
- 凭证插件连接仅供本地测试的 Vault Transit 兼容桩；
- Seed 制品仓库只承载本次运行由临时受信任密钥签名的自举制品；托管测试仓库保留跨运行数据，但仍仅用于本机开发；
- 持久测试签名私钥和所有仓库目录必须保持 `0600/0700` 权限，符号链接或宽权限会使启动 fail-closed；
- 所有运行数据位于 `.vastplan/dev-platform/`，已被 Git 忽略。
- 开发模块只保留最近 8 个 Generation，使用独立 URL 解析入口，不写入不可变插件仓库或正式 Portal 快照。

不要把 `18080` 或内部随机端口暴露到局域网、公网或反向代理。生产环境必须使用真实身份提供方、mTLS/NKey NATS、Vault/KMS、签名制品仓库和受控 Linux Node Agent。

## 4. 运行数据与排障

启动日志：

```text
.vastplan/dev-platform/platformdev.log
```

每次运行的制品、安装目录、状态和临时密钥位于：

```text
.vastplan/dev-platform/runs/<run-id>/
```

跨普通重启保留的测试仓库数据与本地测试签名身份位于：

```text
.vastplan/dev-platform/repositories/testing/
```

普通重启还会保留 Portal Composer/Interaction Broker、Deployment Manager、全局设置、凭证引用、数据库连接、Node Agent ActualState 与 Portal 交付源快照：

```text
.vastplan/dev-platform/state/
```

这些状态的单调 generation 不会因新建 `runs/<run-id>/` 而回退；单次运行的锁、缓存、临时安装目录和会话密钥仍位于当次 run 目录。启动器不会直接相信保留下来的 `active/ready`：只有 ActualState 在本次 Node Agent 启动后重新写入、目标 unit 均无候选事务且全部就绪，才会开放后续 Portal 和代理入口。

该签名身份仅是开发夹具，不是生产发布密钥。`restart --fresh` 与 `clean` 会删除整个开发状态根，因此也会明确删除该测试仓库；普通 `restart` 不会。

`logs` 默认显示最近 100 行，`logs --follow` 持续跟踪；也可用 `logs --lines <行数>` 调整初始行数。后台启动会把编排器的 `[1/6]` 到 `[6/6]` 阶段进度同步显示到终端，失败时自动打印最近日志。

PID 文件失效或 PID 被系统复用时，脚本只清理 PID 文件，不会误杀对应进程；编排器 PID 文件丢失时可从精确二进制路径恢复。异常退出遗留的子进程仅在命令路径位于当前 `.vastplan/dev-platform/runs/` 时才会被回收。

需要删除全部历史运行数据时，先停止平台再清理：

```bash
./engineering/tools/platform-dev.sh down
./engineering/tools/platform-dev.sh clean
```

`clean` 在检测到本项目受管进程仍运行时会拒绝执行。

## 5. 与生产部署的关系

本地启动器验证的是同一套插件清单、Backend/Frontend 双输入组合解析、服务审批与摘要锁、Controller fleet、Node Agent 调度收敛、能力寻址、权限判定、Portal 内容寻址快照和动态模块加载。它有意替换了外部基础设施和登录入口，因此不验证：

- SSH 首次引导、systemd 托管和远端主机故障；
- 生产 NATS 身份、ACL、证书轮换和多节点网络；
- 真实 Vault/KMS、企业身份提供方和签名发布审批；
- 多节点副本、故障转移、容量测试与 Soak。

生产部署流程见《[服务部署控制台](../architecture/服务部署控制台.md)》《[Linux 节点 SSH 首次引导](Linux节点SSH引导.md)》和《[NATS 生产安全](NATS生产安全.md)》。
