# VastPlan 当前系统开发成果（非技术版）

> 状态快照：2026-07-16，代码提交 `47ce523`，Backend 内核版本 `0.1.0`  
> 阅读对象：管理层、产品、业务和项目协作人员  
> 本文只做阶段成果说明，不替代技术设计和验收记录；详细依据见文末“核验依据”。

## 一、结论先行

VastPlan 已经完成一套**可运行、可扩展、可部署、可验证的 Backend 微内核基础设施**。它可以安全地接入插件，把插件分发到不同节点运行，并处理调用、权限、事件、升级、故障恢复、监控诊断和制品安全等平台级问题。

通俗地说，当前已经造好了“平台的发动机、底盘、配电系统和质量检测线”，并用 4 个示例插件证明主要链路可以工作；但面向最终用户的“车身、座舱和具体用途”还没有完成。

因此，当前阶段可以这样定性：

- **Backend 内核主体功能和工程化建设基本完成。**
- **尚不能称为完整产品，也尚未正式发布 Backend 1.0。**
- 下一步最有价值的工作不是继续扩大内核，而是开发有代表性的业务插件，形成一条真实业务闭环，再用真实负载完成 24 小时稳定性验证。

## 二、系统最终要解决什么问题

VastPlan 的目标是为企业提供一套通用 Agent 平台：企业可以在线开发和管理 Agent，把脚本或工作流下发到远端客户端执行，并通过统一插件机制持续增加能力。

系统采用“微内核 + 插件”的方式建设：

- 内核负责稳定、安全、通用的基础能力；
- 权限、审计、Agent、工作流、业务工具和界面等具体功能通过插件实现；
- 新功能可以独立开发、安装、升级和回滚，尽量不修改或重启内核；
- 当前插件均为本方开发和维护，暂不开放第三方不可信插件。

## 三、当前已经完成的成果

### 1. 插件运行基础已经打通

系统已经能够识别插件、检查插件是否兼容、启动独立插件进程、注册插件能力、调用插件、停用插件并清理资源。单个插件发生故障时，内核能够隔离故障并摘除失效能力，避免把问题扩散到整个系统。

已固定 7 类 Backend 插件扩展能力，包括工具、Agent、接口、权限、事件、调用钩子和 Runner 能力。未知或不合规插件会被默认拒绝，而不是带风险运行。

### 2. 插件在线安装、升级和回滚已经具备

节点代理可以按照平台发布的“期望状态”自动安装、启用、停用和升级插件，并持续检查实际运行状态。升级采用“先启动并验证新版本，再切换旧版本”的方式；新版本失败时保留或恢复旧版本。

对需要保存业务状态的插件，系统提供明确的迁移事务和回滚边界，避免内核擅自修改插件私有数据。

### 3. 多节点部署和自动调度已经具备

系统能够把插件副本自动分配到多个节点，并支持：

- 根据节点标签选择部署位置；
- 根据 CPU、内存等容量做资源感知调度；
- 设置亲和或反亲和规则；
- 根据外部指标调整副本数量；
- 节点失联后重新分配；
- 多个控制器之间自动选主和故障接管；
- 节点下线前先停止接收新请求，再等待在途请求完成。

这些能力已经通过自动化测试验证，但还需要在目标生产环境中结合实际网络、证书和容量配置做部署验收。

### 4. 跨节点通信和事件链路已经具备

同一能力可以在本机直接调用，也可以通过网络调用其他节点，业务调用方不需要关心能力具体运行在哪里。系统还支持双向流式通信、持久事件、离线恢复、重复消息控制和调用取消传播。

这为后续接入远端 Runner、日志流、任务进度和跨服务协作提供了基础。

### 5. 远端制品仓库与供应链安全已经具备

插件包可以通过远端仓库发布和下载。系统会同时验证：

- 插件清单和结构是否符合规范；
- 文件内容是否与 SHA-256 摘要一致；
- 发布者 Ed25519 签名是否可信；
- HTTPS 传输是否可信；
- 同一版本是否被非法覆盖。

Backend 发布流程还包含可复现构建、依赖清单（SBOM）、漏洞检查、许可证检查、校验和和来源证明，降低“源码正确但交付物被替换或来源不明”的风险。

### 6. 运行诊断、升级和回滚工具已经具备

Backend 二进制已经内置版本查询、配置预检和脱敏支持包能力。运维人员可以在升级前检查配置兼容性，在异常时收集不含凭证明文和业务载荷的诊断信息，并按标准步骤逐节点升级或回滚。

### 7. 代码质量和工程门禁已经建立

当前工程门禁覆盖代码格式、静态检查、单元测试、真实跨进程测试、架构分层、协议兼容、漏洞、许可证、可复现构建和发布安全。架构规则会自动阻止上下级分层混乱、跨级依赖、公共类型重复定义、生产入口散落以及明显复杂度或重复代码回归。

当前仓库包含 172 个命名测试、7 个性能基准和 3 个模糊测试入口；核心测试已通过竞态检测，真实插件进程和多节点链路也有端到端验证。

## 四、现在可以现场演示什么

当前可以组成一条完整的平台技术演示：

```mermaid
flowchart LR
  A["发布已签名插件"] --> B["远端制品仓库"]
  B --> C["控制面决定部署位置"]
  C --> D["节点自动安装并启动"]
  D --> E["内核调用插件能力"]
  E --> F["权限、Hook 与审计生效"]
  F --> G["故障恢复、升级或回滚"]
```

仓库中已有 4 个示例插件用于证明机制：

| 示例插件 | 可以证明的能力 |
|---|---|
| Hello World | 插件接入、工具调用和插件回调内核 |
| 示例权限 | 多条权限规则按优先级判定，默认拒绝不安全调用 |
| 示例审计 | 事件分发、订阅和审计记录查询 |
| 示例配额 | 调用前限流、调用后计量和有序 Hook 分发 |

这些插件是**平台机制样例**，不是已经完成的企业业务产品。

## 五、当前完成度与边界

| 领域 | 当前状态 | 非技术解释 |
|---|---|---|
| Backend 微内核主体 | 绿色：基本完成 | 主要平台底座已实现并通过工程验证 |
| 插件机制与示例插件 | 绿色：机制完成 | 已证明插件能开发、安装、调用、升级和恢复 |
| 多节点控制与调度 | 绿色偏黄 | 代码和自动化测试完成，仍需目标环境部署验收 |
| 制品与供应链安全 | 绿色偏黄 | 技术链路和发布门禁完成，生产密钥、证书和仓库需按环境配置 |
| Backend 正式 1.0 | 黄色：待最终验证 | 版本仍为 0.1.0，代表性 24 小时稳定性测试尚未执行 |
| 业务 Agent、Workflow、用户系统、LLM 能力 | 灰色：未纳入本阶段 | 应由后续业务插件和应用层实现 |
| Frontend、Runner、Mobile 内核 | 灰色：尚未交付 | 架构方向已确定，但没有达到 Backend 同等实现程度 |
| 第三方插件开放 | 灰色：当前不做 | 当前仅支持本方可信插件，第三方隔离属于未来议题 |

另外，强资源原子预留、跨地域流量治理和有状态服务跨节点迁移等更高级能力尚未实现。这些不影响当前 Backend 基础闭环，但会影响未来超大规模或跨地域部署方案。

## 六、为什么还没有发布 Backend 1.0

正式 1.0 要求冻结候选代码，并在有代表性的持续负载下运行至少 24 小时，证明进程、会话、文件句柄、队列和内存不会持续增长，且插件调用、重启和故障恢复能够长期稳定。

当前只有合成测试链路，不能代表未来真实插件的混合负载。项目已主动决定：先开发代表性插件和调用模型，再重新冻结候选并执行完整 24 小时测试。此前取消的测试运行不作为发布证据。

这不是“功能没有做完”，而是拒绝用缺乏代表性的测试给出错误的生产承诺。

## 七、下一阶段的三种推进方案

### 方案 A：业务闭环优先（推荐）

先选择 2—3 个代表性业务插件，覆盖工具调用、权限、状态持久化、事件、长任务或 Runner 协作；随后补齐最小可用界面或调用入口，形成一条真实业务闭环。业务模型稳定后，再据此设计混合负载并完成 24 小时稳定性测试和 Backend 1.0 发布。

优点是能够尽早验证平台是否真正支撑业务，同时让最终稳定性测试具有代表性；代价是正式 1.0 标签会晚于业务验证开始。该方案与当前“暂缓无代表性 soak”的决定一致。

### 方案 B：Backend 1.0 标签优先

先构造更丰富的合成插件和负载，完成稳定性测试并发布 Backend 1.0，再开始业务插件开发。

优点是内核版本里程碑更早、接口承诺更明确；风险是合成负载仍可能遗漏真实业务模式，第一批业务插件可能暴露新的接口或性能问题，导致 1.0 后只能兼容演进。

### 方案 C：四端并行建设

Backend 继续封板的同时，并行建设 Frontend、Runner、Mobile 和业务插件，尽快展示完整产品形态。

优点是覆盖面扩张最快；代价是协调复杂、人员要求高，多个尚未验证的边界会同时变化，返工和集成风险最大。只有在有独立团队和清晰业务样板时才适合采用。

## 八、推荐实施路径和验收条件

建议采用方案 A，并按以下顺序推进。每一阶段以可验收结果为出口，不以“写了多少代码”为完成标准。

| 阶段 | 主要工作 | 阶段验收条件 |
|---|---|---|
| 1. 业务样板选择 | 明确 2—3 个代表性插件及一条端到端业务流程 | 场景、数据、权限、故障和容量假设形成书面清单 |
| 2. 代表性插件开发 | 实现真实业务插件，补齐所需持久化、事件和长任务链路 | 插件可独立安装、升级、回滚，并通过真实进程测试 |
| 3. 最小产品闭环 | 建设必要的 Frontend 或 Runner 入口 | 非开发人员可完成一次完整业务操作并查看结果 |
| 4. 生产候选验证 | 冻结 Backend 候选，建立真实混合负载 | 全部门禁通过，完整 24 小时报告合格且资源收敛 |
| 5. Backend 1.0 与试点 | 发布可验证制品，在受控环境逐步部署 | 试点环境可升级、回滚、诊断，业务关键路径稳定 |

当前不建议继续无边界地增加内核功能。新增需求应先判断：它是所有业务都需要的稳定基础，还是可以独立替换的业务能力；后者应优先放入插件。

## 九、管理层可采用的阶段验收结论

建议对当前阶段作如下验收表述：

> VastPlan 已完成 Backend 微内核主体能力、插件运行主链、多节点控制、可信制品、安全发布和工程质量门禁建设，具备进入代表性业务插件与最小产品闭环开发阶段的条件。当前成果属于平台内核和工程基础，不等同于完整业务产品；Backend 正式 1.0 将在真实插件负载形成并完成 24 小时稳定性验证后发布。
