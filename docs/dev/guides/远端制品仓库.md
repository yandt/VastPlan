# 远端制品仓库运行指南

本指南是 ADR-0026 与 [ADR-0134](../decisions/ADR-0134-testing候选与stable发布审批绑定.md) 的可执行入口。生产链路固定为“生成发布密钥 → 分发公钥信任文档 → HTTPS 托管仓库 → testing 签名发布 → 异人审批 → 同包 stable 签名发布 → Node Agent 本地复验”。

## 1. 初始化发布密钥

```bash
go run ./engineering/tools/artifactkey \
  -publisher vastplan \
  -key-id release-2026-01 \
  -private-out /secure/release-2026-01.pem \
  -trust-out engineering/deploy/artifact-trust.json
```

私钥文件以 `0600` 创建且工具拒绝覆盖。私钥只进入受控发布流水线；`artifact-trust.json` 分发到制品服务和所有 Node Agent。

## 2. 部署 HTTPS 托管仓库

生产环境必须通过 `cn.vastplan.platform.artifacts.repository` 与 `cn.vastplan.platform.artifacts.storage.file`（或后续受支持 Provider）部署托管仓库，并由 Backend Platform Profile 提供配置、环境白名单和 Leader 运行策略。`engineering/deploy/platform-management-profile.json` 是本地组合示例。

以下 `artifact-server` 只用于单机签名传输兼容和 TLS 联调，不包含 Catalog、发布审批、迁移与 GC，不能作为 production stable 发布入口：

```bash
export VASTPLAN_ARTIFACT_READ_TOKEN='node-agent-read-token'
export VASTPLAN_ARTIFACT_PUBLISH_TOKEN='ci-publish-token'
go run ./core/kernels/backend artifact-server \
  -addr 0.0.0.0:8443 \
  -repository /var/lib/vastplan/artifacts \
  -trust engineering/deploy/artifact-trust.json \
  -tls-cert /etc/vastplan/tls/server.crt \
  -tls-key /etc/vastplan/tls/server.key
```

两个令牌必须不同并由秘密管理系统注入。生产服务不支持明文 HTTP，也不允许空令牌匿名访问。

## 3. testing 候选、审批与 stable 发布

### 3.1 生成一次候选并发布 testing

```bash
export VASTPLAN_ARTIFACT_PUBLISH_TOKEN='ci-publish-token'
go run ./engineering/tools/pluginpackage \
  -source extensions/plugins/cn.vastplan.hello-world \
  -backend-bin bin/cn.vastplan.hello-world \
  -sbom /secure/candidates/cn.vastplan.hello-world-0.1.0.cdx.json \
  -license-file LICENSE \
  -notice-file NOTICE \
  -out /secure/candidates/cn.vastplan.hello-world-0.1.0.tar.gz \
  -channel testing \
  -remote-repository https://artifacts.example.com \
  -remote-ca /etc/vastplan/tls/company-ca.pem \
  -trust engineering/deploy/artifact-trust.json \
  -sign-key /secure/release-2026-01.pem \
  -key-id release-2026-01
```

`-license-file` 与 `-notice-file` 的内容会分别写入清单声明的 `licenseFile`、`noticeFile` 路径；仓库内第一方插件默认使用根 `LICENSE` 和 `NOTICE`。打包器会拒绝声明了法律文件但制品中缺失、重复或为空的包。`-sbom` 接受 CycloneDX JSON 1.5/1.6，主体必须是当前插件 ID/版本；打包器将其写入固定路径并把摘要写入签名清单。候选文件应作为 CI 不可变产物保留，后续不得重新加工。

### 3.2 在平台管理中心审批

一位拥有 `platform.artifacts.publication.submit` 的用户从制品目录提交该 testing 版本；另一位拥有 `platform.artifacts.publication.approve` 的用户在“发布审批”页面批准。仓库将申请精确绑定 testing/stable ref、SHA-256、publisher、key ID 和源证明摘要，同一用户不能完成两步。

申请默认 7 天过期，生产服务可用插件配置 `publication.approvalTTLHours` 在 1–720 小时之间调整。独立审批人可驳回待批或已批申请，原提交人可撤销；两者都必须填写原因。`Rejected`、`Cancelled`、`Expired` 与 `Published` 均不可恢复，要重新晋级应发布新的 testing 候选，而不是复活旧审批。CLI 开始 stable 上传前应重新读取审批状态，但最终仍以仓库写入前的服务端检查为准。

### 3.3 原样复用候选并发布 stable

```bash
export VASTPLAN_ARTIFACT_READ_TOKEN='ci-read-token'
export VASTPLAN_ARTIFACT_PUBLISH_TOKEN='ci-publish-token'
go run ./engineering/tools/pluginpackage \
  -package /secure/candidates/cn.vastplan.hello-world-0.1.0.tar.gz \
  -channel stable \
  -remote-repository https://artifacts.example.com \
  -remote-ca /etc/vastplan/tls/company-ca.pem \
  -trust engineering/deploy/artifact-trust.json \
  -sign-key /secure/release-2026-01.pem \
  -key-id release-2026-01
```

stable 路径强制使用不同的读取和发布令牌。客户端先通过读取令牌拉取并再次验签 testing 候选，比较插件 ID、版本、包 SHA-256、大小、publisher 和 key ID；只有完全一致才生成新的 stable 证明并使用发布令牌上传。服务端随后再次验签并消费精确 `Approved` 记录。`-package` 与 `-source` 互斥，也不能再指定 Backend、Frontend 或 dynamic-go 注入参数，避免审批后改变字节。两类令牌都应通过环境或 CI Secret 注入，不要出现在命令行历史中。

仓库安全默认值要求 stable 制品已经在 testing 包内绑定 SBOM；`-package` 阶段禁止补加或替换。企业若需要改变强制 channel，只能在仓库服务配置中设置 `supplyChain.requiredSBOMChannels`，不要在 CI 客户端放宽服务端准入。

服务端会在授权 stable 写入的同一串行临界区再次收敛过期状态；即便批准后尚未上传，审批人驳回、提交人撤销或有效期到期都会使上传立即失败。

## 4. Node Agent 使用远端仓库

```bash
export VASTPLAN_ARTIFACT_READ_TOKEN='node-agent-read-token'
bin/backend-kernel reconcile \
  -desired engineering/deploy/local.desired-state.json \
  -repository-url https://artifacts.example.com \
  -repository-trust engineering/deploy/artifact-trust.json \
  -repository-ca /etc/vastplan/tls/company-ca.pem
```

Node Agent 验证顺序为：请求 ref 与证明一致 → publisher/keyId 受信任且未撤销 → Ed25519 签名 → 下载大小上限 → SHA-256 → tar 根清单 → 清单与索引绑定。任一步失败均不会进入安装器。

需要在空节点预置仓库基础插件或气隙启动时，可增加签名种子仓库：

```bash
bin/backend-kernel reconcile \
  -nats-url tls://nats.example.com:4222 \
  -node-id node-a \
  -assignment-key deployments/acme/platform/nodes/node-a \
  -bootstrap-repository /opt/vastplan/bootstrap-repository \
  -bootstrap-inventory /var/lib/vastplan/bootstrap/inventory.json \
  -repository-trust /etc/vastplan/artifact-trust.json \
  -repository-url https://artifacts.example.com
```

种子源按精确 ref 优先读取；只有明确 `not found` 才访问远端。Bootstrap Inventory 必须是由部署管理员提供的绝对 `0600` JSON 文件，父目录不可被 group/other 写入；它记录 Seed 全集、LKG 子集和单调 generation。种子文件损坏、权限错误、格式错误或证明不可信都会立即停止，不会通过换源掩盖安全事件。制品源返回的内容始终视为未信任，Node Agent 验证成功后才交给安装器。

生产环境需要允许仓库关键插件在线升级时，再显式增加 `-bootstrap-upgrade`，并通过 `-nats-url` 等参数接入完整控制面。该模式要求 Node Agent 宿主对 Seed 和 Inventory 有窄写权；仓库插件仍无权访问二者。事务顺序固定为候选验签、镜像 Seed、启动与健康、发布 Assignment、推进 LKG、发布 Bootstrap 引用。自动路径拒绝跨 channel 与 SemVer 降级；内核二进制升级仍走 systemd/Guardian 流程。

## 5. 密钥轮换与撤销

1. 生成新 keyId，把新公钥加入信任文档，但保留旧 key。
2. 先更新制品服务和 Node Agent 的信任文档，再切换发布流水线签署密钥。
3. 确认所有仍需安装的版本已经用新 key 重新发布后，把旧 key 标记为 `"revoked": true`。

撤销是 fail-closed 的：旧 key 签署的历史制品将不可再下载安装。紧急泄漏时优先撤销，再处理历史制品重签。

没有可信时间戳服务时，`NotAfter` 到期也会使该 key 的全部历史签名停止通过验证；不得依赖证明中由签名者自行填写的 `signedAt` 延续信任。因此必须在旧 key 到期前完成仍需安装制品的重签、信任文档分发和 Node Agent 验证。
