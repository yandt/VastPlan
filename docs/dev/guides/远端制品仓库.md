# 远端制品仓库运行指南

本指南是 ADR-0026 与 [ADR-0134](../decisions/ADR-0134-testing候选与stable发布审批绑定.md) 的可执行入口。生产链路固定为“生成发布密钥 → 分发公钥信任文档 → HTTPS 托管仓库 → testing 签名发布 → 异人审批 → 同包 stable 签名发布 → Node Agent 本地复验”。

## 1. 初始化发布密钥

```bash
go run ./engineering/tools/artifactkey \
  -publisher vastplan \
  -key-id release-2026-01 \
  -private-out /secure/release-2026-01.pem \
  -trust-out engineering/deploy/artifact-trust.json
```

私钥文件以 `0600` 创建且工具拒绝覆盖。私钥只进入受控发布流水线；`artifact-trust.json` 分发到制品服务和所有 Node Agent。

## 2. 部署 HTTPS 托管仓库

生产环境必须通过 `cn.vastplan.platform.artifacts.repository` 与 `cn.vastplan.platform.artifacts.storage.file`（或后续受支持 Provider）部署托管仓库，并由 Backend Platform Profile 提供配置、环境白名单和 Leader 运行策略。`engineering/deploy/platform-management-profile.json` 是本地组合示例。

以下 `artifact-server` 只用于单机签名传输兼容和 TLS 联调，不包含 Catalog、发布审批、迁移与 GC，不能作为 production stable 发布入口：

```bash
export VASTPLAN_ARTIFACT_READ_TOKEN='node-agent-read-token'
export VASTPLAN_ARTIFACT_PUBLISH_TOKEN='ci-publish-token'
go run ./core/kernels/backend artifact-server \
  -addr 0.0.0.0:8443 \
  -repository /var/lib/vastplan/artifacts \
  -trust engineering/deploy/artifact-trust.json \
  -tls-cert /etc/vastplan/tls/server.crt \
  -tls-key /etc/vastplan/tls/server.key
```

两个令牌必须不同并由秘密管理系统注入。生产服务不支持明文 HTTP，也不允许空令牌匿名访问。

## 3. testing 候选、审批与 stable 发布

### 3.1 生成一次不可变候选

```bash
go run ./engineering/tools/pluginpackage \
  -source extensions/plugins/cn.vastplan.hello-world \
  -backend-bin bin/cn.vastplan.hello-world \
  -license-file LICENSE \
  -notice-file NOTICE \
  -out /secure/candidates/cn.vastplan.hello-world-0.1.0.tar.gz
```

第一方工作区省略 `-sbom` 时，打包器会按实际构建事实自动生成 CycloneDX：Go 读取最终二进制 build info，Node/Frontend 读取构建器保留的 esbuild metafile，Python 读取 PEP 751 `supply-chain/pylock.toml` 的完整传递闭包。Python wheel 必须随包放在 `supply-chain/python-wheels/` 并由锁绑定大小与 SHA-256；运行节点只做离线安装。缺少对应事实会直接失败，不会生成空壳 SBOM。工作区外构建、尚无自动取证器的语言或企业自有生成器继续显式传入 `-sbom <file>`；该文件必须是 CycloneDX JSON 1.5/1.6，主体等于当前插件 ID/版本。两条路径最终都会写入固定路径并把摘要绑定到签名清单。

候选 tar.gz 从此保持字节不可变。自动 SBOM 会改变插件包字节；已经向不可变仓库发布过的相同 ID/版本不能覆盖更新，首次切换到自动 SBOM 的正式发布必须同时升级插件版本。

### 3.2 由外部 Verifier 签发来源证明验证记录

构建系统先为上述最终 tar.gz 生成 DSSE envelope，payload 使用 in-toto Statement v1 + SLSA Provenance v1，subject SHA-256 必须是该 tar.gz。企业静态密钥参考 Provider 可以这样签发 Verification Record：

```bash
go run ./engineering/tools/provenancekey \
  -provider-id platform.artifacts.provenance.static \
  -key-id verifier-2026-01 \
  -private-out /secure/verifier/verifier-2026-01.pem \
  -key-out /secure/verifier/verifier-2026-01.public.json

go run ./engineering/tools/provenanceverify \
  -package /secure/candidates/cn.vastplan.hello-world-0.1.0.tar.gz \
  -provenance /secure/candidates/provenance.dsse.json \
  -builder-trust /secure/policy/builder-trust.json \
  -policy engineering/deploy/provenance-static-policy.example.json \
  -provider-id platform.artifacts.provenance.static \
  -provider-key-id verifier-2026-01 \
  -provider-sign-key /secure/verifier/verifier-2026-01.pem \
  -output /secure/candidates/provenance-verification.json
```

该 Provider 使用 Go，验证 DSSE PAE 上的 Builder Ed25519 签名、builder ID、build type、源码 URI/commit，再签发短时记录。GitHub/Sigstore 或企业 CA Provider 可替换验证实现，但必须输出同一 Record；Repository 和 Node Agent 不信任普通 RPC 的布尔返回。

发布者信任文档可增加来源证明策略。testing 规则使 testing 上传时验证 sidecar；stable 规则复用同一 `policyId`，从而允许原样晋级；只有 stable 列在 `requiredChannels`，本地早期 testing 仍可省略整组 sidecar：

```json
{
  "schemaVersion": "v1",
  "keys": [{ "publisher": "vastplan", "keyId": "release-2026-01", "publicKey": "<base64 Ed25519 public key>" }],
  "provenance": {
    "requiredChannels": ["stable"],
    "maxRecordTtlHours": 8760,
    "keys": [{
      "providerId": "platform.artifacts.provenance.static",
      "keyId": "verifier-2026-01",
      "publicKey": "<base64 Ed25519 public key>"
    }],
    "requirements": [
      {
        "id": "vastplan-release", "channel": "testing", "publisher": "vastplan", "pluginPrefix": "cn.vastplan.",
        "providerIds": ["platform.artifacts.provenance.static"],
        "builderIds": ["https://cdsoft.com.cn/VastPlan/builders/plugin-release/v1"],
        "buildTypes": ["https://cdsoft.com.cn/VastPlan/build/plugin/v1"],
        "sourceUriPrefixes": ["git+https://github.com/yandt/VastPlan"], "requireSourceDigest": true
      },
      {
        "id": "vastplan-release", "channel": "stable", "publisher": "vastplan", "pluginPrefix": "cn.vastplan.",
        "providerIds": ["platform.artifacts.provenance.static"],
        "builderIds": ["https://cdsoft.com.cn/VastPlan/builders/plugin-release/v1"],
        "buildTypes": ["https://cdsoft.com.cn/VastPlan/build/plugin/v1"],
        "sourceUriPrefixes": ["git+https://github.com/yandt/VastPlan"], "requireSourceDigest": true
      }
    ]
  }
}
```

Verifier 私钥与发布者私钥必须分离。记录过期、Verifier key 撤销、issuer/workflow（适用时）不匹配、原始 DSSE 摘要漂移或缺少一半 sidecar 都会 fail-closed。

### 3.3 由外部安全评估 Provider 签发准入记录

企业安全流水线使用最终 tar 和其中已经由签名清单绑定的 CycloneDX SBOM 执行漏洞与许可证扫描，再输出 `AdmissionRecord v1`。Provider 可以组合 Trivy、OSV-Scanner、Grype、ScanCode 或企业商业扫描服务；VastPlan 只接收统一记录，不在内核中绑定扫描器。记录必须包含 tar/SBOM SHA-256、scanner ID/version、漏洞数据库 revision、风险计数、原始漏洞与许可证报告摘要、策略、UTC 有效期和 Provider Ed25519 签名。

信任文档增加独立 `assessment` 段。下面策略表示 stable 必须通过准入，critical/high/拒绝许可证/未知许可证都必须为零；testing 规则允许候选在审批前使用同一策略完成复验和 sidecar 原样晋级：

```json
{
  "assessment": {
    "requiredChannels": ["stable"],
    "maxRecordTtlHours": 168,
    "keys": [{
      "providerId": "platform.artifacts.assessment.enterprise",
      "keyId": "scanner-2026-01",
      "publicKey": "<base64 Ed25519 public key>"
    }],
    "requirements": [
      {
        "id": "vastplan-security", "channel": "testing", "publisher": "vastplan", "pluginPrefix": "cn.vastplan.",
        "providerIds": ["platform.artifacts.assessment.enterprise"],
        "scannerIds": ["enterprise-security-suite"],
        "maximum": { "critical": 0, "high": 0, "deniedLicense": 0, "unknownLicense": 0 },
        "requireReportDigests": true
      },
      {
        "id": "vastplan-security", "channel": "stable", "publisher": "vastplan", "pluginPrefix": "cn.vastplan.",
        "providerIds": ["platform.artifacts.assessment.enterprise"],
        "scannerIds": ["enterprise-security-suite"],
        "maximum": { "critical": 0, "high": 0, "deniedLicense": 0, "unknownLicense": 0 },
        "requireReportDigests": true
      }
    ]
  }
}
```

Provider 私钥必须与发布者、来源证明 Verifier 私钥分离。安全准入记录过期或策略更新后不能改写已批准的不可变 sidecar；持续漏洞数据库更新将由 ADR-0138 定义的只追加复扫状态链处理。

复扫 Worker 使用独立 `VASTPLAN_ARTIFACT_ASSESSMENT_TOKEN` 调用 `POST /v1/catalog/security-status`，请求只包含精确 ref 与 Provider 已签名的 `StatusRecord v1`。记录必须把 sequence 递增一、`previousSha256` 指向前一状态（首条指向准入记录摘要），并继续绑定同一 tar、SBOM 与准入摘要。仓库会保留 pass 和 fail，拒绝跳号、同序号替换或回滚；读取、发布、Bundle token 与 Portal 用户会话都不能追加复扫状态。Node 会验证整条签名链，并在本地保存最高 sequence/digest 防止被仓库返回的旧链回滚。

### 3.4 发布 testing

```bash
export VASTPLAN_ARTIFACT_PUBLISH_TOKEN='ci-publish-token'
go run ./engineering/tools/pluginpackage \
  -package /secure/candidates/cn.vastplan.hello-world-0.1.0.tar.gz \
  -provenance /secure/candidates/provenance.dsse.json \
  -provenance-verification /secure/candidates/provenance-verification.json \
  -security-admission /secure/candidates/security-admission.json \
  -channel testing \
  -remote-repository https://artifacts.example.com \
  -remote-ca /etc/vastplan/tls/company-ca.pem \
  -trust engineering/deploy/artifact-trust.json \
  -sign-key /secure/release-2026-01.pem \
  -key-id release-2026-01
```

### 3.5 在平台管理中心审批

一位拥有 `platform.artifacts.publication.submit` 的用户从制品目录提交该 testing 版本；另一位拥有 `platform.artifacts.publication.approve` 的用户在“发布审批”页面批准。仓库将申请精确绑定 testing/stable ref、SHA-256、publisher、key ID、来源证明摘要和安全准入记录摘要，同一用户不能完成两步。

申请默认 7 天过期，生产服务可用插件配置 `publication.approvalTTLHours` 在 1–720 小时之间调整。独立审批人可驳回待批或已批申请，原提交人可撤销；两者都必须填写原因。`Rejected`、`Cancelled`、`Expired` 与 `Published` 均不可恢复，要重新晋级应发布新的 testing 候选，而不是复活旧审批。CLI 开始 stable 上传前应重新读取审批状态，但最终仍以仓库写入前的服务端检查为准。

### 3.6 原样复用候选并发布 stable

```bash
export VASTPLAN_ARTIFACT_READ_TOKEN='ci-read-token'
export VASTPLAN_ARTIFACT_PUBLISH_TOKEN='ci-publish-token'
go run ./engineering/tools/pluginpackage \
  -package /secure/candidates/cn.vastplan.hello-world-0.1.0.tar.gz \
  -channel stable \
  -remote-repository https://artifacts.example.com \
  -remote-ca /etc/vastplan/tls/company-ca.pem \
  -trust engineering/deploy/artifact-trust.json \
  -sign-key /secure/release-2026-01.pem \
  -key-id release-2026-01
```

stable 路径强制使用不同的读取和发布令牌。客户端先通过读取令牌拉取并再次验签 testing 候选，比较插件 ID、版本、包 SHA-256、大小、publisher 和 key ID；只有完全一致才生成新的 stable 证明并使用发布令牌上传。服务端随后再次验签并消费精确 `Approved` 记录。`-package` 与 `-source` 互斥，也不能再指定 Backend、Frontend 或 dynamic-go 注入参数，避免审批后改变字节。两类令牌都应通过环境或 CI Secret 注入，不要出现在命令行历史中。

如果 testing 已携带来源证明或安全准入记录，stable CLI 会通过读取令牌下载并原样复用同一 DSSE、Verification Record 与 Admission Record，不需要再次指定这些文件；服务端审批记录还会比较三份 sidecar 的精确摘要。审批后重新签发或替换记录会被拒绝。记录过期或 Provider key 撤销后，该不可变 ref 不允许改写，必须升级插件版本并重新走 testing 与审批。

仓库安全默认值要求 stable 制品已经在 testing 包内绑定 SBOM；`-package` 阶段禁止补加或替换。企业若需要改变强制 channel，只能在仓库服务配置中设置 `supplyChain.requiredSBOMChannels`，不要在 CI 客户端放宽服务端准入。

服务端会在授权 stable 写入的同一串行临界区再次收敛过期状态；即便批准后尚未上传，审批人驳回、提交人撤销或有效期到期都会使上传立即失败。

## 4. Node Agent 使用远端仓库

```bash
export VASTPLAN_ARTIFACT_READ_TOKEN='node-agent-read-token'
bin/backend-kernel reconcile \
  -desired engineering/deploy/local.desired-state.json \
  -repository-url https://artifacts.example.com \
  -repository-trust engineering/deploy/artifact-trust.json \
  -repository-ca /etc/vastplan/tls/company-ca.pem
```

Node Agent 验证顺序为：请求 ref 与证明一致 → publisher/keyId 受信任且未撤销 → Ed25519 签名 → 来源证明与安全准入记录预检 → 下载大小上限 → SHA-256 → tar 根清单与 SBOM → 安全准入记录的 tar/SBOM/策略绑定复验。任一步失败均不会进入安装器。

需要在空节点预置仓库基础插件或气隙启动时，可增加签名种子仓库：

```bash
bin/backend-kernel reconcile \
  -nats-url tls://nats.example.com:4222 \
  -node-id node-a \
  -assignment-key deployments/acme/platform/nodes/node-a \
  -bootstrap-repository /opt/vastplan/bootstrap-repository \
  -bootstrap-inventory /var/lib/vastplan/bootstrap/inventory.json \
  -repository-trust /etc/vastplan/artifact-trust.json \
  -repository-url https://artifacts.example.com
```

种子源按精确 ref 优先读取；只有明确 `not found` 才访问远端。Bootstrap Inventory 必须是由部署管理员提供的绝对 `0600` JSON 文件，父目录不可被 group/other 写入；它记录 Seed 全集、LKG 子集和单调 generation。种子文件损坏、权限错误、格式错误或证明不可信都会立即停止，不会通过换源掩盖安全事件。制品源返回的内容始终视为未信任，Node Agent 验证成功后才交给安装器。

生产环境需要允许仓库关键插件在线升级时，再显式增加 `-bootstrap-upgrade`，并通过 `-nats-url` 等参数接入完整控制面。该模式要求 Node Agent 宿主对 Seed 和 Inventory 有窄写权；仓库插件仍无权访问二者。事务顺序固定为候选验签、镜像 Seed、启动与健康、发布 Assignment、推进 LKG、发布 Bootstrap 引用。自动路径拒绝跨 channel 与 SemVer 降级；内核二进制升级仍走 systemd/Guardian 流程。

## 5. 密钥轮换与撤销

1. 生成新 keyId，把新公钥加入信任文档，但保留旧 key。
2. 先更新制品服务和 Node Agent 的信任文档，再切换发布流水线签署密钥。
3. 确认所有仍需安装的版本已经用新 key 重新发布后，把旧 key 标记为 `"revoked": true`。

撤销是 fail-closed 的：旧 key 签署的历史制品将不可再下载安装。紧急泄漏时优先撤销，再处理历史制品重签。

没有可信时间戳服务时，`NotAfter` 到期也会使该 key 的全部历史签名停止通过验证；不得依赖证明中由签名者自行填写的 `signedAt` 延续信任。因此必须在旧 key 到期前完成仍需安装制品的重签、信任文档分发和 Node Agent 验证。
