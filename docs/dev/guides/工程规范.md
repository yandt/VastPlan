# 工程规范

> 状态：v0.1（随项目演进）｜最后更新：2026-07-17
> 决策与理由见《[ADR-0019 工程规范基线](../decisions/ADR-0019-工程规范基线.md)》。本文是**怎么协作、怎么写代码**的单一真相源。
> 标 🤖 的规则由**架构守护测试或 CI 强制**，非君子协定；标 👤 的靠评审。

## 一、提交规范

格式：`<type>(<scope>): <subject>`

| type | 用于 |
|---|---|
| `feat` | 新功能 |
| `fix` | 缺陷修复 |
| `docs` | 文档 |
| `test` | 测试 |
| `refactor` | 重构（不改外部行为） |
| `perf` | 性能 |
| `build` / `ci` / `chore` | 构建 / 流水线 / 杂务 |

`scope` 用组件名：`backend` / `frontend` / `runner` / `mobile` / `protocol` / `registry` / `plugins` / `docs` / `decisions`…

规则：
- 🤖 **格式须合规**（commit-msg 钩子 / CI 校验）。
- 🤖 **不得含任何 AI 生成署名**（`Co-Authored-By: Claude` 等）——CI 正则扫描。
- 👤 **正文说"为什么"和"影响"，不复述"改了什么"**——diff 已经说了改什么。
- 🤖 破坏性变更用 `!` 或 `BREAKING CHANGE:` footer，并**指向对应 ADR**。

示例：

```
feat(protocol): 版本协商改为取交集最高版

原实现取首个匹配，导致双方都支持 v2 时仍降级到 v1。
影响：宿主与插件都需重新构建；协议版本本身不变。
```

## 二、分支与 PR

- **主干开发**：`main` 是唯一长期分支；特性分支短命（理想 <2 天）。
- 🤖 合入前必绿：`gofmt` + `vet` + `lint` + 单元测试 + **架构守护测试**。
- 👤 **PR 描述写：为什么 + 怎么验证**。
- ⚠️ PR 体量建议 <400 行 diff（经验阈值，CI 仅提示不硬卡——避免鼓励无意义的碎 PR）。
- 👤 破坏契约的 PR 必须带 ADR 或指向已有 ADR。

## 三、依赖准入

> 本节只约束**第三方依赖**。VastPlan 项目自身统一采用 `Apache-2.0`，权威决策见 [ADR-0046](../decisions/ADR-0046-Apache开源许可与插件制品声明.md)，完整条款见仓库根目录 `LICENSE`。

引入任何第三方依赖前，先回答三问（ADR-0019 §3）：

1. **真的需要吗**——能否 <100 行自写？（琐事不值得引入供应链风险）
2. **可持续吗**——维护活跃度、单点弃坑风险？
3. **许可证允许吗**——见下。

🤖 **许可证白名单**（可闭源商用）：`Apache-2.0`、`MIT`、`BSD-2/3-Clause`、`ISC`、`MPL-2.0`

🤖 **第三方依赖黑名单**（禁止）：`GPL` / `LGPL` / `AGPL`（强 copyleft）、`BSL` / `SSPL`（非 OSI 开源）、**无许可证**（默认全权保留）

> `MPL-2.0` 是文件级 copyleft：可闭源商用，仅当**修改其源文件**时需开源那些文件。ADR-0008 据此确认 go-plugin 可用。

👤 新增依赖的 PR 须写明**用途 + 许可证**。🤖 `govulncheck` 扫漏洞。

👤 新发布的插件独立制品必须按 ADR-0046 声明 SPDX `license`、包内 `licenseFile`，需要归属告示时同时声明 `noticeFile`；既有未声明 v1 只为兼容读取。只写许可证名但不随包提供相应文本，不视为完成分发声明。

**当前直接依赖**：`google.golang.org/grpc`(Apache-2.0)、`google.golang.org/protobuf`(BSD-3)、`github.com/Masterminds/semver/v3`(MIT)、`github.com/nats-io/nats.go`(Apache-2.0)；E2E 使用 `github.com/nats-io/nats-server/v2`(Apache-2.0) 嵌入式服务——全部在白名单内。

## 四、Go 编码规范

### 4.1 格式与静态检查

🤖 `gofmt`/`goimports` 与 `go vet` 是**硬门禁**，不是建议。格式和低级缺陷不该消耗人类评审注意力。

### 4.2 错误处理二分法（本项目核心，务必遵守）

| 错误类别 | 载体 | 例子 |
|---|---|---|
| **传输层/基础设施** | Go `error`，`%w` wrap 保留链 | 连接断、超时、进程拉起失败 |
| **应用层** | `CallResult.Error`（稳定 code + retryable） | 参数非法、权限拒绝、能力未实现 |

- **应用层错误不得冒泡为 Go error**——否则调用方无法区分"没打通"与"打通了但业务拒绝"。已有 E2E 守护（`TestPluginInvoke_ApplicationErrorsAreNotTransportErrors`）。
- 👤 不吞错误；确需忽略时 `_ =` 旁必须写明理由。
- 错误码必须使用至少两段的小写点分命名空间；内核错误统一引用 `shared/go/errorcode` 常量，禁止在调用链散写字符串。插件自定义 code 不得占用内核登记项。

### 4.3 日志

- 内核日志统一从 `slog` JSON handler 输出，并使用 `component`、`trace_id`、`span_id` 等稳定字段。
- 👤 **禁止打印敏感值**——凭证只以句柄流转，明文永不入日志（对齐《插件契约与协议》§3.7）。
- 具体日志存储、指标和 trace exporter 由部署适配器或可观测插件提供，内核不绑定采集厂商（ADR-0035）。

### 4.4 命名

- 遵循 Go 惯例（驼峰、短包名）。🤖 **不得有 util/common 等垃圾抽屉包**——见 §5.3。
- 🤖 跨内核概念一律用**规范 ID**：`backend`/`frontend`/`runner`/`mobile`（ADR-0015）。
- 四处同名铁律：清单贡献 `id` = 协议注册名 = `CallTarget.capability` = 跨内核寻址名。

### 4.5 注释

- 导出符号必须有 doc comment。
- 👤 **注释写"为什么/什么约束"，不复述代码**。反例 `// 加一` ；正例：
  ```go
  // addrReportTimeout 等待插件回报地址的上限：插件卡住时宿主必须能脱身，
  // 不能被一个坏插件永久拖住（ADR-0004 故障隔离的前提）。
  ```
- 引用决策时写 ADR 编号，便于回溯。

### 4.6 其他

- `context.Context` 作首参，不存进 struct。
- 并发：优先 channel/sync 原语；共享状态必须有明确的锁归属注释。

## 五、代码设计原则（ADR-0020）

### 5.1 高内聚：一个包 = 一个职责

🤖 **每个包必须有 doc comment，写明它的单一职责**（架构守护强制）。

这是**内聚的试金石**，不是文档洁癖：

- 一句话说不清这个包干什么 → 它不内聚，该拆。
- 句子里出现"**以及 / 还有 / 各种 / 相关**" → 同上，那是两个包被塞在一起。

正例（当前 `shared/go/` 全部符合）：

| 包 | 一句话职责 |
|---|---|
| `registry` | 扩展点注册表：登记扩展点与其上贡献 |
| `protocol` | 宿主协议常量的唯一真源 |
| `protocolbus` | 宿主 ↔ 插件的协议总线 |
| `contract` | 不可变契约（proto 生成） |

### 5.2 低耦合：依赖契约，不依赖实现

- **跨组件只经契约通信**：`CallContext/CallTarget/CallResult/CallEvent` + 能力名寻址。
- **消费方定义接口**（Go 惯例 *accept interfaces, return structs*）：接口写在**用它的地方**，不写在实现方——否则消费方被迫依赖实现包。
- **同字段且同语义的契约类型只定义一次**：优先复用 `proto/` 生成类型或 `schemas/*` DTO；版本包可用类型别名保持 API。若 JSON/wire 语义不同则保留独立类型并提供显式转换与往返测试。
- 🤖 依赖方向铁律见 §七。
- **插件间只能经 capability 寻址**——这就是低耦合在本项目的具体形态：**加一个插件，零人需要改别人的代码**。

### 5.3 复用：禁垃圾抽屉，只建有职责的包

🤖 **禁止 `util`/`utils`/`common`/`helper`/`misc`/`base`/`core` 等垃圾抽屉包名**（架构守护强制）。

> **为什么不建"公共工具类"**：testa 有 `shared/`（3,248 LOC），却全库只被 import **86 处**（利用率 **<2%**，工业标准 15-30%），同时背着 **17,000-20,000 LOC 重复代码**（占业务码 13-15%）、5 套并存的权限命名、60 个各自实现 80% 雷同 CRUD 的 Repository。
> **目录建了、没人用、重复照旧疯长。** 根因：无职责的包 = 垃圾抽屉，没人知道什么该放进去，于是谁都不放。它恰恰**违背高内聚**。
> 真正产生复用的是**有职责的机制**（Registry / 契约 / 基类），不是工具包。

**shared/ 准入门槛**——三条同时满足才可进：

1. **≥2 个组件真的在用**（不是"将来可能用"）
2. **能一句话说清职责**，且不与既有包重叠
3. **无反向依赖**（不 import kernels/plugins）

不满足就**先留在原地**。

### 5.4 抽象时机：三次法则

| 次数 | 做法 |
|---|---|
| 第 1 次 | 直接写 |
| 第 2 次 | **复制**，并标注另一处（此时还看不清共性） |
| 第 3 次 | 才提取——三个样本才够看出**真正的**共同点 |

**重复比错误的抽象便宜**：重复是局部的、可见的、易删的；错误的抽象是全局的、隐蔽的、难拆的——后来者会为了复用它而扭曲自己的代码。

> ⚠️ **例外**：契约 / 协议 / 版本这类"单一真源"（§六）**不适用**三次法则——它们从第 1 次就必须唯一。

## 六、单一真源铁律（🤖 全部由架构守护测试强制）

| 规则 | 出处 |
|---|---|
| 契约结构只由 `proto/` 生成，不得手写 | ADR-0016 §6 |
| 协议常量（magic/版本）只在 `shared/go/protocol` | ADR-0017 §3 |
| 内核版本只在 `kernels/<name>/VERSION`，经 ldflags 注入 | ADR-0017 §1 |
| 插件版本只在 `vastplan.plugin.json#version` | ADR-0017 §1 |
| 不得有 `services/<role>/` 目录（服务组合是配置不是代码） | ADR-0016 §3 |
| Backend 普通子包横向依赖只允许已登记边；生产入口不进 `tools/` | ADR-0040 |
| 稳定 DTO 结构只在 Schema 领域真源定义一次 | ADR-0041 |

> 这些不是建议——`arch/` 下的守护测试会让违规**构建失败**。历史教训：ADR-0017 §3 曾被规则作者本人违反（magic 常量写了两份），两轮后才发现。

## 七、依赖方向（🤖 架构守护强制）

```
proto ──► shared ──► kernels
              │          ▲
              └──► sdk   │ （kernels 不得 import plugins）
                    ▲    │
                    └── plugins（插件间不得互相 import，只能经 capability 寻址）
```

- `kernels/*` **不得** import `plugins/*`——内核不认识任何具体插件。
- `plugins/A` **不得** import `plugins/B`——插件间只能经能力名寻址（否则绕过扩展点，架构失效）。
- `shared/*`、`sdk/*` **不得** import `kernels/*` 或 `plugins/*`——共享库不得反向依赖具体组件。
- `shared/go/clientcore` 只可被 `runner`/`mobile` 使用（ADR-0014）。
