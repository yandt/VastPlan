# 插件共享状态服务

本篇是插件跨实例持久状态的权威设计。实例策略与路由见《插件服务集群化设计》；业务配置与凭证状态机见《插件配置与托管凭证》。Shared State 只提供通用版本化字节状态，不理解任何业务字段。

## 1. 边界

插件通过 `state.shared.v1` 的五个内核服务使用 Get、Create、Update、Delete 和分页 List。插件不直接连接 NATS、数据库或文件系统，也不取得基础设施凭证。

请求只包含：

- `scope`：`tenant` 或 `service`；
- 局部 `namespace`；
- `key`、base64url `value` 与 CAS `expectedRevision`；
- List 的 prefix、limit 和不透明于业务的 page cursor。

请求禁止携带 tenant、plugin ID、Runtime scope、Node ID 或物理存储 key。可信宿主从验签 `LaunchPolicy` 取得 plugin ID 与稳定 RuntimeScope，从认证 `CallContext` 取得 tenant，构成：

```text
(scope kind, tenant?, runtime scope, plugin id, local namespace, local key)
```

Node ID 与单次 Instance ID 不进入共享数据主键，否则副本和重启无法读到同一状态。完整 RuntimeIdentity 仍用于认证调用来源。

## 2. 一致性

- Create 是不存在前提的 CAS；已存在返回 `state.conflict`。
- Update/Delete 必须携带非零 revision；旧 revision 返回可重试 `state.conflict`。
- Get/List 返回 Provider revision 和更新时间。
- 单 key CAS 是原子边界。v1 不伪装跨 key 事务；多阶段业务状态机必须把提交点聚合到一个状态文档，或使用显式 Saga/outbox。
- List 是有界、弱一致分页，只用于目录和恢复扫描，不作为事务快照。

每个 value 最大 1 MiB，每页最多 200 条。审计、日志和错误不得输出 value；状态中若包含 CredentialRef 仍只是引用，credential material 禁止写入。凭证插件自身的持久密文迁移需单独安全评审。

## 3. Provider

### NATS KV

生产与集群测试使用 `VASTPLAN_SHARED_STATE_V1`，保留 64 代历史并使用 JetStream revision 做跨节点 CAS。物理 key 对所有身份分量编码；解码后的 key 才可返回插件。Node NKey 可读写该 bucket，但凭证只存在可信 Backend 进程，插件进程没有 NATS 权限。

生产至少三副本；Provider 或 NATS 不可用时返回可重试 `state.unavailable`，绝不回退本地文件。

生产备份使用 backing stream 的签名原生 snapshot，不使用会重编号 revision 的逐 KV 导入。备份同时生成不含 key/value 的整体逻辑摘要，并由领域验证器复核 Credentials Root/chunk；恢复只允许空目标 stream，完成后必须重新计算摘要和领域验证结果。backup/restore 使用两个相互隔离且不能写 `$KV` 的 NATS 身份，完整状态机与取舍见 [ADR-0129](../decisions/ADR-0129-Shared-State签名备份与空目标恢复.md)。

生产 bootstrap 还必须显式配置整个 Shared State bucket 的 `MaxBytes`；warning/critical 阈值与 schema version 进入 stream metadata。容量达到硬上限时保持 KV 的 `DiscardNew`，拒绝新写而不淘汰旧状态。`sharedstatectl capacity` 输出不含业务身份的四级容量快照；Runtime Host Observer 只按固定 operation/outcome 记录调用总数和时延。完整指标与暂不提供 tenant/plugin 配额的理由见 [ADR-0130](../decisions/ADR-0130-Shared-State容量策略与低基数可观测.md)。

### File

File Provider 仅用于单进程开发和单元测试，使用 0600 文件、0700 目录、临时文件、fsync 和原子 rename。它支持相同 CAS API与重启恢复，但不具备跨进程锁和复制能力，不能用于 active-active 或生产集群。

### SQL 与其他 Provider

Store SPI 保持与 NATS 无关。SQL Provider 后续可通过 Database Runtime 或专用受信适配器实现；必须保持相同身份派生、CAS、错误码、大小上限和 fail-closed 语义，不能让插件自带连接串绕过宿主。

## 4. 多语言 SDK

- Go：`extensions/sdk/go/sharedstate`；
- Node：`@vastplan/shared-state`；
- Python：`vastplan_plugin.SharedStateClient`。

三种 SDK 都只构造身份无关请求、严格解析闭合响应，并保留 `state.conflict`、`state.not_found`、`state.unavailable` 稳定错误语义。SDK 是已有 Runtime 内的库，不增加插件进程。

## 5. 迁移顺序

1. `global-settings` 已完成首批迁移：每租户状态聚合为单 key CAS 文档，插件已由 `leader-owned` 调整为 `active-active + external-shared`；业务 setting version 与 Store revision 保持分离。
2. `plugin-settings` 0.12.0 已完成第二批迁移：每租户候选、审批、凭证阶段、激活 Saga 与审计保持一个 CAS 聚合，切换为 active-active；Scoped watch 使用本地通知加一秒 Shared State 观察发现跨实例更新。
3. Portal Composer 1.6.0 已完成第三批迁移：组合治理按 tenant CAS 聚合，PortalPreference 按 tenant + subject 摘要分文档，切换为 active-active。
4. Deployment Manager 0.18.0 已完成第四批账本与副作用 fencing：每租户节点、引导作业、服务组合、Profile 激活 Saga 与测试发布聚合为单 key CAS 文档；运行模型为 `leader + external-shared + leader routing`。Store CAS fence 账本，Runtime Host 复用 Unit Leadership evidence 保护 mutating 回调，SSH 远端用 `flock` 和单调 epoch 拒绝旧 owner。该模型提供 active-passive 接管，不宣称 active-active。
5. Credentials 0.11.0 已完成第五批迁移：每租户以小型 CAS Root 提交内容寻址、多 chunk 的完整安全快照，保持凭证、生命周期审计和维护统计的单事务语义；Material Lease 在 decrypt 后重新读取最新 Root。签名原生备份、空目标恢复和 Root/chunk 双端复核已经完成；当前仍是单 Leader，孤儿 chunk GC、容量基线和 active-active 故障演练继续进入生产工程阶段。
6. File 旧状态到 Shared State 的生产迁移需单独事务；当前开发数据不承诺兼容，不能把删除文件当作生产升级方案。
