# 制品仓库与测试发布

> 状态：架构已采纳，Backend 测试发布闭环已实现｜最后更新：2026-07-21
>
> 本文是 Seed 自举、托管制品仓库、包管理、测试发布和 Storage Provider 迁移的单一真相源。信任决策见 [ADR-0023](../decisions/ADR-0023-插件Schema与可验证制品仓库.md)、[ADR-0026](../decisions/ADR-0026-远端制品仓库与供应链信任.md)、[ADR-0049](../decisions/ADR-0049-制品信任基座与仓库基础插件.md)、[ADR-0091](../decisions/ADR-0091-制品存储Provider供给边界.md) 与 [ADR-0097](../decisions/ADR-0097-测试制品仓库与前端分级热升级.md)。

## 1. 目标

VastPlan 的仓库最终提供类似语言包管理器的体验：发现插件、解析兼容版本、生成精确锁、下载/验证、安装、升级、回退和离线分发。同时它也是测试交付入口：每个测试制品上传后，可以由独立发布控制器安全应用到指定 Backend、Portal、Runner 或 Mobile 测试目标。

仓库不解释“这份代码是否可以运行在某个服务”，也不直接启动插件。仓库负责保存事实，Resolver/发布控制负责授权和锁定，内核执行器负责最终验证和候选切换。

## 2. 三个运行角色

| 角色 | 负责 | 禁止 |
|---|---|---|
| Kernel Trust Base | Schema、签名、撤销、内容校验、安装事务、可信缓存 | 搜索、市场、latest、业务审批 |
| Seed Repository | 空节点自举、基础仓库插件、foundation 最小集、LKG、Bundle | 普通测试上传、在线目录、用户发布 |
| Managed Repository | 不可变对象、目录、流水账、晋级、包解析、GC、管理面 | 写部署期望态、命令 Node Agent、来源自证可信 |
| Test Release Controller | 目标绑定、精确 ref 解析、候选发布、测试、回滚和审计 | 持有签名私钥、绕过 Profile、直接读写仓库对象目录 |

## 3. 制品身份与版本

- `(pluginId, version, channel)` 是不可变键；相同 SHA 重传幂等，不同 SHA 拒绝。
- 测试构建使用唯一 SemVer 预发布版本；`testing` channel 不覆盖已有版本。
- 仓库为成功发布分配单调 `repositoryRevision`，发布事件携带精确 ref、SHA-256、publisher 和发生时间。
- “最近版本”只用于目录展示或 Test Release 解析；最终 Deployment、Activation 和 Lockfile 必须保存精确版本、channel、摘要和发布者。
- channel 晋级生成新的签名证明与审计记录，不改变源 channel 历史。

## 4. Repository Profile

Seed Profile 继续使用部署管理员提供的绝对文件路径。与其配套的 root-owned `Bootstrap Inventory v1` 保存全量 Seed、LKG 子集和单调 generation；Seed HTTPS 服务不读取该清单，也不连接控制面。拥有 Seed 源的 Backend 内核逐项重新验签后，才把两类快照桥接到 Managed Repository。显式启用仓库自升级时，Node Agent 的可信宿主适配器先把已验签候选镜像进 Seed 并以 CAS 扩展 Inventory，只在 Runtime 健康且 Assignment 引用已发布后推进 LKG；仓库插件本身始终无权写 Seed/Inventory。托管仓库 Profile 使用插件配置与托管凭证：普通字段包括监听、Storage Provider、volume、允许 channel、容量和保留策略；TLS 私钥、发布/读取凭证和 Provider 凭证只以部署秘密或 CredentialRef 注入。

开发环境使用两套身份：

- 每次运行生成的 Seed TLS/签名身份，只保护本次自举仓库；
- 保存在 `.vastplan/dev-platform/repositories/testing/` 下的稳定测试发布身份和 File Volume。目录和私钥权限必须不宽于 `0700/0600`，均不进入 Git。

每次启动生成组合信任快照，使 Node Agent 同时信任当前 Seed 和稳定测试发布者。生产环境不得自动生成或持久化生产签名私钥。

## 5. Catalog 与 Publish Journal

目录至少支持按 plugin ID、publisher、命名空间分类、版本、channel、目标内核、操作系统/架构和状态过滤，并返回分页结果。每个版本展示 Manifest 摘要、入口、依赖、签名、许可证、SBOM/来源证明状态、yank/deprecate 状态和活动引用。

Publish Journal 是可重放的单写流水账。索引是派生状态，必须能从不可变 artifact metadata、attestation 和 journal 重建。事件消费者只能把事件当触发器，处理前仍按精确 ref 读取并验证制品。

当前 v1 实现把 Catalog/Journal 放在 Go 仓库插件进程内：Go 可以直接复用内核窄 `SignedRepository` 适配器、文件原子替换与 `fsync` 语义，不需要为了目录功能再增加 Node/Python sidecar。Node.js 更适合后续前端构建 Worker，Python 更适合未来元数据分析或模型增强搜索，但都不适合承担当前单调 durable journal 的第一实现。将来 Catalog 数据量或检索复杂度需要独立扩缩容时，可通过稳定 HTTP/工具契约拆为专用服务。

v1 流水账不是 JSONL，而是 `catalog/journal/<20位revision>.json` 原子事件文件：每次成功发布取得唯一单调 revision；同一精确 ref/SHA/attestation 重传返回原 revision，受控 CLI 则会先按 Catalog 命中原 revision，避免为重试重新生成 `signedAt`；进程在“对象已落盘、事件未落盘”之间崩溃时，重启会逐项通过签名仓库重新验证元数据与发布证明，并生成带 `recovered` 标记的补偿事件。恢复不会读取全部 tar 正文，避免启动成本按仓库总字节数增长；对象在实际下载/安装时仍重新计算摘要并验证包内清单。`catalog/index.json` 只是可删除重建的快照，不代表对象健康检查结果。

查询面包括：

- `GET /v1/catalog/artifacts`：按 plugin ID/前缀、namespace、publisher、version、channel 和目标内核过滤，固定分页上限 100；
- `GET /v1/catalog/journal`：按 `afterRevision` 增量读取，单次上限 200；
- `platform.artifacts.repository/listCatalog` 与 `listPublishJournal`：供 Portal/其他受权平台插件使用，不向浏览器暴露仓库 token。

两个 HTTPS 查询端点只接受仓库读令牌，拒绝明文、未知参数和重复参数。Catalog 条目不返回对象路径、Provider endpoint、token 或私钥。

## 6. 包解析与锁

包解析器输入是根插件约束、目标内核、平台、允许 channel、发布者/隔离策略和 Catalog 快照 revision；输出为不可变锁文件。它必须检测依赖循环、版本冲突、平台不兼容、runtime capability 不满足、yanked/revoked 制品和 namespace/publisher 违规。

Node Agent、Portal Catalog、Runner Builder 和 Mobile Resolver 只消费各自解析产物，不在运行节点重新选择 latest。离线 Bundle 包含锁文件、证明、信任快照和全部对象；导入后仍走相同验证器。

## 7. Test Release

`TestTargetBinding` 由平台管理员预先把插件/分类绑定到测试 tenant、deployment/portal、unit 或允许的 Platform Profile 槽位。上传成功可以同时提交 Test Release 请求，但两者是不同事务。

Test Release 状态固定为：

```text
Queued -> Resolving -> Preparing -> Validating -> Activating -> Ready
             |             |              |             |
             +-------------+--------------+-------> Failed
                                                        |
                                             RollingBack -> RolledBack|Failed
Queued/Resolving/Preparing -> Superseded
```

同一 `target + pluginId` 串行执行；尚未提交的旧请求可以被更新请求合并。Backend 应用插件通过新的 Application Composition revision 进入 Node Agent 候选 unit；Frontend 通过新的 Portal Activation 进入候选 Portal Generation 或 Host Epoch。foundation/platform 只能进入专用测试 Platform Profile，不能由 Application Composition 覆盖。

Backend v1 把 Test Release Controller 作为 deployment-manager 中的独立领域模块实现，而不是仓库模块，也暂不增加一个独立进程。这样它与人工服务发布共享唯一的单调 service revision 分配器、活动修订和审计事务，避免两个控制器争抢同一 Deployment；边界仍通过 `platform.deployment`、`platform.artifacts.repository/listCatalog` 与 `kernel.deployment.*` 稳定契约隔离，将来需要独立扩缩容时可以按契约拆进程。Test Release 的候选引用仍由 testing Catalog 精确锁定；普通服务发布则以可信宿主 `kernel.deployment.preview` 真实读取过的多来源精确 ref/SHA 为准，不再假设所有运行制品都来自托管仓库。

该控制路径使用 Go：相较 Node.js/Python，Go 更适合复用当前强类型组合契约、持久状态原子写入与控制面并发；Node.js 更适合后续管理 UI，Python 更适合未来策略分析，但都不应为了这一步再引入一份 revision 状态或跨语言事务。运行方式与语言独立决策：当前进入 deployment-manager 的可信独立进程，不进入内核，也不与仓库合并。

Backend v1 的安全与恢复约束如下：

- 平台管理员用 CAS 预建 `TestTargetBinding`，只能命中活动 Application Composition 中已经存在的应用插件槽位；`cn.vastplan.foundation.*` 与 `cn.vastplan.platform.*` 明确拒绝；
- 发布请求必须携带精确 `testing` ref、唯一 SemVer 预发布版本、SHA-256 与 `repositoryRevision`，控制器按精确条件重新读取已验证 Catalog，并校验 Backend target 与允许发布者；
- 候选发布使用同一服务 revision 分配器。就绪观察要求 Composition report 的 Deployment revision 精确相等，并要求 Node Actual/Unit 的 applied revision 与当前 assignment generation 相等；旧健康实例不能掩盖新候选失败；
- 候选失败或超时后自动把上一活动组合发布成新的单调 revision，并等待回滚 revision 就绪；控制器重启时，非终态发布 fail-closed 为 `Failed + rollbackRequired`，管理员可调用显式恢复回滚，系统不会盲目重放发布；
- 当前创建操作在一条受认证的 capability 调用中执行到 Ready/RolledBack/Failed，避免把用户调用路径和授权上下文复制给脱离请求的后台 goroutine。后续若改为异步 Worker，必须先增加内核签发的 durable command identity，不能持久化或伪造用户 CallContext。

Frontend v1 复用 Portal Composer 的唯一 Application/Profile/Binding revision 和 Activation 分配器，不创建第二套 Portal 状态：

- `application-plugin` 绑定只能替换当前 Activation 中已属于 Application Composition 的插件槽位；
- `platform-profile-plugin` 绑定不修改共享 Profile，而是克隆当前 Profile 与 PortalBinding，创建 tenant 专用、不可变的测试 Profile/Binding revisions；
- Backend `portaltrust` 窄能力复核精确 testing ref、SHA-256、repository revision、frontend target 和允许的发布者；Composer 不获得仓库 token、证明或对象字节；
- Test Release 在候选验证前先发布精确 `artifact-lock`；普通 Activation 在 CAS 前保护旧活动与新候选并集，CAS 后按“回滚历史、活动引用”收敛，持久 outbox 负责仓库恢复后的幂等重试；
- 候选经完整 Catalog 验证、中央交付快照物化和 `expectedCurrentId` CAS；任一阶段失败时旧 Activation 保持 Current，非终态重启则 fail-closed 并在候选已上线时标记显式回滚需求；
- v1 的 Ready 表示服务端已验证并物化候选 RuntimeSpec。已打开的生产页面仍在用户刷新时获取新 Activation；候选 Portal Generation/Host Epoch 的在线通知、客户端确认和自动回退属于下一阶段。

## 8. Storage Provider 与迁移

Provider 只执行 `probe/provision/describe/migrate/release` 控制路径。对象热路径使用供给后的本地 mount 或 local endpoint，不逐对象调用插件 RPC。File Provider 的 `release` 是原子移入私有 quarantine，真正删除由后续 GC 宽限期处理。

首个迁移控制器已以 File Volume A/B 完成：源/目标 probe、空目标绑定、可取消全量/增量复制、逐文件摘要验证、短暂冻结发布、Catalog 重建校验、原子切换、观察期双写、重启恢复、回滚、确认与隔离 release。失败始终保留可用数据面；相同 migration ID 幂等。只有部署配置已指向目标 volume 后才能 release 源卷。S3/OCI Provider 不在当前阶段实现。

## 9. 生命周期、引用与 GC

- `deprecated`：可继续安装，但解析器给出替代提示；
- `yanked`：禁止新解析和安装，已有运行实例不被仓库主动停止；
- `revoked`：信任或安全事件，所有新验证 fail-closed；
- GC 必须保护 Active Deployment、Assignment、Portal Activation、锁文件、回滚历史、Seed 和 LKG 引用；
- 未引用对象先进入 quarantine，经过配置宽限期和最终复核后删除；
- 仓库关键插件候选在激活前必须进入 Seed，但只能在真实 Runtime 健康且 Assignment 已受保护后替换 LKG；旧 LKG 字节继续保留以供回退。

## 10. 安全与管理面

读取、发布、晋级、yank、迁移、GC 和管理各用独立 operation grant；CLI/CI 发布不等于 Portal 管理员。浏览器不接收仓库 token、信任私钥、mount path、Provider endpoint 或 Credential material。

管理中心提供目录、生命周期、引用、容量、迁移、发布审批和供应链证据工作区。大对象上传仍由 CLI/CI 执行；stable 晋级必须先由一位用户提交精确 testing 候选、另一位用户批准，再由 CI/CLI 原样复用候选 tar.gz、通过独立读取令牌复验 testing 身份，并上传完全匹配的 stable 签名证明与相同包字节。

## 11. 当前实施状态

已实现不可变本地仓库、签名 HTTPS 发布/读取、Node Agent 二次验证、Seed YAML Profile、托管仓库基础插件、本地 File Provider `probe/provision` 和只读状态页。

开发环境现已完成第一阶段分离：

- Seed repository、`local-development` 身份和 Seed 数据仍随单次运行生成；
- 托管测试仓库的 File Volume 与 `local-testing` 身份跨普通重启保留；
- 每次运行生成 Seed-only、testing-only 与包含两类公钥的 Node Agent 组合信任快照，两个仓库不能交叉接受发布身份；
- 每次运行还生成 `seed-inventory.json`，其中 LKG 固定为仓库服务、File Provider 与平台访问策略的精确 Seed 子集；Managed Repository 就绪后以内核受信身份发布，仓库未就绪则保持 Node Agent 未完全收敛并重试；
- 开发 Node Agent 默认启用可信宿主仓库自升级：同通道、非降级候选按 `Seed 镜像 -> Runtime 健康 -> Assignment 引用 -> LKG CAS -> Bootstrap 引用` 顺序提交；任一中断均从 Inventory 与 ActualState 幂等恢复；
- Node Agent 按“本地签名 Seed 优先、托管 HTTPS 仓库后备”解析精确 ref；
- Node Agent 仅在 Runtime 与实际态均收敛后，以 `node-agent/<nodeID>` 受信系统身份发布 `assignment-active` 完整快照；快照使用 120 秒租约、约 40 秒心跳，ActualState v4 持久单调 generation、outbox 与 Bootstrap Inventory 发布进度。仓库失败保留 outbox 并阻止收敛/本机 GC，优雅停机发布永久健康的空快照，异常失联则令仓库 GC fail-closed；
- 消费者完整快照不按“来自 Seed 还是 Managed”裁剪。托管仓库允许保存外部已验证、当前 Catalog 未知的精确 ref+SHA；对本仓已知 ref 继续复核摘要并拒绝 retirement，GC 只对本仓实际存在且 SHA 命中的对象应用保护；
- 独立调度 Controller 与 Node Agent 使用同一托管仓库后备源：只在 Seed 精确 ref 不存在时回退，本地对象破损或验证失败直接 fail-closed，远端证明与正文由客户端再次验证；
- Backend `portaltrust` 也按相同规则将托管仓库作为 Frontend 精确 ref 后备源；明确开发模式可同时验证“无签名本地 Seed + 已签名 testing”，生产仍要求所有来源都携带可信证明；
- 测试签名私钥不进入插件环境，宽权限目录和私钥 fail-closed；
- 开发状态端点只报告两个仓库的 URL 与持久属性，不暴露存储路径或密钥。
- 托管仓库已提供可重建 Catalog、逐 revision 原子 Publish Journal、分页/过滤 HTTP API 和平台工具查询；
- `platform-dev.sh publish-test` 已提供只面向回环开发平台的受控上传：只接受 VastPlan 第一方、`testing` channel 和唯一 `dev.*` SemVer 预发布版本，token 不出现在命令行，并在返回成功前复查 Catalog revision；可选用已有 `--backend-binding`，或以 `--backend-target deployment/unit` 创建开发态预授权绑定并继续提交精确 Test Release。

Backend Test Release Controller 已完成第一条闭环：目标绑定、精确 Catalog 复核、应用组合候选、可信 readiness、自动回滚、重启 fail-closed 恢复和 Portal BFF/TypeScript SDK 均已接通。上传和发布仍是两个独立事务；开发 CLI 可以顺序协调两者，但只有 Controller 会修改 Deployment，仓库本身仍无部署权限。

Frontend Test Release 的服务端闭环已完成。托管仓库已实现按 Catalog revision 的确定性 SemVer 回溯求解、跨内核 `ArtifactLock v1`、独立授权的离线 Bundle、File Volume A/B 在线迁移、带 CAS 审计的 `deprecated/yanked/revoked` 生命周期，以及可恢复的 `plan -> quarantine -> sweep` GC。GC 只处理显式 yanked/revoked 且无精确引用的对象，要求 Seed/LKG 与全部租约引用源健康、迁移结束，并使用至少 24 小时隔离宽限期；普通引用心跳不会令 plan 无意义失效，但任何实际候选集合变化都会使旧 plan 失效。发布前全仓及 namespace/publisher/channel 累积配额、活动/隔离/reclaimed 容量统计，以及目录、容量/配额、引用、GC、存储迁移、stable 发布审批和供应链证据的统一 Workbench 已接通。生命周期原因/替代约束使用 Catalog revision CAS 表单，迁移动作严格投影后端阶段状态；stable 发布按 ADR-0134 精确绑定 testing 候选并执行双人分离，证据由可信仓库逐次复验后返回摘要。已打开页面可按 Profile 选择仅刷新、提示或自动消费新 Activation，并对不兼容边界执行 Host Epoch 预验证与失败回退。
