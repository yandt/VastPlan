# 制品仓库与测试发布

> 状态：架构已采纳，分阶段实施｜最后更新：2026-07-20
>
> 本文是 Seed 自举、托管制品仓库、包管理、测试发布和 Storage Provider 迁移的单一真相源。信任决策见 [ADR-0023](../decisions/ADR-0023-插件Schema与可验证制品仓库.md)、[ADR-0026](../decisions/ADR-0026-远端制品仓库与供应链信任.md)、[ADR-0049](../decisions/ADR-0049-制品信任基座与仓库基础插件.md)、[ADR-0091](../decisions/ADR-0091-制品存储Provider供给边界.md) 与 [ADR-0097](../decisions/ADR-0097-测试制品仓库与前端分级热升级.md)。

## 1. 目标

VastPlan 的仓库最终提供类似语言包管理器的体验：发现插件、解析兼容版本、生成精确锁、下载/验证、安装、升级、回退和离线分发。同时它也是测试交付入口：每个测试制品上传后，可以由独立发布控制器安全应用到指定 Backend、Portal、Runner 或 Mobile 测试目标。

仓库不解释“这份代码是否可以运行在某个服务”，也不直接启动插件。仓库负责保存事实，Resolver/发布控制负责授权和锁定，内核执行器负责最终验证和候选切换。

## 2. 三个运行角色

| 角色 | 负责 | 禁止 |
|---|---|---|
| Kernel Trust Base | Schema、签名、撤销、内容校验、安装事务、可信缓存 | 搜索、市场、latest、业务审批 |
| Seed Repository | 空节点自举、基础仓库插件、foundation 最小集、LKG、Bundle | 普通测试上传、在线目录、用户发布 |
| Managed Repository | 不可变对象、目录、流水账、晋级、包解析、GC、管理面 | 写部署期望态、命令 Node Agent、来源自证可信 |
| Test Release Controller | 目标绑定、精确 ref 解析、候选发布、测试、回滚和审计 | 持有签名私钥、绕过 Profile、直接读写仓库对象目录 |

## 3. 制品身份与版本

- `(pluginId, version, channel)` 是不可变键；相同 SHA 重传幂等，不同 SHA 拒绝。
- 测试构建使用唯一 SemVer 预发布版本；`testing` channel 不覆盖已有版本。
- 仓库为成功发布分配单调 `repositoryRevision`，发布事件携带精确 ref、SHA-256、publisher 和发生时间。
- “最近版本”只用于目录展示或 Test Release 解析；最终 Deployment、Activation 和 Lockfile 必须保存精确版本、channel、摘要和发布者。
- channel 晋级生成新的签名证明与审计记录，不改变源 channel 历史。

## 4. Repository Profile

Seed Profile 继续使用部署管理员提供的绝对文件路径。托管仓库 Profile 使用插件配置与托管凭证：普通字段包括监听、Storage Provider、volume、允许 channel、容量和保留策略；TLS 私钥、发布/读取凭证和 Provider 凭证只以部署秘密或 CredentialRef 注入。

开发环境使用两套身份：

- 每次运行生成的 Seed TLS/签名身份，只保护本次自举仓库；
- 保存在 `.vastplan/dev-platform/repositories/testing/` 下的稳定测试发布身份和 File Volume。目录和私钥权限必须不宽于 `0700/0600`，均不进入 Git。

每次启动生成组合信任快照，使 Node Agent 同时信任当前 Seed 和稳定测试发布者。生产环境不得自动生成或持久化生产签名私钥。

## 5. Catalog 与 Publish Journal

目录至少支持按 plugin ID、publisher、命名空间分类、版本、channel、目标内核、操作系统/架构和状态过滤，并返回分页结果。每个版本展示 Manifest 摘要、入口、依赖、签名、许可证、SBOM/来源证明状态、yank/deprecate 状态和活动引用。

Publish Journal 是可重放的单写流水账。索引是派生状态，必须能从不可变 artifact metadata、attestation 和 journal 重建。事件消费者只能把事件当触发器，处理前仍按精确 ref 读取并验证制品。

## 6. 包解析与锁

包解析器输入是根插件约束、目标内核、平台、允许 channel、发布者/隔离策略和 Catalog 快照 revision；输出为不可变锁文件。它必须检测依赖循环、版本冲突、平台不兼容、runtime capability 不满足、yanked/revoked 制品和 namespace/publisher 违规。

Node Agent、Portal Catalog、Runner Builder 和 Mobile Resolver 只消费各自解析产物，不在运行节点重新选择 latest。离线 Bundle 包含锁文件、证明、信任快照和全部对象；导入后仍走相同验证器。

## 7. Test Release

`TestTargetBinding` 由平台管理员预先把插件/分类绑定到测试 tenant、deployment/portal、unit 或允许的 Platform Profile 槽位。上传成功可以同时提交 Test Release 请求，但两者是不同事务。

Test Release 状态固定为：

```text
Queued -> Resolving -> Preparing -> Validating -> Activating -> Ready
                                      |             |
                                      +--> Failed <-+
Queued/Resolving/Preparing -> Superseded
Ready -> RollingBack -> RolledBack|Failed
```

同一 `target + pluginId` 串行执行；尚未提交的旧请求可以被更新请求合并。Backend 应用插件通过新的 Application Composition revision 进入 Node Agent 候选 unit；Frontend 通过新的 Portal Activation 进入候选 Portal Generation 或 Host Epoch。foundation/platform 只能进入专用测试 Platform Profile，不能由 Application Composition 覆盖。

## 8. Storage Provider 与迁移

Provider 只执行 `probe/provision/describe/migrate/release` 控制路径。对象热路径使用供给后的本地 mount 或 local endpoint，不逐对象调用插件 RPC。

首个迁移控制器以 File Volume A/B 验证：源/目标 probe、目标 provision、全量复制、逐对象摘要验证、发布增量追平、短暂冻结发布、最终校验、候选只读、原子切换、观察窗口和旧 volume release。失败始终保留旧数据面；相同 migration ID 幂等。S3/OCI Provider 不在当前阶段实现。

## 9. 生命周期、引用与 GC

- `deprecated`：可继续安装，但解析器给出替代提示；
- `yanked`：禁止新解析和安装，已有运行实例不被仓库主动停止；
- `revoked`：信任或安全事件，所有新验证 fail-closed；
- GC 必须保护 Active Deployment、Assignment、Portal Activation、锁文件、回滚历史、Seed 和 LKG 引用；
- 未引用对象先进入 quarantine，经过配置宽限期和最终复核后删除；
- 仓库自升级制品在激活前必须写入独立 Seed/LKG 来源，避免重启自举循环。

## 10. 安全与管理面

读取、发布、晋级、yank、迁移、GC 和管理各用独立 operation grant；CLI/CI 发布不等于 Portal 管理员。浏览器不接收仓库 token、信任私钥、mount path、Provider endpoint 或 Credential material。

管理中心逐步提供目录、版本、签名/SBOM、引用、Test Release、容量、迁移和审计工作区。初期浏览器只读目录和发布状态；上传及 stable 晋级在独立权限和审批完成前继续由 CLI/CI 执行。

## 11. 当前实施状态

已实现不可变本地仓库、签名 HTTPS 发布/读取、Node Agent 二次验证、Seed YAML Profile、托管仓库基础插件、本地 File Provider `probe/provision` 和只读状态页。

开发环境现已完成第一阶段分离：

- Seed repository、`local-development` 身份和 Seed 数据仍随单次运行生成；
- 托管测试仓库的 File Volume 与 `local-testing` 身份跨普通重启保留；
- 每次运行生成 Seed-only、testing-only 与包含两类公钥的 Node Agent 组合信任快照，两个仓库不能交叉接受发布身份；
- Node Agent 按“本地签名 Seed 优先、托管 HTTPS 仓库后备”解析精确 ref；
- 测试签名私钥不进入插件环境，宽权限目录和私钥 fail-closed；
- 开发状态端点只报告两个仓库的 URL 与持久属性，不暴露存储路径或密钥。

Catalog/Journal、受控测试上传 CLI、Test Release Controller、包解析/锁、在线迁移、GC/配额和完整管理 UI 尚未完成，不能在文档或产品中宣称可用。
