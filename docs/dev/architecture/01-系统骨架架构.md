# 系统骨架架构（Micro-kernel & Extension Points）

> 状态：设计草案 v0.2｜最后更新：2026-07-14
> 关联：[ADR-0001 微内核+全层扩展点](../decisions/ADR-0001-插件运行模型.md)、[ADR-0003 运行时热装](../decisions/ADR-0003-插件装载模型.md)、[ADR-0004 独立进程+协议总线](../decisions/ADR-0004-插件运行形态.md)、[ADR-0005 骨架与技术栈解耦](../decisions/ADR-0005-骨架与技术栈解耦.md)、[ADR-0006 内核三分区与后端组合](../decisions/ADR-0006-内核分区与后端组合.md)
> 本文是 VastPlan **系统骨架**的单一真相源，是当前第一优先级。全文保持在架构模式/契约层，凡涉具体技术栈处标注"待 ADR"。

## 1. 目标与设计约束

系统的第一要务是**做好骨架**，使功能能在骨架之上以插件形式持续扩展。约束（均已定为 ADR）：

- **微内核 + 一切功能皆插件**：内核最小化，审计/可观测/用户扩展/Studio 模块等都是第一方插件。
- **内核三分区**（ADR-0006）：后端内核、前端内核、客户端内核，各自独立、贴合各自运行环境，但**吃同一套统一插件定义**。
- **后端服务灵活组合**（ADR-0006）：后端内核可组合出 backend、workspace、rs 等多个后端服务，可合可拆。
- **运行时热装**：插件在线安装/启用/停用/卸载/升级，内核不重部署。
- **独立进程 + 协议总线**：插件独立进程运行，经稳定协议与宿主通信；故障隔离、栈自由。
- **第一方可信**：不做第三方不可信代码的安全沙箱；只需**故障隔离**。

## 2. 核心概念（术语单一真相源）

| 概念                       | 定义                                                                                                      |
| ------------------------ | ------------------------------------------------------------------------------------------------------- |
| **内核 Kernel**            | 最小骨架，提供插件宿主、扩展点注册表、生命周期、协议总线、核心横切服务。分三套：后端/前端/客户端内核。不含业务。                                               |
| **后端服务 Backend Service** | 后端内核 + 选定后端插件集 + 部署编排的一次**组合**产物（如 backend / workspace / rs）。同源、可合并或拆分部署。                               |
| **宿主 Host**              | 某套内核对插件的承载与调度运行时。一个后端服务是后端内核的一个宿主实例；前端内核、客户端内核各是一个宿主。                                                   |
| **插件 Plugin**            | 逻辑上一个功能特性，声明其贡献；可为纯 WebUI、纯 backend、或同时含 WebUI+backend+RC。物理上其各面部分由相应内核装配。                              |
| **扩展点 Extension Point**  | 内核/宿主开放的"可被插件填充的位置"，以**注册表 Registry** 实现（借鉴 testa 的 CallerKind/Scene/PermissionChecker/Hook/EventSink）。 |
| **贡献 Contribution**      | 插件对某扩展点的具体填充（一个工具包、一个 UI 视图、一个权限校验器、一个事件订阅…）。                                                           |
| **统一插件定义 Manifest**      | 三套内核共用的一份清单规范，声明插件对各面扩展点的贡献、依赖、装配元数据。见[插件清单规范](../plugins/插件清单规范.md)（待随本篇重构）。                           |
| **协议总线 Protocol Bus**    | 插件进程与宿主之间的稳定通信协议（贡献注册、调用、事件、生命周期指令）。见[插件-宿主协议](../plugins/插件宿主协议.md)（待写）。                               |
| **契约 Contract**          | 跨组件不可变的数据结构（如 CallContext / CallResult / Event），一次定义、所有方共用。                                             |

## 3. 总体结构：三套内核，一套插件定义

```
                         ┌────────────── 统一插件定义（同一份清单规范）──────────────┐
                         │  一个插件可含： webui 贡献 / backend 贡献 / rc 贡献（任意组合）│
                         └───────┬───────────────────┬───────────────────────┬───────┘
                                 │ 各内核只取与己相关的贡献装配                 │
        ┌────────────────────────▼─────┐ ┌───────────▼───────────┐ ┌─────────▼──────────┐
        │        后端内核 Backend Kernel │ │  前端内核 WebUI Kernel │ │ 客户端内核 RC Kernel │
        │  （可灵活组合出多个后端服务）  │ │   （浏览器，单独一套）  │ │  （客户端本地，单独一套）│
        └───┬──────────┬──────────┬─────┘ └───────────┬───────────┘ └─────────┬──────────┘
            │ 组合      │ 组合      │ 组合              │                        │
      ┌─────▼───┐ ┌────▼─────┐ ┌──▼──────┐      ┌──────▼──────┐          ┌──────▼──────┐
      │ backend │ │workspace │ │   rs    │      │ 前端插件组件 │          │  RC 插件进程 │
      │  服务    │ │  服务     │ │  服务   │      │ (视图/编辑器)│          │ (采集/执行) │
      │ +插件集  │ │ +插件集   │ │ +插件集 │      └─────────────┘          └─────────────┘
      └─────────┘ └──────────┘ └─────────┘
      每个后端服务经协议总线承载独立进程的后端插件
```

要点：
- **三套内核彼此独立**：后端内核（服务端）、前端内核（浏览器）、客户端内核（本地）各自贴合运行环境，实现栈可异构（ADR-0005）。
- **后端内核可组合**：backend / workspace / rs 等是同一后端内核的不同组合形态（内核 + 选定插件集 + 部署编排），可合并或拆分部署，同源不分叉。
- **一套插件定义贯通三面**：插件清单同时容纳 webui / backend / rc 贡献；每套内核在装配时只取与自己相关的部分。一个横跨多面的特性逻辑上是一个插件。
- 每套内核内的"宿主↔插件"通信统一走协议总线。

## 4. 微内核职责（三套内核的共性最小集）

每套内核都提供以下最小集（实现可各异，语义一致）；其余一律下沉为插件：

1. **插件生命周期管理器**：装载/激活/停用/卸载/升级的状态机与编排（§7）。
2. **扩展点注册表**：登记本内核有哪些扩展点、每个扩展点上有哪些贡献；提供注册/解绑/查询（§5）。
3. **协议总线**：插件接入、贡献声明、调用路由、事件分发、生命周期指令（§6）。
4. **核心横切服务**（尽量以"内核内置扩展点 + 可被插件替换的默认实现"提供）：身份（统一 Principal）、权限（`(caller, scene, target)` 三元组，校验器为扩展点）、事务/持久化契约、可观测（埋点与 EventSink 出口，审计/可观测实现为插件）、事件总线、配置。

> 三套内核的横切服务**侧重不同**：后端内核最完整；前端内核偏 UI 装配与会话身份；客户端内核偏本地执行与连接管理。共性接口一致，差异在专篇细化。

> 判据：凡"换一种实现是合理需求"的能力（审计策略、权限规则、用户校验、可观测后端），内核只定**扩展点与契约**，默认实现以插件提供。

## 5. 扩展点模型（注册制驱动）

借鉴 testa 应用骨架层的**注册制**：每类可扩展能力对应一个 Registry，插件把贡献注册进去，宿主在调用时查表分发。

- **扩展点 = 一个具名 Registry + 一份贡献契约**。例如：
  - `PermissionCheckerRegistry`（后端）：贡献 = 权限校验器（输入 `(caller, scene, target)`，输出准/拒）。
  - `EventSinkRegistry`（后端）：贡献 = 事件汇（审计、可观测插件在此消费事件）。
  - `HookRegistry`：贡献 = 在关键节点前后插入的钩子。
  - `ToolPackageRegistry`（后端）：贡献 = Agent 工具包（包 + 子命令）。
  - `ViewSlotRegistry`（前端）：贡献 = 挂到某 slot 的视图。
  - `RunnerCapabilityRegistry`（客户端/rs）：贡献 = 一种执行模式/采集能力。
- **场景化三元组**：权限/计量/统计等判断统一基于 `(caller, scene, target)`。
- **动态性**：注册表支持运行时注册/解绑以支撑热装（§7）；解绑需可回收（引用清理、在途调用收敛）。
- **不可变契约**：`CallContext / CallTarget / CallResult / CallEvent` 一次定义、跨面共用，是扩展点协作的稳定接口。

## 6. 插件与宿主的连接：独立进程 + 协议总线

- 插件（进程/组件）启动后**连接**所属内核宿主，握手：上报清单、声明其贡献、协商协议版本。
- 宿主据贡献把插件**接入对应扩展点**（注册进 Registry）。
- 运行时三类消息：
  1. **调用（宿主→插件）**：扩展点被触发时，宿主把 `CallContext` 经总线转给插件，收 `CallResult`。
  2. **事件（双向）**：内核事件下发给订阅插件；插件也可发布事件。
  3. **生命周期指令（宿主→插件）**：激活/停用/优雅关闭等。
- **协议是唯一硬约束**（ADR-0004/0005）：序列化格式、传输方式待 ADR；契约字段先定。
- **故障隔离**：插件崩溃 → 宿主感知连接断开 → 从扩展点摘除其贡献、按策略重启，不影响内核与其他插件。
- **跨面插件的内部协作**：同一插件的前端部分↔后端部分↔RC 部分之间的通信，走平台常规 API/任务通道，不占用宿主-插件协议总线（后者只管本内核内的宿主与插件）。详见 [内核间与服务间通信](02-内核间与服务间通信.md#7-跨面插件各部分的协作通道)。

## 7. 生命周期（运行时热装）

```
  未安装 ──install──▶ 已安装(未激活) ──activate──▶ 已激活 ──deactivate──▶ 已安装(未激活)
     ▲                    │                          │
     └──────uninstall─────┘◀──────uninstall──────────┘
  已激活 ──upgrade──▶ (停旧→装新→迁移状态→起新)
```

- **install**：拉取插件包 → 校验清单与依赖（§8）→ 登记，不启动。
- **activate**：按需（惰性激活事件）拉起插件 → 握手 → 贡献注册进扩展点 → 可用。
- **deactivate**：优雅关闭 → 从扩展点解绑贡献 → 释放资源。
- **uninstall**：deactivate + 删包与登记。
- **upgrade**：停旧 → 装新 → 迁移状态 → 起新；失败可回滚。
- 全程**不重部署内核**；三套内核对各自管辖的插件面独立执行以上流程（一个跨面插件在各内核上的激活可相互独立）。

## 8. 依赖与装配元数据（原"权限"的再定位）

第一方可信语境下，清单里的 `capabilities` 不是安全边界，而是**装配元数据**：

- 声明插件需要哪些**内核能力/资源**（LLM 编排、某凭证、某内核服务），供宿主激活时**装配注入**与**排依赖**。
- 声明对**其他插件**的依赖（版本范围），供生命周期管理器解析依赖树、定序激活、检测环状依赖。
- 后端贡献可标注**面向哪个后端服务角色**（backend/workspace/rs/…），供组合时决定装入哪个服务。
- 不做"安装时逐项授权 + 运行时拒绝"的安全强制（第三方开放后的未来议题）。

## 9. 三内核 / 扩展面速览

| 内核 | 运行环境 | 组合/形态 | 典型扩展点 | 典型第一方插件 |
|---|---|---|---|---|
| 后端内核 | 服务端 | 组合出 backend / workspace / rs 等服务 | 工具包 / Agent / 权限校验器 / 事件汇 / Hook / Runner 调度 | 审计、可观测、用户校验、信息扩充、执行后端 |
| 前端内核 | 浏览器 | 单一前端应用 | 视图 slot / 编辑器 / 命令 / 设置项 | Studio 可开发模块、各业务面板 |
| 客户端内核 | 客户端本地 | 单一 RC 应用 | 本地执行能力 / 采集器 / 运行时适配 | 平台特定采集、脚本运行时 |

> 各内核的详细扩展点清单与契约、后端组合的编排机制，各自展开专篇（待写）。

## 10. 与技术栈的边界（ADR-0005）

本篇定义的一切——内核职责、扩展点=Registry、协议总线三类消息、生命周期状态机、不可变契约字段、统一清单语义——均为**架构模式与契约**，不绑定语言。三套内核实现栈可异构。后续独立 ADR 评估：各内核实现栈、协议序列化/传输、持久化选型、后端组合编排的落地形态。
