# 系统骨架架构（Micro-kernel & Extension Points）

> 状态：设计草案 v0.1｜最后更新：2026-07-14
> 关联：[ADR-0001 微内核+全层扩展点](../decisions/ADR-0001-插件运行模型.md)、[ADR-0003 运行时热装](../decisions/ADR-0003-插件装载模型.md)、[ADR-0004 独立进程+协议总线](../decisions/ADR-0004-插件运行形态.md)、[ADR-0005 骨架与技术栈解耦](../decisions/ADR-0005-骨架与技术栈解耦.md)
> 本文是 VastPlan **系统骨架**的单一真相源，是当前第一优先级。全文保持在架构模式/契约层，凡涉具体技术栈处标注"待 ADR"。

## 1. 目标与设计约束

系统的第一要务是**做好骨架**，使功能能在骨架之上以插件形式持续扩展。约束（均已定为 ADR）：

- **微内核 + 一切功能皆插件**：内核最小化，审计/可观测/用户扩展/Studio 模块等都是第一方插件。
- **全层可插拔**：扩展面覆盖**后端、Web UI、RS、RC** 四层；RS/RC 自身也是可被插件扩展的宿主。
- **运行时热装**：插件在线安装/启用/停用/卸载/升级，内核不重部署。
- **独立进程 + 协议总线**：插件独立进程运行，经稳定协议与宿主通信；故障隔离、栈自由。
- **第一方可信**：不做第三方不可信代码的安全沙箱；只需**故障隔离**。

## 2. 核心概念（术语单一真相源）

| 概念 | 定义 |
|---|---|
| **内核 Kernel** | 最小骨架，提供插件宿主、扩展点注册表、生命周期、协议总线、核心横切服务。不含业务。 |
| **宿主 Host** | 某一层内嵌内核能力、承载并调度该层插件的运行时。四层各一类宿主：Backend Host / WebUI Host / RS Host / RC Host。 |
| **插件 Plugin** | 一个独立进程/组件，声明其贡献点，连接宿主后被接入扩展点。可横跨多层（各层各一份贡献）。 |
| **扩展点 Extension Point** | 内核/宿主开放的"可被插件填充的位置"，以**注册表 Registry** 实现（借鉴 testa 的 CallerKind/Scene/PermissionChecker/Hook/EventSink）。 |
| **贡献 Contribution** | 插件对某扩展点的具体填充（一个工具包、一个 UI 视图、一个权限校验器、一个事件订阅…）。 |
| **清单 Manifest** | 插件的声明式入口，列出它对各层扩展点的贡献、依赖、装配元数据。见[插件清单规范](../plugins/插件清单规范.md)（待随本篇重构）。 |
| **协议总线 Protocol Bus** | 插件进程与宿主之间的稳定通信协议（贡献注册、调用、事件、生命周期指令）。见[插件-宿主协议](../plugins/插件宿主协议.md)（待写）。 |
| **契约 Contract** | 跨组件不可变的数据结构（如 CallContext / CallResult / Event），一次定义、所有方共用。 |

## 3. 总体结构

```
                        ┌──────────────────────── 内核 Kernel（最小骨架）────────────────────────┐
                        │  插件生命周期管理器 · 扩展点注册表 · 协议总线 · 核心横切服务             │
                        │  （身份 / 权限 / 事务 / 可观测 / 事件 / 配置）                          │
                        └───────────────┬───────────────┬───────────────┬───────────────┬───────┘
                                        │ 内嵌于        │               │               │
                   ┌────────────────────▼──┐ ┌──────────▼─────────┐ ┌───▼──────────┐ ┌──▼───────────┐
                   │   Backend Host        │ │   WebUI Host       │ │   RS Host    │ │   RC Host    │
                   │  后端扩展点集合        │ │  前端扩展点集合     │ │  RS 扩展点   │ │  RC 扩展点   │
                   └───────┬───────────────┘ └─────────┬──────────┘ └──────┬───────┘ └──────┬───────┘
                           │ 协议总线              协议总线                协议总线          协议总线
              ┌────────────┼───────────┐             │                     │                 │
        ┌─────▼────┐ ┌─────▼────┐ ┌────▼─────┐   ┌────▼─────┐         ┌─────▼────┐      ┌─────▼────┐
        │ 插件A后端 │ │ 插件B后端 │ │ 审计插件  │   │ 插件A前端 │         │ RS 扩展   │      │ RC 扩展   │
        │ (独立进程)│ │ (独立进程)│ │(可观测)   │   │(独立组件) │         │ (独立进程)│      │ (独立进程)│
        └──────────┘ └──────────┘ └──────────┘   └──────────┘         └──────────┘      └──────────┘
              └──── 同一个功能特性"插件A"可横跨后端+前端+RC，各层各一份贡献 ────┘
```

要点：
- **内核是一份能力**，被四类宿主各自内嵌；宿主是"内核在某一层的落地 + 该层扩展点"。
- **插件横跨多层**：一个功能特性（如"销售助手"）可同时贡献后端逻辑、前端面板、RC 采集脚本，逻辑上是一个插件、物理上是分布在各层宿主下的多个组件。
- 所有"宿主↔插件"通信统一走协议总线，四层同构。

## 4. 微内核职责（最小集）

内核只做"没有它插件就无法协作"的事，其余一律下沉为插件：

1. **插件生命周期管理器**：装载/激活/停用/卸载/升级的状态机与编排（§7）。
2. **扩展点注册表**：登记有哪些扩展点、每个扩展点上有哪些贡献；提供注册/解绑/查询（§5）。
3. **协议总线**：插件进程接入、贡献声明、调用路由、事件分发、生命周期指令下达（§6）。
4. **核心横切服务**（对标 testa 骨架层，但本身也尽量以"内核内置扩展点 + 默认实现可被插件替换"的方式提供）：
   - **身份**：统一 Principal（用户/角色/项目上下文），替代散落的 user_id。
   - **权限**：基于 `(caller, scene, target)` 三元组的判定，校验器（PermissionChecker）本身是扩展点，可由用户系统扩展插件填充。
   - **事务/持久化契约**：统一数据访问抽象（具体存储待 ADR）。
   - **可观测**：Tracing/Metrics/Audit/Statistics 的采集接口——审计与可观测的**实现**是插件，内核只留埋点与事件出口（EventSink 扩展点）。
   - **事件**：内核事件总线，插件通过 Hook/EventSink 订阅与发布。
   - **配置**：插件与内核的配置读写。

> 判据：凡是"换一种实现是合理需求"的能力（审计策略、权限规则、用户校验、可观测后端），内核只定**扩展点与契约**，默认实现以插件提供。

## 5. 扩展点模型（注册制驱动）

借鉴 testa 应用骨架层的**注册制**：每类可扩展能力对应一个 Registry，插件把贡献注册进去，宿主在调用时查表分发。

- **扩展点 = 一个具名 Registry + 一份贡献契约**。例如：
  - `PermissionCheckerRegistry`：贡献 = 一个权限校验器（输入 `(caller, scene, target)`，输出准/拒）。
  - `EventSinkRegistry`：贡献 = 一个事件汇（审计、可观测插件在此消费事件）。
  - `HookRegistry`：贡献 = 在关键节点前后插入的钩子。
  - `ToolPackageRegistry`：贡献 = 一个 Agent 工具包（包 + 子命令）。
  - `ViewSlotRegistry`（WebUI）：贡献 = 挂到某 slot 的视图。
  - `RunnerCapabilityRegistry`（RS/RC）：贡献 = 一种 Runner 侧能力/执行模式。
- **场景化三元组**：权限/计量/统计等判断统一基于 `(caller, scene, target)`，使规则集中、可插拔。
- **动态性**：注册表支持运行时注册/解绑，以支撑热装（§7）。解绑需可回收（引用清理、在途调用收敛）。
- **不可变契约**：`CallContext / CallTarget / CallResult / CallEvent` 等一次定义、跨层共用，是扩展点之间协作的稳定接口。

> 具体有哪些扩展点、各自契约——将逐个展开为专篇；本篇只定"用 Registry 承载扩展点"这一模型。

## 6. 插件与宿主的连接：独立进程 + 协议总线

- 插件进程启动后**连接**所属宿主，完成握手：上报清单、声明其贡献、协商协议版本。
- 宿主据贡献把插件**接入对应扩展点**（注册进 Registry）。
- 运行时交互三类消息：
  1. **调用（宿主→插件）**：某扩展点被触发时，宿主把 `CallContext` 经总线转给插件，收 `CallResult`。
  2. **事件（双向）**：内核事件下发给订阅插件；插件也可发布事件。
  3. **生命周期指令（宿主→插件）**：激活/停用/优雅关闭等。
- **协议是唯一硬约束**（ADR-0004/0005）：序列化格式、传输方式待 ADR；但契约字段先定。
- **故障隔离**：插件进程崩溃 → 宿主感知连接断开 → 从扩展点摘除其贡献、按策略重启，不影响内核与其他插件。

## 7. 生命周期（运行时热装）

插件状态机（内核生命周期管理器编排）：

```
  未安装 ──install──▶ 已安装(未激活) ──activate──▶ 已激活 ──deactivate──▶ 已安装(未激活)
     ▲                    │                          │
     └──────uninstall─────┘◀──────uninstall──────────┘
  已激活 ──upgrade──▶ (停旧→装新→迁移状态→起新)
```

- **install**：拉取插件包 → 校验清单与依赖（§8）→ 登记，但不启动。
- **activate**：按需（惰性激活事件）拉起插件进程 → 握手 → 贡献注册进扩展点 → 可用。
- **deactivate**：优雅关闭进程 → 从扩展点解绑贡献 → 释放资源。
- **uninstall**：deactivate + 删除包与登记。
- **upgrade**：停旧版 → 装新版 → 迁移必要状态 → 起新版；失败可回滚到旧版。
- 全程**不重部署内核**；各层宿主对其管辖插件独立执行以上流程。

## 8. 依赖与装配元数据（原"权限"的再定位）

第一方可信语境下，清单里的 `capabilities` 不再是安全边界，而是**装配元数据**：

- 声明插件需要哪些**内核能力/资源**（如需要 LLM 编排、需要某凭证、需要访问某内核服务），供宿主在激活时**装配注入**与**排依赖**。
- 声明对**其他插件**的依赖（版本范围），供生命周期管理器解析依赖树、定序激活、检测环状依赖。
- 不做"安装时管理员逐项授权 + 运行时拒绝"的安全强制（那是第三方开放后的未来议题）。

## 9. 四层扩展面速览

| 层 | 宿主 | 典型扩展点 | 典型第一方插件 |
|---|---|---|---|
| 后端 | Backend Host | 工具包 / Agent / 权限校验器 / 事件汇 / Hook | 审计、可观测、用户校验、信息扩充 |
| Web UI | WebUI Host | 视图 slot / 编辑器 / 命令 / 设置项 | Studio 可开发模块、各业务面板 |
| RS | RS Host | Runner 调度能力 / 执行模式 / 协议扩展 | 新执行后端、编排策略 |
| RC | RC Host | 本地执行能力 / 采集器 / 运行时适配 | 平台特定采集、脚本运行时 |

> 各层宿主的详细扩展点清单与契约各自展开专篇（待写）。

## 10. 与技术栈的边界（ADR-0005）

本篇定义的一切——内核职责、扩展点=Registry、协议总线三类消息、生命周期状态机、不可变契约字段——均为**架构模式与契约**，不绑定语言。后续独立 ADR 评估：内核实现栈、协议序列化/传输、各层实现技术、持久化选型。
