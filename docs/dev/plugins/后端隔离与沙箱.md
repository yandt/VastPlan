# 后端隔离与沙箱

> 状态：设计草案 v0.1｜最后更新：2026-07-14
> 关联：[ADR-0001 全栈插件](../decisions/ADR-0001-插件运行模型.md)、[插件清单规范](插件清单规范.md)（§3.2 后端贡献点、§5 权限模型）
> 本文是**插件后端代码隔离执行**的单一真相源，是全栈插件里风险最高的一环。

## 1. 定位与威胁模型

后端贡献点（`contributes.backend` 的 tools / agents / apiRoutes / eventHooks）会在 VastPlan 云端运行**第三方插件代码**。插件来自市场，**默认不可信**。隔离的目标：一个恶意或有缺陷的插件，**不能**做到以下任何一件事——

- 触及未在清单 `capabilities` 声明的资源（网络、文件、凭证、其他插件数据）
- 读到其他租户/订阅者的数据（多租户场景）
- 拿到凭证明文或平台密钥
- 逃逸到宿主机、影响平台其他组件的可用性

## 2. 核心原则（来自 testa 的血泪经验，直接沿用）

1. **SDK 软检查不是安全边界。** 插件 SDK 里的网络/文件预检只能拦"诚实代码的意外越界"；恶意代码可以绕过 SDK 直接 `import socket` / `open()`。**真正的越权防护必须在进程外**，插件代码碰不到。SDK 的价值是开发便利与合规提示，不是隔离。
2. **隔离分级，自由度跟着隔离走。** 硬边界（VM/gVisor/容器：越界路径物理不存在、syscall 被拦截）所在之处，SDK 从"必须的门"降为"可选的便利"，插件可用原生 `open`/`requests`；软边界之处 SDK 预检保持 fail-closed。
3. **能力在边界强制。** 清单 §5 声明的 `capabilities` 不是君子协定，而是在沙箱边界上由宿主强制：未授权的出站域名在网络层被拒，未授权的路径在挂载层不存在。
4. **Fail-closed 默认。** 隔离信号缺失、后端未登记、能力未声明——一律按最严处理，绝不放松。
5. **凭证不落盘、只注入、不见明文。** 凭证由宿主加密投递到沙箱内的运行时，插件通过 SDK 取用，日志与输出自动脱敏（对照 testa 凭证信封 + 脱敏）。

## 3. 隔离机制选型（分档，不是单选）

没有单一机制适配所有场景；按"隔离强度 / 启动开销 / 密度 / 平台"分档，插件按其 `capabilities` 与信任级映射到某档。候选对比：

| 机制 | 隔离强度 | 冷启动 | 密度 | 说明 |
|---|---|---|---|---|
| **WASM 运行时** | 中（内存安全，能力按导入函数授予） | 极快(<10ms) | 极高 | 适合纯计算型工具；无原生 syscall，天然最小权限。受限于插件须编译为 WASM |
| **受限进程 / namespace** | 软 | 快 | 高 | seccomp+namespace，成本低但边界弱，仅作低敏档 |
| **容器 (runc)** | 中 | 快(~100ms) | 高 | 标准化打包，但共享宿主内核，需配合 seccomp/AppArmor |
| **gVisor (runsc)** | 硬 | <100ms | 高(2-4× VM) | 用户态内核拦截 syscall；密度与冷启动优，原生 checkpoint/restore；**仅 Linux** |
| **microVM (libkrun/Firecracker)** | 最硬 | 百 ms~秒 | 中 | 独立内核，隔离最强；跨平台（含 macOS 开发态）；密度受限 |

**初步方向（待 ADR 定稿）**：VastPlan 后端主要运行在 **Linux 云端**，因此

- **强隔离主力档 = gVisor**：密度、冷启动、checkpoint/restore 三者兼顾，适合市场第三方插件的默认档。
- **最高敏感档 = microVM**：处理高敏数据或需最强隔离的插件。
- **轻量计算档 = WASM**：无需网络/文件、纯数据处理的工具包，享受最小权限与最高密度。
- 受限进程仅作为无强隔离环境下的**降级兜底**，且 fail-closed（按软边界对待）。

> 具体后端组合与默认策略将各自形成 ADR（gVisor 引入、WASM 运行时选型）。

## 4. 隔离分级模型

沿用 testa 的二值信号，向沙箱内注入隔离档，SDK 与运行时据此决定放松与否：

| 档 | 后端 | 文件系统权威边界 | SDK 文件/网络预检地位 |
|---|---|---|---|
| **hard** | microVM / gVisor / (加固)容器 | 挂载表：未授权路径**物理不存在** | 冗余 → 降为开发提示 |
| **soft** | 受限进程 / namespace | SDK 预检 + 执行器校验 | **主要门，不得放松** |
| （信号缺失） | 直接 exec / 旧环境 | 同 soft | 同 soft（fail-closed） |

**单一真相源**：后端能力注册表中每个后端一个 `IsolatesResources` 谓词，宿主据此折算 `hard/soft` 并注入沙箱环境变量。未登记后端零值 = `false` = soft（fail-closed）。

## 5. 能力（capabilities）在边界的强制点

清单 §5 的每一项能力对应一个**边界强制机制**，而非 SDK 检查：

| capability | 强制机制 |
|---|---|
| `network` 域名白名单 | 沙箱出站流量走**宿主出口代理**，仅放行白名单域名；裸 socket 在 hard 档被网络命名空间/virtio-net 门控 |
| `filesystem` 路径范围 | **挂载表**只挂授权路径，未授权路径在沙箱内不存在 |
| `credentials` | 宿主**加密注入**到沙箱内运行时，SDK 取用，明文永不落盘、输出脱敏 |
| `llm` | LLM 调用不由插件直连，而是经**宿主编排网关**计量、鉴权、限流 |
| `client.exec` | 属客户端面，见 `客户端代码分发.md`（待写） |
| `storage` `plugin-scoped` | 宿主按 `插件id + 租户` 派发隔离存储命名空间 |

## 6. 多租户隔离

已发布插件可能被多个企业客户/订阅者同时使用。要求 **per-租户隔离**：

- 开发者不可见订阅者数据；订阅者之间互不可见。
- 共享型插件的每租户执行实例独立沙箱（隔离档 hard），存储、凭证、日志按租户命名空间隔离。
- 高密度需求下用 gVisor + checkpoint/restore：空闲实例休眠为快照，命中时毫秒级唤醒，兼顾隔离与密度。

## 7. 与清单 / 生命周期的衔接

- **安装时**：宿主读清单 `capabilities`，向管理员逐项授权；据 `activation` 决定惰性加载。
- **激活时**：宿主按插件敏感级选隔离后端，装配挂载表、出口代理白名单、凭证信封、隔离档环境变量，再启动插件后端进程。
- **调用时**：Agent 调用工具包 → 宿主注入执行上下文（用户/租户/项目/凭证句柄）→ 沙箱内执行 → 结果压缩/脱敏/计量/审计。
- **停用/卸载时**：见 `插件生命周期.md`（待写）——回收沙箱、清理命名空间存储、吊销凭证句柄。
