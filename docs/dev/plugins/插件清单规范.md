# 插件清单规范（Manifest & Contribution Points）

> 状态：**待随骨架设计重构**｜最后更新：2026-07-14
> 关联决策：[ADR-0001 微内核 + 全层扩展点](../decisions/ADR-0001-插件运行模型.md)
>
> ⚠️ **重要修订说明**：本篇写于早期误解（把插件当成第三方不可信市场代码）之下。现已明确插件为
> **第一方可信的系统级模块**，扩展面覆盖后端/Web UI/RS/RC 四层。因此：
> - §5 权限模型的"安装时管理员逐项授权 + 运行时安全强制"框架**过度设计**，capabilities 应降级为
>   **依赖与装配元数据**（声明模块需要哪些内核能力/资源以便装配和排依赖），而非安全边界。
> - 三注入面（frontend/backend/client）需扩展为**四层扩展面**，且与骨架的扩展点（Registry）模型对齐。
>
> 待《系统骨架架构》定稿后，本篇将据此重写。以下内容中"贡献点/manifest 声明式注册"的骨干思路仍有效，
> 安全隔离相关表述请忽略。

## 1. 定位

插件清单是插件的**唯一声明入口**。VastPlan 宿主只读清单就能知道：这个插件是谁、需要什么权限、在前端/后端/客户端三处各注入什么能力、何时被激活。运行时代码不参与"我有什么"的声明——**声明与实现分离**（借鉴 VSCode `package.json#contributes` 与 testa `@expose` 的声明式注册）。

设计三原则：

1. **声明式单一真相源**——能力在清单里静态声明，宿主据此建目录、做权限、做审核，不启动插件代码即可掌握全貌。
2. **贡献点分层解耦**——前端、后端、客户端三个注入面各有独立贡献点，任意组合，互不强依赖。
3. **企业安全前置**——权限（capabilities）在清单里显式声明并在安装时授权；插件运行时只能触及已授权的资源。

## 2. 清单文件

- 文件名：`vastplan.plugin.json`（位于插件包根目录）。
- 格式：JSON（配套 JSON Schema 供编辑器校验，路径待定）。

### 2.1 顶层字段

| 字段 | 必填 | 说明 |
|---|---|---|
| `id` | ✓ | 全局唯一，反向域名式：`com.acme.sales-copilot` |
| `name` | ✓ | 展示名 |
| `version` | ✓ | 语义化版本 `MAJOR.MINOR.PATCH` |
| `publisher` | ✓ | 发布者标识（与市场账户绑定，用于签名校验） |
| `description` | ✓ | 一句话描述 |
| `engines.vastplan` | ✓ | 兼容的宿主版本范围，如 `^1.2.0`；宿主据此判断能否安装 |
| `icon` / `categories` / `keywords` | | 市场展示与检索用 |
| `capabilities` | ✓ | 权限声明，见 §5 |
| `activation` | ✓ | 激活事件，见 §4 |
| `contributes` | ✓ | 贡献点集合，见 §3（三个注入面） |
| `dependencies` | | 依赖的其他插件 `{id: versionRange}`，见 §6 |
| `entry` | | 各面的运行时入口文件，见 §3 |

### 2.2 骨架示例

```jsonc
{
  "id": "com.acme.sales-copilot",
  "name": "销售助手",
  "version": "1.0.0",
  "publisher": "acme",
  "description": "为销售流程提供 Agent 工具、工作台面板与客户端采集脚本",
  "engines": { "vastplan": "^1.0.0" },

  "capabilities": {
    "network": ["https://api.acme.com"],
    "llm": true,
    "storage": "plugin-scoped",
    "client.exec": ["python"],
    "credentials": ["acme-crm"]
  },

  "activation": ["onAgentTool:acme.crm", "onView:acme.salesPanel"],

  "entry": {
    "frontend": "dist/frontend.js",
    "backend": "dist/backend/main",
    "client": "client/"
  },

  "contributes": {
    "frontend": { /* §3.1 */ },
    "backend":  { /* §3.2 */ },
    "client":   { /* §3.3 */ }
  }
}
```

## 3. 贡献点模型（三个注入面）

一个插件可在三处注入能力，任意组合。`contributes` 下三个可选键分别对应，未声明的面即不占用。

### 3.1 前端贡献点 `contributes.frontend`

向平台工作台注入 UI。宿主提供固定的挂载位（extension slots），插件声明往哪挂、挂什么。

| 贡献点 | 说明 | 对照 |
|---|---|---|
| `views` | 向侧边栏/面板区注入视图 | VSCode `views` |
| `editors` | 为某类资源提供自定义编辑器（含打开/保存/dirty 生命周期） | testa `EditorProvider` / VSCode `CustomEditorProvider` |
| `commands` | 注册可被菜单/快捷键/命令面板触发的命令 | VSCode `commands` |
| `menus` | 把命令挂到菜单位 | VSCode `menus` |
| `settings` | 声明插件配置项（渲染到设置页，值存宿主） | VSCode `configuration` |

```jsonc
"frontend": {
  "views": [
    { "id": "acme.salesPanel", "title": "销售看板", "slot": "sidebar" }
  ],
  "commands": [
    { "id": "acme.syncCrm", "title": "同步 CRM" }
  ],
  "settings": [
    { "key": "acme.region", "type": "enum", "options": ["cn", "us"], "default": "cn" }
  ]
}
```

> 前端代码如何隔离加载、如何与宿主通信——见 `前端宿主与通信.md`（待写）。

### 3.2 后端贡献点 `contributes.backend`

注入运行在隔离沙箱中的后端能力。核心是 Agent 工具，其模型借鉴 testa 的"**包 + 子命令**"（一个逻辑单元对外是一个包，内部多个子命令），避免工具名散乱、schema 挤占上下文。

| 贡献点 | 说明 | 对照 |
|---|---|---|
| `tools` | 向 Agent 暴露的工具包（`package` + 多个 `subcommand`） | testa `UnifiedToolRegistry` / `@expose` |
| `agents` | 预置 Agent 定义（角色、系统提示、可用工具集） | — |
| `apiRoutes` | 注入受宿主网关代理的 HTTP 端点 | FastAPI router |
| `eventHooks` | 订阅宿主事件（如任务完成、文件变更） | — |

```jsonc
"backend": {
  "tools": [
    {
      "package": "acme.crm",
      "title": "CRM 操作",
      "subcommands": [
        { "name": "query",  "description": "查询客户", "paramsSchema": { /* JSON Schema */ } },
        { "name": "update", "description": "更新客户", "paramsSchema": { /* ... */ } }
      ]
    }
  ]
}
```

> 工具执行时通过宿主注入的执行上下文（用户/项目/凭证/资源，对照 testa `ToolExecutionContext`）访问资源，不允许绕过 §5 权限。后端隔离方案见 `后端隔离与沙箱.md`（待写）。

### 3.3 客户端贡献点 `contributes.client`

声明下发到远程 Runner 本地执行的脚本/工作流。清单只声明"有什么、怎么触发、需要什么运行时"，代码体在 `entry.client` 目录。

| 贡献点 | 说明 |
|---|---|
| `scripts` | 可下发执行的脚本（声明运行时、入口、参数 schema） |
| `workflows` | 由多步骤编排的工作流定义 |
| `runtimeRequirements` | 客户端需具备的运行时（如 `python>=3.10`），宿主据此校验目标机 |

```jsonc
"client": {
  "scripts": [
    {
      "id": "acme.collectLogs",
      "runtime": "python",
      "entry": "client/collect_logs.py",
      "paramsSchema": { /* ... */ }
    }
  ],
  "runtimeRequirements": ["python>=3.10"]
}
```

> 脚本如何签名、下发、在 Runner 侧沙箱执行——见 `客户端代码分发.md`（待写）。

## 4. 激活事件 `activation`

插件默认**不激活**（惰性），仅在声明的事件发生时由宿主唤醒，降低资源占用（借鉴 VSCode activation events）。

| 事件 | 触发时机 |
|---|---|
| `onStartup` | 宿主启动即激活（慎用） |
| `onView:<viewId>` | 对应视图被打开 |
| `onCommand:<commandId>` | 对应命令被调用 |
| `onAgentTool:<package>` | Agent 首次调用该工具包 |
| `onClientScript:<scriptId>` | 该客户端脚本被调度 |

激活的粒度：三个面**可独立激活**——只用到后端工具时不必加载前端代码。

## 5. 权限模型 `capabilities`

企业级 + 插件市场的安全底线。**未在 `capabilities` 声明的能力，运行时一律拒绝**；安装时向管理员展示并逐项授权。

| 键 | 语义 |
|---|---|
| `network` | 允许访问的出站域名白名单（空/未声明 = 禁止联网） |
| `llm` | 是否可调用平台 LLM 编排 |
| `storage` | 存储范围：`plugin-scoped`（隔离）/ `none` |
| `client.exec` | 允许在 Runner 上执行的运行时类型白名单 |
| `credentials` | 可申领的凭证名（值由宿主凭证系统注入，插件永不见明文） |
| `filesystem` | 客户端可访问的路径范围（默认沙箱内） |

> 权限是**声明 → 安装授权 → 运行时强制**三段式，任一段缺失即拒绝。

## 6. 依赖 `dependencies`

插件可依赖其他插件（如共享一个基础工具包）。宿主在安装时解析依赖树、校验版本范围、检测环状依赖，缺失则阻止安装。

```jsonc
"dependencies": { "com.acme.crm-core": "^2.0.0" }
```
