# ADR-0038 Backend 可复现发布与运维交付

- 状态：已采纳
- 日期：2026-07-16
- 关联：[ADR-0017 版本定义与兼容性机制](ADR-0017-版本定义与兼容性机制.md)、[ADR-0026 远端制品仓库与供应链信任](ADR-0026-远端制品仓库与供应链信任.md)、[ADR-0031 Backend 内核 1.0 封板与工程门禁](ADR-0031-Backend内核1.0封板与工程门禁.md)、[ADR-0035 Backend 可观测与健康契约](ADR-0035-Backend可观测与健康契约.md)
- 可执行步骤见《[Backend 发布与运维](../guides/Backend发布与运维.md)》

## 背景

通过测试的源码仍不是可交付内核。若构建嵌入工作目录、时间或随机 build ID，同一提交无法产生同一字节；只有 SHA-256 而没有来源证明，使用方无法确认制品由哪个提交和工作流产生；若升级检查和诊断依赖源码仓库，生产节点就无法独立预检与取证。

Backend 1.0 因此需要把“源码通过门禁”延伸为“可验证制品和可执行运维流程”。这项工作不引入部署平台或 Web 管理面，而是提供部署适配器可以稳定调用的最小交付面。

## 决策

### 1. 构建与版本

1. Backend 正式构建固定使用仓库 `go.mod` 指定的 Go 工具链、`CGO_ENABLED=0`、`-trimpath`、`-buildvcs=false` 和空 Go build ID。同一源码、工具链和目标平台必须逐字节一致。
2. 内核版本只来自 `kernels/backend/VERSION`，以 `-X main.version=<version>` 注入。Git 提交不写进二进制；提交身份由签名来源证明绑定，避免本地 VCS 路径和状态污染可复现字节。
3. 发布标签固定为 `backend-v<VERSION>`。工作流先校验标签与版本文件完全一致，且标签提交必须位于 `main`，任一门禁失败都不创建 Release。
4. 正式矩阵覆盖 Linux/macOS 的 amd64/arm64。每个目标独立构建、独立生成 SBOM、独立签署证明，不能用一个目标的证明覆盖通配符命中的其他文件。
5. 24 小时 soak 运行于版本推广前的冻结提交。推广提交以 `kernels/backend/SOAKED_COMMIT` 绑定该 SHA；从被测提交到标签提交只允许修改 `VERSION`、`SOAKED_COMMIT`、`CHANGELOG.md` 和封板验收状态。Release 工作流下载该提交的成功报告，并重新检查时长、真实调用/重启、pending、goroutine 与文件句柄收敛；只看工作流绿色状态不算完成。

### 2. SBOM、校验和与签名证明

每个 Backend 二进制附带确定性的 CycloneDX 1.5 SBOM。SBOM 从 Go 二进制 build info 读取实际链接模块，并绑定二进制 SHA-256；Release 同时提供覆盖二进制与 SBOM 的 `SHA256SUMS`。

GitHub Release 工作流使用 OIDC 和 Artifact Attestations 为每个二进制生成构建来源证明，并为该二进制绑定其 SBOM。证明主体必须是精确的单个二进制路径，SBOM 文件不得被误登记为可执行制品。使用方同时验证校验和与签名证明；只验证其中一个不算完成供应链校验。

### 3. 运维能力随二进制交付

Backend 二进制内置三个只读运维入口：

- `version [--json]`：报告内核版本、Go 工具链和目标平台；
- `validate`：使用与运行时相同的解析器预检 DesiredState v1、Deployment v2 和 actual-state；actual-state v1 只在内存迁移演练，不写原文件；
- `support-bundle`：打包版本、二进制摘要、脱敏实际态摘要和可选 `kernel.diagnostics` 快照。

支持包不包含 DesiredState 配置、凭证明文、插件安装路径、错误正文、payload 或 metadata。可选诊断再次按敏感字段名防御性脱敏，输出文件权限固定为 `0600`。实时诊断仍由部署适配器经 `kernel.diagnostics` 获取；内核不为了支持包绑定 HTTP 端口或具体编排平台。

### 4. 升级与回滚

升级前必须完成制品验证、配置预检、actual-state 读取迁移演练和备份。二进制以版本目录安装，并通过原子链接切换当前版本；旧版本和旧配置至少保留到观察窗结束。

配置回滚使用旧快照的显式 revision。Node Agent 已允许回退到较小 revision，同时拒绝“同 revision、不同内容”；不得编辑已发布 revision 冒充回滚。actual-state v1 在读取边界迁移为 v2，首次成功保存后只写 v2；不支持把 v2 状态文件反向转换为 v1。回滚旧 Backend 前，应停止进程并恢复升级前备份的 actual-state，避免旧二进制读取未知新格式。

## 备选方案

- **只发布源码或容器构建说明**：无法证明使用方拿到的字节，也让生产预检依赖开发环境。拒绝。
- **把提交哈希和构建时间写入二进制**：信息直观，但破坏跨工作目录、跨时间的可复现性。提交身份改由来源证明承载。
- **只签 `SHA256SUMS`**：可以覆盖多个文件，但不能为每个制品绑定生成工作流和 SBOM。保留校验和用于离线完整性，签名来源证明仍逐制品生成。
- **支持包直接归档原始 actual-state 与日志**：排障信息更多，但会泄露路径、错误上下文和潜在业务数据。拒绝，默认只交付结构化摘要。

## 影响

- 正面：发布字节、依赖清单、生成来源、配置兼容和回滚锚点均可机器验证；生产排障不依赖源码。
- 代价：发布前增加多目标构建、证明生成和运维预检时间；GitHub 签名证明验证需要可访问 GitHub 的环境。
- 约束：气隙环境可离线验证预先带入的 `SHA256SUMS` 与 SBOM，但签名证明必须在入区前完成验证并保存结果，或由部署方替换为企业内部等价证明系统。
