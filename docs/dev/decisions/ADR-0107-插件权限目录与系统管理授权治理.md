# ADR-0107 插件权限目录与系统管理授权治理

- 状态：已采纳，分阶段实施
- 日期：2026-07-22
- 关联：[ADR-0021](ADR-0021-权限判定强制点.md)、[ADR-0068](ADR-0068-分布式平台管理中心与强类型BFF.md)、[ADR-0106](ADR-0106-多端统一身份授权与Runner执行租约.md)

## 背景

平台管理能力已经由 Portal Management Binding、Node BFF 路由和 Backend `permission.checker` 多层保护，但 BFF 与平台访问策略分别维护硬编码 `capability/operation → role` 表，并把 `platform.admin` 当作宽泛旁路。新增插件管理、服务管理和在线角色后，这种方式会产生权限代码漂移、前后端判定不一致和无法即时撤权的问题。

## 决策

1. 权限 code 与 operation guard 由所属插件在签名 Manifest `authorization` 中声明。声明绑定 namespace、scope、risk、assignable、offlineAllowed、精确 operation、访问类型和审批语义。
2. 可信 Catalog 构建器只消费经 Artifact Trust 验证的 Manifest 与制品 SHA-256，拒绝保留命名空间抢占、未知操作、未声明权限、重复 code 和重复 operation owner。安装插件不自动授权。
3. Portal Management Binding 继续决定某 Portal 可寻址的服务和 operation；它不是用户角色。BFF 只做体验预检，Backend PEP 保持最终强制，领域服务继续强制对象状态和职责分离。
4. `authorization-policy` 管理不可变 Permission Catalog revision、Role revision、Subject Binding、撤权序列和签名 Policy Snapshot；`role-management` 只提供管理面，不参与最终判定。
5. 每个内核服务运行 `authorization-enforcer` 本地基础插件。它以平台 audience 校验签名快照并就近判定；策略不可用时按风险和快照 TTL fail-closed。
6. 人的在线角色不授予内部插件/系统回调。系统身份继续使用签名 caller、transport trust、capability allowlist 和 workload grant。
7. `is_admin` 和 `platform.admin` 不得成为长期通用旁路。首次安装 owner 和 break-glass 必须物化为有 revision、有效期、审批和审计的角色/租约；硬编码兼容仅允许存在于 B1→B6 迁移窗口。
8. 系统管理按职责拆分。插件/制品的读取、生命周期、GC、迁移，以及服务/节点的读取、编辑、引导、审批、发布和测试目标授权使用不同权限；高风险操作不得因一个笼统管理员角色合并。
9. platform/tenant/project/resource 是统一 Policy Domain/Scope 契约，不复制成四套权限产品。Root Domain 通过签名 delegation ceiling 向子域委托，子域不能扩大父域授权或覆盖撤权。
10. Policy 下开放 `authorization.store.v1`、`authorization.engine.v1`、`authorization.directory.v1` 和 `authorization.exchange.v1` Provider Protocol。Provider 可以独立插件化和多语言实现，但必须消费/产生统一 Authorization IR、Policy Snapshot 与 Decision/Proof；存储、判定、目录和导入职责不能合并后绕过 Policy 状态机。

## 备选方案

- **直接使用 IdP Group/Token 权限**：部署简单，但角色含义散落、撤权滞后且无法形成统一 revision；拒绝。
- **中央角色插件手工创建权限代码**：UI 简单，但会与插件实际 operation 漂移，也允许授权不存在的能力；拒绝。
- **只靠 Portal 菜单与 BFF**：不能保护 Mobile、Runner、服务间调用或绕过浏览器的请求；拒绝。
- **每次请求同步调用中央 PDP**：一致但把所有服务延迟和可用性绑定到单点；拒绝，采用签名快照加高风险在线复核。

## 影响

- 插件 Manifest Schema 增加可选 `authorization`，但一旦声明就必须闭合、可验证并绑定本插件真实 operation。
- 权限目录属于插件制品供应链的一部分；角色和主体绑定属于独立平台状态，两者不能互相改写。
- Portal、Backend、Mobile 和 Runner 共享 permission code 与 Policy revision，但各自认证载体保持独立。
- 不同 Policy Domain 可以选择不同 Provider Profile；内核使用者仍按发布者策略决定第三方 Provider 是否允许及其隔离强度，critical 根域默认只接受明确批准的实现。
- 迁移期间现有硬编码策略继续作为受测基线；B6 完成后删除通用管理员旁路和重复 operation-role 表。

## 实施记录（2026-07-22）

- 完成 B1 契约：Manifest 权限声明、保留命名空间、作用域/风险、离线限制、operation 绑定、未使用权限和冲突校验。
- 完成确定性 Permission Catalog v1：条目绑定插件 ID、版本、发布者和制品 SHA-256，并生成规范 digest。
- 首批迁移 `cn.vastplan.platform.artifacts.repository` 与 `cn.vastplan.platform.infrastructure.deployment-manager`，覆盖插件/制品和服务/节点管理的 11 个权限、37 个用户管理操作；内部引用发布等 workload operation 不伪装成人员权限。
