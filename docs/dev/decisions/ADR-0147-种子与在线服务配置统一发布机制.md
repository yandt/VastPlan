# ADR-0147 种子与在线服务配置统一发布机制

- 状态：已采纳，第一阶段已实现
- 日期：2026-07-24
- 关联：[ADR-0057](ADR-0057-插件分级管理与双输入组合解析.md)、[ADR-0077](ADR-0077-Backend在线组合与可信发布边界.md)、[ADR-0116](ADR-0116-Backend-Platform-Profile候选Catalog与配置激活.md)、[ADR-0142](ADR-0142-内核启动与业务发布完全分离.md)、[ADR-0143](ADR-0143-插件化应用意图规划与派生依赖.md)

## 背景

Backend 已有两种配置入口：开发/生产首次建立时从受信文件读取平台基础组合，后续由 Deployment Manager 从在线账本读取应用组合。两者最终都生成 Deployment v2，但旧的 `controlplane` 种子路径直接调用 Resolver、生成配置目录并写 Deployment KV；在线路径则通过 `deploymentpublisher` 完成 Catalog 选择、预览摘要锁、Resolver 复验、CAS 和配置目录发布。两条发布实现会使相同服务因为配置来源不同而得到不同的校验、错误和恢复语义。

平台自身不能完全依赖 Deployment Manager 自举，因为 Deployment Manager 尚未运行时无法发布承载它自己的服务。这个循环只能由一个极窄、显式授权的种子入口打破，但种子入口不应成为第二套服务模型或第二套发布器。

## 决策

### 1. 服务执行模型只有一套，配置来源可以有多个

统一链路固定为：

```text
Seed File Source ─┐
                  ├─> Platform Catalog/Profile + Application Composition
Online Source ────┘       └─> deploymentpublisher
                              ├─> trusted preview + digest lock
                              ├─> Composition Resolver
                              ├─> Deployment v2 CAS
                              └─> Configuration Catalog
                                      └─> Controller → Assignment → Node Agent
```

配置来源只负责取得并规范化发布输入，不拥有 Resolver、Deployment KV、Assignment、调度或插件生命周期。来源差异不得进入 Deployment v2，也不得让 Node Agent 分支处理。

### 2. 种子文件是 Source，不是特殊服务类型

种子来源读取受信 `Platform Profile + Application Composition`，把精确 tenant/deployment 与 Profile ref 适配成不可变的单 binding Catalog，再调用与在线路径相同的 `deploymentpublisher.Preview/Publish`。种子服务与在线服务使用相同 Schema、摘要算法、来源锁、制品复验、配置目录和 CAS 规则。

种子来源仅允许在显式 `bootstrap/apply` 中使用。`up`、`restart`、systemd 常驻启动和 Node Agent 恢复不得读取文件后产生发布；它们只消费控制面最后一次已提交的 Deployment。

### 3. 在线配置是另一种 Source，治理流程保持更强

在线来源由 Deployment Manager 的共享账本和后续 Composition Planner 提供。用户编辑 Application Intent，Planner 生成 Application Composition；Deployment Manager 执行草稿、提交、异人审批、预览摘要锁和发布。可信内核每次从持久 Backend Platform Catalog 选择活动 Profile，再调用同一个 `deploymentpublisher`。

种子来源拥有的是部署环境预先授予的 bootstrap 权限，不伪装成人工审批；在线来源不得借用这个权限绕过审批。两者统一的是发布执行机制，不是把权限强度降到相同水平。

### 4. Profile 与 Application 的分级不变

统一机制不等于把平台插件暴露给应用用户。种子平台文件可以声明平台基础服务；在线应用用户仍只编辑 Application Intent/Composition。平台 Profile 的在线调整继续走 ADR-0116 的候选 Catalog、独立权限、readiness 和补偿 Saga。

### 5. Source 元数据只用于审计，不影响运行

发布入口和日志可以记录 `seed-file`、`online-ledger` 等来源，便于审计和诊断；来源不参与 Deployment 摘要和运行分支。相同规范输入必须生成相同 Deployment 内容。未来增加 GitOps、企业 CMDB 或远程配置 Provider 时，只增加来源适配器，不增加发布器或 Controller。

## 否决方案

- **只统一 JSON/YAML 格式**：发布核心仍复制，无法阻止行为漂移。
- **让 Deployment Manager 发布自身**：首次空环境形成循环依赖，无法可靠自举。
- **把种子文件长期当运行真相源**：普通重启会覆盖在线变更，违反启动与发布分离。
- **把在线读取写进内核**：扩大内核并绑定具体数据库/配置中心；在线账本仍由插件管理。

## 实施记录

第一阶段已经完成：

1. `deploymentpublisher.SeedCatalog` 将文件型 Platform Profile 与 Application Composition 适配为精确单 binding Catalog，并严格校验身份。
2. `backend-kernel controlplane` 删除独立的直接 Resolve、配置目录构建和 KV 发布实现；种子发布改为先 Preview，再携带同一摘要调用统一 Publisher。
3. 在线 Deployment Manager 原有 `kernel.deployment.preview/publish` 继续使用同一个 Publisher 和持久 CatalogSource，Wire API、Controller、Node Agent 无需改动。
4. 种子发布日志显式记录 `source=seed-file`；普通启动不创建任何 Source publication。
5. 平台种子 Application 的 Document ID 已与 Deployment name 统一为 `platform-management`；种子 Application 的规范内容纳入开发 source digest，任何 Profile 或 Application 变化都会分配新的单调发布 revision。

后续 Application Intent/Planner 上线时，在线账本会从“直接保存 Application Composition”升级为“保存 Intent、Plan 与已编译 Composition”，但 Publisher 之后的链路不变。

## 影响

正面：平台种子服务和未来在线服务不再有两套 Resolver/CAS 实现；新增配置来源只需适配统一输入；首次自举仍可用；在线权限边界不降低；内核、Controller 和 Node Agent 无需感知来源。

代价：显式 bootstrap 会执行一次可信预览和摘要复验，产生少量控制面开销；种子文件和在线活动 Catalog 仍有不同的所有权与治理状态，运维界面必须清楚标识来源，不能把“机制统一”误解为“任何人都能修改平台基线”。

## 后续边界（2026-07-24）

[ADR-0148](ADR-0148-公共服务基线与种子配置分域.md) 进一步规定：统一发布机制不等于统一配置所有权。Platform Profile `serviceBaselines` 与 Application service 配置分别在线管理；Profile `services` 只保存本地 Seed Service 必要配置，并从在线 Configuration Catalog 排除。三类配置继续进入本 ADR 的同一 Publisher 和运行链路。
