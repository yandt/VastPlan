# ADR-0082 前端工作台组合框架

- 状态：已采纳
- 日期：2026-07-19
- 关联：[ADR-0052](ADR-0052-前端门户内核与多UI设计系统插件.md)、[ADR-0054](ADR-0054-跨端体验契约与交互协调.md)、[ADR-0064](ADR-0064-Portal语义组件契约与动态表单运行时.md)、[ADR-0065](ADR-0065-通用JSON-Schema表单与Arco主题适配.md)、[ADR-0074](ADR-0074-Portal组合Slot与纯布局插件分层.md)、[ADR-0079](ADR-0079-Frontend框架无关多语种契约.md)

## 背景

Portal UI Contract 3.0 已提供框架无关的 Page、Table、FilterBar、Pagination、Dialog、Drawer 与 FormRenderer 基础组件，Arco/MUI 均可实现同一组件面。但它们仍是“积木”：每个功能插件会自行决定筛选状态、分页、列偏好、行/批量操作、卡片懒加载、弹窗提交、表单分组和失败反馈。长期会产生相同业务模式的不同交互、无障碍与权限提示。

把这些模式分别放进 Arco/MUI 适配器会复制状态与策略；让功能插件直接使用框架私有组件会破坏多 UI 适配；把全部页面改成低代码 JSON 则会限制复杂领域视图，并把提交和权限边界暴露到浏览器配置。

## 决策

1. 在语义基础组件与功能插件之间新增 **UI Workbench** 层。它是首方 foundation 插件 `cn.vastplan.foundation.frontend.workflow.workbench`，贡献新的 `ui.workflow.workbench` single 扩展点；Platform Profile 精确固定其制品，并要求它与唯一 `ui.render.adapter` 的 UI Contract 主版本兼容。
2. Workbench 是功能插件唯一的页面 UI 入口。功能插件只能依赖 `@vastplan/workbench-sdk`，每个页面必须是 `WorkbenchPageDefinition`；构建、制品检查和 Portal 组装均拒绝功能插件直接导入 `@vastplan/ui-primitives`、Arco/MUI 或注册裸 React 页面。`@vastplan/ui-primitives` 只允许 Workbench、设计系统和布局/组合基础插件内部使用。
3. Workbench 只组合 `@vastplan/ui-primitives` 的框架无关组件，禁止直接导入 Arco/MUI。它拥有查询、选择、偏好、动作队列、表单提交、Overlay 生命周期、脏数据保护和统一反馈状态机；设计系统适配器仍拥有 DOM、主题、焦点、键盘、虚拟化和框架具体渲染。
4. `@vastplan/ui-contract` 使用 3.x，并一次性迁移当前首方前端插件；开发阶段不保留旧运行时或旧字段别名。新增的可序列化契约包括 `WorkbenchPageDefinition`、`CollectionSpec`、`FilterSpec`、`ColumnSpec`、`ActionSpec`、`FormPresentation`、`FormWorkflow` 与有限条件表达式。数据 Schema、呈现规则和运行代码严格分离。
5. Workbench V1 固定五类通用模式：
   - 数据集合：表格/卡片两种视图、筛选、排序、分页或游标懒加载、列偏好、选择与批量动作；
   - 记录详情：主从布局、详情字段、状态带与上下文动作；
   - 表单工作流：页面/Dialog/Drawer 容器、分组分栏、步骤/标签页、条件显示、校验、提交、确认、成功刷新与错误映射；
   - 动作系统：页面、工具栏、行、卡片、批量和表单操作区的统一位置、优先级、溢出与破坏性确认；
   - 用户偏好：每个用户、Portal 与 collection ID 范围内的列显隐、顺序、宽度、密度和 page size。
6. JSON Schema Draft 7 继续是 `FormSchema.schema` 的唯一数据合法性真相。新 `FormPresentation` 只描述展示；条件显示/禁用使用有限、无副作用的表达式 DSL，禁止 JavaScript、函数、模板字符串、URL 或框架私有 `x-component`。`uiSchema` 继续承载通用 widget/帮助提示，不能成为布局或授权真相。
7. 功能插件提供经过类型化 BFF/能力客户端读取的 `CollectionLoader`、动作处理器和表单提交处理器；这些运行时函数不进入序列化契约。声明式 `ActionSpec` 不能携带 URL、HTTP 方法、权限结论或任意请求模板。服务端仍是权限、对象状态与事务的最终裁决者。
8. 复杂视图不能以业务插件直接使用基础组件的方式逃逸。V1 不提供任意 `custom React block`；新的通用呈现能力必须先作为 Workbench Pattern 进入 foundation Workbench 制品，并同时通过契约、Arco/MUI、无障碍、国际化和安全审查。未来如需独立 Pattern 插件，必须另立 ADR 定义其可信来源、受限 SDK 与 Profile 装配规则。
9. `ui.workflow.workbench` 不负责 Shell、菜单、页面 Slot 拓扑或视觉布局；它与 `ui.structure.composition`、`ui.structure.layout` 保持正交。一个区域的视觉位置由布局决定，区域内列表、卡片、表单和操作行为由 Workbench 决定。
10. Collection 的高数据量查询必须由服务端执行筛选/排序；页码模式与 cursor 模式互斥。Card lazy load 使用 cursor；需要总数、跳页或审计浏览时使用分页。用户偏好只能调整插件白名单中的展示字段，不能改变服务端可见字段、过滤能力或权限。

## 备选方案

- **把组合模式分别塞入每个设计系统插件**：短期改动小，但 Arco、MUI、未来框架以及 Mobile 会分别维护相同的查询、提交和动作状态机，拒绝。
- **功能插件自由组合基础组件**：灵活，但最常见的企业管理交互会持续分叉，无法保证操作区、分页、空态、错误与无障碍一致，拒绝。
- **只规定 Workbench 优先、允许随时逃逸到底层组件**：初期迁移阻力小，但最复杂、最常用的页面会最先绕开规范，最终无法形成一致性，拒绝。
- **所有页面采用完全声明式低代码屏幕 Schema**：一致性最高，却无法优雅承载复杂领域可视化与交互，并会扩大浏览器配置的安全面，暂不采用。
- **让 Layout 插件负责列表和表单**：会把纯视觉布局与领域工作流耦合，且替换布局时可能改变业务行为，拒绝。

## 影响

- 正面：首方功能插件获得统一的企业工作区体验；Arco/MUI/Mobile/Runner 可复用同一数据和交互语义；大部分页面不再重复编写列表、弹窗和提交样板代码。
- 正面：数据、呈现、运行动作和权限边界分别可测试；用户偏好有明确归属，不会污染发布的 Portal 组合。
- 代价：Platform Profile 将增加一个基础单例，Resolver、Catalog、Portal Runtime 和两套设计系统必须同步实现 3.x 契约；当前功能插件需要迁移。
- 代价：Workbench 不是任意页面生成器。新的通用模式必须先进入契约、跨适配器验收和无障碍测试，不能以某个业务插件的临时需求扩张公共面；复杂页面的首次交付会等待 Pattern 能力进入基础层。
- 验收：Arco 与 MUI 对同一 Collection/Form/Action fixture 产生等价键盘、焦点、状态和提交语义；服务端筛选与 cursor 边界、权限隐藏与拒绝、脏数据保护、条件字段、翻译、缩放和窄屏均有自动化覆盖。

## 实施记录

- 2026-07-21：`CollectionWorkbench` 完成共享查询状态机和 Card/Cursor 呈现。`CollectionResult.nextCursor` 是唯一增量游标，加载更多按稳定记录键去重并拒绝重复 cursor；筛选、刷新、取消、选择和 action 继续与 Table/Page 共用。新增跨框架 `DataCard` 语义组件，Arco/MUI 各自实现原生 Card 与选择控件；卡片定义只允许字段键、标准格式、响应式列数和手动/视口加载策略，不开放任意组件或 HTML。
- 2026-07-21：完成 `FormPresentation` / `FormWorkflow`。Workbench 拥有加载、条件投影、脏状态、异步校验、字段错误、单次提交、关闭确认和成功刷新；Render Adapter 在同一完整 JSON Schema 上实现 sections/tabs/steps 与 1–4 列，不拆分验证真相。Collection Action 只引用已登记 form ID，独立页面通过 `defineFormPage` 注册。`credentialRef` widget 强制对应字段为 `format: vastplan-credential-ref + writeOnly`。全局设置 0.3 成为首个非敏感真实 fixture；凭证和数据库连接的一次性秘密输入暂不迁移。
- 2026-07-21：实施时 UI Contract 已由后续基础能力演进到 4.0，Workbench 直接落在当前 4.x，不保留本 ADR 最初规划的 3.x 兼容层。异步校验与提交字段错误使用 `LocalizedText`，由 Workbench 在当前 Portal locale 下解析，避免功能插件或 Arco/MUI Adapter 固化语言。
- 2026-07-21：新增 `secretMaterial` 一次性材料语义，与只传引用的 `credentialRef` 分离。注册门禁强制 `type=string + format=vastplan-secret-material + writeOnly` 且禁止 initialValue；Workbench 不把材料写入 baseline，并在提交、异常、取消或关闭后丢弃引用，loader 回填会被拒绝并清除。凭证 0.5、数据库连接 0.5 已迁移到 Collection/Form Workbench；JavaScript 不承诺无法实现的物理内存清零。
- 2026-07-21：完成动态 `prepare`、状态条件 Action、JSON/表格 Overlay 与统一值格式；Portal Composer 已拆为 Profile、Application、Binding、Activation 四个 Workbench 页面，Deployment Manager 也迁移到相同模式。生产前端构建器与 `engineering/arch` 适应度测试开始同时拒绝功能插件直接导入 React、Arco/MUI、视觉 `ui-primitives` 或调用裸 `context.addPage`。为避免本次改造同时扩张 SDK 拆包范围，暂时只允许从 `ui-primitives` 命名导入非视觉 `PortalControlClient` 和 `Portal*` 数据契约；该例外不提供组件树逃生口。
