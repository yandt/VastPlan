# ADR-0046 Apache-2.0 开源许可与插件制品声明

- 状态：已采纳
- 日期：2026-07-17
- 关联：[ADR-0019 工程规范基线](ADR-0019-工程规范基线.md)、[ADR-0023 插件 Schema 与可验证制品仓库](ADR-0023-插件Schema与可验证制品仓库.md)、[插件契约与协议](../architecture/插件契约与协议.md)

## 背景

VastPlan 已公开源代码，但仓库根目录没有许可证。公开可见不等于获得复制、修改和再分发授权；同时，现有 CI 只检查第三方 Go 依赖的许可证并忽略本仓模块，无法表达 VastPlan 自身的授权方式。

插件还会脱离源码仓库，以独立 gzip tar 制品分发。只在仓库根目录放许可证无法保证制品接收者同时得到授权文本，因此项目级许可和插件制品许可必须分别声明，但不能维护两套互相漂移的规则。

## 决策

1. VastPlan 当前仓库统一采用 **Apache License 2.0**。除某个子目录或文件明确声明其他许可证外，根目录 `LICENSE` 覆盖源代码、Schema、协议、文档、工具和当前第一方示例插件。
2. 版权主体为个人，版权告示使用 `Copyright 2026 zhanghui`，放在根目录 `NOTICE`。Apache-2.0 不授予 VastPlan 名称、标识或商标的额外使用权。
3. `README.md` 必须提供明确、可点击的许可证入口；`LICENSE` 保持 Apache 官方原文，不在正文中加入项目自定义条款，项目归属信息只放 `NOTICE`。
4. 第三方依赖继续由 ADR-0019 的许可证白名单约束。依赖准入规则与项目自身采用何种许可证是两件事；CI 中 `--ignore cdsoft.com.cn/VastPlan` 只表示依赖扫描不重复检查本仓模块，不表示本仓没有许可证。
5. 插件清单 v1 向后兼容地新增可选 `license`、`licenseFile` 与 `noticeFile`：
   - `license` 使用 SPDX 表达式；
   - 一旦声明 `license`，必须同时声明包内相对路径 `licenseFile`，反之亦然；
   - `noticeFile` 在许可证要求传递归属告示时声明包内相对路径；
   - 制品打包和读取必须确认声明的法律文件存在、唯一、非空且大小受限；
   - 当前 `extensions/plugins/` 下第一方插件统一声明 `Apache-2.0`，发布工具把根 `LICENSE` 与 `NOTICE` 注入制品根目录；
   - 未声明这些字段的既有 v1 清单仍可解析，避免破坏已发布测试夹具和旧制品。正式第三方市场启用前，再通过新版本准入策略把声明升级为强制要求。
6. 架构守护必须验证根 `LICENSE`、`NOTICE`、README 入口，以及仓库内第一方插件的许可声明，防止后续重构或打包遗漏。
7. 若未来改为强 copyleft、双许可、开放核心或对特定插件使用不同许可证，必须新增 ADR，并单独处理既有贡献者授权；不得静默覆盖本决策。

## 备选方案

- **MIT**：最简单，但没有 Apache-2.0 同等明确的贡献者专利授权，不利于企业客户审查；否决。
- **MPL-2.0**：能要求修改过的原文件继续开源，但增加逐文件许可边界和分发判断成本；当前优先扩大内核与 SDK 采用，否决。
- **AGPL-3.0 或 AGPL + 商业双许可**：能约束闭源网络服务修改版，但会提高企业采购、依赖兼容和贡献权属治理成本；当前阶段否决，未来商业模式明确后可重新决策。
- **只添加根 LICENSE，不处理插件制品**：源码仓库清晰，但独立插件包仍可能缺少许可证副本；否决。

## 影响

- 正面：任何人都能明确获知使用、修改、再分发和专利授权条件；GitHub 与自动化工具可识别标准许可证；企业客户和贡献者的准入成本降低。
- 正面：第一方插件制品携带可验证的 SPDX 标识与完整许可证文本，源码仓库和制品分发语义一致。
- 代价：再分发者必须保留许可证、变更说明、适用的版权/归属告示和 `NOTICE`；发布链增加许可证文件校验。
- 边界：Apache-2.0 允许商业使用和闭源衍生分发，本决策不阻止第三方提供托管服务；VastPlan 可通过商业插件、托管服务、支持和品牌授权形成商业模式。
