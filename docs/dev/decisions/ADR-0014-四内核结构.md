# ADR-0014 四内核结构：拆出 Runner 内核与移动 Companion 内核

- 状态：已采纳
- 日期：2026-07-15
- 关联：[ADR-0006 内核分区与后端组合](ADR-0006-内核分区与后端组合.md)（内核数由三扩为四）、[ADR-0011 组合是通用内核能力](ADR-0011-组合是通用内核能力.md)、[ADR-0012 APP 内核运行模型](ADR-0012-APP内核运行模型.md)、[ADR-0013 APP 多档能力](ADR-0013-APP多档能力与手机Companion.md)（手机由"APP 的一个档"提升为独立内核）

## 背景

ADR-0013 把手机 Companion 作为"APP 内核的一个受限能力档"。但手机（原生呈现客户端）与桌面/服务器（无头执行器）在**构建装配、运行形态、分发更新、通信、贡献面**上几乎处处不同——是"无头执行器"与"原生呈现客户端"两种东西塞在一个内核名下。按我们"**一个内核对应一种运行环境**"的原则（ADR-0006 拆三内核的同一依据），应把它们拆开。

## 决策

**内核由三分区扩为四**：

| 内核 | 运行环境 | 本质 | 运行/分发 |
|---|---|---|---|
| 后端内核 Backend | 服务端 | 组合出 backend/workspace/rs 服务 | 服务端进程，运行时热装 + 独立进程 |
| 前端内核 WebUI | 浏览器 | React 呈现，组合出门户 Portal | 浏览器；Module Federation 热装 |
| **Runner 内核**（原 APP/RC 桌面档） | 桌面/服务器 mac/win/linux | 无头执行器：常驻连接、后台任务、本地脚本 | 预编译单一二进制 + 自更新（ADR-0012） |
| **移动内核 Companion** | 手机 iOS/Android | 原生呈现客户端：监控/审批/触发 | gomobile bind 进原生外壳；App Store/MDM 分发；APNs/FCM 推送 |

配套：

1. **命名**：桌面/服务器执行器内核定名 **Runner 内核**（客户端执行器；承接原 RC/APP 桌面完整档）；手机内核定名 **移动内核（Companion）**。
   - **为何不叫 "Client 内核"**：桌面执行器与移动 Companion **都装在客户端**，"Client" 只描述位置、无法区分两者（会重蹈 "APP" 撞名）；"Runner" 描述**功能**（无头执行器/跑任务），正是拆内核的执行器 vs 呈现端那条轴，且对齐 testa 的 RS（Runner Service）↔ RC（Runner Client）血缘与保留的 `rc.*` 命名空间。
2. **共享客户端核心库**：契约（CallContext 等）、鉴权、清单模型、配置抽成一个共享库，供 Runner 内核与移动内核共用——拆内核不等于重复造轮子。
3. **组合能力延续**（ADR-0011）：Runner→客户端 App/Profile（预编译，ADR-0012）；移动→Companion App/Profile。四套内核都可"内核 + 选定插件集 + 配置"组合出多个单元。
4. **贡献面**：清单新增移动面（Companion 的监控/审批/触发 UI 贡献）；`rc` 面归 Runner 内核（执行/采集/脚本）。

## 修订的既有决策

- **ADR-0006**：内核分区由三扩为四（后端/前端/Runner/移动）；后端组合部分不变。
- **ADR-0013**：其"桌面完整 runner vs 手机 Companion"的能力划分**仍然有效**，只是承载从"同一 APP 内核的两个档"改为"两套独立内核"。ADR-0012（预编译）归 Runner 内核；ADR-0013 的手机分发/推送归移动内核。

## 备选方案

- **保持三内核、APP 内核含桌面/手机双档**（ADR-0013 现状）：共享代码最多，但一套内核内含两种差异极大的运行形态，与"内核=一种运行环境"原则不符。否决。

## 影响

- 正面：每套内核贴合单一运行环境、无平台分叉；移动作为原生呈现客户端概念清晰（本质近前端，仅栈不同故独立）。
- 代价：从 3 内核变 4 内核，多一套内核定义与贡献面；需抽客户端共享核心库。
- 关注点：
  - 移动内核的贡献面/清单键（`mobile` 面）与可用插件集定稿。
  - 共享客户端核心库的边界（哪些归共享、哪些归各内核）。
  - 三处对客户端内核的历史命名（RC→APP→Runner）在文档中统一，仅保留一次血缘说明。
