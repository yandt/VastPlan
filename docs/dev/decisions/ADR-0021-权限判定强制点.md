# ADR-0021 权限判定的强制点与零校验器语义

- 状态：已采纳
- 日期：2026-07-15
- 关联：[ADR-0001 微内核 + 全层扩展点](ADR-0001-插件运行模型.md)、[ADR-0017 版本定义与兼容性机制](ADR-0017-版本定义与兼容性机制.md)（fail-closed 强制点）、[插件契约与协议 §4.2/§4.3](../architecture/插件契约与协议.md)

## 背景

落地 select 分发（`permission.checker`）时，规格（§4.3）只写了"按 priority 高→低，遇到第一个非 abstain 即定论；**全 abstain 则 fail-closed（deny）**"。但它没回答两个必须回答的问题：

1. **谁来调用校验器？** 一个没人调用的分发器是死代码——权限扩展点的全部价值在于**被强制执行**。
2. **注册了零个校验器时怎么办？** "全 abstain" 说的是已注册的都弃权；零个注册是未定义的。

## 决策

### 1. `Host.Invoke` 是权限强制点

所有经 `Invoke` 的调用（含**插件回调宿主** HostCall）先做权限判定，未获放行即返回应用层错误 `permission.denied`。

- 权限拒绝是**应用层错误**（进 `CallResult.Error`），不是传输层错误——对齐工程规范 §4.2（该文已把"权限拒绝"列为应用层错误示例）。
- 内部另有不校验的 `invoke` 路径，**仅供两处**：① 校验器自身的调用（否则自我递归）；② 事件扇出（内核发起的投递，非用户调用）。业务路径一律走 `Invoke`。

### 2. 零校验器 → fail-closed 拒绝

一个 `permission.checker` 都没注册时，**拒绝所有调用**。

理由：权限校验器是插件（ADR-0001 一切功能皆插件），没装它就等于**没有授权依据**。此时放行会让"忘装权限插件"变成**静默的全开放**——这正是 fail-closed 要防的失败模式。与全系统一致（未定义扩展点拒绝、engines 不符拒绝、能力未注册拒绝）。

代价：内核开箱即"谁也调不动"，必须装权限插件才可用。这是有意的——企业级系统没有授权插件本就不该对外服务。

### 3. 校验器异常一律按拒绝处理

descriptor 非法 → 跳过该校验器**并留痕**（坏策略不得静默失效）；校验器不可达 / 返回错误 / 回答无法解析 → **直接判拒**（不得当作放行）。

## 备选方案

- **零校验器 → 放行（开箱可用）**：便利；但"忘装权限插件 = 全开放"是重大安全隐患，且与全系统 fail-closed 基调矛盾。否决。
- **Invoke 不强制，由调用方自行校验**：灵活；但权限会被遗忘——正是 fail-closed 要防的。且 §1.4 已把权限列为**内核提供的核心横切服务**。否决。
- **内核内置默认校验器（可被插件替换）**：开箱可用且保留可替换性；但"内核内置默认放行策略"本身就是安全footgun，且违背"一切功能皆插件"。否决——不排除未来以显式配置提供。

## 影响

- 正面：权限扩展点从"声明了没人用"变为**真正的强制点**；零校验器的语义明确且安全。
- 关注点：
  - **不测权限的测试必须显式声明放行策略**（如注册一个 allow-all 校验器），而不是偷偷跳过校验——策略摆在明面上。
  - 生产部署必须把权限插件纳入期望态（ADR-0010），否则服务不可用。
  - 目前判定基于 `(caller, scene, target)`；`applies` 预筛已实现，更细的策略（项目/角色维度）随用户系统扩展插件落地。
