# ADR-0090 插件配置与托管凭证闭环

- 状态：已采纳
- 日期：2026-07-20

## 背景

插件运行参数目前来自 `ServiceUnit.config`，但其结构是服务级扁平对象。获得 `kernel.config.get` 的插件可以按 key 读取同一服务内的任意配置，环境变量白名单也会整体传给该服务的每个插件。这不符合插件私有边界。

凭证管理已有独立页面、Vault Transit 密文服务和内核 `CredentialBroker`，但业务插件的配置页面只能要求用户先创建一个凭证，再回来手工填写名称。该流程割裂，且仅有 `name/scope` 的引用不足以表达所有者、用途和版本。

制品仓库等平台插件还需要在本地文件、S3、OCI 等存储实现间选择。若每种存储都写进仓库主插件，主插件会不断增长；若把前端布局或 Renderer 也一律拆成远端 Provider，又会增加无必要的进程与网络开销。

## 决策

### 1. 统一插件配置契约

签名插件清单可声明 `configuration`：

- `scope`：`service / tenant / user`；
- `applyMode`：`restart / hot`；
- `schema`：只描述非敏感值的闭合 JSON Schema，根必须为 `object` 且 `additionalProperties=false`；
- `managedCredentials`：声明由该插件配置页采集、但由平台托管的秘密字段，只包含 `id/title/description/purpose/required`。

远端 `$ref` 被禁止，避免清单校验或表单渲染发生网络取数、漂移与供应链注入。

### 2. 服务配置使用隔离信封

`ServiceUnit.config` 只允许三个顶层区域：

```json
{
  "plugins": {
    "cn.example.plugin-a": { "retries": 3 }
  },
  "environment_allowlist": {
    "cn.example.plugin-a": ["EXAMPLE_ENDPOINT"]
  },
  "partition_keys": ["tenant-a"]
}
```

控制面和 Node Agent 都校验 key 必须是本单元已安装插件。Backend Host 根据已认证 `caller.id` 投影 `kernel.config.get`，插件不能在请求体中指定目标插件。环境变量同样逐插件授权，不再按 service unit 广播。

### 3. 凭证在业务插件内录入，由凭证插件托管

用户在数据库、制品仓库等业务插件自己的配置表单里直接填写密码、token 或私钥，不需要先跳转到独立凭证页面创建名称。浏览器提交由两部分组成：

- `values`：符合插件 JSON Schema 的非敏感配置；
- `secrets`：只写的 `managedCredentials` 输入，永不合并进 `values`。

配置协调器把秘密直接交给凭证托管插件，返回的运行时引用至少包含：

- 不透明 `handle`；
- `scope`；
- 受签名清单约束的 `owner`（插件 ID）；
- `purpose`；
- `version`；
- 可选人类可读 `name`，但授权不得依赖它。

业务插件可以在自己的页面发起创建、替换、轮换、撤销和状态查看；它读取的是引用与元数据，不是明文或密文。独立凭证页面保留为平台安全管理员的审计、应急撤销和全局治理视图，不再是普通配置流程的前置步骤。

### 4. 使用耐久 Saga，不伪装跨服务 ACID

配置与凭证可能由不同插件、不同存储甚至不同节点持有，不能承诺单数据库事务。采用以下状态机：

1. 凭证托管器创建 staged 版本；
2. 配置存储以 CAS 保存 `Preparing` candidate；运行时仍读取旧 `Active`；
3. 激活本 candidate 的全部凭证版本；
4. candidate 切换为 `Active`；
5. 任一步失败则 candidate 标记 `Failed`，未激活凭证 abort；已激活但不可达的版本由 candidate ID 驱动 GC。

因此运行插件永远看不到“配置已生效、凭证尚未生效”的半状态。明文可变缓冲区在 stage 返回后立即清零。

### 5. Provider 只用于重型后端能力

普通参数使用统一配置契约。S3、OCI、文件存储等有独立依赖、连接、探测、迁移与故障语义的能力使用 Provider 插件；制品仓库只选择一个受平台策略允许的 Provider。切换 Provider 必须经过 probe、数据迁移/校验和事务式切换，不能作为普通用户即时偏好。

前端 Shell 模板、Renderer 和主题仍作为已签名插件内部目录，由用户即时选择并按需加载；不因形式统一而拆成远端 Provider。

### 6. 自举根凭证是明确例外

凭证插件自身启动所需的 Vault 工作负载 token 不能再依赖凭证插件，否则形成启动环。它必须由 systemd credential、只读秘密挂载或等价的部署层机制注入，且不得进入 DesiredState、日志或 Portal。

## 备选方案

- **全部配置继续放服务级扁平 map**：实现最少，但同服务插件可横向读取，拒绝。
- **用户先在凭证中心创建，再手工填写 CredentialRef**：安全边界尚可，但产品流程割裂、容易引用错误，拒绝作为默认流程。
- **插件自己保存或加密秘密**：会产生多套密钥、轮换和审计实现，拒绝。
- **给所有插件返回凭证明文**：扩大泄漏面，违反最小权限，拒绝。
- **配置与凭证宣称为跨服务 ACID 事务**：在分布式插件模型中无法诚实保证，拒绝；采用可恢复 Saga。
- **所有差异都拆成 Provider 插件**：会把轻量 UI 偏好变成进程和网络开销，拒绝。

## 影响

- 同一 Backend 服务中的插件配置和环境授权真正隔离。
- 插件可拥有完整配置体验，但秘密仍由统一凭证能力托管。
- CredentialRef 从“可猜名字”演进为受所有者、用途和版本约束的不透明句柄。
- 配置管理需要 candidate 状态、GC、审计和幂等实现，复杂度高于简单 CRUD。
- 旧扁平 `ServiceUnit.config` 不再接受；项目仍在开发阶段，不提供兼容迁移层。
- 第一阶段已实现清单契约、配置信封、内核投影和 Saga 领域强制点；Portal/BFF 配置协调器、凭证托管持久适配器和制品存储 Provider 属于后续增量实现。
