# ADR-0017 版本定义与兼容性机制

- 状态：已采纳
- 日期：2026-07-15
- 关联：[ADR-0003 运行时热装](ADR-0003-插件装载模型.md)、[ADR-0010 插件服务与部署编排](ADR-0010-插件服务与部署编排.md)、[ADR-0014 四内核结构](ADR-0014-四内核结构.md)、[ADR-0015 命名规范](ADR-0015-内核与贡献面命名规范.md)、[ADR-0016 单仓与布局](ADR-0016-单仓与代码目录布局.md)

## 背景

系统里有多种独立演进的东西需要版本：四内核、插件、宿主协议、契约（proto）、能力/贡献、清单 schema、期望态、系统整体。它们的兼容关系必须**可协商、可校验、fail-closed**，否则热装（ADR-0003）与灰度会失控。

## 决策

### 1. 各对象的版本方案与单一真源

| 对象 | 方案 | 单一真源 | 协商/校验点 |
|---|---|---|---|
| **内核**（四套，各自独立版本） | SemVer `MAJOR.MINOR.PATCH` | `core/kernels/<name>/VERSION`，构建经 ldflags 注入 | 插件激活时按 `engines` 校验 |
| **插件** | SemVer | `vastplan.plugin.json#version` | 制品仓库按版本存；`dependencies` 解析 |
| **宿主协议** | **单调整数**（非 SemVer） | `core/shared/go/protocol`（唯一常量集） | 握手取交集，无交集拒绝 |
| **契约（proto）** | **包级** `v1`/`v2` | `contracts/proto/<domain>/vN/` | 只增不改；破坏性→`vN+1`，双版本并存 |
| **能力/贡献** | `id@MAJOR` | 清单 `contributes[].id` | `CallTarget.version` 指定，新旧并存灰度 |
| **清单 schema** | 单调整数 | schema 文件 | 宿主按 schema 校验清单 |
| **期望态** | `version`(schema) + `revision`(单调) | DesiredState 文档 | reconcile 对齐 revision；回滚=回退 revision |
| **系统整体** | **CalVer Release Train**（如 `2026.1.0`） | `engineering/deploy/trains/<calver>.yaml` | 钉一组内核+插件版本 = 可复现系统版本 |

> **四内核各自独立版本，不强制同版**；需要"一组可复现组合"时由 Release Train 钉住（借鉴 testa train）。

### 2. SemVer 在本项目的精确含义

**内核**：
- `MAJOR`：破坏扩展点契约 / 协议不再兼容 / 清单 schema 破坏性变更。
- `MINOR`：新增扩展点或能力，向后兼容。
- `PATCH`：修复，不改契约。

**插件**：
- `MAJOR`：破坏其**对外能力契约**（capability descriptor / paramsSchema / 语义）。
- `MINOR`：新增子命令或贡献，向后兼容。
- `PATCH`：修复。

> 判据统一为"**是否破坏别人依赖的契约**"，不看内部实现改动大小。

### 3. 协议版本用整数、不用 SemVer

宿主协议只需回答"能不能通话"，握手**取交集**即可；SemVer 的 MINOR/PATCH 语义对它无意义。协议版本集在**一处**定义（`core/shared/go/protocol`），宿主与 SDK 共用——**禁止两处各自声明**。

### 4. 五个 fail-closed 强制点

版本机制不是君子协定，在这五处强制：

1. **握手**：协议版本无交集 → 拒绝接入。
2. **激活**：插件 `engines.<kernel>` 与本内核版本不符 → 拒绝装载；**未声明本内核的 engines 亦拒绝**（说明它不该被装进这个内核）。
3. **注册**：扩展点未定义 / single 语义重复 id → 拒绝该贡献。
4. **调用**：`capability@version` 解析不到 → 拒绝。
5. **装配**：期望态 revision 不匹配 → 不对齐。

### 5. 版本记录在仓库，不在记忆/技能

本机制属**项目工件**：决策记于本 ADR，强制点固化在代码，活的约定见 `guides/`。不放进 AI 记忆或跨项目 skill——单一真相源在仓库（ADR-0016 精神）。

## 备选方案

- **全局统一版本号（四内核+插件同版）**：省心但强耦合，任一组件发版拖动全体；与"四内核各自贴合环境独立演进"（ADR-0014）矛盾。否决——改由 Release Train 表达"可复现组合"。
- **协议也用 SemVer**：MINOR/PATCH 对"能否通话"无意义，徒增歧义。否决。
- **契约用字段级版本**：复杂且易错。否决——包级 vN + 字段只增不改更简单可靠。

## 影响

- 正面：各组件独立演进；兼容性在五处 fail-closed 强制而非靠约定；Release Train 提供可复现系统版本与回滚锚。
- 关注点：
  - `engines` 校验需要 SemVer 范围求解（引入 constraint 库）。
  - Release Train 的组装与 bundle（气隙）留待编排落地（ADR-0010 预留）。
  - 契约 `v1→v2` 并存期的双版本维护成本，需明确弃用节奏。
