# ADR-0144 组合服务代际热升级与插件动态调试

- 状态：已采纳，P1 已实施
- 日期：2026-07-24
- 关联：[ADR-0024](ADR-0024-单节点自动装配与回滚语义.md)、[ADR-0044](ADR-0044-全局依赖编排与本地自治启动管理.md)、[ADR-0045](ADR-0045-插件实例化策略与服务集群化边界.md)、[ADR-0089](ADR-0089-Runtime-Provider与共享Host池.md)、[ADR-0097](ADR-0097-测试制品仓库与前端分级热升级.md)、[ADR-0142](ADR-0142-内核启动与业务发布完全分离.md)

## 背景

Backend 已能以完整 service unit 创建候选宿主，在插件握手、依赖、配置、迁移、贡献和能力租约全部就绪后原子切换路由，并 Drain 旧宿主。dynamic-go 已经通过新 Go Runtime Host generation 解决 Go 模块不可卸载问题；测试仓库也能把唯一 `dev.*` 制品提交给 Backend Test Release。

缺口有两类：一是 Node Worker 与 Python 子解释器的共享 Pool 过去不包含 service generation，活动代与候选代可能在兼容条件相同时共用物理 Host，候选崩溃会破坏“失败保留上一可用代”；二是 Backend 尚无依赖感知的源码监听、增量构建和语言中立调试入口，开发者只能手工打包、上传和提交 Test Release。

系统需要同时获得快速开发反馈、生产等价验证、供应链边界、低进程数量和可回滚升级。不能为开发再建立一套绕开 Node Agent、Runtime Manager 和候选状态机的加载器。

## 决策

### 1. 热升级采用完整 Service Generation，不做通用原地代码替换

`ServiceGeneration` 是一个 service unit 在某一时刻的完整、不可变运行快照，至少绑定：Deployment/Assignment revision、unit fingerprint、精确插件 ref 与摘要、配置摘要、依赖图、Runtime Provider 兼容摘要和状态迁移计划。

service unit 继续作为最小事务边界。只重新编译发生变化的插件，但激活时创建完整候选 unit，禁止逐插件原地替换形成新旧混合组合。热升级表示调用流量在候选就绪后无感切换，不承诺所有语言都能在同一个物理进程中卸载代码。

### 2. 开发和真实测试共用一个候选执行器

提供两个制品入口，但只保留一套 Generation 状态机：

- `workspace-fast`：开发控制器从工作区生成内容寻址的临时候选制品，只能进入明确标记的本地开发节点；不写 stable/testing Catalog，也不获得生产等价证明；
- `repository-verify`：生成唯一 SemVer 预发布版本，完成签名、测试仓库上传、Test Release、Deployment/Assignment 和 Node Agent 复验；作为提交前和集成验收路径。

两条路径最终都必须经过 Node Agent 的安装、配置投影、Runtime Provider、握手、依赖/readiness、候选注册、原子路由切换、Drain 和失败保活。仓库仍不取得部署权限，内核启动仍不构建、发布或选择新插件。

### 3. 所有托管 Runtime Provider 都按 Service Generation 隔离候选

Node Worker、Python 子解释器、dynamic-go 及未来 Java/.NET/WASM 托管 Provider 的 Pool key 都必须包含 service generation。活动代和候选代在重叠窗口内不得共享物理 Host；同一 generation 内，只有服务 scope、Provider、隔离等级、发布者信任域、平台、ABI 和执行要求全部兼容的逻辑插件才可共享。

候选提交后释放旧代 lease，最后一个逻辑执行单元退出时回收旧 Host。`dedicated` 仍由精确插件、发布者和节点默认策略决定；generation 隔离不能被配置关闭。缺少 generation 的托管启动请求 fail-closed。

native/隔离进程天然形成候选进程边界；dynamic-go 继续禁止在 Backend 内 `plugin.Open`。Go 模块不可卸载不再成为服务热升级限制，因为每次变化都进入新 Go Host generation。

### 4. 开发动态调试由开发控制器和 Provider Debug Adapter 协作

后续增加开发态 `Plugin Dev Controller`，由 `platform-dev` CLI 和可选开发 Profile 使用，不成为内核启动职责。它必须：

1. 从 `vastplan.plugin.json` 自动发现 Backend 入口和语言，不维护插件名白名单；
2. 计算源码、Manifest、依赖锁和本地 SDK 的内容摘要，只构建变化插件及受影响本地依赖；
3. 对连续保存做 debounce，取消尚未提交的旧构建，使用单调本地 generation 防止过期候选覆盖新候选；
4. 构建失败、候选失败或调试器退出时保留上一可用 generation；
5. 让开发者显式选择 `workspace-fast` 或 `repository-verify`，不得把临时开发制品晋级为生产证据。

Runtime Provider 通过稳定 Debug Adapter 描述启动参数、附加端点和就绪门闩。首批适配为 Go/Delve、Node Inspector 和 Python/debugpy；其他语言由对应 Provider 提供。运行方式与开发语言分开决策，新增插件仍按效率、资源、生态和场景在 Go、Node、Python 或其他语言中选择。

### 5. 调试候选强制专用 Host

带断点的执行会暂停事件循环、线程或解释器，因此调试候选无论生产 hosting 策略为何都必须进入临时 dedicated Host，不能冻结同池插件或活动代。候选默认不接收普通业务流量；开发者附加并放行后，才进入受限测试路由。写操作不得默认使用流量镜像。

调试端点只绑定 loopback，远端开发使用 SSH tunnel。Debug Lease 必须短期有效，并绑定开发主体、节点、service unit、plugin、generation 和 Provider；日志、状态与支持包不得包含调试密钥、环境秘密或 Credential material。生产 Profile 和生产节点拒绝 workspace 制品、开发信任根、Debug Lease 和调试端口。

### 6. 多节点升级由发布控制器编排，本地原子性仍归 Node Agent

- `per-kernel`：各节点分别执行本地 generation 切换；
- `active-active`：预取后 Canary/分批滚动，受 `maxUnavailable` 与 readiness 控制；
- `leader`：候选先完成健康、迁移和只读准备，再通过新 epoch/fencing token 交接；首版跨节点热 standby 仍是待实现能力；
- `partitioned`：逐分区转移 owner 和 fencing，不一次撤销全部旧 owner。

跨节点发布控制器只编排批次、观察和回滚，不复制 Runtime Manager、Node Agent ActualState 或能力目录。网络分区期间不声称跨节点原子事务；每个本地 unit 保持原子切换，集群以 generation/readiness/fencing 最终收敛。

### 7. 配置、状态与内核升级分流

- `applyMode=hot` 的配置继续走语言中立配置候选控制器，不重建插件制品；
- `service + restart` 配置生成新 Service Generation；
- 插件或 Runtime 兼容摘要变化生成新 Service Generation；
- 可逆状态迁移可自动回滚，不可逆迁移必须显式阻断自动回滚并进入人工审批；
- Backend 内核二进制、Runtime Provider 自身和系统级启动信任链升级仍由 systemd/节点滚动流程处理，不伪装成普通插件热加载。

## 备选方案

### 直接从工作区向 Runtime 注入源码

反馈最快，但绕过不可变制品、身份、安装复验和生产状态机；Go 不可可靠卸载，Node/Python 也会遗留模块与全局状态。只允许工作区构建器产生临时候选制品，不允许 Runtime 直接信任源码路径。

### 所有保存都经过托管测试仓库

生产等价性最好，但签名、上传、Catalog 和 Test Release 会拖慢内部开发循环。保留为 `repository-verify` 验收通道，不作为唯一循环。

### 候选与活动代继续共享语言 Host

进程更少，但候选 Host 崩溃会同时破坏活动代，违反失败保活语义。拒绝；只在同一 generation 内共享。

### 把热升级与调试全部做进内核

会把文件监听、语言构建器、IDE 协议、仓库发布和开发身份带入封板内核，也违反启动/发布分离。拒绝；内核只保留既有候选执行和 Runtime Manager 强制点。

## 影响

正面：开发快速循环和真实测试共享最终执行语义；候选故障不能通过共享 Host 摧毁活动代；插件升级不要求重建或重启内核；多语言调试可以按 Provider 扩展。

代价：升级窗口会短暂多一个候选 Host；开发控制器需要维护增量构建、取消、日志归并和 Debug Lease；有状态集群仍需逐策略完成跨节点编排。

约束：完整 unit 是切换边界，但不是完整重编译边界；未变化制品必须复用内容寻址缓存。生产热升级不等于 same-process reload，也不允许以开发便利降低第三方隔离下限。

## 实施阶段

1. P0：固定本文、权威架构和开发指南；
2. P1：所有托管 Runtime Pool 加入强制 service generation，并覆盖同代共享、跨代隔离测试；
3. P2：实现 Backend 工作区发现、依赖感知增量构建和 `workspace-fast` 候选入口；
4. P3：增加统一 Debug Adapter、短期 Debug Lease、专用候选 Host 与开发状态接口；
5. P4：把 Test Release 扩展为多节点 Canary、滚动升级和自动回滚；
6. P5：覆盖 native、Node、Python、dynamic-go、leader、partitioned、状态迁移和故障恢复 E2E。
