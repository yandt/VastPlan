# ADR-0060 五大区域仓库布局与根目录收敛

- 状态：已采纳
- 日期：2026-07-18
- 关联：[ADR-0016 单仓与代码目录布局](ADR-0016-单仓与代码目录布局.md)、[ADR-0018 测试布局与分层](ADR-0018-测试布局与分层.md)、[ADR-0042 复杂度分层与 CI 质量门禁](ADR-0042-复杂度分层与CI质量门禁.md)
- 活的目录树见《[代码目录结构](../guides/代码目录结构.md)》

## 背景

随着四内核、多语言 SDK、合同、插件、E2E 和发布工具逐步落地，原先按具体角色平铺的根目录已不能清晰表达宏观边界。新组件一多，根目录就会同步增长，导致代码、契约、扩展和工程辅助内容混杂。

## 决策

### 1. 保持 monorepo，收敛为五大区域

可演进的项目内容只进入以下五个顶层区域：

| 区域 | 职责 | 主要内容 |
|---|---|---|
| `core/` | 产品内核与共享运行时 | `kernels/`、`shared/` |
| `extensions/` | 内核之外的可扩展面 | `plugins/`、`sdk/` |
| `contracts/` | 机器可执行的契约真源 | `proto/`、`schemas/` |
| `engineering/` | 开发、测试、发布和部署辅助 | `arch/`、`e2e/`、`deploy/`、`tools/`、`internal/` |
| `docs/` | 架构、ADR、指南和插件文档 | `dev/` |

根目录只保留单仓级配置与法务文件，例如 `go.mod`、`package.json`、`pnpm-workspace.yaml`、`.npmrc`、`LICENSE` 和 CI 配置。

### 2. 用架构测试防止根目录再次扩张

`engineering/arch` 对顶层目录实施显式白名单。新增一级业务目录必须先经过架构决策，不允许为新服务、新语言或新工具继续平铺根目录。

### 3. Node.js 是前端工具链，不是 Backend 运行时

仓库根的 pnpm workspace 统一管理 Portal 内核、TypeScript SDK 和插件前端。`node_modules/` 是可重建的本地依赖入口，必须 Git 忽略，不得进入源码、插件制品或 Backend 发布包。项目级 `.npmrc` 把 pnpm 内容仓库固定到 `.vastplan/cache/node/content-store`，把虚拟依赖树固定到 `.vastplan/cache/node/virtual-store`；根 `node_modules/` 只保留标准 Node 解析与 workspace 链接。只有构建或测试前端产物时需要 Node.js/pnpm。

本地运行状态和工程缓存统一收敛到 Git 忽略的 `.vastplan/`，与 `bin/`、`node_modules/` 同属派生目录白名单；不得在根目录为单个服务、插件或包管理器另建状态目录。

### 4. Go `internal` 边界与物理布局一致

供 E2E 和工程命令共用、但不对仓外承诺兼容的 Go 包放在 `engineering/internal/`。不放入 `core/internal/`，因为 Go 的 `internal` 导入规则会禁止 `engineering/e2e` 和 `engineering/tools` 使用它。

## 备选方案

- **保持根目录平铺**：迁移成本最低，但目录数量会随能力线性增长，不能解决当前问题。
- **按语言分区**：方便单语言团队，但会拆散同一插件的四面内容，违背全栈插件内聚设计。
- **拆分多仓**：物理边界更硬，但会增加 proto、Schema、SDK 和四内核的跨仓原子变更成本。

## 影响

- 根目录变得稳定，新能力先选择区域，再选择内部模块。
- Go 导入路径、pnpm workspace、CI、脚本、部署样例与文档引用需一次性迁移。
- ADR-0016 的 monorepo、不设 per-service 目录、插件四面同处和契约单一真源决策继续有效；其顶层物理布局由本 ADR 细化并取代。
