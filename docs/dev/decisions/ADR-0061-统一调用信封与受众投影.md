# ADR-0061 统一调用信封与受众投影

- 状态：已采纳
- 日期：2026-07-18
- 关联：[ADR-0017 版本定义与兼容性机制](ADR-0017-版本定义与兼容性机制.md)、[ADR-0036 Backend 核心 SPI 边界](ADR-0036-Backend核心SPI边界.md)、[ADR-0048 发布者级插件运行策略](ADR-0048-发布者级插件运行策略.md)

## 背景

`CallContext` 已成为本地调用、进程插件、dynamic-go、NATS、gRPC 和多语言 SDK 的共同信封。拆成多套上下文会造成身份、追踪和凭证语义漂移；继续让所有消费者读取完整对象，又会扩大敏感信息传播面。现有 `metadata` 还承载了 delegation token 与 transport identity，dynamic-go 处理器也能拿到可变的原始指针，均与“不可变、最小披露”的约束不符。

## 决策

1. 保留唯一的 Wire `CallContext`。协议字段只增不改，所有传输与 SDK 共用同一 Protobuf 定义；不为本地、远端或某种语言另造上下文。
2. 内核内部以 `TrustedContext` 持有 Wire 快照和不可序列化的 `Provenance`。只有认证入口、已验证传输层和宿主派生器可以创建可信上下文；业务代码及插件不能自行提升可信度。
3. 每次跨组件或跨信任边界前执行受众投影。有效访问集为：扩展点上限 ∩ 插件签名清单申请 ∩ 发布者策略上限 ∩ 运行形态/信任边界上限 ∩ 上游委托上限。任一必需字段不在交集内则 fail-closed。
4. 业务组件优先依赖 `ScopeView`、`CallerView`、`SubjectView`、`AuthorizationView`、`TraceView`、`RequestControlView`、`GrantView` 与 `BaggageView` 等只读窄视图；视图是最小使用面，不单独充当安全边界。
5. `metadata` 仅作为非敏感、命名空间化的 baggage。`vastplan.internal.*` 与 `vastplan.transport.*` 为宿主保留前缀，不接受外部输入，也不投影给插件。delegation token 移入 Plugin-Host `InvokeRequest` 专用字段；transport identity 仅保存在宿主 provenance。
6. `tenant_id` 是范围的规范值。兼容期内 `Principal.tenant_id` 可以保留在 Wire，但入口必须校验两者一致并由宿主规范化，消费者不得自行选择其中之一。
7. 凭证只传播 `CredentialRef`/grant reference，不传播 material。引用必须继续受 audience、purpose 与生命周期约束，实际 material 仅能进入 `CredentialBroker.WithCredential` 的可信回调。
8. `call_path`、deadline 与 trace 属于宿主管理的传播/控制信息。插件可读取投影结果，但不能借回调覆盖宿主保存的可信基线。
9. 预留 `Context Handle SPI`：未来第三方隔离运行时可只收到短期 opaque handle，并通过受控查询读取已授权视图；当前 v1 不开放该查询协议。
10. SYSTEM caller 默认不能在线上自报，接收端按签名 transport identity 重建。少数宿主控制器需要稳定子身份时，必须由可信宿主构造，并同时受 transport trust 的精确 `allowedSystemCallers` 约束；该例外只保留 Caller，不保留 Principal，也不改变插件不能创建 `TrustedContext` 的规则。

## 字段所有权

| 语义组 | 当前 Wire 字段 | 写入者 | 典型读取者 | 默认传播 |
|---|---|---|---|---|
| Subject | `principal` | 认证边界 | 权限策略、审计 | 需申请并投影 |
| Scope | `tenant_id/project_id` | 认证边界/宿主 | 租户与项目隔离组件 | tenant 通常必需 |
| Caller | `caller/scene` | 宿主逐跳派生 | 权限、计量、路由 | 按扩展点上限 |
| Trace | `trace` | 可观测宿主 | 日志、追踪、事件 | 默认可投影 |
| Request control | `deadline/idempotency_key` | 调用入口/宿主 | 调度、去重 | 按需投影 |
| Grant reference | `credentials` | 凭证授权边界 | 可信执行适配器 | 默认不投影 |
| Baggage | `metadata` | 已验证入口 | 显式声明的消费者 | 前缀白名单 |
| Propagation | `call_path` | 宿主 | 路由与环保护 | 宿主管理 |
| Provenance | 不进入 Wire | 认证/传输/宿主 | 投影器、审计 | 永不序列化 |

## 备选方案

- 按传输或语言拆成多套 `CallContext`：短期类型更窄，但会引入转译、版本和安全语义分叉，否决。
- 只在 SDK 提供 getter：第三方仍可直接解析 Protobuf，不能形成安全边界，否决。
- 对所有第一方插件无条件透传完整上下文：会把发布者信任误当成业务最小权限，否决；第一方只能获得更高的策略上限，仍需清单申请。
- 立即用 handle 完全替代上下文：会增加一次远程查询和故障面，且当前第三方隔离驱动尚未开放，暂缓。

## 影响

- 正面：协议继续统一，多语言与跨服务兼容不分叉；敏感字段按受众收敛。
- 正面：可信来源、派生和投影有单一实现，可由架构门禁与单元测试验证。
- 代价：插件清单需声明 `contextAccess`，发布者策略需维护上限；缺失必需授权会在启动或调用时明确失败。
- 迁移：先保持 Wire 字段编号不变，迁移控制信息与宿主内部 API；未来若字段长期分化，再在同一 `CallEnvelope` 内新增子消息，而不是创建互不兼容的上下文。
