# ADR-0012 APP 内核运行模型：预编译 + 整体热升级

- 状态：已采纳
- 日期：2026-07-15
- 关联：[ADR-0003 运行时热装](ADR-0003-插件装载模型.md)、[ADR-0004 独立进程+协议总线](ADR-0004-插件运行形态.md)、[ADR-0009 技术栈](ADR-0009-内核技术栈选型.md)、[ADR-0010 插件服务与部署编排](ADR-0010-插件服务与部署编排.md)、[ADR-0011 组合是通用内核能力](ADR-0011-组合是通用内核能力.md)

## 背景

APP 内核（客户端，原 RC 内核）跑在**客户环境**（常在 NAT 后、可能离线/气隙、受企业安全管控），与服务端"我们自己机房"的前提根本不同。其编译型插件的实质运行模型有两种：预编译进单一二进制，或像服务端那样运行时下载插件进程。这决定客户端的分发、安全审批与运维形态。

## 决策

**APP 内核采用预编译模型**：

1. **编译型插件预编译进单一签名二进制**（per-profile）：一个客户端 App/Profile = APP 内核 + 选定编译型插件集，编译为**一个签名二进制**，编译型插件在**进程内**运行（非独立进程）。
2. **换插件集 = 整体原子热升级**：改 profile/插件集 → 插件服务发对应**预构建二进制** → 客户端 Launcher **符号链接原子切换 + 失败回滚**（借鉴 testa RC upgrader）。不做单个编译型插件的运行时热插拔。
3. **内容仍运行时下载**：`rc.script` / `rc.workflow` 等**内容型**贡献（脚本/工作流，非编译代码）照常运行时下载，由 APP 内核内置运行时（如 Python）执行——内容热加载，非代码热加载。

## 理由

- **客户安全审批**：白名单一个签名可执行文件易过审；"运行时下载并执行插件二进制"的 agent 极难过企业安全审。
- **离线/气隙**：单文件直接交付；热加载需制品库可达或额外 bundle。
- **Go 动态性弱**：Go 无稳定进程内动态加载；热加载只能 go-plugin 多子进程，在无人管客户机上开销与运维复杂度高。
- **testa 先例**：RC 是单一签名二进制、执行引擎编译进去、整体原子升级，从不运行时装插件进程。
- **仍足够动态**：多 App via config 成立（配置选 profile → 发预构建二进制 → 原子热升级）；内容型脚本/工作流仍运行时下载。

## 备选方案

- **运行时热加载（同服务端）**：最动态，但客户机跑多子进程、运行时下载可执行代码，安全审批/离线/运维皆劣。否决为默认；受管/在线客户端未来可选，另议。
- **两者都支持**：灵活但两套机制、复杂度高。暂不做。

## 影响

- 与其他决策的**分歧**（有意为之）：APP 内核**偏离** ADR-0003（单插件运行时热装）与 ADR-0004（独立进程 + 协议总线）——客户端编译型插件是**进程内、编译进二进制**、走**整体热升级**。服务端三内核之外的后端仍保持运行时热装 + 独立进程。这是"同一插件定义、不同内核不同装配"的体现。
- 正面：客户端分发/审批/离线/运维最优；升级自动原子 + 回滚。
- 代价：加/减一个**编译型**客户端插件需重构建 + 重分发整个二进制（但热升级自动，体验可接受）；不如服务端能运行时单独热插拔。
- 关注点：
  - per-profile 二进制的构建流水线（插件服务侧按 profile 预构建、签名、登记制品）。
  - 进程内插件的故障隔离弱于独立进程——编译型客户端插件质量需保证（第一方可信下可接受）。
  - 内容型（脚本/工作流）与编译型（插件）的边界与各自分发通道要清晰。
