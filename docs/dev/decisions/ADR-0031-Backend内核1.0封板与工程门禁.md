# ADR-0031 Backend 内核 1.0 封板与工程门禁

- 状态：已采纳
- 日期：2026-07-16

## 背景

Backend 微内核的注册表、协议总线、跨服务寻址、持久事件、节点装配、控制器高可用、资源调度和可信制品主链已经跑通。但“主链能运行”不等于“公共内核可以发布 1.0”：扩展点 descriptor 仍有通用对象占位，生命周期缺少完整状态与升级迁移契约，可观测与资源边界未统一，CI 也尚未证明兼容性、故障恢复和长期稳定性。

继续让应用边开发边倒逼内核修改，会把尚未冻结的内部实现变成事实公共 API。Backend 1.0 必须先定义可机器验证的封板门槛，再允许 `core/kernels/backend/VERSION` 从 `0.x` 升为 `1.0.0`。

## 决策

### 1. 封板范围

Backend Kernel 1.0 只覆盖 Backend 微内核及其共享运行基础：插件生命周期、扩展点注册与分发、插件协议总线、跨服务寻址与事件、控制面装配、核心横切契约和发布运维门禁。

以下不作为 Backend Kernel 1.0 的完成条件：Agent/Workflow/用户系统/LLM 等应用实现，Frontend/Runner/Mobile 三套内核，具体权限策略、审计后端和可观测后端插件。测试夹具可以实现最小参考插件，但不得把具体业务实现固化进内核。

### 2. 公共扩展点目录封闭

Backend 1.0 允许插件贡献且必须具有完整 descriptor Schema 的扩展点固定为：

- `tool.package`
- `agent`
- `api.route`
- `permission.checker`
- `event.sink`
- `hook`
- `runner.capability`

`kernel.service` 是宿主内部能力命名空间。插件只能在 `capabilities.kernelServices` 声明依赖并通过 HostCall 调用，不得注册 `kernel.service` contribution。未知、尚未发布或内部扩展点一律 fail-closed；不能再以“任意 JSON 对象”兼容未来扩展点。新增扩展点属于 Backend `MINOR` 版本变更，必须同时更新 Schema、目录常量、文档和契约测试。

跨扩展点引用 `capabilityRef` 固定为与 `CallTarget` 同构的对象：`{extensionPoint, capability, version?, operation?}`，不使用难以无歧义解析的拼接字符串。`when` 条件表达式不进入 Backend 1.0 descriptor；在语法、求值上下文和安全边界形成单独 ADR 前，不接受一个“可写但不会执行”的伪契约。

每条 Backend manifest contribution 必须显式声明 `service_role`，不以缺省值猜测它属于 backend/workspace/rs；`runner.capability` 只能声明为 `rs`。运行态 descriptor 位于已经确定角色的插件单元内，除 `api.route` 和 `runner.capability` 等路由必需场景外不强制重复该字段。

### 3. 生命周期和升级是显式状态机

插件单元必须以机器可观察状态表达未安装、已安装未激活、激活中、已激活、排空中、停用中、失败和已移除。所有状态转换必须幂等，且明确新调用门闩、在途调用处理、超时、崩溃和重试结论。

升级继续采用候选实例验证后原子切换。无持久业务状态的插件直接切换；需要状态迁移的插件必须显式声明迁移能力和状态格式版本。未声明迁移能力时，内核不得猜测或复制插件私有状态；迁移失败必须保留或恢复旧版本并留下可诊断实际态。

### 4. 协议和横切边界必须有资源上限

Backend 1.0 必须冻结协议/内核/插件/Schema 的兼容矩阵、稳定错误码、CallContext 传播和跨信任边界裁剪规则，并对 payload、metadata、并发、队列、deadline 和 drain 设置可配置的有界默认值。传输错误与 `CallResult.Error` 应用错误继续严格分层。

内核统一提供结构化日志、trace/metric 埋点、健康与就绪状态、诊断快照、配置读取以及 persistence/transaction SPI。具体日志存储、指标后端、权限策略、审计落库和数据库产品仍由插件或部署适配器提供。

### 5. 1.0 工程门禁

`1.0.0` 发布前必须同时满足：

1. 全量 lint、架构守护、`go test -race ./...`、真实跨进程 E2E、contracts/proto/schema 同步、漏洞和许可证门禁通过。
2. 公共协议、descriptor、清单和旧版插件夹具有向后兼容测试；破坏性变更能被 CI 阻断。
3. 协议解析、descriptor、状态转换和流式帧具备 fuzz 种子与持续门禁。
4. 插件崩溃、网络中断、控制器切换、重复/迟到消息、升级失败和存储短暂不可用有故障注入证明。
5. 核心调用、注册、路由、调度和事件链路有 benchmark 基线及回归阈值。
6. 发布候选完成至少 24 小时稳定负载 soak；goroutine、会话、文件句柄和队列无持续增长。
7. 构建可复现，制品带版本、校验、SBOM 和签名；升级、回滚、配置迁移和诊断 runbook 可执行。

版本文件只有在上述证据齐全后才能更新为 `1.0.0`，不得以“功能看起来完成”代替封板审计。

## 备选方案

- **只冻结 v0.1 契约，生产加固后补**：能更早开始应用开发，但无法给出生产内核承诺。拒绝作为最终目标；可作为本 ADR 的第一个实施检查点。
- **应用与内核并行，遇到问题再改接口**：短期最快，却会让第一批应用变成隐式兼容包袱。拒绝。
- **把默认权限、审计、Agent 和 Workflow 一并纳入内核**：交付表面更完整，但破坏微内核可替换边界。拒绝。

## 影响

- 正面：Backend 1.0 的“完成”由可审计证据定义；应用和插件团队可依赖稳定公共面；内部能力不会被误开放为插件贡献。
- 代价：1.0 前需要补齐契约、状态、诊断、故障和性能工程，发布周期长于只做功能主链。
- 约束：封板期间仍维持 `0.x`；任何新增公共面都必须同步机器 Schema、测试和文档，不能只改 Go 常量或设计文本。
