# ADR-0097 测试制品仓库与前端分级热升级

- 状态：已采纳，分阶段实施
- 日期：2026-07-20
- 关联：[ADR-0023](ADR-0023-插件Schema与可验证制品仓库.md)、[ADR-0049](ADR-0049-制品信任基座与仓库基础插件.md)、[ADR-0078](ADR-0078-Frontend事务式热替换与插件生命周期.md)、[ADR-0086](ADR-0086-单Shell插件与可切换布局模板.md)、[ADR-0087](ADR-0087-统一Render-Adapter与可切换Renderer.md)、[ADR-0091](ADR-0091-制品存储Provider供给边界.md)

## 背景

本地平台已经能够启动签名 Seed 仓库、托管制品仓库、Node Agent 候选运行单元和开发态 Portal Generation，但这些能力尚未形成一条完整的测试发布链路：Seed 仓库仍承载全部开发制品，托管仓库随单次运行消失，仓库没有目录、发布流水账或测试目标绑定，上传制品也不会形成新的 Backend Deployment 或 Portal Activation。

前端现有热替换也把不同风险的变化混在一起。功能插件源码变化会重建完整 Portal Generation；Shell 的 `standard` 与 `top-navigation` 同时打进一个模块并用 React 组件分支切换；Arco/MUI 切换则先保存偏好再刷新页面。前者不能证明页面子树不会被重新挂载，后者在候选 Renderer 损坏时可能重复启动失败。

测试仓库、发布控制和前端运行时必须分层，否则仓库会取得部署权限，浏览器会获得制品解析权，或为了追求“热加载”而破坏不可变制品、平台基线和失败回滚语义。

## 决策

### 1. Seed、托管仓库与测试发布控制器分离

1. Seed 仓库只负责空节点自举、仓库基础插件、必须的 foundation 插件、气隙 Bundle 和仓库自身最后可用版本。它不提供普通测试上传、目录搜索、latest 或用户发布功能。
2. `cn.vastplan.platform.artifacts.repository` 是普通制品的托管仓库。开发环境为它提供跨运行持久化的本地 File Volume 和稳定测试发布身份；生产环境仍由部署方配置真实存储、TLS、信任和凭证。
3. 仓库只保存不可变制品并发布内容中立的发布事实。它不得直接写 Deployment、Assignment、Portal Activation 或命令 Node Agent。
4. 独立 Test Release Controller 把精确制品绑定到预授权测试目标。它调用内核窄发布能力，Node Agent 与 Portal Edge 继续是唯一执行和物化强制点。

### 2. 测试版本与 latest 解析

1. `(pluginId, version, channel)` 继续是不可变键，测试环境不得覆盖相同版本。
2. 测试构建器从源码版本派生唯一 SemVer 预发布版本，例如 `0.4.0-dev.20260720.3.a81c33f`，并让 Manifest、Backend 握手身份和前端 RuntimeSpec 使用同一个 BuildInfo。
3. `testing` 是发布通道，不是可变 URL。仓库发布流水账为每次成功发布分配单调 revision；Test Release Controller 可以把“最近接受的 testing 发布”解析成精确 ref 与摘要，但 Node Agent 和浏览器只消费锁定结果。
4. `testing -> candidate -> stable` 采用新证明和审计记录形成新的不可变 channel 引用，不修改历史对象。

### 3. 仓库产品能力独立演进

托管仓库按阶段增加 Catalog Index、Publish Journal、搜索分页、包依赖解析、锁文件、channel 晋级、yank/deprecate/revoke、引用保护、GC、配额、SBOM/来源证明状态、离线 Bundle 和管理 UI。这些能力属于仓库领域插件，不进入内核信任基座。

Storage Provider 仍只在供给和迁移路径运行。首个在线迁移闭环使用两个本地 File Volume 验证 probe、provision、复制、摘要校验、增量追平、候选切换、观察和回退；本阶段不实现 S3/OCI Provider。

### 4. 前端采用三级运行代

1. **选择态**：主题、密度和不改变模块或页面所有权的轻量选项在当前 Generation 内原子更新。
2. **Portal Generation**：功能插件、Workbench、兼容 Shell Library、同 Renderer 家族的兼容制品升级，在后台完整装配候选 Generation；成功后提交，失败时保留活动 Generation。
3. **Host Epoch**：Renderer 家族切换、React/vendor、`ui-primitives`、`ui-contract` major、Portal Kernel 或静态宿主变化，先预验证候选，再受控刷新整个文档。偏好采用 `pending/active/lastKnownGood`，失败不得形成刷新循环。

测试环境可通过本机事件自动应用新的 Portal Activation；生产默认只在用户刷新时读取新 Activation，后续可由 Platform Profile 选择“仅刷新、提示更新、自动更新”。生产默认不得通过周期轮询增加稳定负载。

### 5. Shell Catalog 与按需 Shell Library

本 ADR 取代 ADR-0086 中“所有可选模板必须作为同一 Shell 模块内代码实现”的范围，但保留其稳定 Slot、统一 Composition、平台允许范围和用户偏好边界。

1. 唯一可信 Shell Catalog 拥有 Slot、页面/导航归并、活动路径和已签名 Shell Library 目录，不包含具体侧栏或顶部导航实现。
2. `standard`、`top-navigation` 成为独立、内容寻址、可延迟加载的 Shell Library 模块。RuntimeSpec 锁定允许模块的精确 ref 与摘要，浏览器只下载当前选择项。
3. Platform Profile 决定默认 Library、允许集合和是否允许用户选择；用户只能选择已锁定 ID，不能提交 URL、版本、摘要、CSS 或 DOM。
4. Portal Kernel 持有稳定 `PageOutlet`。Shell Library 只提供 Chrome、导航表现和受限布局计划，不能拥有页面、Workbench 或表单生命周期。兼容 Library 切换通过候选 Portal Generation 完成，成功后才保存偏好。
5. 无法消费统一 Shell Surface 的实现属于不同信息架构，只能通过新的 Platform Profile/Activation 发布，不能伪装成用户即时偏好。

### 6. 开发 HMR 与真实制品测试是两条链路

1. 开发态继续使用仅回环可用的源码监听、增量构建和 SSE。构建器按插件依赖图只重建变化插件及受影响模块；候选失败时当前页面不变。
2. 真实制品测试必须经过唯一测试版本、签名上传、Test Release、Backend Deployment 或 Portal Activation，再进入同一候选运行机制。
3. 开发 URL、未签名模块和源码覆盖不得写入托管仓库的稳定 channel，也不得被生产 RuntimeSpec 接受。

## 备选方案

- **上传后由仓库直接重启服务**：把存储和部署权限合并，无法表达目标授权、候选失败、平台插件分级和多 Portal，拒绝。
- **覆盖同一 testing 版本实现 latest**：破坏不可变 ref 和供应链审计，拒绝。
- **每种 Shell 都成为相互竞争的完整基础插件**：用户偏好会变成 Platform Profile 变更，且容易产生不同 Slot 语义，拒绝。
- **所有前端变化都尝试无刷新替换**：不同 Renderer、共享 React 身份和宿主 ABI 无法安全共存，拒绝。
- **所有前端变化都刷新页面**：实现简单，但丢失开发反馈速度和兼容插件的状态恢复能力，拒绝。

## 影响

- 正面：仓库成为可独立演进的插件包服务，同时不取得部署和信任解释权。
- 正面：测试制品能够自动进入明确的测试目标，失败时 Backend 和 Frontend 都保留上一可用运行代。
- 正面：Shell 可以由用户在线选择并按需下载，功能插件仍只依赖统一 Slot 和 UI 语义。
- 代价：UI Contract 需要升级 major；Shell、RuntimeSpec、偏好和测试发布都要增加新状态机。
- 代价：本地开发需要维护临时 Seed 身份与持久测试身份两套信任材料，并在每次运行生成组合信任快照。

## 实施顺序

1. 建立本文与《制品仓库与测试发布》权威设计；
2. 分离临时 Seed 与持久化本地测试仓库；
3. 增加 Catalog/Publish Journal 和测试发布器；
4. 增加 Test Release Controller 的 Backend 闭环；
5. 升级 UI Contract，拆分 Shell Catalog/Library 和稳定 PageOutlet；
6. 增量开发 HMR 与真实前端制品热升级；
7. 包解析/锁文件、File Provider 在线迁移、生命周期/GC 和管理 UI；
8. 最后覆盖 foundation、仓库自升级、Runner/Mobile 与未来共享存储 Provider。

## 实施记录

- 2026-07-20：完成第 2 步。开发编排器将单次运行 Seed 与跨运行托管测试 File Volume 分离；分别生成 Seed-only、testing-only 与 Node Agent 组合信任快照，两个仓库不交叉接受发布身份；Node Agent 使用本地签名 Seed 作为 bootstrap source、托管 HTTPS 仓库作为普通远端 source。测试发布私钥仅保存在开发状态根的私有目录，不注入仓库插件或 Node Agent。
- 2026-07-20：完成第 3 步的仓库侧能力。托管仓库增加可从签名制品恢复的 Catalog、逐 revision 原子 Publish Journal、分页/过滤查询与平台工具契约；本地测试上传器只允许回环开发平台、第一方 `dev.*` 预发布制品和 `testing` channel，并在成功后复查 Catalog revision。Test Release Controller 仍为下一阶段。
- 2026-07-21：完成第 4 步 Backend 闭环。Test Release Controller 作为 deployment-manager 内部独立领域模块与人工发布共享唯一 service revision 分配器，但仍通过仓库查询与内核窄服务保持边界；新增 CAS 目标绑定、精确 testing 制品复核、候选 readiness、失败后单调 revision 自动回滚、重启 fail-closed 与显式恢复回滚。修正 Composition readiness，只有 Actual/Unit applied revision 匹配当前 assignment generation 才计为就绪，新候选失败不再被旧健康进程掩盖。
- 2026-07-21：补齐 Backend 开发使用入口。`publish-test` 保持上传与部署两个事务，但可在确认精确仓库回执后，用已有 binding 或 `deployment/unit` 开发目标提交 Test Release；自动建绑定使用本地管理员会话，发布使用独立发布者会话，且整个入口继续被回环、开发模式和非生产等价状态约束。
- 2026-07-21：真实闭环验证暴露并修正 Controller 只读单次运行 Seed 的问题。Controller 现与 Node Agent 一样仅在 Seed 精确 ref 不存在时读托管测试仓库，对远端签名、摘要和包内清单再验证；本地损坏不会被远端同名对象掩盖。
- 2026-07-21：为 Frontend Test Release 补齐 Portal Edge 制品源前置。Portal Edge 仅在 Seed 精确 ref 不存在时读托管 HTTPS 仓库，对本地读取或内容验证错误直接 fail-closed；本地开发混合验证器可接受无签名 Seed，但对携带证明的 testing 制品仍强制验签。
- 2026-07-21：完成 Frontend Test Release 服务端闭环。Portal Composer 新增 tenant 隔离的 CAS 目标绑定和发布状态；Application 插件仅克隆 Application revision，Platform Profile 插件则克隆 tenant 专用 Profile 与 Binding revisions。Portal Edge 可信宿主复核精确 Catalog/Journal 回执并物化候选，再以 Activation CAS 上线；中断恢复 fail-closed。真实 `global-settings` testing 制品验证已从托管仓库 revision 2 生成 Activation 2，RuntimeSpec 同时锁定 testing ref、模块摘要与包摘要。
