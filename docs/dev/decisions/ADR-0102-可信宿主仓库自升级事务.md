# ADR-0102 可信宿主仓库自升级事务

- 状态：已采纳并实施
- 日期：2026-07-21
- 关联：[ADR-0094](ADR-0094-操作系统Guardian与独立进程故障收敛.md)、[ADR-0100](ADR-0100-制品生命周期引用保护与垃圾回收.md)、[ADR-0101](ADR-0101-离线Bootstrap-Inventory与LKG推进.md)

## 背景

托管仓库插件也必须通过托管仓库取得新版本，这形成天然自举环。仓库插件不能获得 Seed 目录、Bootstrap Inventory 或 root 权限，否则一个被攻陷的仓库实例可以把任意制品写成最后可用版本。单独为此增加常驻升级服务又会复制 Node Agent 已有的制品验证、安装、候选切换、实际态和健康判断。

## 方案比较

1. **仓库插件自行升级**：路径最短，但跨越 Seed 离线边界并形成“被升级者决定信任”的循环，拒绝。
2. **全部交给 systemd/外部 Upgrader**：适合内核二进制，却会为插件复制组合、引用和 Runtime 健康事务，拒绝作为插件升级主路径。
3. **Node Agent + 可信宿主适配器**：复用现有候选事务，只有内核宿主拥有向 Seed 镜像和原子推进 Inventory 的窄能力，采用。

控制器使用 Go。它需要直接复用 Go 制品证明类型、Node Agent 事务、文件锁、`fsync` 和 Runtime 健康事实；Node.js/Python 在此没有生态优势，反而会引入跨语言事务与新进程。该选择不改变普通插件可使用多语言 Runtime 的原则。

## 决策

显式启用 `-bootstrap-upgrade` 后，一次仓库关键栈升级按以下顺序执行：

1. Node Agent 从远端候选源取得精确制品，在固定验证点校验内容、清单与发布者证明；
2. 可信宿主适配器把当前 LKG 中同 ID 的候选及原始已验证证明幂等写入本机签名 Seed；
3. 以文件锁和 generation CAS 原子扩展 Bootstrap Inventory 的 Seed 集合，LKG 保持不变；
4. Node Agent 安装候选并通过真实 Runtime `Apply` 与健康状态完成事务式替换；
5. 先发布新的 Assignment 活动引用，确保 Managed Repository GC 已保护运行制品；
6. 可信宿主再以新的 generation 把对应 LKG 项替换为健康候选，随后发布新 Seed/LKG 快照；
7. 旧 LKG 字节继续保留在 Seed，自动事务不执行 Seed 删除。

自动路径只接受与当前 LKG **同 channel 且 SemVer 不低于当前版本**的候选。跨通道和已提交后的人工降级必须走显式运维恢复，不得伪装成普通 reconcile。升级能力只处理关键插件；内核可执行文件仍由 systemd/Guardian 的版本目录与原子切换流程管理。

## 权限与文件边界

- 仓库插件进程仍不获得 Seed、Inventory、信任私钥或宿主写权限；
- 适配器只接收内核构造的 `VerifiedArtifact`，证明和包字节不暴露给业务插件；
- Inventory 必须是规范绝对 `.json` 路径、`0600` 普通文件，父目录不可被 group/other 写入；锁文件拒绝符号链接并使用 owner-only 权限；
- 临时文件写入、文件 `fsync`、原子 rename 与目录 `fsync` 构成持久提交；generation CAS 防止同机多个 Agent 丢失更新；
- 该能力必须同时配置签名 Seed、远端候选仓库、Bootstrap Inventory 和控制面引用发布器，缺一即启动前拒绝。

## 崩溃与回滚语义

- **复制前失败**：Inventory 与 LKG 不变；
- **Seed/Inventory 已提交、Runtime 未健康**：Seed 多一个不可变候选，LKG 仍是旧版本，旧实例继续运行；
- **Runtime 已健康、LKG 未提交时进程退出**：重启后从 ActualState 恢复已安装候选，复核其存在于 Seed 后再提交 LKG；
- **Assignment 引用失败**：不得推进 LKG；
- **LKG 已提交、引用快照尚未发布时退出**：Inventory 是本地真源，下一轮按更高 generation 重发；
- **rename 已完成但调用方未观察到成功**：相同目标 Inventory 的精确重试视为成功；不同内容继续 CAS 冲突；
- 任一候选启动或健康失败都不得改写旧 LKG，也不得回退 generation。

## 当前实现

`core/kernels/backend/bootstrapupgrade` 实现控制器与安全文件 Store；Node Agent 在 prepare/health/reference/commit 各阶段接入。`platform-dev` 默认启用该事务，使日常启动持续覆盖其配置与恢复边界。生产部署仍需由安装器创建受保护 Inventory 路径并只授予 Node Agent 宿主写权。
