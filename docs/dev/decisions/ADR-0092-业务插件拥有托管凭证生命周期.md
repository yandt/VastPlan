# ADR-0092 业务插件拥有托管凭证生命周期

- 状态：已采纳
- 日期：2026-07-20

## 背景

原有页面要求管理员先在独立凭证插件中创建一个名称，再回到数据库等业务插件填写 `CredentialRef`。这暴露了底层实现概念，容易填错租户、用途或名称，也让业务资源与凭证的更新、删除无法形成一个可恢复操作。

凭证明文不能写入普通插件配置、部署组合、URL、日志或响应；凭证所有者也不能由浏览器声明。与此同时，数据库插件和凭证插件可能运行在不同 Backend 服务中，不能假装存在跨服务 ACID 事务。

## 决策

业务插件拥有其业务资源的凭证生命周期，凭证插件拥有密文和安全状态：

1. 业务插件页面直接提供只写秘密字段；用户无需创建或填写凭证名称。
2. 业务插件通过 `platform.credentials` 调用 `stageManaged`。凭证插件从宿主认证后的 `CallContext.caller` 确定 owner，只接受 purpose、resource 和秘密，不接受 payload 指定 owner。
3. 凭证插件加密并持久化 `Preparing` 候选，只返回随机 `stageId` 和不透明 `ManagedCredentialRef`。
4. 业务插件先持久化自己的 pending 变更，再调用 `activateManaged`；成功后才将业务资源切为正式版本。重启或下次调用时继续收敛 pending。
5. 替换或删除资源后，业务插件通过 `retireManaged` 退役旧句柄。候选可通过 `abortManaged` 终止。
6. 业务 API 读取只返回“已托管、版本”等状态，不向浏览器返回句柄、秘密或密文；编辑时秘密字段留空表示保留当前句柄。

`CredentialRef` 是能力句柄，不是用户维护的名称。句柄至少绑定 tenant、owner、purpose 和 version；可信宿主在使用 material 时必须再次校验这些约束。

## 备选方案

- **继续由用户预创建凭证再引用**：实现简单，但体验差、易错，无法可靠绑定业务资源生命周期，否决。
- **浏览器依次调用凭证 API 和业务 API**：浏览器崩溃或网络失败会留下半完成状态，而且可伪造 owner，否决。
- **把凭证明文写入业务插件配置后由内核提取**：秘密会进入普通配置和部署路径，否决。
- **要求所有业务插件与凭证插件同进程并使用本地事务**：破坏跨服务部署与集群能力，否决。

## 影响

- 数据库等插件的配置体验更自然，资源与凭证具有可恢复的 Saga，而非虚假的分布式事务。
- 凭证插件新增仅限已认证插件调用的托管生命周期操作；独立凭证页保留为审计、轮换和应急撤销入口。
- 状态格式升级；当前处于开发阶段，不提供旧状态迁移。
- 本决策不授权普通插件读取明文。可信宿主 material lease/解密适配器已由 [ADR-0093](ADR-0093-可信宿主加密Material-Lease.md) 落地；`CredentialRef wire v2`、多语言 SDK 和过期候选 GC 仍需单独完成。
