# ADR-0146 开发构建锁一致性与稳定制品身份账本

- 状态：已采纳并实施
- 日期：2026-07-24
- 关联：[ADR-0023](ADR-0023-插件Schema与可验证制品仓库.md)、[ADR-0098](ADR-0098-制品依赖解析锁与离线Bundle.md)、[ADR-0135](ADR-0135-插件SBOM可信摄取与外置来源证明.md)、[ADR-0142](ADR-0142-内核启动与业务发布完全分离.md)

## 背景

插件 SBOM 应记录 esbuild 实际消费的物理依赖。现有收集器却以 npm 包名作为 map key：同一 bundle 同时包含 `react-is@18/19` 或 `fast-uri@3/4` 时，Go map 遍历会随机让其中一个版本覆盖另一个，导致相同源码、锁和模块图产生不同 SBOM 与包摘要。不可变仓库最终会拒绝它，但故障出现太晚，且表面看起来像 Portal Activation 多 SHA 冲突。

开发构建的源码摘要还会有意排除 `node_modules/`，并以 `pnpm-lock.yaml` 作为 Node 依赖真源。若构建前不重新对齐工作区链接，SBOM 即使准确描述实际输入，也可能证明了一个不符合锁文件的依赖树。这是独立于同名多版本覆盖的第二条可复现性风险。

开发者也可能修改已进入 Seed 的插件源码而忘记提升版本。清理运行目录或构建缓存不能成为覆盖稳定版本的隐式方法。

## 决策

1. Node SBOM 不再按包名覆盖同名多版本，而按 npm PURL 保存每个实际 `name + version`；例如同一 bundle 中的 `fast-uri@3.1.3/4.1.0` 和 `react-is@18/19` 必须同时存在。回归测试必须反复生成同一多版本 metafile 并得到完全相同的 CycloneDX 字节。
2. `platform-dev.sh` 在任何 Node 构建前执行 `pnpm install --offline --frozen-lockfile`；`platformdev` 在生成前端产物、Node 后端和 SBOM 前再次执行同一门禁。离线 store 不完整时 fail-closed，并提示开发者显式执行联网的 `pnpm install --frozen-lockfile`。构建仍从实际 metafile 生成 SBOM，不允许用锁文件伪造“预期依赖”来掩盖实际输入。
3. `platformdev` 对本次 Seed 候选仓库中所有 `stable` 制品建立精确身份账本，普通键为 `pluginId + version + channel`，值为最终未签名包 SHA-256。同一键后续只能幂等出现相同摘要；不同摘要在装配 Activation/Deployment 前直接失败，并要求提升插件 SemVer。ADR-0051 定义的 dynamic-go 包必须与 Backend 共同构建，额外以签名 Manifest 中的 64 位构建 fingerprint 作为平台/工作区 variant；不同 fingerprint 可并存，同 fingerprint 不同字节仍拒绝。
4. 账本保存在工作区 `.vastplan/stable-package-identities.json`，使用进程锁、原子替换和 `0600` 权限。它不进入 Git，也不随 `.vastplan/dev-platform` 的 `--fresh` 或 `clean` 删除；清缓存只影响性能，不能改变稳定制品身份。
5. `testing`、`workspace` 和唯一 `dev.*` 预发布候选继续走测试发布协议，不写入 stable 账本。正式远端仓库仍以自身 Catalog、证明、审批和不可变存储为权威；本地账本只是更早的开发反馈，不是生产信任根。
6. 因真实 SBOM 改变，受影响插件均提升补丁版本；统一 Render Adapter 与其 Renderer 精确引用保持同代升级。首次迁移还对比既有 Portal Activation 与当前 Seed，一次性清理此前已产生同 ref 不同摘要的平台 UI 插件历史债务。

## 备选方案

### 只清理 `node_modules` 和开发状态

能解除当前冲突，但无法阻止下一次旧依赖链接或漏升版本；否决。

### 只从 `pnpm-lock.yaml` 生成 SBOM

摘要稳定，但当构建实际消费了另一版本时会生成虚假供应链证据；否决。锁用于约束安装，metafile 用于证明实际输入。

### 允许 stable 同版本覆盖

开发更方便，却会破坏精确引用、回滚、签名证明和缓存的一致性；否决。高频开发使用 workspace/testing 与唯一预发布版本。

## 影响

正面：依赖漂移在构建开始时被修复或明确拒绝；忘记提升 SemVer 会在 Seed 装配前失败；`--fresh` 不再掩盖稳定版本覆盖；SBOM 继续反映真实输入。

代价：普通开发启动增加两次快速的离线 frozen install 检查；首次安装或本地 store 不完整时必须先显式联网安装；任何会改变稳定包字节的源码、依赖或工具链升级都需要提升插件版本。

约束：身份账本不可手工编辑来规避版本升级；若账本损坏应从可信历史重建，而不是静默覆盖。生产发布仍必须通过远端仓库和供应链准入，不能把开发账本作为发布证明。
