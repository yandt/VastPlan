# ADR-0142：内核启动与业务发布完全分离

- 状态：已采纳
- 日期：2026-07-24
- 关联：[ADR-0044](ADR-0044-全局依赖编排与本地自治启动管理.md)、[ADR-0077](ADR-0077-Backend在线组合与可信发布边界.md)、[ADR-0078](ADR-0078-Frontend事务式热替换与插件生命周期.md)

## 背景

开发编排器曾在一次 `up` 中同时启动基础进程、发布 Platform Profile、创建 Portal Activation、发布 `hello-service`，并等待全部平台和业务插件收敛后才开放 Portal。这样把进程生命周期与业务变更事务绑定在一起：普通重启会产生或推进 revision，发布链任何一步失败都会关闭已经健康的前后端内核，也无法区分“宿主未启动”和“业务组合未发布”。

## 决策

1. `up`、`restart` 和生产服务管理器只负责启动进程、恢复持久期望态及开放最小宿主，不得创建或修改 Deployment、Platform Profile、Application Composition、Portal Profile/Application/Binding/Activation、角色策略或业务服务 revision。
2. Backend Node Agent 可以先于首次 Deployment 发布加入控制面。目标 key 不存在是正常 `WaitingForDesiredState`，只记录一次等待状态；传输错误、损坏文档和同 revision 冲突仍然 fail-closed。
3. Controller 在普通启动中只 watch 已授权 key，并从最近持久 Deployment 恢复 Assignment；只有显式 `bootstrap/apply` 才可携带组合输入并调用发布 CAS。
4. Node Portal Kernel 的最小就绪只证明 HTTP 宿主、身份入口和恢复界面可访问，不证明 Composer、制品仓库或任意业务页面已经 Ready。插件组合状态通过独立状态接口呈现。
5. 首次平台建立与主动基线更新使用显式 `bootstrap`。该命令可以发布平台 Deployment 与初始 Portal Activation，但不得顺带创建示例业务服务。示例和真实服务必须通过 Deployment Manager 页面或独立业务发布命令走草稿、提交、异人审批与发布流程。
6. 普通启动不得因为工作区源码摘要变化分配新的业务 revision，也不得重编译并发布权限策略；开发签名、临时 TLS、Seed Inventory 物化和构建缓存属于宿主运行材料，不获得控制面业务发布权。
7. 内核就绪门与插件就绪门分开。显式 `bootstrap/apply` 可以等待其发布事务收敛并返回失败；普通 `up` 不因某个平台或业务插件暂时不可用而关闭最小内核。
8. Controller 从签名 Manifest 推导出的 `strong/data` 依赖 DAG 必须写入节点 Assignment。不能只用作者填写的 `depends_on`，否则 Node Agent 会退回 unit ID 顺序并形成级联超时。
9. Node Lease、Capability Lease、ActualState、恢复产生的 Assignment generation、制品引用租约及心跳属于运行事实，不是业务发布。它们可以在启动与恢复期间更新，但不得反向生成或修改 Deployment、Activation、应用组合和审批账本。

## 命令语义

- `platform-dev.sh up`：零发布启动，恢复已有状态；空环境进入最小内核等待态。
- `platform-dev.sh restart`：零发布重启。
- `platform-dev.sh bootstrap`：显式发布或更新开发平台基础组合，然后保持内核运行。
- `publish-test` 与管理中心发布操作：业务层操作，不属于启动路径。

## 影响

正面：普通重启可预测且不改业务状态；内核故障与发布故障可以独立诊断；空控制面也能安全启动；Portal 不再因示例服务发布失败而整体退出。

代价：首次空环境只显示最小宿主或恢复入口，开发者必须明确执行一次 `bootstrap`；内核就绪不再等价于全部插件可用，状态页和运维文档必须区分两者。
