# ADR-0042 复杂度分层与 CI 质量门禁

- 状态：已采纳
- 日期：2026-07-16
- 关联：[ADR-0019 工程规范基线](ADR-0019-工程规范基线.md)、[ADR-0031 Backend 内核 1.0 封板与工程门禁](ADR-0031-Backend内核1.0封板与工程门禁.md)、[ADR-0040 Backend 生产入口与包边界](ADR-0040-Backend生产入口与包边界.md)、[ADR-0041 Go 契约类型与 CAS 模板单一真源](ADR-0041-Go契约类型与CAS模板单一真源.md)

## 背景

质量审计发现 `Reconciler.Reconcile` 圈复杂度 61，Backend `runReconcile` 圈复杂度 39。两者把参数、依赖装配、状态阶段、失败记录、运行和关闭揉在一个函数中，局部修改需要同时理解整条控制流。现有 golangci 配置不会阻止复杂度或复制代码回升，Shell 脚本也只有语法检查；安全 job 使用 `@latest`，同一提交在不同时间可能安装不同工具。

## 决策

1. Node Agent reconcile 拆成加载与冲突校验、目标筛选、逐 unit 候选装配、失败记录、旧 unit 淘汰、收敛判定等阶段函数；每个副作用前后的实际态 checkpoint 顺序保持不变。
2. Backend reconcile 命令拆成参数预检、制品源构造、NATS 控制面接入、节点租约守护、运行与关闭。根 main 仍是组合根，helper 不创建第二运行入口。
3. golangci 启用 `gocyclo`（生产函数阈值 30）与 `dupl`（token 阈值 100）。测试表格排除 gocyclo，但不排除生产代码；生成文件排除两者。
4. 架构守护增加 Backend 同级包白名单、稳定 DTO 单一结构体真源和 `engineering/tools/` 生产入口禁令。
5. CI 使用 ShellCheck `v0.11.0` 检查全部受版本控制的 Shell 脚本，并校验官方 Linux x86_64 制品 SHA-256。
6. `govulncheck` 钉为 `v1.1.4`；`go-licenses` 升级并钉为 `/v2@v2.0.1`。工具升级必须显式改版本并走评审，不使用 `@latest`。

## 备选方案

- **只给大函数加注释**：降低不了分支耦合，也不能单独测试阶段。拒绝。
- **提高复杂度阈值以让现状通过**：把已发现的问题编码成例外。拒绝。
- **gocyclo 同时约束测试函数**：大型表格测试会产生高分支但不承载生产控制流，信噪比低。拒绝。
- **依赖 ubuntu-latest 预装 ShellCheck**：版本会随 runner 镜像漂移，无法复现实验。拒绝。
- **继续使用 Go 工具 `@latest`**：供应链和行为不可复现。拒绝。

## 影响

- 正面：大函数回升、复制粘贴、未登记横向依赖、脚本缺陷和工具版本漂移都会在 CI 阻断；状态机阶段更易局部验证。
- 代价：新增复杂分支或 Backend 横向依赖需要先拆分/登记；CI 多一次约 2.6 MiB 的 ShellCheck 下载。
- 约束：复杂度数字是报警线，不是质量得分；低于 30 仍需评审职责与副作用顺序。
