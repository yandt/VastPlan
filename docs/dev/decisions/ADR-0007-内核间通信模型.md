# ADR-0007 内核间与服务间通信模型

- 状态：已采纳
- 日期：2026-07-14

## 背景

三套内核（后端/前端/客户端）与后端内核组合出的多个服务（backend/workspace/rs）之间存在多条通信链路，形态差异很大：后端服务间可能同机也可能跨网、前端在浏览器、RC 在客户端本地且常处 NAT/防火墙之后。需要一个统一而分层的通信模型，尤其要让后端服务的"灵活组合（合/拆）"不影响调用方代码。

## 决策

按"**统一消息契约 + 分类连接拓扑 + 位置透明**"设计：

1. **统一消息契约**：三条平面共用同一套不可变契约（`CallContext / CallResult / CallEvent`，见骨架文档），语义一致；差异只在**传输与连接拓扑**，不在调用心智。
2. **位置透明的后端服务寻址**：后端服务间调用经一个**内部服务寻址层**解析——合并部署时走进程内直调，拆分部署时走网络 RPC，**调用方代码不变**。这是"灵活组合"能成立的前提：组合改变的是部署拓扑，不是代码。
3. **三类连接拓扑**（按发起方与网络位置分）：
   - **后端服务 ↔ 后端服务**：对等内网，位置透明的请求/响应 + 跨服务事件（pub/sub）。
   - **WebUI ↔ 后端**：客户端→服务端，经**稳定边缘入口**路由到组合出的后端服务；含请求/响应与实时推送两个子通道；前端不感知后端组合拓扑。
   - **RS ↔ RC**：RC 在客户端本地、常在 NAT 后，由 **RC 主动回连 RS**（反向持久连接）；RS 向下推指令/任务，RC 向上回流结果/日志；带签名客户端 + 令牌的接入信任链。

## 备选方案

- **各链路各自为政、各定协议**：短期省事，长期协议爆炸、契约不一致、可观测割裂。被否决。
- **前端直连每个后端服务**（testa 现状：前端既连 backend 又连 RS）：性能上流式可直连，但让前端感知后端拓扑，与"灵活组合"冲突。改为默认经边缘入口，仅特定流式端点可选直连。
- **RS 主动连 RC**：RC 在 NAT/防火墙后不可达，不成立。必须 RC 回连。

## 影响

- 正面：调用心智统一；后端合/拆自由；前端与拓扑解耦；RC 穿透 NAT。
- 关注点：
  - 内部服务寻址层需处理服务发现、路由、超时、重试、背压，且进程内/跨网两条路径语义一致。
  - 反向连接需解决断线重连、离线缓冲、心跳、多副本 RS 下的连接亲和与会话迁移。
  - 具体传输与序列化（HTTP/gRPC/WS/MQ 等）待各自 ADR（ADR-0005 栈解耦）。
