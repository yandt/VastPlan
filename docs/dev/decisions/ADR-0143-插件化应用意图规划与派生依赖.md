# ADR-0143 插件化应用意图规划与派生依赖

- 状态：已采纳，待实施
- 日期：2026-07-24
- 关联：[ADR-0044](ADR-0044-全局依赖编排与本地自治启动管理.md)、[ADR-0057](ADR-0057-插件分级管理与双输入组合解析.md)、[ADR-0077](ADR-0077-Backend在线组合与可信发布边界.md)、[ADR-0090](ADR-0090-插件配置与托管凭证闭环.md)、[ADR-0098](ADR-0098-制品依赖解析锁与离线Bundle.md)、[ADR-0142](ADR-0142-内核启动与业务发布完全分离.md)

## 背景

Backend 在线组合当前让浏览器直接编辑接近完整的 `deployment/v2.ServiceUnit`。因此应用配置人员不仅选择应用插件和配置，还能填写 `depends_on`、`instance_policy`、`state_model`、`routing`、`logical_service` 和分片等执行字段。这把三个本应分离的职责混在一起：

1. 插件开发者在签名 Manifest 中声明包依赖、运行时 capability 依赖和故障策略；
2. 应用配置人员决定某个内核服务启用哪些应用插件、功能与配置；
3. 内核执行已经锁定的 Deployment、Assignment 和本地依赖 DAG。

服务依赖若由用户手工填写，会产生漏依赖、伪依赖、循环依赖和预览/运行漂移。若把依赖求解、动态配置表单、Provider 选择和冲突解释继续扩进内核，又会把应用管理策略固化到微内核，并重复 ADR-0098 已经放在仓库插件中的版本求解能力。

系统处于开发阶段，不需要保留现有在线草稿或前期组合输入的兼容行为；但已完成的 Deployment v2、Controller、Node Agent、能力租约和故障恢复链应保持稳定。

## 决策

### 1. 增加插件化组合规划层，不扩张内核职责

采用以下分层：

```text
Application Intent               用户可写：根应用插件、Feature、配置、受限运维参数
        │
        ▼
Deployment Manager               草稿、审批、审计、发布工作流
        │
        ▼
Composition Planner Plugin       依赖闭包、配置计划、策略推导、解释报告
        │
        ├── Artifact Repository   确定性版本求解、Artifact Lock、已验证描述
        └── Configuration Catalog 配置 Schema、CredentialRef 与生效路径
        │
        ▼
Application Composition          已编译提案，不是浏览器编辑模型
        │
        ▼
Existing Kernel Resolver         Profile 选择、来源/制品复验、摘要锁、CAS 发布
        │
        ▼
Deployment v2 → Controller → Assignment v1 → Node Agent
```

规划层由新的平台基础插件 `cn.vastplan.platform.infrastructure.composition-planner` 提供稳定 capability `platform.composition.plan`。它是确定性、无外部副作用的 `active-active + external-shared + cluster + queue` 服务，由平台管理 Profile 部署，但不进入不可插件化的 Seed Core，也不扩大仅用于仓库栈自救的 Bootstrap LKG 子集。其制品可以保存在 Seed 全集中供管理面恢复，只有托管仓库和配置 Provider 就绪后才发布能力。

Planner 故障只阻止新的草稿解析和发布，不影响已提交 Deployment、Controller、Node Agent 或运行中的数据面。内核启动不得自动调用 Planner 或产生新业务 revision。

### 2. 用户输入改为插件协议 Application Intent

`Application Intent` 是 Deployment Manager 与 Planner 之间的插件层协议，不成为 Kernel/Node Agent 的执行契约。它只允许：

- 目标部署和平台允许的 service class；
- 内核服务 ID；
- 根应用插件的精确 ref；
- 插件 Manifest 预先声明的 Feature；
- 根插件及自动引入的应用级依赖插件配置；
- 独立授权并受 Platform Profile 上限约束的副本、资源和放置参数。

它不得包含 `depends_on`、`logical_service`、`instance_policy`、`state_model`、`visibility`、`routing`、`routing_domain`、平台 Provider 实现、Foundation/Platform 插件或明文秘密。

Application Composition 保留为 Planner 生成并交给现有内核发布入口的已编译提案。在线与普通离线配置入口都提交 Intent；离线工具通过同一 Planner capability 或受控本地插件宿主生成提案。只有随签名 Seed/发布 Bundle 交付的预编译平台提案可以直接进入恢复路径。前期开发数据不迁移；旧的浏览器或普通 CLI 直编 Application Composition 路径直接删除。

### 3. 服务依赖只来自签名开发期契约

依赖继续分为两类：

- Manifest `dependencies`：同一内核服务中的包依赖和版本范围；Planner 调用仓库求解传递闭包并锁定精确版本。
- Manifest `runtime.requires`：跨插件或跨服务 capability 依赖；Planner 根据同一组合中的 provider 和 Platform Profile 的共享服务绑定生成只读服务 DAG。

应用用户不能增加、删除或重定向依赖边。Deployment v2/Assignment v1 可以继续携带 `depends_on` 作为内部执行字段，但它只能由 Planner/Controller 根据已验证 Manifest 派生。Controller 继续独立读取完整 Manifest 并二次校验，不能信任 Planner 的布尔结论或浏览器预览。

包依赖默认与消费者共置。如果某能力必须作为独立共享服务运行，开发者必须使用 `runtime.requires`，由 Platform Profile 或同一组合中的 provider 提供，不能把包依赖偷偷转换成远程服务。

### 4. 条件依赖使用显式 Feature

插件契约增加可选的组合 Feature 描述。每个 Feature 可以声明额外包依赖、runtime requirement 和配置 Schema 条件。用户只能启用已签名 Manifest 中存在的 Feature；不得使用任意 JSON Pointer、脚本或配置值表达式临时创建依赖。

Feature 开关变化必须重新执行完整解析、配置检查和摘要计算。未启用 Feature 的依赖和配置不进入发布产物。

新增字段属于共享插件契约/SDK 的演进，不增加内核调度、生命周期或运行时策略。Kernel 和 Node Agent 仍只消费解析后的既有执行字段。

### 5. 配置按管理级别分流

- 根应用插件和自动包依赖插件：应用配置人员使用其签名 Configuration Schema 配置；
- 共享 Platform Provider：只显示绑定、版本和 readiness，不在消费服务中重复配置；
- Foundation/安全插件：由 Platform Profile 固定，应用不可见或只读；
- 敏感值：继续走 CredentialRef、候选凭证窗口和 Configuration Activation，不进入 Intent、Planner 日志或解析报告；
- Runtime 派生值：由可信宿主在 Assignment/运行时注入，不能由应用覆盖。

Planner 输出 `ConfigurationPlan`，说明每个配置对象的来源、可编辑性、Schema digest、缺失字段和依赖路径。Deployment Manager 负责持久草稿与审批；Planner 不持有 Credential material，也不成为新的配置真相源。

### 6. 解析结果必须可解释且可复现

Planner 返回：

- 已规范化的 Application Composition 提案；
- 仓库生成的精确 Artifact Lock 及 Catalog revision；
- 根插件、自动依赖、Feature 和 capability Provider 绑定；
- 派生服务 DAG；
- Configuration Plan；
- 警告、冲突路径和确定性 `planDigest`。

相同 Intent、Platform Profile、Catalog revision 和配置摘要必须得到相同结果。提交审批时绑定 Intent、Plan、Profile、Artifact Lock 与配置摘要；任一输入变化都使草稿 stale 并要求重新解析、重新审批。

### 7. Planner 是建议生成者，内核仍是最终强制点

Planner 不获得 NATS KV、Deployment CAS、SSH、Assignment、仓库签名私钥、信任根或 Credential material。它只能通过窄插件 capability 读取已验证目录/解析结果和配置描述，并输出提案。

现有 `kernel.deployment.preview/publish` 保持最终权限边界：重新选择活动 Platform Profile，复验精确制品与来源，检查摘要后发布 Deployment。Controller 继续验证 package dependency、runtime requirement、策略兼容和依赖环；Node Agent 继续复核制品并执行本地 DAG。

因此本决策不引入第二套调度器、第二套实际态或第二套依赖执行器。

### 8. 运行策略由 Manifest 派生，不由用户选择

Planner 从每个非辅助贡献的签名 runtime policy 生成 service unit 的实例策略、状态模型、可见性与路由。一个内核服务中的非辅助插件策略必须兼容；不兼容时要求用户拆分为两个内核服务，Planner 不自动改变故障域或偷偷拆分服务。

副本、资源和放置属于运维意图，可在 Platform Profile 允许范围内由独立权限修改；它们不改变插件固有实例策略。例如应用用户不能把 `leader` 插件改成 `active-active`。

## 语言与运行方式

Planner 首个实现使用 Go，运行方式与语言分开选择：

- Go 可直接复用现有 SemVer、规范 JSON、配置契约 DTO 和插件 SDK，资源占用低；
- Node.js 适合 Portal 动态表单，但不应承担后端确定性求解和发布摘要；
- Python 适合后续策略分析 Provider，但引入解释器不能改善当前确定性图算法或信任边界；
- 第一方签名、构建指纹匹配且部署方允许时，可进入既有 generation-scoped dynamic-go Runtime Host；否则使用现有 native trusted-process 路径。不得为 Planner 增加新的内核驱动或专用执行通道。

Portal/Workbench 部分继续使用 TypeScript，只负责 Application Intent 编辑和 Resolution Report 呈现。

## 修订关系

- 修订 ADR-0057：在线用户可写资源从接近完整的 Application Composition 收窄为插件层 Application Intent；Application Composition 成为 Planner 的已编译输出。
- 修订 ADR-0077 第 2 条：应用管理员不再手写依赖和实例策略，只能配置根应用插件、Feature、插件配置以及受限运维参数。
- 保持 ADR-0044：运行时依赖、readiness、Controller/Node Agent 本地自治和故障语义不变。
- 保持 ADR-0098：版本依赖求解继续属于制品仓库插件；Planner 编排该能力，不复制求解算法。
- 保持 ADR-0142：启动只恢复已提交状态，Planner 只在显式草稿/发布流程中运行。

## 备选方案

### 直接扩展内核 Composition Resolver

强制点集中，但会把 Feature、动态配置、仓库选择和用户工作流带入内核，扩大封板后内核；拒绝。

### 全部加入 Deployment Manager

文件最少，但节点引导、审批、测试发布、依赖求解、配置计划和页面编排会继续堆入一个巨型插件，也无法被 Runner/Mobile 复用；拒绝。

### 仅在 Portal 隐藏 `depends_on`

改动最小，但 API 仍可提交内部字段，后端也不能自动引入依赖或解释冲突；拒绝。

### 独立外部微服务

便于独立扩容，但引入新的部署、身份、网络和可用性负担。Planner 当前是低频、确定性控制路径，先作为平台插件运行；未来可在不改变 capability 的前提下切换到独立进程或远程 Provider。

## 影响

正面：用户界面只表达业务意图；依赖在开发阶段声明并自动计算；内核继续只验证和执行；Planner 可独立升级、替换、扩容并复用于其他内核。

代价：需要新增插件层 Intent/Plan 契约、Planner 插件、Repository 描述接口和 Deployment Manager 解析状态；发布前多一次低频控制面调用。

约束：Planner 不是授权或信任根；远端平台 Provider 不会因为某个应用声明依赖而自动安装；不兼容运行策略不会自动拆分服务；P0 只固定文档，代码实施按后续阶段推进。

## 实施阶段

1. P0：固定 ADR、架构单一真相源、控制台边界和索引；
2. P1：定义插件层 Application Intent、Resolution Report、Configuration Plan 与 Feature 契约；
3. P2：实现 Composition Planner 插件及仓库/配置 Provider 适配；
4. P3：Deployment Manager 接入解析快照、stale 判定和摘要审批；
5. P4：Portal 删除内部执行字段，增加根插件、Feature、依赖配置和只读解析图；
6. P5：用真实插件覆盖传递依赖、版本冲突、Feature、Provider 歧义、配置缺失、撤销和 Node Agent 收敛 E2E。
