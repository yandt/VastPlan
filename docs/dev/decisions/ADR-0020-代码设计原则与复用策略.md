# ADR-0020 代码设计原则与复用策略

- 状态：已采纳
- 日期：2026-07-15
- 关联：[ADR-0016 单仓与代码目录布局](ADR-0016-单仓与代码目录布局.md)、[ADR-0019 工程规范基线](ADR-0019-工程规范基线.md)
- 活的细则见《[工程规范](../guides/工程规范.md)》§五

## 背景

此前的约束覆盖了**结构**（四内核、依赖方向、目录布局）与**流程**（提交、依赖准入、CI），但缺**代码设计**层面的要求：高内聚低耦合怎么落地、共享代码放哪、什么时候该抽象。

同时需要正面回答一个常见诉求："建一个公共工具类/基础工具类来提升复用"。

## 关键证据：建了公共目录 ≠ 有复用

testa **有** `shared/`（3,248 LOC），结果：

| 指标 | testa 实际 |
|---|---|
| 共享代码全库被 import | **86 处** |
| **共享利用率** | **< 2%**（工业标准 15-30%） |
| 重复代码 | **17,000-20,000 LOC**，占业务码 **13-15%** |
| 并存的权限命名 | **5 套** |
| 未继承基类的 Repository | **60 个**，各自实现 ~80% 雷同 CRUD |

**目录建了、没人用、重复照旧疯长。** 根因：无职责的"公共工具"包是个**垃圾抽屉**——没人知道什么该放进去，于是谁都不放；复制粘贴永远比"先想清楚放哪"省事。它恰恰**违背高内聚**（一个包塞进互不相关的东西 = 内聚为零）。

testa 补救方案（应用骨架层）真正奏效的不是"工具包"，而是**有职责的机制**：Registry + 基类 + 统一契约 + 场景化三元组。

## 决策

### 1. 高内聚判据：一个包 = 一个职责，且能一句话说清

🤖 **每个包必须有 doc comment，写明它的单一职责**（架构守护强制）。

这不是文档洁癖，而是**内聚的试金石**：如果一句话说不清这个包干什么，或者句子里出现"以及/还有/各种"，说明它已经不内聚，该拆。

### 2. 低耦合手段：依赖契约，不依赖实现

- **跨组件只经契约通信**：`CallContext/CallTarget/CallResult/CallEvent` + 能力名寻址。这已是架构基座。
- **消费方定义接口**：Go 惯例"accept interfaces, return structs"。接口定义在**用它的地方**，不在实现方——否则消费方被迫依赖实现包。
- 🤖 **依赖方向铁律**（已由架构守护强制）：内核↛插件、插件间不互依、共享↛组件。
- **插件间只能经 capability 寻址**——这就是低耦合在本项目的具体形态：加一个插件，零人需要改别人的代码。

### 3. 复用策略：禁垃圾抽屉，只建有职责的包

🤖 **禁止 `util` / `utils` / `common` / `helper` / `helpers` / `misc` / `base` / `core` 等垃圾抽屉包名**（架构守护强制）。

替代做法：**按能力命名，一包一职责**。当前 `shared/go/` 全部符合——`registry`（扩展点注册表）、`protocol`（协议常量真源）、`protocolbus`（宿主↔插件总线）、`contract`（契约）。没有一个叫 util。

**共享代码准入（shared/ 的 charter）**——同时满足才可进：

1. **≥2 个组件真的在用**（不是"将来可能用"）；
2. **能一句话说清职责**，且不与既有包重叠；
3. **无反向依赖**（不 import kernels/plugins）。

不满足就**先留在原地**。

### 4. 抽象时机：三次法则；错误的抽象比重复更贵

- **第 1 次**：直接写。
- **第 2 次**：复制，并**标注**另一处（此时还看不清共性）。
- **第 3 次**：才提取——三个样本才够看出**真正的**共同点。

理由：过早抽象会锁定**错误的**共性，后来者为了复用它而扭曲自己的代码，代价远大于当初的重复。**重复比错误的抽象便宜**——重复是局部的、可见的、易删的；错误的抽象是全局的、隐蔽的、难拆的。

> 例外：**契约/协议/版本**这类"单一真源"由 ADR-0016/0017 强制，不适用三次法则——它们从第 1 次就必须唯一。

## 备选方案

- **建 `common`/`util` 基础工具包**（原始诉求）：直觉上提升复用；但 testa 实证其失败（<2% 利用率、17-20K 重复照旧），且违背高内聚、与本项目 §4.4 及 Go 惯例冲突。**否决**——复用靠有职责的包，不靠垃圾抽屉。
- **强制包扇出/圈复杂度阈值**：看似客观；但阈值武断、噪声大，且低耦合真正靠的是依赖方向与契约。暂不做（可作后续待决）。
- **只写原则不强制**：正是本轮之前的状态——ADR-0019 的教训是君子协定连规则作者都拦不住。**否决**：内聚（包注释）与垃圾抽屉（包名）两条必须机器强制。

## 影响

- 正面：内聚有了可执行判据（包注释）；垃圾抽屉在**长出来之前**就被禁掉（当前尚未出现，正是立规则的最佳时机）；复用有了准入门槛而非口号；抽象时机有了明确标准。
- 关注点：
  - 三次法则与"单一真源"是**两条不同规则**，勿混用——契约类从第 1 次就唯一。
  - 包扇出/复杂度阈值暂不强制，若日后耦合失控再议。
  - 包注释判定须用 AST（`ast.File.Doc`，即"紧贴 package 子句上方的注释块"），**不可用 grep `// Package X`**——`package main` 的惯例是直接描述该命令，grep 会产生假阳性（本轮已踩过）。
