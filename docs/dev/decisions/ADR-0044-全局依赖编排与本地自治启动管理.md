# ADR-0044 全局依赖编排与本地自治启动管理

- 状态：已采纳
- 日期：2026-07-17
- 关联：[ADR-0010 插件服务与部署编排](ADR-0010-插件服务与部署编排.md)、[ADR-0011 组合是通用内核能力](ADR-0011-组合是通用内核能力.md)、[ADR-0024 单节点自动装配与回滚语义](ADR-0024-单节点自动装配与回滚语义.md)、[ADR-0025 NATS 控制面、能力寻址与多节点调度](ADR-0025-NATS控制面寻址与多节点调度.md)、[ADR-0028 控制器选主与 Drain 收敛](ADR-0028-控制器选主与Drain收敛.md)、[ADR-0032 Backend 插件生命周期与实际态 v2](ADR-0032-Backend插件生命周期实际态v2.md)、[ADR-0035 Backend 可观测与健康契约](ADR-0035-Backend可观测与健康契约.md)

## 背景

后端服务可能由多个平台级插件组成。例如：服务 A 负责全局设置，服务 B 负责凭证管理，服务 C 负责数据库能力，业务插件再依赖 C。仅按进程启动顺序或 unit ID 排序，无法表达“进程已启动但能力尚未就绪”，也无法定义服务崩溃后依赖方应停止、降级还是继续使用缓存。

现有系统已经具备全局 Deployment、节点 assignment、Node Agent reconcile、候选实例原子切换、能力租约和进程退出后重试。但这些能力尚未形成跨服务的运行时依赖模型。

需要同时满足两类稳定性：

1. 全局控制面能够校验依赖图、统一发布计划并审计组合状态。
2. 控制面或远端服务短暂故障时，已有健康实例仍可本地运行和恢复，不能把控制面变成所有数据面的同步单点。

## 决策

### 1. 采用“全局编排 + 本地自治”的两级模型

全局控制面负责声明式计划，不直接管理每个插件进程：

- 读取全局部署组合并解析跨服务依赖；
- 检查缺失能力、版本冲突、作用域不匹配和环状依赖；
- 生成部署/升级/回滚顺序并观察各节点实际态；
- 汇总服务组合的 `Ready`、`Degraded`、`Blocked` 和 `Failed` 状态；
- 负责审计、灰度、扩缩容和全局期望态版本。

内核启动管理是内核能力，由 Node Agent/Runtime 执行本地 DAG、插件重启、能力租约和故障隔离。启动管理器不是普通插件，不能依赖自身启动；未来若需要替换编排策略，也只能作为由内核加载和监管的可信策略模块。

### 2. 全局依赖与本地依赖分层

- **同一内核/同一 service unit**：使用本地硬依赖 DAG；候选实例只有在本地依赖全部就绪后才可原子激活。
- **不同 Backend 服务或不同内核**：使用跨服务 capability readiness，不建立全局同步进程顺序。
- **控制面与数据面**：控制面负责期望态和观察，数据面在短暂失去控制面时继续执行已有健康计划。

跨服务不使用全局两阶段提交。全局只对“计划和组合状态”提供一致性，本地对“进程和能力激活”提供原子性。

### 3. 运行时依赖必须与制品依赖分开

插件清单中的包依赖只描述制品解析和兼容性；运行时依赖描述服务启动和调用时需要的能力，不能把两者混为同一字段。

后续契约采用以下概念（字段名可在实现时调整）：

```json
{
  "runtime": {
    "provides": [
      {"capability": "platform.settings", "version": "1.0"}
    ],
    "requires": [
      {
        "capability": "platform.credentials",
        "version": "^1.0",
        "scope": "remote",
        "kind": "strong",
        "ready": "readiness",
        "failurePolicy": "degrade"
      }
    ]
  }
}
```

依赖至少区分：

- `strong`：未就绪时禁止激活；运行中丢失时进入策略指定状态；
- `soft`：允许启动，但必须报告降级；
- `lazy`：首次调用时检查，不阻塞启动；
- `data`：等待 schema/迁移/数据版本就绪后才激活。

依赖还必须标注作用域（`same-node`、`same-kernel`、`remote`）和就绪条件，不能只声明“进程存在”。

### 4. 就绪以能力租约为准

服务只有在健康检查、配置加载、凭证访问、连接池和必要迁移完成后，才能发布 capability readiness lease。租约至少包含：

- capability、版本、服务角色、unit/instance、节点；
- generation/fencing token；
- `ready`、`degraded` 或 `draining` 状态；
- TTL、续租时间和诊断原因。

能力租约失效或 generation 不匹配时，寻址层必须撤销旧实例，禁止调用继续流入已失效实例。

### 5. A/B/C 平台服务的依赖关系

推荐定义如下：

```text
A platform.settings
├──> B platform.credentials
└──> C platform.database
       B ────────────────┘

业务插件 ──strong──> platform.database
```

- A 不依赖 C，避免基础配置服务反向依赖数据库服务；
- B 强依赖 A；
- C 强依赖 A，并在需要动态凭证时强依赖 B；
- 业务插件只依赖其真正使用的能力，不重复声明 B 的传递依赖。

### 6. 故障与恢复语义

服务状态采用封闭状态集合：`Pending`、`Starting`、`Ready`、`Degraded`、`DependencyLost`、`Failed`、`Draining`、`Stopped`。

- A 失效：撤销 `platform.settings` 租约；B/C 进入 `DependencyLost` 或按策略降级，不自动重启整个内核；
- B 失效：C 可继续使用现有连接，但暂停新凭证获取和连接刷新；
- C 失效：撤销 `platform.database` 租约，业务插件快速失败、熔断、重试或进入只读模式；
- 插件进程崩溃：本地 Runtime 立即报告事件，Node Agent 以指数退避和抖动重新对账；
- 控制面失效：已有健康实例继续运行，本地只暂停需要新全局计划的操作；
- 内核进程崩溃：由外部进程监管器拉起，Node Agent 根据持久化期望态和真实进程状态重新收敛。

恢复必须具备冷却时间、重试上限、generation fencing、旧租约撤销和依赖恢复顺序，禁止无条件级联重启。

### 7. 全局组合状态不是全局事务

控制面可以报告“某组合尚未就绪”，但不能假设 A、B、C 在网络分区期间能够完成跨进程原子启动。远端依赖采用最终一致的 readiness 观察；本地 unit 仍保持候选启动、原子切换、Drain 和回滚语义。

控制面本身必须按 ADR-0028 运行多副本、短租约选主、fencing 和可恢复状态；它是可替换和可恢复的协调者，不是业务调用热路径的强制中心。

### 8. 安全边界

- 依赖声明和就绪状态不是授权本身，真正调用仍须经过能力寻址和现有身份/权限校验；
- B 不向普通插件暴露明文凭证，C 使用受限凭证句柄或宿主托管回调；
- readiness lease 不承载秘密；
- 控制面、Node Agent 和服务实例使用现有 NATS mTLS、NKey、Subject ACL 和租约身份。

## 备选方案

### 方案 A：固定顺序 + 重试

实现成本最低，但不能表达 readiness、远端依赖和级联故障策略；长期会把错误变成隐式超时，拒绝作为目标架构。

### 方案 B：声明式依赖 + 本地 DAG + 跨服务能力就绪

故障隔离最好，控制面与数据面边界清晰，但全局审计和升级编排需要额外状态汇总。适合作为基础能力，已被本 ADR 吸收。

### 方案 C：全局同步启动工作流

全局顺序和一致性最强，但控制面容易成为恢复单点，网络分区时需要全局补偿事务，且会把远端服务故障放大成全局阻塞。拒绝纯同步实现。

### 方案 C+：全局计划/观测 + 本地自治执行

保留 C 的依赖图校验、审计、灰度和组合状态，同时保留 B 的本地恢复、租约和故障隔离。作为本 ADR 的最终选择。

## 影响

- 正面：可统一描述 A/B/C 及未来数据库、对象存储、消息服务的依赖；控制面故障不会自动扩大为全系统停机；服务崩溃有明确的状态、租约和恢复边界。
- 代价：需要扩展 Deployment/Plugin runtime schema、能力目录、实际态和健康契约；需要新增环依赖、版本冲突、租约过期和网络分区测试。
- 约束：不能承诺跨服务全局原子启动；调用方必须遵守超时、熔断、幂等和降级契约。

## 后续落地顺序

1. 固定 runtime `provides/requires` schema 和依赖类型语义；
2. 在 Node Agent 内实现本地 DAG、readiness gate 和依赖状态机；
3. 扩展 capability lease 的 readiness、TTL 和 fencing；
4. 控制面增加跨服务依赖校验、组合状态和升级计划；
5. 增加 A/B/C 崩溃、控制面分区、租约过期、循环依赖和恢复顺序 E2E；
6. 在代表性平台插件完成后，再进行长时间 soak 和级联恢复测试。
