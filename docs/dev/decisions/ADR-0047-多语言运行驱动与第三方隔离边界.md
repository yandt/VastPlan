# ADR-0047 多语言运行驱动与第三方隔离边界

- 状态：已采纳
- 日期：2026-07-17
- 关联：[ADR-0004 插件运行形态](ADR-0004-插件运行形态.md)、[ADR-0009 内核技术栈选型](ADR-0009-内核技术栈选型.md)、[ADR-0043 插件启动授权](ADR-0043-插件启动授权与签名时间边界.md)

## 背景

Backend 已通过独立进程和 gRPC/Protobuf 获得语言中立的协议边界，但启动路径仍直接执行 `entry.backend`，只有 Go SDK。若直接把 Python、OCI、WASM 等分支写入宿主，内核会逐步绑定具体语言；若把启动管理本身交给普通插件，又会形成“启动插件之前先启动启动插件”的循环依赖。

当前插件均为第一方可信代码，只要求故障隔离；未来第三方插件必须拥有更强的文件、网络、进程和凭证隔离。第三方代码不能通过自己签署的清单宣称“我是第一方”或主动降低隔离等级。

## 决策

1. 启动管理继续属于 Backend 内核能力；内核新增 `PluginRuntimeDriver` SPI，把“选择运行时”与“协议宿主和生命周期”解耦。
2. v1 Manifest 向后兼容新增 `execution.backend`：声明驱动、无 shell 参数、运行时要求、目标平台、协议特性和最低隔离等级。缺少该字段的旧插件等价于 `native + trusted-process`。
3. 驱动名使用可扩展标识，不封闭为 Go 枚举。当前提供 `native` 与 `python`；未来 `oci`、`wasm`、操作系统沙箱或厂商驱动注册到同一 SPI。
4. 隔离采用双重下限：签名清单只能提出最低要求；节点侧 `ExecutionPolicy` 根据发布者再次提高要求，取二者较强者。驱动隔离能力不足时 fail-closed，绝不静默回退。
5. 生产 Node Agent 默认把 `vastplan` 视为第一方，并要求其他发布者至少 `process-sandbox`。在隔离驱动落地前，第三方制品即使签名有效也不能启动。
6. 第一方 native/python 插件继续使用受限环境变量、一次性启动令牌、签名清单贡献授权、资源上限和独立进程故障隔离；stdout/stderr 由宿主接管进入统一日志。
7. 语言 SDK 位于 `sdk/<language>`，只能依赖公开 Schema/Proto，不得反向依赖 Backend 内核实现。跨语言一致性由同一套协议测试覆盖。

## 备选方案

- 在宿主中硬编码 Python/OCI/WASM 分支：初期简单但持续污染内核，否决。
- 把启动管理实现为普通插件：产生引导循环且失去 fail-closed 强制点，否决。
- 立即强制所有插件容器化：隔离最好，但会显著增加当前第一方开发和本地调试成本，保留为第三方开放阶段。
- 只提供 Python 启动脚本、不定义驱动 SPI：最快但下一种语言会再次修改主干，否决。

## 影响

- 正面：内核只理解稳定的驱动 SPI 与隔离等级，不理解语言生态细节；多语言和未来沙箱可以独立演进。
- 正面：第三方扩展空间已进入强制策略路径，当前没有隔离驱动时默认拒绝未知发布者，不留下“以后再补”的绕过路径。
- 代价：Manifest、安装实际态和运行时增加执行契约；SDK 与 CI 需维护跨语言生成物和一致性测试。
- 边界：`trusted-process` 不是恶意代码安全边界。第一方安全指供应链验证、协议授权、资源与故障隔离；第三方开放仍以至少一个真正的隔离驱动和外部安全评估为前置条件。
