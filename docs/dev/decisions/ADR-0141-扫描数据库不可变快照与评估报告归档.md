# ADR-0141：扫描数据库不可变快照与评估报告归档

- 状态：已采纳，分阶段实施
- 日期：2026-07-24
- 关联：[ADR-0091](ADR-0091-制品存储Provider供给边界.md)、[ADR-0138](ADR-0138-插件安全准入与可追加复扫状态.md)、[ADR-0139](ADR-0139-安全评估Provider与持续复扫控制器.md)

## 背景

Assessment Provider 已在每次扫描前复核 `db/metadata.json + db/trivy.db` 的内容摘要，并固定使用 Trivy `--offline-scan --skip-db-update`，但尚无可信任务负责准备和原子切换数据库快照。原始扫描报告虽然按摘要写入 Provider 本地目录，但 active-active 副本之间不可共享，Repository 接受签名 Admission/Status 时也无法证明其引用的报告仍真实存在。

Trivy 官方把漏洞数据库作为 OCI 制品分发，支持 `--download-db-only` 或手工放置 `metadata.json + trivy.db`。官方签名文档明确覆盖 Trivy 二进制和容器镜像，不能由此推断企业环境取得的每个数据库快照都自动满足 VastPlan 信任策略。

## 决策

### 1. 下载来源与运行快照分离

数据库 Source 只负责把候选字节送入受控 staging；首个 Source 是本地文件，后续可增加内部 OCI、S3 或离线 Bundle。Snapshot Materializer 不信任路径或来源名称，只信任已验证服务配置中的精确 SHA-256 revision。

`cn.vastplan.platform.artifacts.assessment.database.file` 是每内核本地基础插件。它在 ACTIVATE 阶段读取私有 staging 的 `db/metadata.json + db/trivy.db`，拒绝符号链接、特殊文件、空文件和大小超限，复制到同一文件系统的 candidate，逐文件同步后计算规范 revision；只有与配置钉死值相同才原子发布到 `snapshots/<revision>`。已有同 revision 目录必须重新复核，不能覆盖。

该插件选择 Go 原生独立进程：文件边界、摘要、原子 rename 和低资源启动检查是 Go 的优势；Node/Python 没有生态收益，Rust 会增加首版工程成本。插件是 per-kernel 本地物化缓存，不持有集群权威，不需要自主 HostCall。未来 OCI Source 可调用 Trivy、ORAS 或企业镜像工具，但不得把联网更新带入 Assessment Provider。

### 2. Provider 只读精确 revision

Assessment Provider 配置使用精确 `trivySnapshotDirectory`，目录必须符合 `.../snapshots/<databaseRevision>`。Provider 启动和每次扫描都复核内容摘要；扫描仍固定 `--offline-scan --skip-db-update`。数据库换代由平台发布新的已验证配置/服务 generation 完成，旧 generation 在途扫描继续使用旧不可变目录，不能跟随可变 `current` 软链接。

### 3. 报告归档是共享数据面，不走插件 RPC

原始报告通过 ADR-0091 的 Storage Provider 供给一个独立共享 volume，并把私有 mount 投影给 Provider 与 Repository。Provider 使用内容寻址 `reports/<sha256>.json` 落盘：先写私有临时普通文件、同步内容、以不覆盖方式原子发布并同步目录；相同摘要幂等，不同内容冲突 fail-closed。

Repository 在接受带报告摘要的 AdmissionRecord 或 StatusRecord 前，从同一归档重新打开普通文件、限制大小、流式计算 SHA-256。报告缺失、部分写入、符号链接、超限或摘要漂移时拒绝记录。Repository 不在 Portal Catalog 中暴露 mount path，报告读取以后续一次性审计 Ticket 提供；首版先封闭写入与存在性验证。

### 4. 生命周期与故障语义

- candidate 复制、进程崩溃或摘要不符不会改变任何已发布 snapshot；残留临时目录可安全回收。
- 已发布 snapshot 和报告只按内容摘要寻址，不原地修改；删除必须基于配置 generation、Admission/Status 链与保留期做可达性 GC。
- 数据库不可用时 Provider fail-closed，Controller 退避但不伪造新记录；报告归档不可用时 Provider 不返回签名记录，Repository 也拒绝孤立引用。
- 当前仍不做 soak；用确定性大/小文件、符号链接、崩溃残留、摘要冲突和双写验证覆盖边界。

## 否决方案

- **Provider 在线自更新数据库**：扫描结果不可复现，并把外网与注册表凭证带进持有签名能力的进程，否决。
- **用可变 current 软链接切换**：检查与扫描之间可能换代，且软链接扩大路径逃逸面，否决。
- **报告通过 HostCall/RPC 上传**：64MiB 报告进入控制面消息和内存，否决。
- **仅相信签名记录中的报告摘要**：无法区分报告已归档与永远不存在，审计证据不闭环，否决。

## 实施顺序

1. 落地通用私有内容寻址报告归档和 Repository 二次存在性验证。
2. 落地 File Database Snapshot 插件、精确目录契约及确定性原子物化测试。
3. 更新开发环境 volume/config，完成 Provider 与 Repository 进程级验收。
4. Portal 配置阶段再增加数据库 Source/Profile、快照状态和一次性报告读取 Ticket；OCI/S3 Source 独立实现，不改变本 ADR。

## 实施记录

- 2026-07-24：通用 `artifactreport` 私有内容寻址归档落地；Provider 0.3.0 使用共享归档并绑定精确 `snapshots/<revision>`，Repository 0.31.0 在接受 Admission/Status 前二次复核报告。离线 Bundle 同时去重携带准入与复扫报告、只追加状态链；导入端先恢复被引用报告，再由目标仓库按原顺序重新验签和验链。
- 2026-07-24：File Database Snapshot 0.1.0 落地 per-kernel 原子物化、大小/权限/符号链接/摘要边界；真实子进程 E2E 证明插件只在物化完成后 Ready。当前按计划未实现 OCI/S3 Source、报告读取 Ticket 或 soak。
