# ADR-0100 制品生命周期、引用保护与垃圾回收

- 状态：已采纳，分阶段实施
- 日期：2026-07-21
- 关联：[ADR-0098](ADR-0098-制品依赖解析锁与离线Bundle.md)、[ADR-0099](ADR-0099-File-Volume在线迁移与可回滚切换.md)

## 背景

不可变发布解决了版本被覆盖的问题，但没有解决版本失效、磁盘容量和安全删除。仓库若直接扫描 Deployment、Portal、Runner、Mobile 的私有状态，会跨越内核边界；各控制器自行删除，又无法得到完整引用集合。GC 在引用模型完成前启动会误删活动、回滚或离线制品。

## 决策

1. 仓库 leader 独占生命周期事件、引用快照和 GC 状态机。消费者通过稳定契约主动提交自己拥有的精确引用，不允许仓库解析其他内核的内部状态。
2. 生命周期是追加式 Catalog 事件，并进入同一单调 revision：`active`、`deprecated`、`yanked`、`revoked`。更新必须携带 `expectedRevision` CAS 和审计原因；`revoked` 不可逆。
3. `deprecated` 仍可解析，生成的 ArtifactLock 必须携带原因和可选替代约束；`yanked/revoked` 不参加新的解析和交付。历史 Catalog revision 按当时生命周期求解，不能用当前状态改写旧快照。
4. 引用发布者使用宿主验证的 owner kind/owner ID，提交完整快照而非增量计数。首批 owner 为 Active Deployment、Assignment、Portal Activation、ArtifactLock、回滚历史、Seed、LKG；快照有单调 generation、租约和摘要，过期不直接等价于可删除。
5. GC 固定为 `plan -> quarantine -> sweep`。plan 是无副作用 dry-run；quarantine 再次复核全量引用与 Catalog revision 后原子移出活动命名空间；sweep 只删除超过宽限期且再次无引用的隔离对象。任何不确定引用、仓库迁移、镜像偏差或引用源不健康都 fail-closed。
6. 配额在发布前按 namespace/publisher/channel 做 admission，容量统计只读取已验证 Catalog 和 quarantine 元数据。配额拒绝不能触发 GC，也不能把 Seed/LKG 当作普通可回收对象。
7. 控制面使用现有仓库 leader 的 Go 实现，不新增进程；管理 Workbench 使用 TypeScript。Go 适合复用签名制品 DTO、CAS、原子文件和锁，Node/Python 不为本状态机另建一份持久事务。

## 实施顺序

1. 生命周期事件、历史快照解析、CAS 管理 API；
2. 引用快照契约与各控制器发布适配器；
3. dry-run、quarantine、宽限期 sweep 与故障恢复；
4. 发布配额和容量统计；
5. 目录、引用、生命周期、GC 和配额 Workbench。

- 2026-07-21：第二阶段完成规范化引用快照 DTO、owner kind、generation、TTL、内容摘要、发布者绑定、持久聚合、过期 fail-closed、管理查询和迁移观察期双卷镜像。各控制器发布适配器尚在接入，GC 因必需 owner 未齐仍保持关闭。
- 2026-07-21：Deployment Manager 完成首个控制器发布适配器。服务切换前先提交旧活动与新候选的并集快照，切换后按“回滚历史、活动精确引用”顺序收敛；持久化 `referencePending` outbox 在仓库恢复和控制器重启后幂等重试。Backend Test Release 在候选激活前提交精确 `artifact-lock`，引用仓库不可用时 fail-closed。Portal Activation、Assignment、Seed/LKG 发布者仍待接入，GC 继续关闭。
- 2026-07-21：Portal Composer/Edge 完成第二个控制器发布适配器。Edge 从已验签且已物化的前端包生成精确引用，Composer 以两阶段 generation 保护 Activation 与回滚历史，并用持久 outbox 重试；Frontend Test Release 在目录验证和候选激活前提交精确 `artifact-lock`。跨进程写入只经过认证 Composer、tenant 与 `portal/*` owner 的窄内核桥。Assignment、Seed/LKG 发布者仍待接入，GC 继续关闭。
- 2026-07-21：Node Agent 完成 Assignment 发布适配器。只有实际 Runtime 与 ActualState 收敛后才发布精确安装引用；ActualState v3 在仓库调用前持久化 generation/outbox，使用 120 秒租约和 40 秒心跳，重试保持同 generation/同 digest。优雅停机提交空快照；异常失联或仓库不可用会使引用源过期并阻止 GC。仓库与访问策略只允许 addressing 认证的 `node-agent/<nodeID>` 写入自身 `assignment/*/<nodeID>`。Seed/LKG 发布者仍待接入，GC 继续关闭。
- 2026-07-21：按 ADR-0101 完成 Seed/LKG 发布者第一阶段。部署工具生成 root-owned Bootstrap Inventory；Backend 内核逐项从 Seed 重新验签后，以稳定 `bootstrap-inventory/<repositoryId>` 身份发布非过期 Seed/LKG 完整快照。Seed 服务仍不依赖控制面。Deployment、Portal、Assignment、Seed/LKG 必需引用源已经齐备；Runner/Mobile owner 尚未接入，通用 GC 仍保持关闭。
- 2026-07-21：第三阶段完成仓库 leader 内的 GC 状态机。只有显式 `yanked/revoked` 且无精确引用的制品进入无副作用 plan；quarantine 复核候选、Seed/LKG、全部租约源与迁移状态后原子移出活动命名空间，sweep 在至少 24 小时后再次复核。持久化 `quarantining/quarantined/sweeping/swept` 支持崩溃恢复，Catalog 仅容忍 retirement 精确记录的物理缺失；已隔离 ref 禁止重发、重新激活或新增引用。plan 身份不绑定无内容变化的心跳 revision，但候选变化会使旧 plan 失效。独立 `platform.artifacts.gc` 保护隔离/清扫，Runner/Mobile 引用未接入前生产策略仍不得把其制品纳入候选。
