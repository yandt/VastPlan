# ADR-0100 制品生命周期、引用保护与垃圾回收

- 状态：已采纳，分阶段实施
- 日期：2026-07-21
- 关联：[ADR-0098](ADR-0098-制品依赖解析锁与离线Bundle.md)、[ADR-0099](ADR-0099-File-Volume在线迁移与可回滚切换.md)

## 背景

不可变发布解决了版本被覆盖的问题，但没有解决版本失效、磁盘容量和安全删除。仓库若直接扫描 Deployment、Portal、Runner、Mobile 的私有状态，会跨越内核边界；各控制器自行删除，又无法得到完整引用集合。GC 在引用模型完成前启动会误删活动、回滚或离线制品。

## 决策

1. 仓库 leader 独占生命周期事件、引用快照和 GC 状态机。消费者通过稳定契约主动提交自己拥有的精确引用，不允许仓库解析其他内核的内部状态。
2. 生命周期是追加式 Catalog 事件，并进入同一单调 revision：`active`、`deprecated`、`yanked`、`revoked`。更新必须携带 `expectedRevision` CAS 和审计原因；`revoked` 不可逆。
3. `deprecated` 仍可解析，生成的 ArtifactLock 必须携带原因和可选替代约束；`yanked/revoked` 不参加新的解析和交付。历史 Catalog revision 按当时生命周期求解，不能用当前状态改写旧快照。
4. 引用发布者使用宿主验证的 owner kind/owner ID，提交完整快照而非增量计数。首批 owner 为 Active Deployment、Assignment、Portal Activation、ArtifactLock、回滚历史、Seed、LKG；快照有单调 generation、租约和摘要，过期不直接等价于可删除。
5. GC 固定为 `plan -> quarantine -> sweep`。plan 是无副作用 dry-run；quarantine 再次复核全量引用与 Catalog revision 后原子移出活动命名空间；sweep 只删除超过宽限期且再次无引用的隔离对象。任何不确定引用、仓库迁移、镜像偏差或引用源不健康都 fail-closed。
6. 配额在发布前按 namespace/publisher/channel 做 admission，容量统计只读取已验证 Catalog 和 quarantine 元数据。配额拒绝不能触发 GC，也不能把 Seed/LKG 当作普通可回收对象。
7. 控制面使用现有仓库 leader 的 Go 实现，不新增进程；管理 Workbench 使用 TypeScript。Go 适合复用签名制品 DTO、CAS、原子文件和锁，Node/Python 不为本状态机另建一份持久事务。

## 实施顺序

1. 生命周期事件、历史快照解析、CAS 管理 API；
2. 引用快照契约与各控制器发布适配器；
3. dry-run、quarantine、宽限期 sweep 与故障恢复；
4. 发布配额和容量统计；
5. 目录、引用、生命周期、GC 和配额 Workbench。

## 当前实施记录

- 2026-07-21：第一阶段完成 Catalog 追加式生命周期事件、历史 revision 求解、deprecated 锁提示、yanked/revoked 新解析与交付拒绝、CAS 管理 BFF 和独立 `platform.artifacts.lifecycle` 权限。引用保护与 GC 尚未启用，因此当前不会删除制品字节。
