# ADR-0145 本地测试与正式远端制品仓库双协议

- 状态：已采纳，P0 契约已实施
- 日期：2026-07-24
- 关联：[ADR-0049](ADR-0049-制品信任基座与仓库基础插件.md)、[ADR-0091](ADR-0091-制品存储Provider供给边界.md)、[ADR-0097](ADR-0097-测试制品仓库与前端分级热升级.md)、[ADR-0144](ADR-0144-组合服务代际热升级与插件动态调试.md)

## 背景

当前开发环境把正式 Managed Repository 的 HTTPS 协议部署在回环地址，并配以本地 File Provider。它虽然能完成测试发布，却把“仓库协议”和“存储位置”混为一谈：开发快速循环只能继续走远端上传、Catalog 和远端令牌语义，或者再增加一个临时 Workspace Repository。后者会形成第三种通用仓库形态，扩大 Controller、Node Agent、信任和垃圾回收分支。

系统只需要两种普通制品仓库形态：开发时的本地测试仓库，以及正式运行时的远端仓库。Seed 只负责空节点启动和 LKG，是内核启动信任源，不提供普通包管理、测试发布或 workspace 生命周期，因此不计为第三种通用仓库协议。

## 决策

### 1. 统一制品语义，固定两种精确协议

新增协议契约：

- `artifact.repository.local-test.v1`：本地开发测试；
- `artifact.repository.remote.v1`：正式远端仓库。

两者共享不可变 `pluginId + version + channel`、精确读取、发布者证明、内容摘要、Catalog 快照和候选回执语义。Controller、Node Agent、Portal 与发布控制器继续只消费精确 ref 和已验证 Envelope，不因协议不同复制制品身份或信任判断。

协议选择必须来自受信 Repository Profile，并按 exact protocol 匹配 Adapter；未知协议、major 不同或 Adapter 缺失均 fail-closed，不允许从 remote 自动降级 local-test，也不允许反向回退。

### 2. local-test 是唯一开发仓库协议

local-test 只允许 `developmentOnly=true`，首版使用规范绝对 Unix Socket，避免监听可被局域网访问的 TCP 端口。它至少支持 `testing`，可选支持 `workspace`：

- `workspace` 使用内容寻址候选、短期 lease、TTL 和数量上限；覆盖或同 ref 不同摘要仍禁止；
- `testing` 保存唯一 `dev.*` 预发布版本和可重复验证回执；
- workspace 到 testing 是重新签发的受控提交，不原地改 channel；
- 本地身份、socket、File Volume 和签名私钥不得进入 Git，也不能被生产 Profile 接受。

Backend/Frontend 保存后的快速候选和动态调试都发布到同一个 local-test 仓库的 workspace 命名空间，不再创建每次运行的第三仓库。提交前 `repository-verify` 进入同仓库 testing 通道，再由既有 Test Release 状态机验证和回滚。

### 3. remote 是唯一正式仓库协议

remote 只接受 HTTPS endpoint，凭证由可信宿主单独注入，不允许 URL 内嵌身份。它支持 testing/candidate/stable、Publish Journal、解析锁、晋级、retire、供应链证据、引用保护和 GC；明确拒绝 workspace channel、开发信任根、Debug Lease 和自动生成私钥。

remote 的对象存储仍由 Storage Provider 供给，可以是 File、S3、OCI 或其他实现；Provider 不改变 remote 协议。开发 local-test 也可使用 File Provider，但这不使它变成 remote。

### 4. Test Release 使用带协议身份的仓库回执

候选回执后续增加 `repositoryId + protocol + exact ref + SHA-256 + repositoryRevision/lease`。Test Release Controller 先从活动 Repository Profile 选择 Adapter，再复核回执：

- local-test/workspace 仅允许本地开发、预授权测试目标和短期候选；
- local-test/testing 用于开发环境完整验证；
- remote/testing/candidate 用于正式测试或晋级流程；
- 任何协议、仓库 ID、Profile digest、channel 或摘要漂移都使请求 stale。

仓库仍不写 Deployment、Assignment 或 Portal Activation。启动仍只恢复已提交状态，不自动构建或发布候选。

### 5. 协议实现继续采用 Go，构建器按插件语言选择

Repository Adapter 首版使用 Go：当前不可变对象、签名复验、文件原子提交、HTTP 服务和 Catalog/Journal 都在 Go；新增 Unix Socket Adapter 不值得引入 Node/Python 状态进程。远端搜索、扫描和 UI 仍可由其他语言 Provider 扩展。

插件开发构建器继续按插件自身场景在 Go、Node、Python 或其他语言中选择；构建语言不决定仓库协议。

## 备选方案

### 在本机运行 remote HTTPS 协议

部署简单，但开发快速循环继续承担远端令牌、上传和治理开销，也无法从协议层拒绝生产 workspace；否决为最终形态，保留为迁移期兼容。

### 增加独立 Workspace Repository

能快速落地，却形成 Seed、local testing、workspace、remote 多套来源和生命周期，Node Agent 与 Controller 分支持续增加；否决。workspace 是 local-test 的短期命名空间，不是仓库形态。

### 两套仓库完全独立建模

会复制 ArtifactRef、Envelope、摘要、签名和 Catalog 语义，最终产生开发能跑、生产不兼容；否决。两协议只在传输、生命周期和治理能力上不同。

## 影响

正面：仓库形态固定为两种；开发热升级不会污染正式协议；远端生产边界可明确拒绝 workspace；Node Agent 保持统一验证器和候选执行器。

代价：需要增加协议 Profile、Adapter Registry、Unix Socket local-test 服务、带协议身份的发布回执，并把当前回环 HTTPS 开发仓库迁移到 local-test。

约束：Seed 不能承接普通开发上传；File/S3 只是 Storage Provider，不代表仓库协议；local-test 不能因部署在服务器而升级成生产仓库；remote 不能因部署在 localhost 而被当作 local-test。

## 实施阶段

1. P0：固定双协议 ADR、Profile 契约、操作矩阵和架构单一真相源；
2. P1：实现 Repository Adapter Registry 与 `local-test.v1` Unix Socket Server/Client；
3. P2：平台开发环境迁移到 local-test Profile，保留 remote 回环路径的短期兼容诊断；
4. P3：Test Release 回执绑定 repository ID/protocol/Profile digest，并接入 workspace TTL；
5. P4：Backend/Frontend 保存后构建、候选取消、热切换和 Debug Lease；
6. P5：remote 生产 Profile、TLS/Vault/企业 CA、迁移和故障矩阵验收。
