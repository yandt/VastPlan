# ADR-0006 内核三分区与后端服务灵活组合

- 状态：已采纳
- 日期：2026-07-14

## 背景

早期骨架稿把系统描述为"一套内核内嵌到 Backend/WebUI/RS/RC 四个平级宿主"。实际运行环境差异很大——后端诸服务同处服务端、前端在浏览器、RC 在客户端本地——用单一内核套四层并不贴合，且掩盖了"后端多个服务其实同源、应可灵活组合"的诉求。

## 决策

内核分为**三套**，各自独立，但**吃同一套统一插件定义**：

1. **后端内核（Backend Kernel）**：一套内核，可**灵活组合**出多个后端服务——backend、workspace、rs 等。每个后端服务 = 后端内核 + 选定的后端插件集 + 部署编排；服务之间可合并部署、也可拆分独立部署。RS、workspace **不是独立内核**，而是后端内核的不同组合形态。
2. **前端内核（WebUI Kernel）**：单独一套，运行于浏览器，承载前端插件。
3. **客户端内核（RC Kernel）**：单独一套，运行于客户端本地，承载 RC 插件。

**统一插件定义**：三套内核共用同一份清单规范。一个插件可为纯 WebUI、纯 backend、或同时含 WebUI + backend + RC（任意组合）；每套内核只装配清单中与自己相关的贡献部分。

## 备选方案

- **单一内核套四层**（原稿）：忽略了三类运行环境的本质差异，也没表达后端服务的可组合性。被否决。
- **每个后端服务各自独立内核**：backend/workspace/rs 各造一套，重复且无法灵活组合。被否决——它们应共享后端内核。

## 影响

- 正面：
  - 后端部署拓扑灵活——按需把 backend/workspace/rs 合并或拆分，同源不分叉。
  - 三类环境各用贴合自身的内核实现，互不牵制（呼应 ADR-0005 栈解耦：三套内核实现栈可异构）。
  - 插件"一次定义、三处可用"，跨面特性仍是逻辑上的一个插件。
- 关注点：
  - **统一插件定义**必须同时容纳三套内核的贡献语义，且后端部分需能标注"面向哪个后端服务角色"。
  - 跨面插件的多个部分（前端↔后端↔RC）之间如何协作/通信，需在协议与 API 层明确。
  - 后端"灵活组合"需要一套**组合/部署编排**机制（选插件集、定服务边界、连接关系）。
