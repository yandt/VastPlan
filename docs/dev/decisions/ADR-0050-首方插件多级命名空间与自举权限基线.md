# ADR-0050 首方插件多级命名空间与自举权限基线

- 状态：已采纳
- 日期：2026-07-17
- 关联：[ADR-0021 权限判定强制点](ADR-0021-权限判定强制点.md)、[ADR-0044 全局依赖编排与本地自治启动管理](ADR-0044-全局依赖编排与本地自治启动管理.md)、[ADR-0047 多语言运行驱动与第三方隔离边界](ADR-0047-多语言运行驱动与第三方隔离边界.md)

## 背景

首个正式基础插件需要在 `platform.settings` 尚未启动时提供权限依据。既有示例 ID 如 `com.vastplan.demo-permission` 只能标识发布组织，不能稳定表达插件所处层级和功能域；如果再把命名空间字符串直接当成授权依据，第三方还可能伪造首方前缀。

## 决策

1. 新增首方生产插件统一使用 `com.vastplan.<layer>.<category...>.<component>`：
   - `layer` 固定为 `foundation | platform | product | integration | example`；
   - `category` 至少一级，表示功能域，如 `security | configuration | data | artifacts | observability`，并可继续细分子域；
   - `component` 是最后一段的具体组件名。例如 `com.vastplan.platform.data.relational.connection-manager` 可解析为 `platform / data.relational / connection-manager`。
2. `com.vastplan.*` 与 `publisher=vastplan` 双向绑定。命名空间提供分类信息，但只有已经通过 Manifest、制品和发布者信任链验证的身份才能使用该分类参与策略。
3. 历史 demo/hello 插件保持兼容，不作为新命名样例；架构门禁禁止新增未分类的首方生产插件。
4. 首个正式插件命名为 `com.vastplan.foundation.security.bootstrap-policy`。它是 `per-kernel + local-ephemeral + local + direct` 的无状态本地插件，不依赖 settings、credentials、database 或持久化。
5. bootstrap-policy 对 `platform.settings` 提供两级校验：
   - 最高优先级写保护：只有 system 或直接登录的管理员用户可以执行非只读操作；插件不能继承管理员 principal 获得写权限，未知操作按写操作处理；
   - 最低优先级基线：system/管理员可访问，已验证的 `foundation`/`platform` 首方插件可执行 `get/list/changesSince`，其他身份默认拒绝。
6. 更高优先级的正式动态策略可以收紧或扩展读取授权，但不能依赖 bootstrap-policy 自身从 settings 取配置；静态自举规则随签名制品发布。

## 备选方案

- 只增加一级，如 `com.vastplan.security-*`：无法区分自举基础层、共享平台层和产品层，否决。
- 增加独立 category 字段但保持扁平 ID：查询时需要同时读取清单，不能仅凭命名空间路由和审计，保留为未来补充元数据而非主分类。
- bootstrap-policy 从 `kernel.config.get` 动态读取规则：权限校验期间回调宿主会重新触发权限链并形成递归风险，否决。
- 允许插件继承管理员调用上下文写设置：会形成 confused-deputy 风险，否决。

## 影响

- 正面：插件 ID 可直接提取层级、功能域和组件，部署、审计、市场分类与默认策略可共享同一解析器。
- 正面：settings 启动前仍有最小权限基线，且第三方不能抢占首方分类前缀。
- 代价：历史示例 ID 暂时存在兼容例外；后续重做示例时迁入 `com.vastplan.example.*`。
- 约束：命名空间是可信身份的分类属性，不替代制品签名、发布者策略和调用授权。
