# ADR-0076：Portal Edge 分布式快照交付

- 状态：已接受
- 日期：2026-07-19
- 扩展：ADR-0073 的单 Edge 私有交付目录

## 背景

Portal Edge 可以有多个副本并运行在不同服务器。发布节点已经物化的前端快照若只留在本机，其他 Edge 会缺少活动 revision；若每个浏览器请求重新从插件制品仓库下载、验签和解包，又会恢复原有的高延迟和供应链攻击面。普通 Backend 服务节点不需要浏览器模块，也不应同步这些数据。

## 决策

1. 前端交付分为可信中央 origin 与每个 Portal Edge 私有 cache。origin 保存发布边界生成的不可变 Runtime 快照和 SHA-256 内容对象，可由受控共享卷或后续对象存储适配器提供。
2. 发布节点先完成制品验证、入口提取、对象写入和快照提交，再激活 revision。对象先写，快照最后原子提交；不完整 revision 不可见。
3. 每个 Portal Edge 后台轮询其平台 Catalog 中租户的活动 revision，从 origin 校验 PortalSpec、入口摘要和包摘要后预取到本机。普通 Backend 服务节点不启动预取器。
4. 新扩容 Edge 或错过后台轮询时，首个 runtime 请求允许从可信 origin 冷填充本机 cache；该路径只复制已物化的快照/内容对象，不访问插件仓库、不重新验签、不解包，也不执行前端代码。后续 runtime 与模块请求只读本地 cache。
5. 模块仍只能通过活动 revision 的 Runtime 快照授权；浏览器不能把 origin 或 cache 当成任意内容仓。恢复 revision 使用相同机制按服务端选择的历史锁冷填充。
6. origin 不替代签名制品仓库和信任根。对象存储、CDN、清理、容量配额和跨区域复制都位于交付端口之后，不改变 Portal 组合契约。

## 否决方案

- 所有 Edge 各自重新物化插件包：重复验签解包，发布延迟和攻击面随副本数增长。
- 浏览器直接访问制品仓库或 origin：绕过活动 revision 授权、会话和同源 CSP。
- 把前端快照同步到所有 Backend 节点：浪费存储并扩大可读范围。
- 发布等待所有 Edge 一致确认：节点扩缩容会阻塞配置发布；后台预取加受限冷填充可以在最终一致调度下保持可用。

## 结果

多 Edge 部署共享一次可信物化成本，稳态请求只访问本机内容寻址对象；新节点也不会出现活动 Portal 无法打开的窗口。代价是生产环境必须为 origin 提供高可用持久存储，并监控预取失败、本地 cache 容量和冷填充频率。
