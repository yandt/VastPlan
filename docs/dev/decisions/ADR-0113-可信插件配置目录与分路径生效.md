# ADR-0113 可信插件配置目录与分路径生效

- 状态：已采纳
- 日期：2026-07-23
- 关联：[ADR-0090](ADR-0090-插件配置与托管凭证闭环.md)、[ADR-0092](ADR-0092-业务插件拥有托管凭证生命周期.md)、[ADR-0107](ADR-0107-插件权限目录与系统管理授权治理.md)

## 背景

插件清单已经能够声明非敏感 `configuration.schema`、`service/tenant/user` 作用域、`restart/hot` 生效方式和只写 `managedCredentials`，但 Portal 仍没有通用配置闭环。把清单 Schema 直接交给浏览器并写回任意 `ServiceUnit.config` 会产生四个问题：

1. 浏览器可以伪造插件、Schema、作用域或凭证 owner；
2. Application Composition 无权修改 Platform Profile 中的基础/平台插件；
3. `hot` 与 `restart` 不是同一种发布事务；
4. 通用协调器若以自身身份调用 `stageManaged`，凭证 owner 会错误地变成协调器，而不是实际使用凭证的插件。

`global-settings` 是租户隔离的普通 JSON 键值能力。它不知道插件制品、配置 Schema、部署来源和生效状态，不应演变为插件生命周期控制器。

## 决策

### 1. 独立配置协调器，不扩张内核或 global-settings

新增第一方平台插件 `cn.vastplan.platform.configuration.plugin-settings`，对外提供 `platform.plugin-configuration`。它保存候选事务、审计和 `tenant/user` 热配置真源，但不保存插件包、不解析不可信清单，也不直接重启进程。

Backend Kernel 只增加窄的可信目录与授权强制点；`global-settings` 继续只管理普通平台键值。协调器后端采用 Go，便于复用现有 Schema、CAS、部署和凭证契约；Portal/BFF 采用 TypeScript。Python/Node 后端在该强事务、强类型控制路径上没有生态优势，暂不增加第二份实现。协调器作为 leader-owned 的可信独立插件进程运行，不内嵌内核。

### 2. 配置目录只能来自已验签制品

Backend Resolver 在生成 Deployment 时，同时从它已经读取并验证的精确制品中生成 `ConfigurationCatalog v1`。初始 controlplane 发布和 Deployment Manager 在线发布都必须在 Deployment CAS 成功后，把目录写入同一个控制面 sidecar key；内核读取时重新匹配活动 Deployment 的 tenant、revision 与 digest，不匹配时隐藏旧目录并等待同代目录补齐。目录项绑定：

- 认证 tenant、deployment、unit 和插件；
- 插件来源 `platform-profile/application`；
- 精确版本、channel 和 SHA-256；
- 清单中的 scope、applyMode、闭合 JSON Schema 与 managed credential 目录；
- 当前非敏感值、Deployment revision/digest 和 Schema digest。

浏览器只使用不可猜用途的 `cfg_<96-bit>` 资源 ID；不得提交插件 ID、artifact digest、Schema、scope、applyMode、owner 或 purpose 作为授权事实。每次写入都必须匹配当前活动目录 revision/digest，插件升级或组合换代会令旧候选失效。

### 3. 一个界面，三条真实生效路径

配置协调器统一呈现和审计，但不伪造单一存储事务：

| 配置类型 | Active 真源 | 生效路径 |
|---|---|---|
| `service + restart`、application 来源 | Application Composition / Deployment | 由 deployment-manager 创建受治理 revision，经预览、审批、发布、readiness 后切换 |
| `service + restart`、platform-profile 来源 | Backend Platform Profile | 由专用 Platform Profile Controller 创建新 Profile/Catalog revision；禁止写进 Application Composition |
| `tenant/user + hot` | plugin-settings 的租户/主体隔离 Store | 目标插件通过受认证的 resolve/watch 窄端口读取自己的 Active revision |
| `service + hot` | 目标插件的标准配置控制端口 | 协调器调用 `configuration.v1 prepare/commit/abort/status`，目标插件自行持久化并报告状态 |

`restart` 只允许 `service` scope；`tenant/user` 必须使用 `hot`。仅在清单写 `applyMode=hot` 不足以在线编辑：制品必须同时实现标准 `configuration.v1` 控制端口或 resolve/watch 读取端口，否则目录将其标记为只读并说明原因。

### 4. 托管凭证使用内核签发的 Owner Delegation

协调器不能冒充目标插件。写入候选前，它向可信宿主申请短时、一次性 `ConfigurationAuthority`，绑定 tenant、配置资源、精确制品、Schema digest、候选 ID、目标插件 owner，以及清单允许的 credential field/purpose。

凭证插件增加只允许精确配置协调器调用的 delegated stage 操作，并独立验证 Authority。owner 和 purpose 取自 Authority，不取自浏览器或普通 payload。返回的 handle 只进入协调器私有候选与宿主投影；浏览器只看到字段是否已托管、版本和状态。目标插件仍只能解析自己的字段，其他插件和配置协调器都不能取得 material。

### 5. 可恢复候选状态机

候选状态固定为：

```text
Draft -> Preparing -> Publishing -> Activating -> Ready
             |             |             |
             +-------------+-------> Failed -> RollingBack -> RolledBack|Failed
```

协调器先 stage 凭证，再持久化候选，然后驱动对应生效路径；目标 readiness 成功后才激活凭证并提交 Active。进程重启从耐久检查点继续查询状态，未知外部副作用不盲目重放。CAS 冲突、制品换代、Schema 变化、审批拒绝或目标插件故障均 fail-closed。

### 6. 权限和审批不被配置中心绕过

权限至少拆为 `platform.plugin-configuration.read/write/publish`。配置写入仍受当前 Management Binding、在线角色和对象来源约束；Platform Profile、关键基础插件、秘密轮换及生产发布可要求不同主体审批。配置中心只能请求已有控制器执行动作，不能自行签名制品、改 KV、写 Profile 文件或命令 Node Agent。

## 备选方案

- **扩展 global-settings**：会把普通 KV、制品清单、部署发布和凭证 Saga 混在一个插件中，否决。
- **只做 Portal 动态表单**：没有可信真源、CAS、恢复和真实生效事务，否决。
- **所有配置都写 Application Composition**：无法修改 Platform Profile 插件，也不适用于 tenant/user 热配置，否决。
- **配置协调器持有全部凭证**：目标插件 owner 错误且形成高权限秘密代理，否决。
- **协调器直接重启目标进程**：绕过 Deployment、Node Agent 候选切换和 readiness，否决。

## 影响与实施顺序

1. 先实现由 Backend Resolver 生成并签定摘要的 `ConfigurationCatalog v1`；
2. 统一初始与在线 Deployment 的目录 sidecar，随后实现 plugin-settings 的只读目录、候选 Store、CAS 和审计；
3. 实现 `ConfigurationAuthority` 与凭证 delegated stage；
4. 分别接入 Application Deployment、Platform Profile 和 hot configuration 控制端口；
5. 接入 Node BFF、TypeScript SDK、Workbench 动态表单和状态页面；
6. 以 global-settings/File Storage Provider 验证无秘密路径，再以带托管秘密的 Provider 验证完整 Saga。

在第 4 步完成前，配置页面不得把“保存候选”显示为“已生效”。
