# ADR-0148 公共服务基线、在线服务配置与种子配置分域

- 状态：已采纳，已实现
- 日期：2026-07-24
- 关联：[ADR-0057](ADR-0057-插件分级管理与双输入组合解析.md)、[ADR-0113](ADR-0113-可信插件配置目录与分路径生效.md)、[ADR-0116](ADR-0116-Backend-Platform-Profile候选Catalog与配置激活.md)、[ADR-0147](ADR-0147-种子与在线服务配置统一发布机制.md)

## 背景

统一 Seed 与在线发布机制后，仍需明确三类配置的所有权。公共安全、运行时和治理策略要应用到同类在线服务，但不能由单个服务覆盖；应用插件配置属于具体服务；平台管理中心、制品仓库等 Seed Service 又必须在在线管理面不可用时从本地恢复。若三类配置都塞进 Platform Profile `services`，管理界面会把种子自举配置误当在线配置；若公共基线复制到每个 Application Composition，则会产生漂移、重复维护和越权覆盖。

## 决策

### 1. 三个配置域、两个在线入口

Backend Platform Profile 明确分为：

- `serviceBaselines`：平台所有、按 `serviceClass` 注入在线应用服务的公共插件与配置；
- `services`：种子服务的显式本地定义，只保存平台自举和恢复所需的插件配置；
- Application Composition `units`：具体在线服务及其应用插件配置。

平台管理中心分别提供“公共基线配置”和“服务配置”入口。两者使用同一可信 Configuration Catalog、候选、异人审批与发布基础设施，但按所有权分流；不建立混合列表，也不允许应用配置覆盖公共基线。Seed Service 不进入在线 Configuration Catalog，因此不能从普通在线配置页修改。

### 2. 公共基线是 Profile 一等资源

旧 `attachments` 更名为 `serviceBaselines`，并增加稳定 `id` 与 `config`。同一插件只能属于一个基线，基线必须引用 Profile 已声明的 `serviceClass`。Resolver 按 service class 把基线插件与配置注入 Application unit，并在 Deployment resolution 中记录 `plugin_baselines`，让在线配置目录能够可靠区分公共基线与服务配置，而不靠插件 ID、命名空间或 UI 猜测。

基线配置是受保护层：服务可以增加自身配置分支，不能覆盖基线已有叶子；多个同类基线之间也不能静默覆盖。冲突在发布预览阶段 fail-closed。Resolver 输出仍是完全物化的 Deployment v2，Controller、Scheduler、Node Agent 和 Runtime 不需要理解基线。

### 3. 在线发布沿用两条既有 Saga

- 公共基线配置属于 Platform Profile，沿用 ADR-0116 的 Profile Candidate/Catalog/Deployment 双重单调激活与回滚；
- 服务配置属于 Application Composition，沿用 ADR-0115 的 Application Deployment 激活，或在插件声明并实现控制器时走 Service Hot；
- 两类入口都只允许编辑精确验签插件清单声明的 Configuration Schema 和托管凭证槽。

基线成员、精确插件版本和 service class 仍由 Platform Profile 治理，不开放给普通服务配置用户。当前“公共基线配置”页面只列出声明了 `service + restart` 可信 Configuration 契约的基线插件，并按 `baseline + plugin` 去重，不随服务单元数量重复展示。公共基线 Hot 会绕过 Profile 真相源且需要多目标原子 fan-out，当前 fail-closed；Service Hot 只用于具体服务配置。不具备在线配置契约的基线项继续通过受信 Profile 管理，而不是生成可随意编辑的通用表单。

### 4. Seed Service 只接受本地必要配置

Profile `services` 是 Seed Service 平面。其配置来自受信本地 Seed 文件，通过 ADR-0147 的统一 Publisher 显式 bootstrap；普通 `up/restart` 只恢复最后提交状态。Configuration Catalog 生成时排除所有“不属于 service baseline 的 platform-profile 插件”，防止 plugin-settings、浏览器或在线账本接管自举凭证、仓库路径和管理面恢复参数。

这不是第二套运行模型：Seed 与在线服务仍生成相同 Deployment v2，走相同 Resolver、CAS、Controller、Assignment 和 Node Agent。不同点仅是配置所有权和变更入口。

### 5. 不增加第三个内核输入

公共基线保留在 Platform Profile，服务配置保留在 Application Composition。Publisher 仍只接受这两份输入。引入独立 Baseline Controller 或第三份内核输入会扩大事务、摘要锁和回滚组合，且没有运行时收益，因此否决。

## 备选方案

- **公共基线复制进每个服务**：服务可独立修改，但会漂移、难以整体升级并扩大越权面，否决。
- **所有配置都在线管理**：会让管理面故障影响 Seed 自救，并把自举秘密暴露给普通配置工作流，否决。
- **所有配置都本地管理**：无法满足多服务、集群和租户的在线治理，否决。
- **新增独立 Baseline Deployment 输入**：边界表面更清楚，但需要三输入 CAS、预览、恢复和回滚，否决。

## 实施记录

1. Backend Platform Profile Schema 已以 `serviceBaselines[] { id, serviceClass, plugins, config }` 替代开发期 `attachments`；项目处于开发阶段，不保留旧字段兼容分支。
2. Resolver 已实现受保护深合并、重复插件拒绝、Seed Service 不继承基线，并把 `plugin_baselines` 写入 Deployment v2 resolution。
3. Configuration Catalog 已携带 `serviceBaselineId`；公共基线进入 Profile Activation，应用配置进入服务工作流，Seed Service 配置被排除。
4. plugin-settings 0.13.0 已将 Workbench 拆成“公共基线配置”和“服务配置”两个页面；TypeScript SDK 使用同一显式标记分流。
5. 平台管理 Seed Profile 只保留本地显式服务；受管在线服务 Profile 已增加应用授权公共基线，后续新服务自动注入且不能覆盖。

## 影响

正面：公共策略不再逐服务复制；服务用户只能管理自身配置；Seed 自举边界保持离线可恢复；发布、运行和调度内核不增加分支；未来 GitOps、CMDB 或在线 Profile 编辑器可复用同一 Profile 契约。

代价：公共基线变更会生成新的 Profile、Catalog 与 Deployment revision；同一插件不能同时作为 Seed Service 插件和公共基线插件；开发环境中持久化的旧 `attachments` Catalog 需要显式 fresh bootstrap，生产形成历史数据前必须另行设计迁移。
