# ADR-0030 资源感知、亲和与指标自动伸缩调度

- 状态：已采纳
- 日期：2026-07-16
- 关联：[ADR-0025 NATS 控制面、能力寻址与多节点调度](ADR-0025-NATS控制面寻址与多节点调度.md)、[ADR-0028 控制器选主与 Drain 收敛](ADR-0028-控制器选主与Drain收敛.md)

## 背景

ADR-0025 的调度器只知道精确标签和整数副本。同一节点上多个不同 unit 不会累计资源，运维只能靠手工标签粗分机器；SSD 等优选条件只能写成硬 selector；副本数变化也必须修改 Deployment。企业部署需要可审计的资源硬约束、稳定的软放置和与监控系统解耦的伸缩输入。

## 决策

1. **节点租约上报规范化容量**。Node Agent 用 `cpu_millis`、`memory_bytes`、`gpu` 三个非负整数发布可分配容量。NodeRecord schema version 升为 2；控制器兼容读取无容量的 v1 记录，此类节点只可承载零资源请求的旧规格。
2. **资源请求同时存在于 v2 输入和 v1 assignment**。v2 `resources.requests` 是调度输入；控制器把它复制进节点 v1 快照。每次计划先扫描其他 Deployment 已提交 assignment 的请求，再累计本次计划内请求，任何维度不足都在写入前 fail-closed。
3. **硬约束先过滤、软偏好后计分**。`nodeSelector` 与每个 `affinity.required` 必须匹配；匹配任一 `antiAffinity.required` 即排除。preferred 项按 1–100 加减分，偏好分相同再使用 rendezvous hash 和 node ID 破平，保证拓扑不变时结果稳定。
4. **自动伸缩使用短租约外部指标**。`VASTPLAN_AUTOSCALING_V1` KV 保存监控适配器发布的非负有限值，TTL 为 2 分钟。queue depth、CPU demand、定时预测等来源都映射为一个命名指标，控制器不依赖具体 Prometheus/KEDA SDK。
5. **统一按每副本目标值计算**。`desired = ceil(value / target_value_per_replica)`，随后 clamp 到 `min_replicas..max_replicas`；零值仍保持 min。配置 autoscaling 后，`replicas` 必须位于 min/max，作为规格的初始/人工基线，但运行计划以当前指标为准。
6. **指标异常 fail-closed**。指标缺失、TTL 过期、损坏或指标 KV 未配置时，不采用可能过时的 `replicas` 静默回退；本轮计划不写入，最近健康 assignment 保持运行。Controller 最长在默认 5 秒轮询内响应指标变化。
7. **指标写权限与调度权限分离**。runtime/node 身份可写 autoscaling 指标，controller 只读；bootstrap 负责 bucket 生命周期。`engineering/tools/autoscalingmetric` 是监控适配器和运维验证共用的最小发布入口。

## 备选方案

- **资源使用字符串单位**：对人友好，但每种语言都要重复实现 `m/Mi/Gi` 解析和溢出规则，拒绝；展示层负责格式化。
- **把 Prometheus 或 KEDA 写死进 Controller**：减少一个适配器，却把核心控制面绑定到外部系统和查询语法，拒绝。
- **指标缺失时使用上次指标或静态 replicas**：提高可用性但会在观测失效时继续错误扩缩容，拒绝。
- **只看节点容量、不扣 assignment**：会让不同 unit 和不同 Deployment 互相超卖，拒绝。

## 影响与边界

- 正面：调度计划具有可测试的资源硬上限、稳定软偏好和统一伸缩入口；失败不会破坏已运行计划。
- 代价：监控适配器必须至少每 2 分钟续报；严格顺序的 unit 计划会让前面的 unit 在容量竞争中优先，Deployment 中应按业务优先级排序。
- 边界：跨 Deployment 占用基于已提交 assignment 快照；两个不同 Deployment leader 在同一瞬间首次抢占同一剩余容量仍是乐观并发窗口。需要强资源配额/预留时，应增加每节点 CAS reservation 或统一 admission 层，不能把当前快照过滤宣称为分布式原子预留。
