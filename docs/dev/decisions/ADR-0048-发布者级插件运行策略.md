# ADR-0048 发布者级插件运行策略

- 状态：已采纳
- 日期：2026-07-17
- 关联：[ADR-0047 多语言运行驱动与第三方隔离边界](ADR-0047-多语言运行驱动与第三方隔离边界.md)、[ADR-0026 远端制品仓库与供应链信任](ADR-0026-远端制品仓库与供应链信任.md)

## 背景

ADR-0047 为未知发布者固定要求至少 `process-sandbox`，适合作为安全默认值，但无法覆盖企业私有插件、经过线下审计的合作方插件、应急封禁发布者等部署差异。把“第三方”永久等同于“不可信”也会迫使内核代码承载部署方的信任判断。

运行策略必须由内核使用者决定，同时不能让插件发布者通过自己的清单降低隔离要求，也不能因配置迁移意外放宽现有安全默认值。

## 决策

1. Node Agent 提供全局三态策略：
   - `require-isolation`：至少要求 `process-sandbox`；驱动能力不足时 fail-closed；
   - `allow-trusted`：允许 `trusted-process` 驱动；
   - `deny`：无论驱动隔离能力如何都拒绝启动。
2. 提供 `publisher=policy` 发布者级规则，优先级高于全局策略。发布者匹配使用已验签清单中的精确 `publisher`，不接受插件进程自报身份。
3. 最终约束顺序为：发布者级规则（若有）否则全局规则 → 与签名清单 `minimumIsolation` 取更强下限 → 校验运行驱动能力。`allow-trusted` 只是不额外提高下限，永远不能降低清单下限。
4. 生产默认继续使用 `require-isolation`；`vastplan` 通过兼容信任名单隐式映射为 `allow-trusted`。显式发布者规则优先于兼容名单，因此可以封禁或提高第一方发布者的隔离要求。
5. 旧参数 `-require-third-party-isolation=true|false` 分别迁移为 `require-isolation|allow-trusted`；与新全局参数同时配置视为歧义并在启动前拒绝。旧 `-first-party-publishers` 暂保留为 `allow-trusted` 快捷映射。
6. `allow-trusted` 是部署方明确承担风险的信任决策，不代表 `trusted-process` 成为恶意代码安全边界。开放不可信第三方市场仍以真正隔离驱动和独立安全评估为前置条件。

本决策修订 ADR-0047 第 4、5 点中“未知发布者必须隔离”的固定部署策略；ADR-0047 的驱动 SPI、清单下限和 fail-closed 能力校验继续有效。

## 备选方案

- 只保留布尔开关：无法表达封禁，也不能按合作方或私有发布者建立差异化信任，否决。
- 只按发布者配置、没有全局策略：新增发布者容易遗漏，缺少安全默认值，否决。
- 允许插件清单自行声明可信：信任主体与受益主体相同，可被恶意制品直接绕过，否决。
- 由远端市场统一下发策略：不适合气隙环境，也会让部署方失去最终控制权，保留为未来可选配置分发方式，不作为信任根。

## 影响

- 正面：同一内核可支持严格隔离、私有可信插件和发布者封禁，不再把部署信任写死在代码中。
- 正面：发布者覆盖、清单下限和驱动能力形成可测试的单向收紧链路。
- 代价：运维方需要维护发布者策略；错误选择 `allow-trusted` 可能在宿主权限下执行不可信代码。
- 兼容：现有默认行为不变，旧布尔参数可迁移；新旧全局参数冲突时 fail-closed。
