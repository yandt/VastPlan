# ADR-0057 插件分级管理与双输入组合解析

- 状态：已采纳
- 日期：2026-07-18
- 关联：[ADR-0010 插件服务与部署编排](ADR-0010-插件服务与部署编排.md)、[ADR-0011 组合是通用内核能力](ADR-0011-组合是通用内核能力.md)、[ADR-0044 全局依赖编排与本地自治启动管理](ADR-0044-全局依赖编排与本地自治启动管理.md)、[ADR-0050 首方插件多级命名空间](ADR-0050-首方插件多级命名空间与自举权限基线.md)

## 背景

`deployment/v2.units[].plugins` 当前同时承担两种职责：应用配置人员选择插件的输入，以及 Controller/Node Agent 执行的完整插件集合。随着 bootstrap policy、全局设置、凭证、数据库、制品仓库、Portal 设计系统和 Interaction Broker 等基础插件出现，这会让应用配置人员能够删除、替换或降级平台基线，也会迫使每个服务重复理解基础插件应该本地附着还是作为共享服务运行。

首方插件 ID 已采用 `cn.vastplan.<layer>.<category...>.<component>`，其中 `foundation/platform/product/integration/example` 是可信制品身份的固有分类。分类可以复用，但它只说明插件是什么，不能单独表达谁有权选择插件、在哪个环境中把第三方插件提升为平台组件。

## 决策

1. 插件固有分类与组合来源分离：
   - `foundation/platform` 默认属于平台管理面；
   - `product/integration` 和外部发布者插件默认属于应用管理面；
   - `example` 只允许显式开发模式；
   - 平台管理员可在 Platform Profile 中固定任意已验证插件，包含经评审的第三方平台适配器，但插件不能通过 Manifest 自报获得平台权限。
2. 不在 Manifest 中增加可自报的 `tier`。首方固有分类由已验证的 ID、publisher 和保留命名空间推导；组合来源由控制面生成并锁定。
3. 引入两个版本化输入：
   - **Platform Profile**：平台管理员管理，声明目标 kernel、允许的 service class、本地附着插件和独立共享平台服务；
   - **Application Composition**：应用配置人员管理，只声明应用 service unit 和应用插件，不引用或覆盖基础插件。
4. Composition Resolver 以环境已绑定的 Platform Profile 和 Application Composition 生成完整 `deployment/v2`。输出记录两份输入的 `id + revision + digest`、仅限开发环境的插件策略状态，以及每个插件的 `platform-profile/application` 来源。`deployment/v2` 是 Scheduler 的内部稳定输入，不再作为服务配置人员直接提交的资源。
5. Platform Profile 的本地附着插件注入匹配 service class 的应用 unit；共享平台服务作为独立 unit 进入完整部署，不能机械复制到每个应用服务。
6. 分级只决定管理权限与组合来源，不决定固定启动阶段。启动顺序仍由包依赖、runtime capability requirement 和 readiness DAG 决定。
7. 应用只能声明平台 capability 及兼容版本，不能指定基础插件实现或强迫平台插件降级。基础插件与应用插件使用独立 revision；应用回滚不能移除或回滚平台基线。
8. 强制点覆盖 Schema、Resolver、Controller 和 Node Agent。UI 隐藏基础插件只是呈现规则，不能替代服务端校验。

## 取代关系

- ADR-0010/0011 中“选择插件集进行组合”仍然成立，但修订为：平台控制面选择 Platform Profile，应用控制面只选择应用插件，Resolver 产生完整插件集。
- ADR-0044/0045 的依赖排序、实例策略、副本和放置规则保持不变；它们作用于解析后的 Deployment。
- `deployment/v1` 仍是每节点完整执行快照，Node Agent 不接收 Application Composition，也不自行解析平台基线。
- Runner `app/v1` 是预构建产物的完整锁定 Profile，不是普通用户直接选择基础插件的入口；后续构建控制面同样从平台 Profile 与应用组合生成它。
- Portal 由 Portal Platform Profile 固定设计系统，应用组合只选择功能插件；当前直接保存 `designSystem` 插件引用的实现应直接改造，不保留旧草稿兼容分支。
- 项目处于开发阶段，不迁移前期 Deployment、Portal 草稿或 App Profile 数据；旧格式直接由新格式取代。

## 备选方案

- **只增加 `tier` 字段**：插件可自报提升，而且分类、管理权限和部署来源混为一体；拒绝。
- **在同一 Deployment 中分 `basePlugins/applicationPlugins`**：应用仍能修改基础列表，无法形成独立权限、版本和回滚边界；拒绝。
- **按命名空间自动向所有服务注入基础插件**：无法区分本地策略与共享平台服务，也会造成重复实例和状态冲突；拒绝。

## 影响

- 正面：服务配置人员只能管理应用插件；平台基线可独立升级、灰度、审计和回滚。
- 正面：同一分类模型可用于 Backend、Portal、Runner 和 Mobile，而各内核仍可采用不同装配方式。
- 代价：需要新增组合 Schema、Resolver、来源锁并调整生产 CLI；部署调试需要同时观察平台基线和应用 revision。
- 约束：Platform Profile 必须来自受授权的平台控制面；Resolver 读取的 Manifest 必须先经过制品身份与内容验证。
