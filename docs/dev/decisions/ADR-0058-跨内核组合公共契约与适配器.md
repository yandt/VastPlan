# ADR-0058 跨内核组合公共契约与内核适配器

- 状态：已采纳
- 日期：2026-07-18
- 关联：[ADR-0011 组合是通用内核能力](ADR-0011-组合是通用内核能力.md)、[ADR-0057 插件分级管理与双输入组合解析](ADR-0057-插件分级管理与双输入组合解析.md)

## 背景

ADR-0057 确立四类内核都使用 Platform Profile 与 Application Composition 双输入，但首个 Backend 实现把跨内核身份字段、来源锁与 Backend `ServiceUnit` 放在同一个 `schemas/composition/v1` 包中。直接让 Frontend、Runner 和 Mobile 使用该包会把服务调度 DTO 泄漏到门户模块图、客户端构建和移动发布；分别复制全部规则又会造成发布者校验、来源授权和摘要算法漂移。

## 决策

1. 组合能力分为三层：
   - `schemas/composition/common/v1` 定义四内核共用的文档身份、`target.kernel`、来源枚举、解析引用和摘要语义；
   - `schemas/composition/<kernel>/vN` 定义各内核自己的 Platform Profile、Application Composition 和载荷；
   - `kernels/<kernel>/compositionresolver` 负责生成内核专用解析产物。
2. 两类输入统一使用 `target.kernel`，不再允许某些输入使用根级 `kernel`。当前处于开发阶段，旧 `schemas/composition/v1` 和旧字段直接删除，不提供兼容解析。
3. 精确制品读取、Manifest 身份一致性、发布者/命名空间管理分类、开发插件策略和来源授权放入 `shared/go/compositioncore`。Backend 与 Runner 可直接共享；其他语言依据公共 JSON Schema 和同一组一致性样例实现薄适配。
4. 不建立一个输出无关的万能 Resolver。Backend 生成 Deployment/Assignment，Frontend 生成 Portal revision，Runner 生成签名 App Profile/二进制，Mobile 生成原生或 MDM Profile。
5. `deployment/v2` 的组合引用和来源枚举引用公共组合契约，但服务副本、放置、资源与依赖字段继续只属于 Backend 部署领域。

## 备选方案

- **四内核共用 Backend Resolver**：会迫使 Portal、Runner 和 Mobile 伪装成 service unit，拒绝。
- **四套完全独立实现**：短期直接，但安全分类和来源规则容易漂移，拒绝。
- **立即引入 WASM/Rust 万能解析器**：跨语言执行统一，但显著增加构建、调试和供应链复杂度；当前没有足够收益，拒绝。

## 影响

- 正面：统一治理规则，同时保持四类内核产物和运行方式独立。
- 正面：Frontend/Runner/Mobile 可以新增自身 Schema，不再依赖 Backend 调度结构。
- 代价：非 Go 内核需要实现小型语言适配层，并通过公共一致性测试证明行为相同。
- 约束：公共层不得引用任一内核专用 DTO；只有至少两个内核语义完全一致的规则才能进入公共层。
