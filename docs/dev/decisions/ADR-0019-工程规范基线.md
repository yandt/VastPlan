# ADR-0019 工程规范基线

- 状态：已采纳
- 日期：2026-07-15
- 关联：[ADR-0016 单仓与代码目录布局](ADR-0016-单仓与代码目录布局.md)、[ADR-0017 版本定义与兼容性机制](ADR-0017-版本定义与兼容性机制.md)、[ADR-0018 测试布局与分层](ADR-0018-测试布局与分层.md)
- 活的细则见《[工程规范](../guides/工程规范.md)》

## 背景

项目此前只有**架构**决策，缺**工程**约束：无提交/分支规范、无依赖与许可证准入、无编码规范、无基准规范。而我们对插件坚持 fail-closed，对自身约束却全靠自觉——**已被现实打脸**：ADR-0017 §3 明写"协议常量禁止两处声明"，而 MVP 里 `MagicCookie`/`ProtocolVersion` 恰恰在 `protocolbus` 与 `sdk` 各写了一份，两轮后才发现。**君子协定连规则作者都拦不住。**

本 ADR 定工程规范基线；**强制手段（架构守护测试 + CI 门禁）另行落地**，本 ADR 为其提供靶子。

## 决策

### 1. 提交规范：Conventional Commits，中文正文，无 AI 署名

`<type>(<scope>): <subject>`，type ∈ `feat|fix|docs|test|refactor|perf|build|ci|chore`。

- **正文说"为什么"与"影响"，不复述"改了什么"**（diff 已经说了改什么）。
- **不得包含任何 AI 生成署名**（Co-Authored-By 等）——本条写进仓库而非仅存于工具侧记忆，确保团队与工具无关地一致。
- 破坏性变更用 `!` 或 `BREAKING CHANGE:` footer，并须指向对应 ADR。

### 2. 分支策略：主干开发（trunk-based）+ 短生命周期分支

`main` 为唯一长期分支；特性分支短命（理想 <2 天）、小步合入。

理由：单仓（ADR-0016）+ 当前无多版本维护负担 + 设计需要频繁被代码验证。待 Release Train（ADR-0017）真正对外发版、需并行维护多版本时，再评估是否引入发布分支。

### 3. 依赖准入：三问 + 许可证白名单

引入任何第三方依赖前必须回答：

1. **真的需要吗**——能否以 <100 行自写替代？（避免为琐事引入供应链风险）
2. **可持续吗**——维护是否活跃、有无单点弃坑风险？
3. **许可证允许吗**——见下。

**许可证白名单**（可闭源商用）：`Apache-2.0`、`MIT`、`BSD-2/3-Clause`、`ISC`、`MPL-2.0`（文件级 copyleft，仅修改其源文件时需开源该文件；ADR-0008 已据此确认 go-plugin 可用）。

**黑名单**（禁止）：`GPL`/`LGPL`/`AGPL`（传染性，与闭源商用冲突）、`BSL`/`SSPL`（非 OSI 开源，商用受限）、**无许可证**（默认全权保留，法律上不可用）。

新增依赖的 PR 须在描述中写明**用途 + 许可证**。

### 4. 错误处理二分法（本项目特有，务必成文）

- **传输层/基础设施错误** → Go `error`，用 `%w` wrap 保留链。
- **应用层错误** → 进 `CallResult.Error`（带稳定 code + retryable），**不冒泡为 Go error**。

二者严格区分是协议的既定语义（《插件契约与协议》§2.7），已有 E2E 守护。混用会让调用方无法区分"没打通"与"打通了但业务拒绝"。

### 5. 格式化与静态检查强制

`gofmt`/`goimports` 与 `go vet` 为**硬门禁**（非建议）；`golangci-lint` 作为质量基线。理由：格式与低级缺陷不该消耗人类评审注意力。

CI 固定 `golangci-lint` 的精确版本；该版本自身的构建 Go 版本不得低于 `go.mod` 声明的目标版本，否则 linter 会在加载配置阶段拒绝执行。升级项目 Go 版本时，必须同步验证并按需升级 linter。

### 6. 规范进仓库，不进工具侧记忆或跨项目 skill

工程规范是**项目工件**，须与代码一同演进、被 CI 执行、对全体成员（与任何 AI 工具）一致生效。放记忆/skill 会造成"只对某个工具生效"的隐性不一致。

## 可机器强制清单（为架构守护与 CI 提供靶子）

| 规则 | 可否自动化 | 手段 |
|---|---|---|
| 提交信息格式 | ✅ | commit-msg 钩子 / CI |
| 提交无 AI 署名 | ✅ | CI 正则扫描 |
| gofmt / vet / lint | ✅ | CI |
| 依赖许可证白名单 | ✅ | `go-licenses` / CI |
| 依赖漏洞 | ✅ | `govulncheck` |
| 协议常量单一真源 | ✅ | 架构守护测试（扫字面量） |
| 依赖方向（内核不依赖插件等） | ✅ | 架构守护测试（扫 import 图） |
| codegen 与 proto 同步 | ✅ | CI：跑 codegen 后 `git diff` 须为空 |
| 文档无死链 | ✅ | 测试/CI |
| PR 体量 <400 行 | ⚠️ 提示 | CI 警告（不硬卡） |
| "内核不含业务" | ⚠️ 部分 | 可查 import；语义部分靠评审 |
| 提交正文说"为什么" | ❌ | 靠评审 |

## 备选方案

- **Git Flow / 发布分支**：适合并行维护多版本；当前无此负担，徒增流程。否决（Release Train 落地后再评估）。
- **不限许可证、按需评估**：灵活；但企业级闭源商用一旦误引 AGPL 代价极高，事后清理成本远大于事前门禁。否决。
- **规范只写文档不做强制**：正是当前状态，且已被证明失效（见背景）。否决。

## 影响

- 正面：工程约束从"自觉"变为"可被机器检查"；许可证风险前置拦截；错误处理二分法成文，不再靠口传。
- 关注点：
  - 本 ADR 只是**基线与靶子**，真正价值在后续的架构守护测试 + CI 门禁落地——**否则它只是第 11 条君子协定**。
  - `golangci-lint` 规则集需按项目调，避免噪声淹没真问题。
  - PR 体量阈值是经验值，不硬卡以免鼓励拆分出无意义的碎 PR。
