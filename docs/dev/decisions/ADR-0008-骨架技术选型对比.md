# ADR-0008 骨架技术选型对比：go-plugin / Dapr / NATS

- 状态：已采纳（方向性结论）
- 日期：2026-07-14
- 关联：[ADR-0004 独立进程+协议总线](ADR-0004-插件运行形态.md)、[ADR-0005 骨架与技术栈解耦](ADR-0005-骨架与技术栈解耦.md)、[ADR-0007 内核间通信模型](ADR-0007-内核间通信模型.md)、[内核间与服务间通信](../architecture/02-内核间与服务间通信.md)

## 背景

骨架涉及几层需要"轮子"的能力：①独立进程插件宿主 + 协议总线；②内部服务寻址层（位置透明：合=进程内、拆=网络 RPC）；③控制面（服务发现、事件 pub/sub、RC 反向连接）。对三个候选做了联网深调，评估它们各自覆盖哪一层、以及相对我们**硬需求**的差距。

**两条不可退让的硬需求**（用于判定）：
- **H1 位置透明**：同一份代码，**合并部署=进程内零开销**、拆分部署=网络 RPC。
- **H2 RC 穿 NAT**：RC 在客户现场防火墙后，需 **RC 主动外连**、服务端把指令**可靠路由到持有该连接的节点**。

## 核心结论（先给要点）

**三者不是同一槽位的竞品，而是分属不同层**：

| 候选 | 对应我们的层 | 采纳结论 |
|---|---|---|
| **go-plugin** | ①插件宿主 + 协议总线 | **作为范式/候选传输**；是否直接用取决于后端内核是否用 Go（留待栈 ADR） |
| **Dapr** | ②内部服务寻址层 | **否决**（H1、H2 双双硬伤）；其 durable workflow 可作为**独立的、后置的**可选项另议 |
| **NATS** | ③控制面（事件 + 服务 RR + RC 连接） | **采纳为主选方向**；尤其 RC 反向连接是最强理由 |

**H1 位置透明这一层，三者都给不了"进程内零开销"**——那薄薄一层寻址（本地 Registry 命中即直调、否则走远程传输）**我们自建**，远程传输可挂在 NATS request-reply 或 gRPC 之上。

## 逐项对比

| 维度 | go-plugin | Dapr | NATS |
|---|---|---|---|
| 是什么 | Go 的进程外插件系统（子进程 + gRPC/net-rpc） | 分布式应用运行时（sidecar 边车） | 单二进制消息系统（pub/sub + streaming + KV） |
| 许可证 | **MPL-2.0**（库，非 BSL），可闭源商用 | **Apache-2.0** | **Apache-2.0** |
| 成熟度/治理 | Terraform/Vault/Nomad 在用；低频维护、稳定（v1.8.0, 2024-04）；归属 IBM | **CNCF 已毕业**（2024-10）；活跃 | **CNCF 已毕业**；2025 治理插曲已和解，仍 CNCF/Apache-2.0 |
| H1 进程内零开销合并 | ❌ 插件**永远进程外**，即使合并也走本机 IPC | ❌ **每次调用必过 sidecar 一跳网络**，同机也不例外 | ❌（非其职责；远程传输尚可，进程内需自建快路径） |
| H2 RC 穿 NAT + 精准路由 | ❌ **仅限本机**，不跨机器 | ❌ 无 NAT 穿透/服务端推送；边缘/客户端不适配 | ✅ **client 恒 dial-out** 穿 NAT；**兴趣路由**把消息送达持有连接的节点；**leaf node** 正为"现场部署、回连中心"设计 |
| 跨语言 | 插件可跨语言（gRPC），**宿主必须 Go** | 多语言 SDK | 30+ 语言客户端 |
| 服务发现 | ❌ 自建 | ✅ 按 app-id + mTLS | ✅ queue group + NATS Micro（注册/发现/健康/负载均衡） |
| 事件 pub/sub | ❌ | ✅ 但**需外接 broker** | ✅ 原生；JetStream 提供持久化/至少一次/KV/ObjectStore |
| 插件生命周期 | 运行时启停单插件、**reattach 支持宿主热升级**；但注册表/编排/依赖/自动重启需自建 | N/A | N/A |
| 故障隔离 | ✅ 插件崩溃不拖垮宿主 | 进程级（边车） | 消息级解耦 |
| 主要短板 | 宿主限 Go、仅本机、无编排 | H1/H2 双硬伤、每服务一边车、运维重、对合并部署无收益 | 非 DB/工作流引擎/强类型流式 RPC；JetStream 有运维心智 + Jepsen 2025-12 发现极端场景丢写风险 |

## 决策

1. **控制面采纳 NATS 为主选方向**：跨服务**事件 pub/sub** + 服务间 **request-reply**（含 queue group 负载均衡、NATS Micro 发现）+ **RC↔RS 反向连接**（leaf node + 兴趣路由，直接解 H2）。RC 结果/日志回流走 JetStream Stream/ObjectStore。多租户用 NATS accounts（subject 命名空间隔离），安全用 TLS→NKey/JWT。
   - **约束**：JetStream 不当强一致数据库用（关键数据 replicas≥3、监控积压与磁盘、评估 fsync）；关系型/事务数据仍用数据库。
2. **否决 Dapr 作为内部服务寻址层**：违背 H1（sidecar 恒过网络，合并部署零收益）与 H2（无 NAT 穿透/服务端推送）。其 **durable workflow** 引擎作为"崩溃可恢复的长任务编排"是独立议题，若未来需要可单独评估，不进核心骨架。
3. **go-plugin 作为独立进程插件宿主的范式与候选传输**：其"进程外 + gRPC + 崩溃隔离 + reattach 热升级"正对 ADR-0004。**是否直接采用取决于后端内核语言**——它要求宿主为 Go 且仅限本机子进程；跨机器插件不在其能力内（跨机走 ②寻址层）。此项**输入到后续栈 ADR**，本 ADR 不锁定内核语言。
4. **H1 的位置透明快路径自建**：本地 Registry 命中→进程内直调；未命中→经寻址层走远程传输（NATS request-reply 或 gRPC）。这层薄适配是我们自研的核心，任何现成轮子都不代劳。

## 影响

- 正面：控制面有了强主选（NATS 一件覆盖事件+RR+RC，且唯一优雅解 H2）；排除了会带来 H1/H2 硬伤的 Dapr，避免走回中心边车的弯路；插件宿主有成熟范式可循。
- 关注点/待验证：
  - go-plugin 的"宿主限 Go"会实质影响后端内核选栈——需在栈 ADR 权衡。
  - NATS 的 JetStream 运维与耐久边界需要监控与容量规划；治理层面 Synadia 关系长期留意。
  - 强类型/大流式/大 payload 的服务间调用保留 gRPC，与 NATS request-reply 并存，边界需在寻址层专篇明确。
  - 本 ADR 是方向性结论；落地前应做一次小基准（进程内直调 vs NATS RR vs gRPC 的时延/吞吐）与 RC leaf 拓扑 PoC。
