# ADR-0011 组合是通用内核能力（服务 / 门户 / 客户端App）

- 状态：已采纳
- 日期：2026-07-15
- 关联：[ADR-0006 内核分区与后端组合](ADR-0006-内核分区与后端组合.md)（本 ADR 将其"组合"从后端专属上升为通用能力）、[ADR-0010 插件服务与部署编排](ADR-0010-插件服务与部署编排.md)、[ADR-0012 APP 内核运行模型](ADR-0012-APP内核运行模型.md)

## 背景

ADR-0006 只给**后端内核**定义了"灵活组合出多个服务（backend/workspace/rs）"，前端与客户端内核被描述为"单一应用"。但同样的诉求存在于另两套内核：同一前端内核应能配出多个产品面（运维控制台 / Studio / 终端门户 / 按 edition·租户定制），同一客户端内核应能配出多个客户端形态（纯采集端 / 脚本执行端 / 全功能）。应把"组合"统一为三套内核的通用能力。

同时明确一处改名：**客户端内核 RC 内核 → APP 内核**（它在客户端"运行 App"，APP 命名更贴切）。

## 决策

1. **组合是通用内核能力**：三套内核共用同一组合模型——**组合单元 = 内核 + 选定插件集 + 配置**。产出的单元类型按内核而异：

   | 内核 | 组合单元 | 组合的轴 | 运行形态 |
   |---|---|---|---|
   | 后端内核 | **服务 Service**（backend/workspace/rs） | 部署/伸缩/职责 | 服务端进程，多副本，NATS queue group |
   | 前端内核 | **门户 Portal**（admin/Studio/门户…） | 产品面、edition/租户定制 | 浏览器；外壳 + Module Federation 远程插件集 |
   | APP 内核（客户端） | **客户端 App / Profile** | 部署场景/客户/角色 | 客户机二进制，按 profile 组合（预编译，见 ADR-0012） |

2. **改名**：客户端内核统一称 **APP 内核**；其组合单元称 **客户端 App / Profile**。前端内核的组合单元称 **门户 / Portal**，以与客户端 App 区分、避免"app"撞名。

3. **命名空间范围（本次不改，避免契约 churn）**：清单三面键仍为 `webui / backend / rc`（`rc` 面 = APP 内核贡献面），`rc.*` 技术命名空间与 RS↔RC 通信轴暂保留（RC = 客户端作为任务 runner 的角色名）。如需彻底统一为 `app.*` 另开 ADR。

4. **插件服务统一管理三类组合**（扩展 ADR-0010）：期望态配置除 `service`（后端）外，新增 `portal`（前端）与 `app`（客户端 profile）两类组合声明；"指定配置→自动装配"对三套内核一致——后端节点装服务、前端服务装门户、APP 装 profile。

## 备选方案

- **仅后端可组合，前端/客户端单一**（ADR-0006 原状）：与实际产品面/客户端形态多样性不符。被否决。
- **不改名，沿用 RC 内核**：客户端"运行 App"用 APP 命名更直观，且未来对外表述更清晰。采纳改名。

## 影响

- 正面：三套内核组合心智统一；插件服务/期望态成为三类组合的统一控制面；前端多门户、客户端多 profile 有了正式表达。
- 关注点：
  - 前端"门户"= 外壳 + Module Federation 远程插件集，可运行时热装（ADR-0003）；与后端进程模型不同，细化见架构文档。
  - 客户端"App/Profile"的运行模型是**预编译**（ADR-0012），与服务端的运行时热装不同。
  - `rc` 命名空间暂留，未来若统一为 `app` 需评估契约兼容。
