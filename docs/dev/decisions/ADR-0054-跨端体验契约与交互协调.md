# ADR-0054 跨端体验契约与交互协调

> 实现更新（2026-07-22）：本文的 Portal Edge 身份边界已由 Node Portal Kernel 的 OIDC Code+PKCE、密封 BFF Session 与 CSRF 实现；Mobile 仍使用独立令牌和设备边界。

## 背景

Frontend Portal、Runner 与 Mobile 都可能需要人机交互，但它们的运行环境和分发安全边界不同：Portal 是可信 Web 模块组合，Runner 是预编译的无头执行器，Mobile 是经商店/MDM 分发的原生 Companion。现有 `@vastplan/ui-primitives` 含 React 组件与回调，不能成为 Mobile 或 Runner 的运行时依赖。

统一插件清单已经有 `frontend`、`runner`、`mobile` 三个贡献面；问题不是增加新的插件面，而是让三面以稳定、可审计的语义协作，并避免 Runner 直接访问浏览器或 Mobile 直接加载 Web 模块。

## 决策

1. 定义两个独立的、可序列化的跨端契约：
   - `ui.contract`：Web 与 Mobile 共用的表单、操作、数据、反馈和渲染能力语义；不得包含 React、DOM、原生控件或回调函数。
   - `interaction.contract`：Runner 或后端能力发起的确认、表单、审批、通知与进度交互；定义请求、响应、状态机、超时、取消和幂等性。
2. `@vastplan/ui-primitives` 定位为 Web Adapter：它消费 `ui.contract` 并提供 React/设计系统集成；Mobile 以原生 Adapter 消费同一契约；Runner 不消费一般 UI 组件契约。
3. Runner 不直接调用 Portal 或 Mobile。它通过 Backend 的 `platform.interaction-broker` capability 创建并观察持久化交互任务；Portal、Mobile 或受管 Runner 本地 Renderer 通过受权限约束的接口呈现并提交一次性响应。
4. Interaction Broker 是平台级基础插件，不并入 Portal Composer。Portal Composer 继续只负责 Portal revision 的草稿、审批、发布和回滚。
5. 四面清单保持 `backend/frontend/runner/mobile` 不变。后续仅收紧各面 contribution descriptor，增加契约版本、所需渲染能力、体验逻辑名和交互声明。
6. Web、Mobile 的身份边界不同：Portal Edge 延续 Cookie/CSRF；Mobile 使用独立的移动端认证与设备绑定模型。Broker 只接受已验证 Principal，不信任客户端声称的用户、租户或任务归属。

## 备选方案

- **让 Runner/Mobile 调用 Portal Runtime**：会把 React、Cookie/CSRF 和浏览器运行时泄漏到原生/无头内核，且不适用于离线和商店分发，拒绝。
- **三个内核各自定义表单与审批模型**：短期实现快，但权限、字段语义、审计和回退策略必然漂移，拒绝。
- **将交互任务塞入 Portal Composer**：混合门户配置治理与运行中工作流协调，扩大其状态和权限边界，拒绝。

## 影响

- 正面：同一业务插件可为 Web、Mobile 提供不同代码实现，但复用表单/操作/审批语义；Runner 保持无头和可审计；跨面通信继续复用统一 addressing 与 `CallContext`。
- 正面：移动端和 Runner 仍遵守预编译、签名和整体升级模型，不引入客户端下载可执行 UI 代码。
- 代价：需要新增 Schema、跨语言 SDK、Broker、移动适配器和端到端兼容测试。
- 约束：交互响应必须具备来源绑定、过期、幂等和一次性终态；`secretRef` 只能在响应中传递凭证引用，真实密钥仍经凭证能力的短期委托取得。
