# ADR-0010 插件服务与部署编排模型

- 状态：已采纳
- 日期：2026-07-14
- 关联：[ADR-0003 运行时热装](ADR-0003-插件装载模型.md)、[ADR-0004 独立进程+协议总线](ADR-0004-插件运行形态.md)、[ADR-0006 内核分区与后端组合](ADR-0006-内核分区与后端组合.md)、[ADR-0008 骨架技术选型对比](ADR-0008-骨架技术选型对比.md)、[插件服务与部署编排](../architecture/03-插件服务与部署编排.md)

## 背景

ADR-0006 留了待决项"后端灵活组合/部署编排机制"。现需落定：①**插件服务**（集中管理插件制品 + 期望状态配置）；②**配置共享**（多节点/多副本配置一致）；③**集群化**（同一插件-服务多副本）；④**自动装配**（指定服务器放配置，节点自动下载插件、启动服务）。

对 testa 的只读调研表明：其 release-service（单表 + 声明式 Kind + sha256）、endpoints.yaml 握手下发、RC 自举/自升级、Edition-Train-Bundle 均为成熟范式；但"**服务端读配置自动装配启动**"与"**集群化**"是 testa 空白——这正是 VastPlan 要自建的核心价值。

## 决策

1. **设立"插件服务 Plugin Service"为一等组件**（后端内核组合出的核心服务），承担两职：
   - **插件制品仓库**：存插件包 + manifest + 版本 + channel；下载 **sha256 强校验、fail-closed**。借鉴 testa release-service 的"**单表 + 声明式可插拔 Kind 注册表**"。同时充当插件市场后端。
   - **期望状态配置（desired-state）**：以 config-as-code、版本化方式声明"哪些插件组合成哪个服务、副本数、落在哪些节点、连接关系"，是集中真源。
2. **自建"节点代理 Node Agent" + reconcile 循环**（每后端节点一份，随内核）：取本节点期望态 → 下载所需插件（sha256 校验）→ 本地宿主拉起插件进程 → 注册进 mesh（NATS + 寻址层）→ 持续对齐期望态 vs 实际态。这是 testa 空白的补齐。
3. **不走纯 k8s 镜像重部署**：自管 reconcile **可裸机可 k8s**。纯 k8s 重部署与 ADR-0003 运行时热装冲突，且不适配对私/气隙/边缘。k8s 只作为可选承载平台，不作为编排前提。
4. **集群化用 NATS queue group**（ADR-0008 已定）：同一插件-服务的 N 副本加入同一服务身份/queue group，天然负载均衡、无状态水平伸缩；副本数由期望态配置声明。
5. **配置共享经 NATS KV 推送，寻址走 mesh**：期望态集中存于插件服务、版本化，经 **NATS KV（watch）** 推送到各节点代理触发 reconcile；**服务寻址走 mesh（NATS 兴趣路由 + 寻址层），不让节点轮询集中存储要地址**——沿用 testa"寻址与身份分离 + 握手下发"的教训。
6. **插件自升级借鉴 RC upgrader**：版本目录 + 符号链接原子切换 + 失败回滚 + 后台 checker。
7. **Edition/Train/Bundle 预留**：对公对私（edition 是数据非代码分叉）、可复现系统版本（train 钉一组插件版本）、气隙离线（bundle 内容寻址 sha256）——本轮预留，不展开。

## 备选方案

- **纯 k8s + Helm + GitOps + KEDA**（testa 集群化设计稿方向）：生态成熟、伸缩现成；但要求 k8s 无处不在、对私/气隙/边缘重、且镜像重部署与运行时热装心智冲突。否决为**默认**方案，保留为自管 reconcile 之上的可选承载。
- **中心配置存储让节点轮询要地址**：与 mesh 寻址重复且引入寻址中心。否决——寻址走 mesh，集中存储只放期望态。
- **插件服务兼做部署执行器（把编排耦合进制品仓库）**：testa 明确让 release-service"只发布不部署"。VastPlan 中"制品仓库"与"期望态配置"同属插件服务但**职责内部分层**，部署执行下放节点代理。

## 影响

- 正面：补齐 testa 空白（服务端自动装配 + 集群化）；"指定配置→自动下载启动"闭环成立；裸机/k8s/气隙统一心智；集群化复用已定的 NATS。
- 关注点：
  - 节点代理 reconcile 的健壮性（期望态与实际态对齐、部分失败、回滚、在途任务 drain）需细化。
  - 期望态配置的 schema、版本化与冲突解决；NATS KV 作配置面的容量与一致性边界。
  - 插件制品仓库与"统一插件定义/清单"的衔接（manifest 中的服务角色标注驱动装配）。
  - 有状态依赖（DB/对象存储/消息）不纳入本编排范围，另行处理。
