# ADR-0001 插件化架构模型：微内核 + 全层扩展点

- 状态：已采纳
- 日期：2026-07-14
- 修订：2026-07-14 —— 纠正早期"第三方不可信市场插件"的误解，明确为第一方系统级模块的可插拔骨架

## 背景

VastPlan 的核心诉求是**先做好系统骨架，让功能通过插件在骨架之上持续扩展**。这里的"插件"指**系统级功能模块**——对标 testa 的审计系统、可观测系统、Studio 可开发模块、用户系统扩展模块（用户校验、信息扩充）等，而非面向终端用户的小挂件。

关键约束：

- 插件当前**全部由本方开发（第一方、可信）**，暂不开放第三方。
- 扩展面必须覆盖**四层**：后端功能、Web UI、RS（Runner Service）、RC（Runner Client）。
- **RS/RC 自身也要通过插件机制扩展**，而不是只有主后端可插拔。

## 决策

采用**微内核 + 全层扩展点**架构：

1. **微内核**：内核只提供最小骨架——插件宿主、扩展点注册表（Registry）、生命周期、模块间通信、以及少量横切基础（身份、权限、事务、可观测、事件）。
2. **一切功能皆插件**：审计、可观测、用户系统扩展、Studio 模块等都是骨架之上的第一方插件，通过注册制挂到扩展点，不写死在内核里。
3. **全层可插拔**：后端、Web UI、RS、RC 各自暴露扩展点；一个功能特性可以横跨多层（如一个模块同时贡献后端逻辑、Web UI 面板、RC 侧执行能力）。RS/RC 与主后端一样是可被插件扩展的宿主。

借鉴 testa 应用骨架层的**注册制驱动**经验（CallerKind / Scene / PermissionChecker / Hook / EventSink 等 Registry），但为全新技术栈重新设计。

## 备选方案

- **单体内聚、无扩展点**：功能写死在核心，加特性即改核心。被否决——违背"骨架上扩展"的根本目标。
- **仅后端可插拔**：RS/RC 固定。被否决——RS/RC 的能力扩展是明确诉求。
- **面向第三方的不可信插件市场 + 沙箱隔离**：这是早期误解。第一方可信插件不需要 gVisor/多租户沙箱那套安全隔离；相关设计（原《后端隔离与沙箱》）已废弃，留待未来若开放第三方再议。

## 影响

- 正面：功能与内核解耦，加特性 = 加插件，四层一致的扩展心智。
- 关注点：内核必须把扩展点、生命周期、模块间契约设计得足够稳，否则插件会反向侵蚀内核。**骨架设计是当前第一优先级。**
- 安全隔离**暂不作为重点**（第一方可信）；仅需模块级**故障隔离**（一个插件出错不拖垮内核），归入骨架设计。
