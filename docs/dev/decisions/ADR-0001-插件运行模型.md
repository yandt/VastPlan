# ADR-0001 插件运行模型：全栈插件

- 状态：已采纳
- 日期：2026-07-14

## 背景

VastPlan 的核心差异化是插件化扩展。插件的运行位置与边界有三种可能：仅服务端扩展点、仅下发到客户端执行、或前后端全栈。这个选择决定整个插件运行时的架构复杂度与扩展能力上限。

## 决策

采用**全栈插件模型**：一个插件可同时包含
1. 前端 UI（注入平台工作台）
2. 后端逻辑（工具 / Agent 能力 / LLM 编排 / API，运行在隔离沙箱）
3. 客户端脚本/工作流（下发到远程 Runner 本地执行）

三部分通过统一的插件清单（manifest）声明，任意组合。这是最接近 VSCode 完整扩展体验的形态。

## 备选方案

- **服务端为主**：实现简单，但插件无法自带 UI，扩展体验受限。
- **客户端为主**：贴近 testa 的 runner-client 模式，但无法扩展平台自身能力。

均被否决，因为都无法支撑"通过插件持续增强整个系统"的目标。

## 影响

- 正面：扩展能力上限最高，插件可提供端到端完整功能。
- 代价：运行时最复杂——需同时解决前端隔离与宿主通信、后端沙箱隔离、客户端代码分发三套机制。这些将各自展开为 `plugins/` 下的设计篇。
