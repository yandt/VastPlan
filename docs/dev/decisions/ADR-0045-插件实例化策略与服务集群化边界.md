# ADR-0045 插件实例化策略与服务集群化边界

- 状态：已采纳
- 日期：2026-07-17
- 关联：[ADR-0044 全局依赖编排与本地自治启动管理](ADR-0044-全局依赖编排与本地自治启动管理.md)、[ADR-0025 NATS 控制面、能力寻址与多节点调度](ADR-0025-NATS控制面寻址与多节点调度.md)、[插件服务集群化设计](../architecture/插件服务集群化设计.md)

## 背景

ADR-0044 已区分同一内核内的本地依赖和跨服务的远端 capability 依赖，但还没有回答两个容易混淆的问题：

1. 一个插件是否允许在多个内核中独立重复启动；
2. 一个插件实例发布的 capability 是只供本内核使用，还是要加入跨服务集群。

如果只用插件 ID、service role 和 capability 名称表达这些语义，本地基础插件可能被错误发布到全局 mesh；必须单例的插件也可能被错误扩成 active-active 副本。

## 决策

### 1. 分离五种身份

插件运行身份由以下字段共同构成，不能用 `plugin_id` 单独推导集群关系：

- `plugin_id/version`：不可变制品身份；
- `kernel_id`：运行所在内核实例；
- `service_role`：Backend 组合角色，如 `backend/workspace/rs`；
- `logical_service`：参与同一服务集群的逻辑服务名；
- `instance_id`：一次进程/宿主实例的唯一身份。

只有 `logical_service`、capability、作用域和路由域都相同的实例，才允许加入同一服务副本组。

### 2. 固定四种实例化策略

插件清单声明实例策略和状态模型，部署期望态只能在清单允许的范围内选择副本数和放置方式：

- `per-kernel`：每个内核独立运行；通常只发布本地 capability，不加入全局 queue group；
- `active-active`：允许多个实例并行提供同一 capability；要求无状态或共享状态、幂等和可重试；
- `leader`：允许多个候选实例，但同一作用域只有一个持有有效 fencing token 的服务实例；
- `partitioned`：按稳定分片键分配所有权；同一分片同一时刻只有一个有效 owner。

`singleton` 不作为独立运行模式，而是 `leader` 在指定作用域内副本基线为 1 的部署约束。

### 3. 能力必须声明可见范围和路由方式

运行时 capability descriptor 必须区分：

- `visibility`：`local`（本内核）、`service`（同一 Backend 服务）、`cluster`（跨节点同一服务集群）、`global`（跨服务/跨内核）；
- `routing`：`direct`（本地直调）、`queue`（active-active queue group）、`leader`（leader/fencing）、`shard`（分片路由）。

约束如下：

- `local` 只能进入本地 Registry，不得写入全局 capability 目录；
- `service/cluster` 才能登记 mesh，但必须带 `logical_service` 和实例租约；
- `leader` 必须带 leader epoch/fencing token；
- `shard` 必须声明稳定路由键和 owner 租约；
- `routing=queue` 只能用于 `active-active`，不能代替选主或数据复制。

### 4. 清单声明与部署声明分层

插件清单负责声明“我支持什么”：

```json
{
  "runtime": {
    "instancePolicy": "active-active",
    "stateModel": "external-shared",
    "provides": [
      {
        "capability": "platform.database",
        "visibility": "cluster",
        "routing": "queue"
      }
    ]
  }
}
```

部署期望态负责声明“这次如何部署”：

```json
{
  "logical_service": "platform.database",
  "replicas": 2,
  "instance_policy": "active-active",
  "placement": {"nodeSelector": {"tier": "platform"}}
}
```

部署不能把 `per-kernel` 强行改成 `active-active`，也不能把 `leader` 当成普通 queue 副本。

为兼容已有 v1 插件，缺少 `runtime` 的旧清单按历史行为解释为 `active-active + external-shared + cluster + queue`。新开发的本地基础插件必须显式声明 `per-kernel + local + direct`，不能依赖兼容默认。

### 5. 状态模型决定集群模式

- `local-ephemeral`：只能用于 `per-kernel`；
- `external-shared`：可用于 `active-active`，状态由外部一致存储负责；
- `leader-owned`：用于 `leader`，写入和迁移必须带 fencing；
- `partition-owned`：用于 `partitioned`，每个分片必须有明确 owner 和恢复协议。

内核不自动为插件复制业务状态，也不把启动两个进程视为数据集群。

### 6. A/B/C 的推荐部署

- A 全局设置：读取能力可以 `active-active`；写入或版本发布操作可单独使用 `leader`；
- B 凭证管理：凭证读取可以 `active-active`；密钥轮换、撤销和迁移使用 `leader`；
- C 数据库服务：连接代理可以 `active-active`，数据库 schema migration 使用 `leader`；
- 业务插件：通过 `remote`/`cluster` capability 依赖 C，不直接依赖 C 的某个进程实例。

### 7. 运行时强制边界

能力目录记录必须扩展逻辑服务、可见范围、实例策略、路由方式、generation 和健康状态。Node Agent/Runtime 在发布前校验：

1. 运行时声明与签名清单一致；
2. 部署副本数与实例策略兼容；
3. `local` capability 不进入全局目录；
4. `leader`/`partitioned` 实例具有有效租约和 fencing；
5. 同一逻辑服务的版本满足兼容窗口。

### 8. 故障和升级

- `per-kernel` 实例故障只影响本内核及其本地调用者；
- `active-active` 实例故障由租约过期和 queue group 摘流处理，调用方必须允许重试；
- `leader` 故障先撤销旧 fencing，再选出新 leader；
- `partitioned` 故障先完成 owner 转移，再恢复分片流量；
- 升级必须先验证策略、状态模型和契约兼容，再按副本逐步 Drain/替换；
- 控制面短暂不可用时，已有实例按本地租约继续服务，新编排暂停。

## 备选方案

- **只按 plugin ID 聚合**：无法区分本地基础插件和跨服务集群，拒绝。
- **所有 capability 都发布到全局 mesh**：拓扑简单，但扩大故障面并破坏内核边界，拒绝。
- **只用 NATS queue group 代表集群**：能做请求分发，不能表达单例、选主、分片和状态一致性，拒绝。
- **把实例策略全部交给部署者**：容易把不支持 active-active 的插件错误扩容，拒绝；策略必须由签名清单声明、部署期望态选择。

## 影响

- 正面：同一插件可以安全地按 `per-kernel`、`active-active`、`leader` 或 `partitioned` 运行；本地能力不会意外泄漏到跨服务 mesh；集群行为和状态所有权可审计。
- 代价：需要扩展插件 runtime schema、Deployment v2、Announcement/能力目录、Node Agent 状态机和路由测试。
- 边界：本 ADR 不引入通用 Raft 或自动业务数据复制；有状态集群协议仍由具体平台插件或外部基础设施负责。

## 实施澄清（2026-07-17）

Backend 1.0 将 `leader` 的部署副本约束为 `replicas=1`：这是一个活动 owner，不是数据副本数。节点租约失效后 Controller 将 owner 重新放置；Runtime 用持久 epoch 和 fencing token 防止旧 owner 写入。热 standby 多候选需要独立的非阻塞候选状态机，否则会阻塞 Node Agent 接收新期望态，因此不伪装成已实现能力。

`partitioned` 的 `replicas` 表示期望 owner 进程数，`partition_keys` 是完整所有权集合。每个 key 同时只分配给一个节点；节点不足时允许剩余 owner 接管全部 key 并由组合状态报告副本降级。业务数据恢复仍由插件的 partition-owned 协议负责。

服务发现不是授权。生产 TransportIdentity 必须携带 capability allowlist，并按 visibility 追加 service role、logical service 或 global 授权校验；目录查询和实际调用使用同一过滤规则。
