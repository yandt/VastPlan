# ADR-0055 交互访问策略作为独立基础插件

- 状态：已采纳
- 日期：2026-07-18
- 关联：[ADR-0021 权限判定的强制点与零校验器语义](ADR-0021-权限判定强制点.md)、[ADR-0053 门户访问策略作为独立基础插件](ADR-0053-门户访问策略插件.md)、[ADR-0054 跨端体验契约与交互协调](ADR-0054-跨端体验契约与交互协调.md)

## 背景

Portal Edge 需要以浏览器会话代理 `platform.interaction-broker`，但 Broker 同时面向 Runner/后端来源与 Portal/Mobile 呈现端。现有 `portal-access-policy` 的职责仅是 Portal Composer 组合治理；若把运行中交互权限加进去，会把两种不同的对象、角色和生命周期耦合为一个策略域。

## 决策

新增首方基础插件 `cn.vastplan.foundation.security.interaction-access-policy`，作为 `permission.checker` 的粗粒度入口策略：

1. `open`、`cancel` 仅允许可信 `plugin`、`runner` 或 `system` 调用方；`system` 可用于受审计的运维取消。
2. `list`、`get`、`present`、`respond` 仅允许经验证的 `user` 调用方；Portal Edge 和未来 Mobile Gateway 都必须投影服务端验证的 Principal，不能从 JSON 接收用户、角色或 tenant。
3. Broker 对任务来源、租户、允许呈现面、`eligibleSubjects`、过期和一次性终态继续做对象级判定；策略插件不复制这些可变状态。
4. `kernel.config.get` 仅允许 Broker 插件自身调用，以读取其受控状态文件位置。
5. Portal Edge 同时启动 Composer 策略与交互策略；未匹配请求仍由 ADR-0021 的零/全弃权 fail-closed 语义拒绝。

## 备选方案

- **扩展 `portal-access-policy`**：实现较快，但把 Portal revision 治理和运行中人机交互混为同一个授权域；拒绝。
- **让 Edge 自行做交互授权**：会绕过 Backend 的强制权限点，且 Mobile/Runner 需要复制规则；拒绝。
- **只依赖 Broker 不注册权限策略**：违反 fail-closed 基线，漏装策略时所有调用都会被拒绝或形成部署歧义；拒绝。

## 影响

交互策略可独立演进为组织、项目或设备级规则；Broker 保持为唯一任务状态与对象授权真相源。Portal Edge 的部署需要增加经过验签的交互策略和 Broker 制品，以及 Broker 状态文件配置。
