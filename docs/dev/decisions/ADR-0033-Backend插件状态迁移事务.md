# ADR-0033 Backend 插件状态迁移事务

- 状态：已采纳
- 日期：2026-07-16
- 关联：[ADR-0024 单节点自动装配与回滚语义](ADR-0024-单节点自动装配与回滚语义.md)、[ADR-0031 Backend 内核 1.0 封板与工程门禁](ADR-0031-Backend内核1.0封板与工程门禁.md)、[ADR-0032 Backend 插件生命周期与实际态 v2](ADR-0032-Backend插件生命周期实际态v2.md)

## 背景

候选进程启动成功并不代表有状态插件可以安全升级。插件状态可能位于数据库、对象存储或插件自有目录，内核既不了解其结构，也不能用复制目录等猜测行为替代业务迁移。另一方面，只让插件自行在启动时修改旧状态，会让失败候选破坏仍在服务的当前实例，无法兑现 ADR-0031 的失败回滚承诺。

迁移契约必须同时满足：状态格式显式、旧实例在候选验证期间可继续服务、操作可重试、多个插件可按事务回滚，并且不把数据库产品或插件私有结构固化进微内核。

## 决策

### 1. 清单显式声明状态格式

Backend 插件需要持久业务状态时，在 `state.backend` 声明：

```json
{
  "format": "com.acme.crm.state",
  "formatVersion": 2,
  "migration": {
    "protocol": "lifecycle.v1",
    "from": [{ "format": "com.acme.crm.state", "formatVersion": 1 }]
  }
}
```

`format + formatVersion` 共同构成状态身份。首次引入状态时可以没有 `migration`；已有状态身份发生变化时，新制品必须以 `migration.from` 精确列出旧身份，否则 Node Agent 在启动候选前 fail-closed。已有状态的同一插件不能通过删除 `state.backend` 隐式变成无状态插件。

Schema 只发布 `backend` 字段；其他运行面在各自内核封板时追加。安装器把验签清单里的状态契约写入 `InstalledPlugin`，后续升级只比较已验证的旧、新制品，不重新信任外部文本。

### 2. 插件负责私有迁移，内核负责事务编排

插件 SDK 提供 `OnMigration`，协议 v1 以只增字段和枚举的方式加入三个生命周期操作：

- `MIGRATION_PREPARE`：构建与旧视图隔离的候选状态；
- `MIGRATION_COMMIT`：提交候选视图，但在候选取得路由所有权前仍必须可回滚；
- `MIGRATION_ROLLBACK`：逆转本事务的 prepare/commit，不影响旧实例视图。

消息携带 `transaction_id/from_state_format/from_state_version/to_state_format/to_state_version`。事务 ID 由 unit、目标 fingerprint、插件和状态身份确定性生成；三个操作以及崩溃后的重复投递都必须按该 ID 幂等。

内核不读取、复制或转换插件私有状态。插件可以在迁移处理器中使用自身存储，也可以经后续 persistence/transaction SPI 获取部署适配器提供的事务能力，但必须兑现 copy-on-write 和可回滚约束。

### 3. 升级顺序

单个 unit 的候选升级顺序固定为：

1. 安装并校验全部候选制品；
2. 拉起隔离候选宿主，校验进程自报 ID/版本与验签清单一致；
3. 对所有迁移计划执行 `PREPARE`；
4. 对所有已准备计划执行 `COMMIT`；
5. 以 `starting` 租约预注册全部候选 capability，此时本地路由和 NATS handler 门闩保持关闭；
6. 全部租约可发布后，将整组改为 `healthy`、一次打开调用门闩，并在同一 Runtime 临界区切换运行所有权；
7. 排空并回收旧宿主。

第 6 步门闩打开以前任何失败都对已经 prepare 的插件逆序发送 `ROLLBACK`，候选宿主随后回收，旧运行实例和实际态保持不变。逐条预注册不得直接公开 handler，避免“前几条能力已接流量、后一条发布失败”的半切换。rollback 自身失败必须进入错误链和诊断实际态，不能只写日志。取得路由所有权后不再回滚到旧代码；因此所有可能失败的状态事务和能力准备都必须发生在所有权切换前。

### 4. 兼容边界

新增 proto 字段和枚举值保持 wire v1 的只增兼容；不声明状态迁移的旧插件不会收到迁移操作。新清单声明需要迁移但二进制未实现处理器时，SDK 明确拒绝，候选启动失败并保留旧版本。

卸载或从 unit 移除插件只停止运行与解除引用，不自动删除其私有业务状态。数据保留/销毁属于显式运维动作，不能与代码卸载隐式绑定。

## 备选方案

- **内核复制插件数据目录**：无法覆盖远端数据库，也会猜测插件一致性边界。拒绝。
- **候选启动时自行就地迁移**：实现最简单，但失败候选可能破坏旧实例正在读取的数据。拒绝。
- **只发一个 `MIGRATE` 操作**：无法区分准备、取得所有权和回滚窗口，不能编排多插件失败。拒绝。
- **把迁移实现注册成新的公共扩展点**：会扩大 Backend 1.0 插件贡献目录；迁移是候选自身的生命周期能力，不是供业务寻址的公共 capability。拒绝。

## 影响

- 正面：内核能够证明格式变化经过显式、幂等、可回滚事务，失败升级保留旧代码与旧状态视图。
- 代价：有状态插件作者必须实现 copy-on-write 三阶段处理器，并维护可接受的来源格式目录。
- 风险：外部存储是否真正隔离仍由插件/部署适配器负责；内核通过协议顺序、失败回滚和 E2E 夹具验证编排，不假装理解业务数据。
- 后续：persistence/transaction SPI 将为迁移处理器提供标准宿主能力，但不能改变本 ADR 的事务顺序和失败语义。
